# DDexec / EverythingExec

{{#include ../../../../banners/hacktricks-training.md}}

## Контекст

У Linux, щоб запустити програму, вона має існувати як файл і бути доступною певним чином через ієрархію файлової системи (саме так працює `execve()`). Цей файл може зберігатися на диску або в оперативній пам’яті (tmpfs, memfd), але вам потрібен шлях до файлу. Це значно спрощує контроль над тим, що запускається в Linux-системі, виявлення загроз та інструментів зловмисника або повне блокування їхньої спроби щось виконати (_наприклад_, заборона непривілейованим користувачам розміщувати виконувані файли будь-де).

Але ця техніка покликана все це змінити. Якщо ви не можете запустити потрібний процес... **тоді ви захоплюєте вже наявний**.

Ця техніка дає змогу **обійти поширені методи захисту, як-от read-only, noexec, whitelisting імен файлів, whitelisting хешів...**

## Залежності

Для роботи фінальний скрипт залежить від наведених нижче інструментів; вони мають бути доступні в системі, яку ви атакуєте (типово всі вони є практично всюди):
```
dd
bash | zsh | ash (busybox)
head
tail
cut
grep
od
readlink
wc
tr
base64
```
## Техніка

Якщо ви можете довільно змінювати пам'ять процесу, то можете взяти його під контроль. Це можна використати для перехоплення вже наявного процесу та заміни його іншою програмою. Цього можна досягти або за допомогою syscall `ptrace()` (для цього потрібно мати можливість виконувати syscalls або мати `gdb` у системі), або, що цікавіше, записуючи до `/proc/$pid/mem`.

Файл `/proc/$pid/mem` є однозначним відображенням усього адресного простору процесу (_напр._ від `0x0000000000000000` до `0x7ffffffffffff000` в x86-64). Це означає, що читання з цього файлу або запис до нього за зміщенням `x` еквівалентні читанню або зміні вмісту за віртуальною адресою `x`.

Тепер маємо чотири основні проблеми:

- Загалом лише root і власник файлу можуть його змінювати.
- ASLR.
- Якщо спробувати прочитати або записати адресу, не відображену в адресному просторі програми, ми отримаємо помилку I/O.

Ці проблеми мають такі рішення, які, хоча й не є ідеальними, працюють:

- Більшість shell-інтерпретаторів дозволяють створювати файлові дескриптори, які потім успадковуються дочірніми процесами. Ми можемо створити fd, що вказує на файл `mem` shell із правами на запис... тому дочірні процеси, які використовують цей fd, зможуть змінювати пам'ять shell.
- ASLR навіть не є проблемою: ми можемо перевірити файл `maps` shell або будь-який інший файл із procfs, щоб отримати інформацію про адресний простір процесу.
- Отже, нам потрібно виконати `lseek()` у файлі. Із shell це неможливо зробити без використання сумнозвісної `dd`.

### Детальніше

Кроки відносно прості, і для їх розуміння не потрібні спеціальні знання:

- Проаналізувати binary, яку ми хочемо запустити, і loader, щоб з'ясувати, які mappings їм потрібні. Потім створити "shell"code, який, загалом, виконуватиме ті самі кроки, що й kernel під час кожного виклику `execve()`:
- Створити зазначені mappings.
- Прочитати binaries у них.
- Налаштувати permissions.
- Нарешті ініціалізувати stack аргументами для програми та розмістити auxiliary vector (потрібний loader).
- Перейти до loader і дозволити йому виконати решту (завантажити libraries, потрібні програмі).
- Отримати з файлу `syscall` адресу, до якої процес повернеться після syscall, що виконується.
- Перезаписати це місце, яке буде executable, нашим shellcode (через `mem` ми можемо змінювати unwritable pages).
- Передати програму, яку ми хочемо запустити, до stdin процесу (цей "shell"code виконає `read()`).
- На цьому етапі loader має завантажити необхідні libraries для нашої програми та перейти до неї.

**Ознайомтеся з tool за адресою** [**https://github.com/arget13/DDexec**](**https://github.com/arget13/DDexec**)

## EverythingExec

Існує кілька альтернатив `dd`, однією з яких є `tail` — наразі це стандартна програма для виконання `lseek()` у файлі `mem` (саме для цього й використовувалася `dd`). Такими альтернативами є:
```bash
tail
hexdump
cmp
xxd
```
Установивши змінну `SEEKER`, ви можете змінити використовуваний seeker, _наприклад_:
```bash
SEEKER=cmp bash ddexec.sh ls -l <<< $(base64 -w0 /bin/ls)
```
Якщо ви знайдете інший дійсний seeker, не реалізований у скрипті, ви все одно можете використовувати його, задавши змінну `SEEKER_ARGS`:
```bash
SEEKER=xxd SEEKER_ARGS='-s $offset' zsh ddexec.sh ls -l <<< $(base64 -w0 /bin/ls)
```
Блокуйте це, EDR.
