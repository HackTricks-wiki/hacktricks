# DDexec / EverythingExec

{{#include ../../../../banners/hacktricks-training.md}}

## Πλαίσιο

Στο Linux, για να εκτελεστεί ένα πρόγραμμα, πρέπει να υπάρχει ως αρχείο και να είναι προσβάσιμο με κάποιον τρόπο μέσω της ιεραρχίας του file system (έτσι λειτουργεί το `execve()`). Αυτό το αρχείο μπορεί να βρίσκεται στον δίσκο ή στη RAM (tmpfs, memfd), αλλά χρειάζεστε ένα filepath. Αυτό έχει κάνει πολύ εύκολο τον έλεγχο του τι εκτελείται σε ένα Linux system, καθώς και τον εντοπισμό threats και εργαλείων του attacker ή την αποτροπή της προσπάθειάς τους να εκτελέσουν οτιδήποτε δικό τους (_π.χ._ να μην επιτρέπεται σε unprivileged users να τοποθετούν executable files οπουδήποτε).

Όμως αυτή η τεχνική έρχεται για να αλλάξει όλα αυτά. Αν δεν μπορείτε να εκκινήσετε το process που θέλετε... **τότε κάνετε hijack ένα που υπάρχει ήδη**.

Αυτή η τεχνική σάς επιτρέπει να **παρακάμπτετε κοινές τεχνικές προστασίας, όπως read-only, noexec, file-name whitelisting, hash whitelisting...**

## Εξαρτήσεις

Το τελικό script εξαρτάται από τα ακόλουθα εργαλεία για να λειτουργήσει· πρέπει να είναι προσβάσιμα στο system που κάνετε attack (από προεπιλογή, θα βρείτε όλα αυτά σχεδόν παντού):
```
dd
bash | zsh | ash (busybox)
head
tail
cut
grep
od
readlink
wc
tr
base64
```
## Η τεχνική

Αν μπορείτε να τροποποιήσετε αυθαίρετα τη μνήμη μιας διεργασίας, τότε μπορείτε να την καταλάβετε. Αυτό μπορεί να χρησιμοποιηθεί για την κατάληψη μιας ήδη υπάρχουσας διεργασίας και την αντικατάστασή της με ένα άλλο πρόγραμμα. Μπορούμε να το επιτύχουμε είτε χρησιμοποιώντας το syscall `ptrace()` (το οποίο απαιτεί να έχετε τη δυνατότητα εκτέλεσης syscalls ή να υπάρχει διαθέσιμο το gdb στο σύστημα) είτε, πιο ενδιαφέροντα, γράφοντας στο `/proc/$pid/mem`.

Το αρχείο `/proc/$pid/mem` είναι μια αντιστοίχιση ένα-προς-ένα ολόκληρου του address space μιας διεργασίας (_π.χ._ από `0x0000000000000000` έως `0x7ffffffffffff000` σε x86-64). Αυτό σημαίνει ότι η ανάγνωση από ή η εγγραφή σε αυτό το αρχείο στη θέση `x` είναι ισοδύναμη με την ανάγνωση ή την τροποποίηση των περιεχομένων στην virtual address `x`.

Τώρα, έχουμε τέσσερα βασικά προβλήματα να αντιμετωπίσουμε:

- Γενικά, μόνο ο root και ο owner του αρχείου μπορούν να το τροποποιήσουν.
- ASLR.
- Αν προσπαθήσουμε να διαβάσουμε ή να γράψουμε σε μια address που δεν είναι mapped στο address space του προγράμματος, θα λάβουμε σφάλμα I/O.

Αυτά τα προβλήματα έχουν λύσεις οι οποίες, αν και δεν είναι τέλειες, είναι αποτελεσματικές:

- Οι περισσότεροι shell interpreters επιτρέπουν τη δημιουργία file descriptors οι οποίοι στη συνέχεια κληρονομούνται από child processes. Μπορούμε να δημιουργήσουμε ένα fd που δείχνει στο αρχείο `mem` του shell με write permissions... έτσι, τα child processes που χρησιμοποιούν αυτό το fd θα μπορούν να τροποποιούν τη μνήμη του shell.
- Το ASLR δεν αποτελεί καν πρόβλημα· μπορούμε να ελέγξουμε το αρχείο `maps` του shell ή οποιοδήποτε άλλο αρχείο από το procfs, ώστε να αποκτήσουμε πληροφορίες σχετικά με το address space της διεργασίας.
- Επομένως, πρέπει να κάνουμε `lseek()` στο αρχείο. Από το shell αυτό δεν μπορεί να γίνει παρά μόνο με τη χρήση του διαβόητου `dd`.

### Πιο αναλυτικά

Τα βήματα είναι σχετικά εύκολα και δεν απαιτούν κανενός είδους εξειδίκευση για να τα κατανοήσετε:

- Κάντε parse το binary που θέλουμε να εκτελέσουμε και τον loader, ώστε να διαπιστώσετε ποια mappings χρειάζονται. Στη συνέχεια, δημιουργήστε "shell"code που θα εκτελεί, σε γενικές γραμμές, τα ίδια βήματα με αυτά που εκτελεί ο kernel σε κάθε κλήση του `execve()`:
- Δημιουργία των συγκεκριμένων mappings.
- Ανάγνωση των binaries μέσα σε αυτά.
- Ρύθμιση των permissions.
- Τέλος, αρχικοποίηση του stack με τα arguments του προγράμματος και τοποθέτηση του auxiliary vector (που απαιτείται από τον loader).
- Μετάβαση στον loader και άφησέ τον να κάνει τα υπόλοιπα (να φορτώσει τις libraries που χρειάζεται το πρόγραμμα).
- Λήψη, από το αρχείο `syscall`, της address στην οποία θα επιστρέψει η διεργασία μετά το syscall που εκτελεί.
- Αντικατάσταση αυτής της θέσης, η οποία θα είναι executable, με το shellcode μας (μέσω του `mem` μπορούμε να τροποποιούμε unwritable pages).
- Μεταφορά του προγράμματος που θέλουμε να εκτελέσουμε στο stdin της διεργασίας (θα γίνει `read()` από το συγκεκριμένο "shell"code).
- Σε αυτό το σημείο, εναπόκειται στον loader να φορτώσει τις απαραίτητες libraries για το πρόγραμμά μας και να μεταβεί σε αυτό.

## EverythingExec

Υπάρχουν αρκετές εναλλακτικές του `dd`, μία από τις οποίες, το `tail`, είναι επί του παρόντος το προεπιλεγμένο πρόγραμμα που χρησιμοποιείται για το `lseek()` μέσα στο αρχείο `mem` (κάτι που ήταν ο μοναδικός λόγος χρήσης του `dd`). Οι συγκεκριμένες εναλλακτικές είναι:
```bash
tail
hexdump
cmp
xxd
```
Ορίζοντας τη μεταβλητή `SEEKER`, μπορείτε να αλλάξετε τον seeker που χρησιμοποιείται, _π.χ._:
```bash
SEEKER=cmp bash ddexec.sh ls -l <<< $(base64 -w0 /bin/ls)
```
Αν βρείτε κάποιο άλλο έγκυρο `seeker` που δεν έχει υλοποιηθεί στο script, μπορείτε και πάλι να το χρησιμοποιήσετε ορίζοντας τη μεταβλητή `SEEKER_ARGS`:
```bash
SEEKER=xxd SEEKER_ARGS='-s $offset' zsh ddexec.sh ls -l <<< $(base64 -w0 /bin/ls)
```
Μπλοκάρετέ το, EDRs.
