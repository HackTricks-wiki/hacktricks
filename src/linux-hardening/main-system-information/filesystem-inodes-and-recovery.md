# Файлова система, inode та відновлення

{{#include ../../banners/hacktricks-training.md}}

Зловживання файловою системою часто полягає в заплутуванні взаємозв’язку між видимим шляхом і об’єктом, що стоїть за ним. Образи дисків можуть приховувати іншу файлову систему, монтування з правом запису можуть використовуватися привілейованими завданнями, hardlinks можуть надавати доступ до того самого inode через інше ім’я, а видалені файли все ще можуть бути доступними для читання через відкритий файловий дескриптор.

Ця сторінка зосереджена на техніці, а не на конкретній лабораторній роботі чи цілі.

## Образи дисків і Loop Mounts

Звичайний файл може містити повну файлову систему. Тому backup-образи, скопійовані блокові пристрої, артефакти VM або перейменовані blobs можуть містити облікові дані, скрипти, SSH-ключі, конфігураційні файли або flags, навіть якщо зовні вони не здаються корисними.

Визначте ймовірні образи:
```bash
file ./candidate
ls -lh ./candidate
blkid ./candidate 2>/dev/null
strings -a ./candidate | head -n 50
```
Якщо монтування дозволено, спочатку монтуйте невідомі образи лише для читання:
```bash
mkdir -p /tmp/imgmnt
sudo mount -o loop,ro ./candidate /tmp/imgmnt
find /tmp/imgmnt -maxdepth 3 -type f -ls 2>/dev/null
sudo umount /tmp/imgmnt
```
Якщо монтування недоступне, безпосередньо перевірте метадані файлової системи:
```bash
debugfs -R 'ls -l /' ./candidate 2>/dev/null
debugfs -R 'stat /' ./candidate 2>/dev/null
```
Ця техніка корисна, оскільки перетворює файл, схожий на звичайний, на друге дерево файлової системи. Розглядайте її як спосіб відновлення прихованих даних, а не як самостійне підвищення привілеїв.

## Зловживання записуваним монтуванням

Записуване монтування стає небезпечним, коли привілейований контекст згодом довіряє чомусь усередині нього. Важливе питання полягає не лише в тому, «чи можу я сюди записувати?», а й у тому, «хто згодом читає, виконує, імпортує або завантажує дані звідси?».

Знайдіть монтування, доступні для запису, і підозрілих споживачів:
```bash
findmnt -o TARGET,SOURCE,FSTYPE,OPTIONS
find /mnt /media /srv /opt -xdev -type d -writable -ls 2>/dev/null
find /mnt /media /srv /opt -xdev -type f -writable -ls 2>/dev/null | head -n 50
grep -RniE 'cron|systemd|ExecStart|backup|hook|plugin|sh |bash |python' /mnt /media /srv /opt 2>/dev/null | head -n 50
```
Поширені шаблони зловживання:

- Привілейований cron або systemd unit запускає скрипт із mount, доступний для запису.
- Привілейований сервіс завантажує plugins, config, templates або helper binaries із mount.
- Mount містить SUID-файли й дозволяє їх модифікацію, заміну або маніпуляції з path.
- Container або chroot відкриває host-backed path, доступний для запису з restricted environment.

Загальний шаблон перевірки:
```bash
find /mnt /media /srv /opt -xdev -perm -4000 -type f -ls 2>/dev/null
find /mnt /media /srv /opt -xdev -type f -writable -ls 2>/dev/null | head -n 50
```
Під час демонстрації впливу в авторизованій лабораторії зберігайте payload спостережуваним і мінімальним, наприклад записуючи вивід `id` у тимчасовий файл. Основна техніка полягає у відкладеному виконанні через довірене доступне для запису розташування.

## Іноді та плутанина шляхів

Інод — це об'єкт файлової системи; шлях — лише ім'я, що на нього вказує. Це важливо, оскільки два різні шляхи можуть вказувати на один і той самий інод, а видалення імені шляху не завжди означає, що дані зникли.

Порівнюйте файли за інодом і пристроєм:
```bash
ls -li /path/a /path/b
stat -c 'dev=%d inode=%i links=%h mode=%A owner=%U:%G path=%n' /path/a /path/b
```
Знайдіть кожен видимий шлях до того самого inode:
```bash
find / -xdev -samefile /path/to/file -ls 2>/dev/null
```
Шукайте безпосередньо за номером inode, коли маєте лише метадані:
```bash
find / -xdev -inum <inode_number> -ls 2>/dev/null
```
Ця техніка корисна, коли файл має неочікуване ім’я, коли application перевіряє один path, але використовує інший, або коли privileged wrapper взаємодіє з inode, до якого також можна дістатися в іншому місці.

## Hardlink Abuse

Hardlinks створюють кілька імен для одного inode. Вони не вказують на target path, як symlinks; це рівнозначні імена для одного file object.

Знайти SUID files із кількома hardlinks:
```bash
find / -xdev -perm -4000 -type f -links +1 -ls 2>/dev/null
```
Перевірте один підозрілий файл:
```bash
stat /path/to/suspicious
find / -xdev -samefile /path/to/suspicious -ls 2>/dev/null
```
Чому це важливо:

- Конфіденційний файл може бути доступний через менш очевидний шлях.
- SUID-обгортка може бути прихована за назвою, яка не виглядає привілейованою.
- Очищення, під час якого видаляється один шлях, може залишити активним інший hardlink.

Сучасні kernel і mount options можуть обмежувати створення hardlink, щоб зменшити цей клас зловживань, але наявні hardlink усе ще варто перевіряти.

## Відновлення видалених файлів через відкриті FD

Коли процес утримує файл відкритим, дані файлу можуть залишатися доступними навіть після видалення шляху. Linux показує ці відкриті дескриптори в `/proc/<pid>/fd/`.

Знайти видалені відкриті файли:
```bash
ls -l /proc/*/fd/* 2>/dev/null | grep ' (deleted)' | head -n 50
lsof 2>/dev/null | grep deleted | head -n 50
```
Відновіть дані, якщо це дозволяють дозволи:
```bash
readlink /proc/<pid>/fd/<fd>
cp /proc/<pid>/fd/<fd> /tmp/recovered-file
file /tmp/recovered-file
```
Це практична техніка для відновлення видалених журналів, тимчасових секретів, скинутих бінарних файлів, ротованих файлів або скриптів, видалених після виконання.

## Відновлення ext за допомогою debugfs

У файлових системах ext `debugfs` може перевіряти метадані inode та іноді вивантажувати вміст файлів із образу файлової системи. За можливості працюйте з копією або образом, доступним лише для читання.

Перелічіть записи та перевірте inode:
```bash
debugfs -R 'ls -l /' ./disk.img
debugfs -R 'stat <inode_number>' ./disk.img
debugfs -R 'ncheck <inode_number>' ./disk.img
```
Дамп відомого inode:
```bash
debugfs -R 'dump <inode_number> /tmp/recovered.bin' ./disk.img
file /tmp/recovered.bin
```
Це не гарантує відновлення. Результат залежить від стану файлової системи, того, чи були блоки повторно використані, і чи все ще існують метадані. Ця техніка залишається цінною, оскільки дає змогу перевіряти стан на рівні inode, не покладаючись на звичайний обхід шляхів.

## Вичерпання та порядок inode

Вичерпання inode відбувається, коли у файловій системі закінчуються файлові об’єкти, навіть якщо вільне місце на диску ще залишається. Зазвичай це спричиняє збої надійності, але також може пояснити дивну поведінку під час реагування на інцидент або triage у лабораторії.

Перевірте навантаження на inode:
```bash
df -h
df -i
find /var /tmp /home -xdev -printf '%h\n' 2>/dev/null | sort | uniq -c | sort -n | tail
```
Номери inode та часові мітки також можуть допомогти відтворити активність у простих лабораторних середовищах:
```bash
find /path -xdev -printf '%i %TY-%Tm-%Td %TH:%TM %p\n' 2>/dev/null | sort -n | tail -n 50
find /path -xdev -newermt '2026-01-01' -ls 2>/dev/null
```
Розглядайте порядок як підказку, а не як доказ. Операції копіювання, розпакування архівів, тип файлової системи, відновлення та одночасні записи можуть змінювати шаблони розподілу.

## Примітки щодо захисту

- Під час аналізу підключайте невідомі образи лише для читання.
- Зберігайте привілейовані скрипти, службові модулі, плагіни та шляхи до допоміжних файлів поза монтуваннями, доступними для запису користувачам.
- Використовуйте `nosuid`, `nodev` і `noexec`, якщо це операційно доцільно, але не вважайте їх повноцінним бар’єром.
- За можливості обмежуйте доступ до `/proc/<pid>/fd`, метаданих процесів і засобів перегляду процесів інших користувачів.
- Відстежуйте точки монтування, доступні для запису, неочікувані hardlinks на привілейовані файли та конфіденційні файли, які були видалені, але залишаються відкритими.
