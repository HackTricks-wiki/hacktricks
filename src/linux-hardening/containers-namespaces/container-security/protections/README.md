# Επισκόπηση Προστασιών Container

{{#include ../../../../banners/hacktricks-training.md}}

Η σημαντικότερη ιδέα στο hardening των container είναι ότι δεν υπάρχει ένας μοναδικός έλεγχος που ονομάζεται "container security". Αυτό που αποκαλούν οι περισσότεροι απομόνωση container είναι στην πραγματικότητα το αποτέλεσμα αρκετών μηχανισμών ασφάλειας και διαχείρισης πόρων του Linux που λειτουργούν μαζί. Αν η τεκμηρίωση περιγράφει μόνο έναν από αυτούς, οι αναγνώστες τείνουν να υπερεκτιμούν την ισχύ του. Αν η τεκμηρίωση παραθέτει όλους τους μηχανισμούς χωρίς να εξηγεί πώς αλληλεπιδρούν, οι αναγνώστες αποκτούν έναν κατάλογο ονομάτων αλλά όχι ένα πραγματικό μοντέλο. Αυτή η ενότητα προσπαθεί να αποφύγει και τα δύο λάθη.

Στο κέντρο του μοντέλου βρίσκονται τα **namespaces**, τα οποία απομονώνουν όσα μπορεί να δει το workload. Παρέχουν στη διεργασία μια ιδιωτική ή εν μέρει ιδιωτική προβολή των filesystem mounts, των PIDs, της δικτύωσης, των αντικειμένων IPC, των hostnames, των αντιστοιχίσεων χρηστών/ομάδων, των διαδρομών cgroup και ορισμένων clocks. Ωστόσο, τα namespaces από μόνα τους δεν καθορίζουν τι επιτρέπεται να κάνει μια διεργασία. Εκεί εισέρχονται τα επόμενα επίπεδα.

Τα **cgroups** ελέγχουν τη χρήση πόρων. Δεν αποτελούν πρωτίστως boundary απομόνωσης με την ίδια έννοια όπως τα mount ή PID namespaces, αλλά είναι κρίσιμα σε επιχειρησιακό επίπεδο, επειδή περιορίζουν τη μνήμη, την CPU, τα PIDs, το I/O και την πρόσβαση σε συσκευές. Έχουν επίσης σημασία για την ασφάλεια, επειδή ιστορικές τεχνικές breakout εκμεταλλεύτηκαν writable λειτουργίες των cgroup, ιδιαίτερα σε περιβάλλοντα cgroup v1.

Τα **Capabilities** διαχωρίζουν το παλιό, πανίσχυρο μοντέλο του root σε μικρότερες μονάδες προνομίων. Αυτό είναι θεμελιώδες για τα container, επειδή πολλά workload εξακολουθούν να εκτελούνται ως UID 0 μέσα στο container. Επομένως, το ερώτημα δεν είναι απλώς "είναι η διεργασία root;", αλλά "ποια capabilities διατηρήθηκαν, μέσα σε ποια namespaces και υπό ποιους περιορισμούς seccomp και MAC;" Γι' αυτό μια διεργασία root σε ένα container μπορεί να είναι σχετικά περιορισμένη, ενώ μια διεργασία root σε άλλο container μπορεί στην πράξη να είναι σχεδόν αδιάκριτη από το root του host.

Το **seccomp** φιλτράρει τα syscalls και μειώνει την επιφάνεια επίθεσης του kernel που εκτίθεται στο workload. Αυτός είναι συχνά ο μηχανισμός που αποκλείει προφανώς επικίνδυνες κλήσεις, όπως οι `unshare`, `mount`, `keyctl` ή άλλα syscalls που χρησιμοποιούνται σε αλυσίδες breakout. Ακόμη και αν μια διεργασία διαθέτει ένα capability που διαφορετικά θα επέτρεπε μια λειτουργία, το seccomp μπορεί να αποκλείσει τη διαδρομή του syscall πριν ο kernel την επεξεργαστεί πλήρως.

Τα **AppArmor** και **SELinux** προσθέτουν Mandatory Access Control πάνω από τους κανονικούς ελέγχους filesystem και προνομίων. Είναι ιδιαίτερα σημαντικά, επειδή εξακολουθούν να έχουν σημασία ακόμη και όταν ένα container διαθέτει περισσότερα capabilities από όσα θα έπρεπε. Ένα workload μπορεί να διαθέτει το θεωρητικό προνόμιο να επιχειρήσει μια ενέργεια, αλλά παρ' όλα αυτά να εμποδίζεται από την εκτέλεσή της, επειδή το label ή το profile του απαγορεύει την πρόσβαση στη σχετική διαδρομή, το αντικείμενο ή τη λειτουργία.

Τέλος, υπάρχουν πρόσθετα επίπεδα hardening που λαμβάνουν λιγότερη προσοχή αλλά έχουν συχνά σημασία σε πραγματικές επιθέσεις: `no_new_privs`, masked διαδρομές procfs, read-only διαδρομές συστήματος, read-only root filesystems και προσεκτικά runtime defaults. Αυτοί οι μηχανισμοί συχνά σταματούν το "τελευταίο βήμα" ενός compromise, ιδιαίτερα όταν ένας attacker προσπαθεί να μετατρέψει την εκτέλεση κώδικα σε ευρύτερη απόκτηση προνομίων.

Ο υπόλοιπος φάκελος εξηγεί καθέναν από αυτούς τους μηχανισμούς με περισσότερες λεπτομέρειες, συμπεριλαμβανομένου του τι κάνει στην πραγματικότητα το kernel primitive, πώς μπορείτε να τον παρατηρήσετε τοπικά, πώς τον χρησιμοποιούν τα συνηθισμένα runtimes και πώς οι operators τον αποδυναμώνουν κατά λάθος.
