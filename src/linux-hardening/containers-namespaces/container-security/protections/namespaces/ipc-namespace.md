# Простір імен IPC

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен IPC ізолює **об’єкти System V IPC** і **черги повідомлень POSIX**. Це включає сегменти спільної пам’яті, семафори та черги повідомлень, які інакше були б видимі для непов’язаних процесів на host. На практиці це не дає контейнеру безперешкодно під’єднуватися до IPC-об’єктів, що належать іншим workloads або host.

Порівняно з mount-, PID- або user-просторами імен, простір імен IPC часто згадується рідше, але це не означає, що він неважливий. Спільна пам’ять і пов’язані з нею IPC-механізми можуть містити надзвичайно корисний стан. Якщо простір імен IPC host доступний, workload може отримати видимість об’єктів або даних для міжпроцесної координації, які ніколи не призначалися для виходу за межі контейнера.

## Робота

Коли runtime створює новий простір імен IPC, процес отримує власний ізольований набір ідентифікаторів IPC. Це означає, що такі команди, як `ipcs`, показують лише об’єкти, доступні в цьому просторі імен. Якщо натомість контейнер приєднується до простору імен IPC host, ці об’єкти стають частиною спільного глобального представлення.

Це особливо важливо в середовищах, де applications або services активно використовують спільну пам’ять. Навіть якщо контейнер не може безпосередньо виконати breakout лише через IPC, простір імен може leak-ати інформацію або уможливити втручання в роботу інших процесів, що суттєво допоможе під час подальшої атаки.

## Лабораторія

Ви можете створити приватний простір імен IPC за допомогою:
```bash
sudo unshare --ipc --fork bash
ipcs
```
І порівняйте поведінку під час виконання з:
```bash
docker run --rm debian:stable-slim ipcs
docker run --rm --ipc=host debian:stable-slim ipcs
```
## Використання під час виконання

Docker і Podman типово ізолюють IPC. Kubernetes зазвичай надає Pod власний IPC namespace, спільний для контейнерів у тому самому Pod, але не типовим чином із host. Спільне використання IPC з host можливе, однак його слід розглядати як суттєве зниження рівня ізоляції, а не як незначну опцію runtime.

## Неправильні конфігурації

Очевидною помилкою є `--ipc=host` або `hostIPC: true`. Це може робитися для сумісності зі застарілим програмним забезпеченням або для зручності, але суттєво змінює модель довіри. Інша поширена проблема — просто не враховувати IPC, оскільки він здається менш небезпечним, ніж host PID або host networking. Насправді, якщо workload працює з браузерами, базами даних, науковими workload або іншим програмним забезпеченням, яке активно використовує shared memory, поверхня IPC може бути дуже важливою.

## Зловживання

Коли host IPC є спільним, attacker може перевіряти або втручатися в об’єкти shared memory, отримувати додаткову інформацію про поведінку host або сусідніх workload, а також поєднувати отримані там відомості з visibility процесів і можливостями на кшталт ptrace. Спільне використання IPC часто є допоміжною слабкістю, а не повним шляхом до breakout, але допоміжні слабкості важливі, оскільки вони скорочують і стабілізують реальні attack chains.

Перший корисний крок — визначити, які IPC-об’єкти взагалі видимі:
```bash
readlink /proc/self/ns/ipc
ipcs -a
ls -la /dev/shm 2>/dev/null | head -n 50
```
Якщо IPC namespace хоста є спільним, великі сегменти спільної пам’яті або цікаві власники об’єктів можуть одразу розкрити поведінку застосунку:
```bash
ipcs -m -p
ipcs -q -p
```
У деяких середовищах сам вміст `/dev/shm` може leak-ати назви файлів, артефакти або токени, які варто перевірити:
```bash
find /dev/shm -maxdepth 2 -type f 2>/dev/null -ls | head -n 50
strings /dev/shm/* 2>/dev/null | head -n 50
```
IPC sharing рідко саме по собі одразу надає host root, але може розкрити канали даних і координації, які значно спрощують подальші атаки на процеси.

### Повний приклад: відновлення секретів із `/dev/shm`

Найреалістичніший повний сценарій зловживання — крадіжка даних, а не прямий escape. Якщо IPC хоста або broad shared-memory layout відкриті, конфіденційні артефакти іноді можна безпосередньо відновити:
```bash
find /dev/shm -maxdepth 2 -type f 2>/dev/null -print
strings /dev/shm/* 2>/dev/null | grep -Ei 'token|secret|password|jwt|key'
```
Вплив:

- вилучення секретів або даних сесій, залишених у спільній пам’яті
- отримання відомостей про застосунки, які наразі активні на хості
- краще визначення цілей для подальших атак на основі PID namespace або ptrace

Тому спільний доступ до IPC краще розглядати як **підсилювач атаки**, а не як самостійну примітивну техніку втечі з хоста.

## Перевірки

Ці команди призначені для того, щоб визначити, чи має workload приватне представлення IPC, чи видимі значущі об’єкти спільної пам’яті або повідомлень, а також чи відкриває сам `/dev/shm` корисні артефакти.
```bash
readlink /proc/self/ns/ipc   # Namespace identifier for IPC
ipcs -a                      # Visible SysV IPC objects
mount | grep shm             # Shared-memory mounts, especially /dev/shm
```
Що тут цікаво:

- Якщо `ipcs -a` показує об'єкти, власниками яких є неочікувані користувачі або сервіси, namespace може бути не таким ізольованим, як очікується.
- За великими або незвичними сегментами спільної пам'яті часто варто простежити додатково.
- Широке монтування `/dev/shm` не є автоматично помилкою, але в деяких середовищах воно витікає іменами файлів, артефактами та тимчасовими секретами.

IPC рідко отримує стільки ж уваги, як важливіші типи namespace, але в середовищах, де він активно використовується, спільне використання з host є цілком свідомим рішенням щодо безпеки.
