# Простір імен UTS

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен UTS ізолює **ім’я хоста** та **ім’я домену NIS**, які бачить процес. На перший погляд це може здаватися незначним порівняно з просторами імен монтування, PID або користувачів, але саме це частково дає змогу контейнеру виглядати як окремий хост. Усередині простору імен робоче навантаження може бачити, а іноді й змінювати ім’я хоста, локальне для цього простору імен, а не глобальне для машини.

Сам по собі цей простір зазвичай не є центральним елементом breakout-сценарію. Однак якщо простір імен UTS хоста спільно використовується, достатньо привілейований процес може впливати на налаштування, пов’язані з ідентичністю хоста, що може мати операційне, а іноді й безпекове значення.

## Лабораторна робота

Ви можете створити простір імен UTS за допомогою:
```bash
sudo unshare --uts --fork bash
hostname
hostname lab-container
hostname
```
Зміна hostname залишається локальною для цього простору імен і не змінює глобальний hostname хоста. Це простий, але ефективний приклад властивості ізоляції.

## Використання під час виконання

Звичайні контейнери отримують ізольований UTS namespace. Docker і Podman можуть приєднуватися до UTS namespace хоста через `--uts=host`, а подібні шаблони спільного використання хоста можуть застосовуватися в інших runtime та системах оркестрації. Однак здебільшого приватна ізоляція UTS є звичайною частиною налаштування контейнера й майже не потребує уваги оператора.

## Вплив на безпеку

Хоча UTS namespace зазвичай не є найнебезпечнішим для спільного використання, він усе одно сприяє забезпеченню цілісності межі контейнера. Якщо UTS namespace хоста доступний, а процес має необхідні привілеї, він може отримати змогу змінювати пов’язану з hostname інформацію хоста. Це може вплинути на моніторинг, журналювання, операційні припущення або скрипти, які приймають рішення про довіру на основі даних про ідентичність хоста.

## Зловживання

Якщо UTS namespace хоста спільно використовується, практичне питання полягає в тому, чи може процес змінювати налаштування ідентичності хоста, а не лише читати їх:
```bash
readlink /proc/self/ns/uts
hostname
cat /proc/sys/kernel/hostname
```
Якщо контейнер також має необхідний привілей, перевірте, чи можна змінити ім’я хоста:
```bash
hostname hacked-host 2>/dev/null && echo "hostname change worked"
hostname
```
Це насамперед проблема цілісності та операційного впливу, а не повний escape, але вона все одно показує, що контейнер може безпосередньо впливати на глобальну властивість host.

Вплив:

- підроблення ідентичності host
- плутанина в логах, моніторингу або автоматизації, які довіряють hostname
- зазвичай не є повним escape сам по собі, якщо не поєднується з іншими слабкими місцями

У середовищах на кшталт Docker корисний шаблон виявлення на стороні host має такий вигляд:
```bash
docker ps -aq | xargs -r docker inspect --format '{{.Id}} UTSMode={{.HostConfig.UTSMode}}'
```
Контейнери, у яких указано `UTSMode=host`, спільно використовують UTS namespace хоста, тому їх слід ретельніше перевірити, якщо вони також мають capabilities, що дають змогу викликати `sethostname()` або `setdomainname()`.

## Перевірки

Цих команд достатньо, щоб визначити, чи має workload власне представлення hostname, чи спільно використовує UTS namespace хоста.
```bash
readlink /proc/self/ns/uts   # UTS namespace identifier
hostname                     # Hostname as seen by the current process
cat /proc/sys/kernel/hostname   # Kernel hostname value in this namespace
```
Що тут є цікавого:

- Збіг ідентифікаторів namespace із хостовим процесом може вказувати на спільне використання UTS namespace із хостом.
- Якщо зміна hostname впливає не лише на сам контейнер, workload має більший вплив на ідентичність хоста, ніж повинен.
- Зазвичай це менш пріоритетна знахідка, ніж проблеми з PID, mount або user namespace, але вона все одно підтверджує, наскільки ізольованим насправді є процес.

У більшості середовищ UTS namespace найкраще розглядати як допоміжний рівень ізоляції. Він рідко є першим, що перевіряють під час breakout, але все одно є частиною загальної узгодженості та безпеки представлення контейнера.
