## Що таке контейнер насправді

Практичний спосіб визначити контейнер такий: контейнер — це **звичайне дерево процесів Linux**, яке запущене за певною конфігурацією в стилі OCI так, що воно бачить контрольовану файлову систему, контрольований набір ресурсів ядра та обмежену модель привілеїв. Процес може вважати себе PID 1, може думати, що має власний мережевий стек, власний hostname та IPC ресурси, і навіть може працювати як root всередині власного user namespace. Але під капотом це все ще процес хоста, який ядро планує як будь-який інший.

Саме тому безпека контейнерів фактично вивчає, як ця ілюзія конструюється і як вона дає збій. Якщо mount namespace слабкий, процес може побачити host filesystem. Якщо user namespace відсутній або вимкнений, root всередині контейнера може занадто точно відображатися на root хоста. Якщо seccomp неконфігурований і набір capabilities занадто широкий, процес може отримати доступ до syscall-ів і привілейованих можливостей ядра, які мали залишитися недосяжними. Якщо runtime socket змонтований всередині контейнера, контейнеру може навіть не знадобитися kernel breakout, тому що він може просто попросити runtime запустити більш потужний sibling контейнер або змонтувати host root filesystem прямо.

## Чим контейнери відрізняються від віртуальних машин

VM зазвичай має власне kernel і межу абстрагування обладнання. Це означає, що гостьове ядро може впасти, панікувати або бути експлуатованим без автоматичного отримання прямого контролю над ядром хоста. У контейнерах робоче навантаження не отримує окреме ядро. Натомість воно отримує ретельно відфільтрований і іменований (namespaced) вигляд того самого ядра, яке використовує хост. Внаслідок цього контейнери зазвичай легші, швидше стартують, їх простіше щільно пакувати на машині і вони краще підходять для короткоживучого розгортання застосунків. Плата полягає в тому, що межа ізоляції залежить значно пряміше від правильної конфігурації хоста та runtime.

Це не означає, що контейнери «ненадійні», а VMs — «надійні». Це означає, що модель безпеки інша. Добре налаштований стек контейнерів з rootless виконанням, user namespaces, дефолтним seccomp, суворим набором capabilities, без спільного доступу до host namespace-ів і з жорстким SELinux або AppArmor може бути дуже стійким. Навпаки, контейнер, запущений з `--privileged`, спільним host PID/network, з Docker socket змонтованим всередині і з записуваною bind mount `/`, функціонально набагато ближчий до доступу root хоста, ніж до безпечного ізольованого sandbox. Різниця виникає з шарів, які були ввімкнені або вимкнені.

Існує також проміжна зона, яку читачам варто розуміти, оскільки вона з’являється все частіше в реальних середовищах. **Sandboxed container runtimes** такі як **gVisor** і **Kata Containers** навмисно підвищують межу безпеки порівняно з класичним `runc` контейнером. gVisor поміщає userspace kernel шар між робочим навантаженням і багатьма інтерфейсами host kernel, тоді як Kata запускає робоче навантаження всередині легковагової віртуальної машини. Вони все ще використовуються через екосистеми контейнерів і оркестраційні робочі процеси, але їхні властивості безпеки відрізняються від звичайних OCI runtime-ів і не слід ментально ставити їх в один ряд із «звичайними Docker контейнерами», ніби все поводиться однаково.

## Стек контейнерів: кілька шарів, а не один

Коли хтось каже «цей контейнер небезпечний», корисне уточнення: **який шар зробив його небезпечним?** Контейнеризоване робоче навантаження зазвичай є результатом роботи кількох компонентів разом.

На верхньому рівні часто є **image build layer** такий як BuildKit, Buildah або Kaniko, який створює OCI image і metadata. Над низькорівневим runtime-ом може бути **engine або manager** такий як Docker Engine, Podman, containerd, CRI-O, Incus або systemd-nspawn. В кластерних середовищах також може бути **orchestrator** типу Kubernetes, який визначає бажану безпекову позицію через конфігурацію робочого навантаження. І нарешті, **kernel** — це те, що фактично застосовує namespaces, cgroups, seccomp і MAC policy.

Ця багаторівнева модель важлива для розуміння дефолтів. Обмеження може бути запрошене Kubernetes, перекладене через CRI containerd або CRI-O, конвертоване в OCI spec обгорткою runtime-а, і лише потім застосоване `runc`, `crun`, `runsc` або іншим runtime-ом щодо ядра. Коли дефолти відрізняються між середовищами, це часто через те, що один з цих шарів змінив фінальну конфігурацію. Тож той самий механізм може виглядати в Docker або Podman як CLI flag, в Kubernetes як поле Pod або `securityContext`, і в нижчорівневих runtime-стеках як OCI конфігурація, згенерована для робочого навантаження. З цієї причини приклади CLI в цьому розділі слід читати як **runtime-specific syntax для загальної концепції контейнера**, а не як універсальні прапори, підтримувані кожним інструментом.

## Справжня межа безпеки контейнера

На практиці безпека контейнерів походить від **перекриваючихся контролів**, а не від одного ідеального контролю. Namespaces ізолюють видимість. cgroups керують і обмежують використання ресурсів. Capabilities зменшують те, що процес з привілеями може фактично робити. seccomp блокує небезпечні syscalls до того, як вони дійдуть до ядра. AppArmor і SELinux додають Mandatory Access Control поверх звичайних DAC перевірок. `no_new_privs`, masked procfs paths і read-only system paths ускладнюють звичні ланцюги підняття привілеїв і зловживання proc/sys. Сам runtime також має значення, оскільки він вирішує, як створюються mounts, sockets, labels і joins у namespace-ах.

Ось чому багато документації з безпеки контейнерів здається повторюваною. Той самий шлях втечі часто залежить від кількох механізмів одночасно. Наприклад, записуваний host bind mount — погано, але стає набагато гірше, якщо контейнер також працює як реальний root на host, має `CAP_SYS_ADMIN`, не обмежений seccomp і не контролюється SELinux або AppArmor. Так само спільний host PID — серйозна вразливість, але вона стає драматично кориснішою для атакуючого, коли поєднується з `CAP_SYS_PTRACE`, слабкими procfs захистами або інструментами входу в namespace такі як `nsenter`. Тому правильний підхід до документації — не повторювати одну й ту саму атаку на кожній сторінці, а пояснювати, що кожен шар додає до фінальної межі.

## Хороший початковий підхід до енумерації

При оцінці контейнеризованої цілі набагато корисніше задати невелику кількість точних технічних питань, ніж негайно переходити до відомих PoC втеч. Спочатку ідентифікуйте **стек**: Docker, Podman, containerd, CRI-O, Incus/LXC, systemd-nspawn, Apptainer або щось більш спеціалізоване. Потім визначте **runtime**: `runc`, `crun`, `runsc`, `kata-runtime` або іншу OCI-сумісну реалізацію. Після цього перевірте, чи середовище **rootful або rootless**, чи активні **user namespaces**, чи є спільні **host namespaces**, які **capabilities** залишилися, чи увімкнений **seccomp**, чи реально застосовується **MAC policy**, чи присутні **небезпечні mounts або sockets**, і чи процес може взаємодіяти з container runtime API.

Ці відповіді говорять набагато більше про реальну безпекову позицію, ніж назва базового image. У багатьох оцінках ви можете передбачити ймовірну сім’ю breakout-ів ще до читання жодного файлу застосунку, просто зрозумівши фінальну конфігурацію контейнера.
