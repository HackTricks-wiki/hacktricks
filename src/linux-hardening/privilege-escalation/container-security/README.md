## What A Container Actually Is

Практичний спосіб визначити контейнер такий: контейнер — це **звичайне дерево процесів Linux**, яке було запущено з певною OCI-style конфігурацією так, щоб воно бачило контрольовану файлову систему, контрольований набір ресурсів ядра та обмежену модель привілеїв. Процес може вважати себе PID 1, може думати, що має власний мережевий стек, власний hostname та IPC-ресурси, і навіть може працювати як root всередині власного user namespace. Але під капотом це все ще хост-процес, який ядро планує як і будь-який інший.

Саме тому безпека контейнерів — це, по суті, вивчення того, як ця ілюзія конструюється і як вона дає збій. Якщо mount namespace слабкий, процес може бачити файлову систему хоста. Якщо user namespace відсутній або вимкнений, root всередині контейнера може занадто тісно відображатися на root хоста. Якщо seccomp неконфігурований і набір capabilities занадто широкий, процес може досягати syscalls і привілейованих функцій ядра, які мали б залишатися недосяжними. Якщо runtime socket змонтовано всередині контейнера, контейнер може й зовсім не потребувати kernel breakout, бо він просто може попросити runtime запустити більш потужний sibling container або змонтувати root файлову систему хоста прямо.

## How Containers Differ From Virtual Machines

VM зазвичай несе власне ядро і межу апаратної абстракції. Це означає, що гостьове ядро може впасти, запанікувати або бути експлуатованим без автоматичного контролю над ядром хоста. У контейнерах робоче навантаження не отримує окремого ядра. Натомість воно отримує ретельно відфільтрований і іменований вид того самого ядра, яке використовує хост. Внаслідок цього контейнери зазвичай легші, швидше стартують, їх легше щільно упакувати на машині і вони краще підходять для короткоживучого розгортання застосунків. Ціна — ізоляційна межа значно більше залежить від правильної конфігурації хоста і runtime.

Це не означає, що контейнери «небезпечні», а VM — «безпечні». Це означає, що модель безпеки відрізняється. Добре налаштований стек контейнерів з rootless виконанням, user namespaces, дефолтним seccomp, суворим набором capabilities, без спільного використання namespace з хостом і з сильною примусовою політикою SELinux або AppArmor може бути дуже стійким. Навпаки, контейнер, запущений з `--privileged`, зі спільним PID/мережею хоста, з Docker socket змонтованим всередині та записуваним bind mount`ом `/` функціонально набагато ближчий до доступу root на хості, ніж до безпечного ізольованого sandbox. Відмінність виникає через шари, які були ввімкнені або вимкнені.

Існує також середня зона, яку читачі повинні розуміти, оскільки вона з'являється все частіше в реальних середовищах. **Sandboxed container runtimes** такі як **gVisor** і **Kata Containers** навмисно загартовують межу понад класичний `runc` контейнер. gVisor поміщає userspace kernel шар між робочим навантаженням і багатьма інтерфейсами ядра хоста, тоді як Kata запускає робоче навантаження всередині легковагової віртуальної машини. Їх все ще використовують через екосистеми контейнерів і оркестраційні workflow-и, але їхні властивості безпеки відрізняються від звичних OCI runtimes і їх не слід ментально групувати з «звичайними Docker containers», ніби все поводиться однаково.

## The Container Stack: Several Layers, Not One

Коли хтось каже «цей контейнер небезпечний», корисне уточнення: **який шар зробив його небезпечним?** Контейнеризоване робоче навантаження зазвичай є результатом роботи кількох компонентів разом.

На верхньому рівні часто є шар **image build** такий як BuildKit, Buildah або Kaniko, що створює OCI image і метадані. Над низькорівневим runtime може бути **engine or manager** такий як Docker Engine, Podman, containerd, CRI-O, Incus або systemd-nspawn. У кластерних середовищах також може бути **orchestrator** такий як Kubernetes, який визначає бажану безпекову позицію через конфігурацію робочого навантаження. Нарешті, **ядро** — це те, що фактично застосовує namespaces, cgroups, seccomp та MAC політику.

Ця багатошарова модель важлива для розуміння дефолтів. Обмеження може бути запрошене Kubernetes, трансльоване через CRI контейнерd чи CRI-O, перетворене у OCI spec runtime-обгорткою і лише потім застосоване `runc`, `crun`, `runsc` або іншим runtime проти ядра. Коли дефолти відрізняються між середовищами, це часто через те, що один з цих шарів змінив кінцеву конфігурацію. Той самий механізм тому може з'являтися в Docker або Podman як CLI прапорець, у Kubernetes як поле Pod або `securityContext`, і в нижчорівневих runtime-стеках як OCI конфігурація, згенерована для робочого навантаження. Саме з цієї причини CLI приклади в цьому розділі слід читати як **runtime-specific syntax для загальної концепції контейнера**, а не як універсальні прапорці, що підтримуються кожним інструментом.

## The Real Container Security Boundary

На практиці безпека контейнерів походить від **перекриваючихся контролів**, а не від одного досконалого контролю. Namespaces ізолюють видимість. cgroups управляють і обмежують використання ресурсів. Capabilities зменшують те, що привілейований на вигляд процес насправді може зробити. seccomp блокує небезпечні syscalls до того, як вони досягнуть ядра. AppArmor і SELinux додають Mandatory Access Control поверх звичайних DAC перевірок. `no_new_privs`, masked procfs paths і read-only system paths ускладнюють звичайні ланцюги зловживання привілеями і proc/sys. Сам runtime також важливий, бо він вирішує, як створюються mounts, sockets, labels і joins для namespace.

Ось чому багато документації з безпеки контейнерів здається повторюваною. Той самий ланцюг втечі часто залежить від декількох механізмів одночасно. Наприклад, записуваний host bind mount — це погано, але воно стає набагато гіршим, якщо контейнер також працює як реальний root на хості, має `CAP_SYS_ADMIN`, не обмежений seccomp і не контролюється SELinux або AppArmor. Так само, спільний PID хоста — серйозна вразливість, але вона стає набагато кориснішою для атакуючого, коли поєднується з `CAP_SYS_PTRACE`, слабкими захистами procfs або інструментами входу в namespace такими як `nsenter`. Правильний спосіб документувати тему тому не полягає в повторенні тієї самої атаки на кожній сторінці, а в поясненні того, що кожен шар додає до кінцевої межі.

## A Good First Enumeration Mindset

Під час оцінювання контейнеризованої цілі набагато корисніше ставити невелику кількість точних технічних питань, ніж одразу кидатися до відомих PoC втечі. Спочатку ідентифікуйте **stack**: Docker, Podman, containerd, CRI-O, Incus/LXC, systemd-nspawn, Apptainer або щось більш спеціалізоване. Потім ідентифікуйте **runtime**: `runc`, `crun`, `runsc`, `kata-runtime` або іншу OCI-compatible реалізацію. Далі перевірте, чи середовище **rootful чи rootless**, чи активні **user namespaces**, чи будь-які **host namespaces** спільні, які **capabilities** залишились, чи увімкнений **seccomp**, чи **MAC policy** справді застосовується, чи присутні **небезпечні mounts або sockets**, і чи процес може взаємодіяти з container runtime API.

Ці відповіді скажуть вам набагато більше про реальну безпекову позицію, ніж назва базового образу. У багатьох оцінках ви можете передбачити ймовірну сімейство втечі ще до того, як прочитаєте хоча б один файл застосунку, просто зрозумівши кінцеву конфігурацію контейнера.
