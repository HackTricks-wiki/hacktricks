# Безпека образів, підписування та секрети

{{#include ../../../banners/hacktricks-training.md}}

## Огляд

Безпека контейнерів починається до запуску робочого навантаження. Образ визначає, які бінарні файли, інтерпретатори, бібліотеки, скрипти запуску та вбудована конфігурація потраплять у продуктивне середовище. Якщо образ містить бекдор, є застарілим або побудований із вбудованими в нього секретами, подальше підсилення безпеки середовища виконання вже працюватиме з скомпрометованим артефактом.

Саме тому походження образу, сканування вразливостей, перевірка підписів та обробка секретів мають бути частиною тієї самої розмови, що й namespaces and seccomp. Вони захищають іншу фазу життєвого циклу, але помилки тут часто визначають поверхню атаки, яку середовище виконання пізніше має стримувати.

## Реєстри образів та довіра

Образи можуть надходити з публічних реєстрів, таких як Docker Hub, або з приватних реєстрів, які експлуатує організація. Питання безпеки полягає не тільки в тому, де зберігається образ, а в тому, чи може команда встановити його походження та цілісність. Завантаження непідписаних або слабо відстежуваних образів з публічних джерел підвищує ризик потрапляння шкідливого або підробленого контенту в продуктивне середовище. Навіть внутрішні реєстри потребують чіткого володіння, перевірки та політики довіри.

Docker Content Trust історично використовував концепції Notary і TUF для вимоги підписаних образів. Конкретна екосистема змінювалася, але стале правило залишається корисним: ідентичність і цілісність образу повинні бути перевіреними, а не прийматися за замовчуванням.

Приклад історичного робочого процесу Docker Content Trust:
```bash
export DOCKER_CONTENT_TRUST=1
docker pull nginx:latest
tar -zcvf private_keys_backup.tar.gz ~/.docker/trust/private
```
Сенс цього прикладу не в тому, що кожна команда повинна використовувати однакові інструменти, а в тому, що підписування та управління ключами — це операційні завдання, а не абстрактна теорія.

## Сканування вразливостей

Сканування образів допомагає відповісти на два різні питання. По-перше, чи містить образ відомі вразливі пакети або бібліотеки? По-друге, чи містить образ непотрібне програмне забезпечення, що розширює площу атаки? Образ, переповнений інструментами для налагодження, оболонками, інтерпретаторами та застарілими пакетами, одночасно легше піддається експлуатації і складніший для аналізу.

Приклади поширених сканерів включають:
```bash
docker scan hello-world
trivy -q -f json alpine:3.19
snyk container test nginx:latest --severity-threshold=high
clair-scanner -w example-alpine.yaml --ip YOUR_LOCAL_IP alpine:3.5
```
Результати, отримані цими інструментами, слід інтерпретувати обережно. Вразливість у невикористовуваному пакеті має інший рівень ризику порівняно з відкритим RCE шляхом, але обидва мають значення для рішень щодо зміцнення безпеки.

## Секрети під час збірки

Одна з найстаріших помилок у pipeline-ах збірки контейнерів — вбудовувати секрети безпосередньо в образ або передавати їх через змінні середовища, які згодом стають видимими через `docker inspect`, логи збірки або відновлені шари. Секрети під час збірки слід підключати тимчасово під час збірки, замість того щоб копіювати їх у файлову систему образу.

BuildKit покращив цю модель, дозволивши спеціалізовану обробку секретів під час збірки. Замість запису секрету у шар, крок збірки може тимчасово споживати його:
```bash
export DOCKER_BUILDKIT=1
docker build --secret id=my_key,src=path/to/my_secret_file .
```
Це важливо, тому що шари образу є стійкими артефактами. Коли секрет потрапляє до зафіксованого шару, подальше видалення файлу в іншому шарі насправді не видаляє початкового розкриття з історії образу.

## Секрети під час виконання

Секрети, необхідні для запущеного робочого навантаження, також слід уникати ad hoc підходів, таких як прості змінні середовища, коли це можливо. Томи, спеціалізовані інтеграції для управління секретами, Docker secrets і Kubernetes Secrets — поширені механізми. Жоден із них не усуває всіх ризиків, особливо якщо атакуючий уже має виконання коду в робочому навантаженні, але вони все одно кращі за постійне зберігання облікових даних в образі або їх випадкове розкриття через інструменти інспекції.

Проста декларація секрету у стилі Docker Compose виглядає так:
```yaml
version: "3.7"
services:
my_service:
image: centos:7
entrypoint: "cat /run/secrets/my_secret"
secrets:
- my_secret
secrets:
my_secret:
file: ./my_secret_file.txt
```
У Kubernetes, Secret objects, projected volumes, service-account tokens і cloud workload identities створюють ширшу й потужнішу модель, але водночас вони відкривають більше можливостей для випадкового розкриття через host mounts, надміру широкі RBAC або слабкий дизайн Pod.

## Зловживання

Під час огляду цілі мета — з'ясувати, чи secrets були вбудовані в image, leaked into layers, або змонтовані в передбачувані runtime locations:
```bash
env | grep -iE 'secret|token|key|passwd|password'
find / -maxdepth 4 \( -iname '*.env' -o -iname '*secret*' -o -iname '*token*' \) 2>/dev/null | head -n 100
grep -RniE 'secret|token|apikey|password' /app /srv /usr/src 2>/dev/null | head -n 100
```
Ці команди допомагають розрізнити три різні проблеми: витоки конфігурації застосунку (application configuration leaks), витоки шару образу (image-layer leaks) та runtime-injected secret files. Якщо секрет з'являється під `/run/secrets`, у projected volume або на cloud identity token path, наступним кроком є з'ясувати, чи надає він доступ лише до поточного workload або до значно ширшої control plane.

### Повний приклад: вбудований секрет у файловій системі образу

Якщо конвеєр збірки скопіював файли `.env` або облікові дані у фінальний образ, post-exploitation стає простим:
```bash
find / -type f -iname '*.env*' 2>/dev/null
cat /usr/src/app/.env 2>/dev/null
grep -iE 'secret|token|jwt|password' /usr/src/app/.env 2>/dev/null
```
Вплив залежить від застосунку, проте вбудовані ключі підпису, секрети JWT або хмарні облікові дані можуть легко перетворити компрометацію контейнера на компрометацію API, латеральне переміщення або підробку довірених токенів застосунку.

### Повний приклад: Build-Time Secret Leakage Check

Якщо є побоювання, що історія образу зафіксувала шар, який містить секрети:
```bash
docker history --no-trunc <image>
docker save <image> -o /tmp/image.tar
tar -tf /tmp/image.tar | head
```
Такий огляд корисний, оскільки секрет міг бути видалений з кінцевого вигляду файлової системи, але все ще залишатися в попередньому шарі або в метаданих збірки.

## Перевірки

Ці перевірки призначені для з'ясування, чи ймовірно, що образ і конвеєр обробки секретів збільшили поверхню атаки перед виконанням.
```bash
docker history --no-trunc <image> 2>/dev/null
env | grep -iE 'secret|token|key|passwd|password'
find /run /var/run /var/lib/kubelet -type f -iname '*token*' 2>/dev/null | head -n 50
grep -RniE 'secret|token|apikey|password' /etc /app /srv /usr/src 2>/dev/null | head -n 100
```
Що тут цікаво:

- Підозріла історія збірок може виявити скопійовані облікові дані, SSH-матеріали або небезпечні кроки збірки.
- Secrets під шляхами проєкційних томів можуть надати доступ до кластера або хмари, а не лише до локального застосунку.
- Велика кількість конфігураційних файлів з обліковими даними у відкритому тексті зазвичай вказує, що образ або модель розгортання несуть більше матеріалів довіри, ніж потрібно.

## Налаштування за замовчуванням

| Runtime / платформа | Стан за замовчуванням | Поведінка за замовчуванням | Поширені ручні послаблення |
| --- | --- | --- | --- |
| Docker / BuildKit | Підтримує безпечні монтовані секрети під час збірки, але не автоматично | Секрети можна тимчасово змонтувати під час `build`; підписування та сканування образів вимагають явного вибору workflow | копіювання секретів в образ, передача секретів через `ARG` або `ENV`, відключення перевірок provenance |
| Podman / Buildah | Підтримує OCI-native builds та secret-aware workflows | Доступні надійні workflow для збірки, але оператори мають свідомо їх обирати | вбудовування секретів у Containerfiles, надто широкі контексти збірки, дозволяючі bind mounts під час збірки |
| Kubernetes | Нативні об'єкти Secret і проєкційні томи | Доставка секретів у рантаймі є першокласною функцією, але ризик розкриття залежить від RBAC, дизайну pod та host mounts | занадто широкі монтування Secret, неправильне використання service-account token, доступ через `hostPath` до томів, що керуються kubelet |
| Registries | Інтегритет опціональний, якщо його не примусити | Публічні та приватні реєстри залежать від політик, підписування та рішень admission | вільне завантаження непідписаних образів, слабкий admission control, погане управління ключами |
