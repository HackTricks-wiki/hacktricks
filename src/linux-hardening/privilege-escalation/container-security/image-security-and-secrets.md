# Безпека образів, підписування та секрети

{{#include ../../../banners/hacktricks-training.md}}

## Огляд

Безпека контейнерів починається ще до запуску робочого навантаження. Образ визначає, які бінарні файли, інтерпретатори, бібліотеки, скрипти запуску та вбудовані конфігурації потраплять у продуктивне середовище. Якщо образ містить бекдор, застарілий або зібраний із вбудованими секретами, то подальше посилення безпеки під час виконання вже працює зі скомпрометованим артефактом.

Саме тому питання походження образу, сканування на вразливості, перевірки підписів та обробки секретів належать до тієї ж дискусії, що й namespaces і seccomp. Вони захищають іншу фазу життєвого циклу, але помилки тут часто визначають поверхню атаки, яку пізніше має обмежувати середовище виконання.

## Реєстри образів та довіра

Образи можуть надходити з публічних реєстрів, таких як Docker Hub, або з приватних реєстрів, які обслуговує організація. Питання безпеки — не лише де зберігається образ, а чи може команда встановити його походження та цілісність. Завантаження непідписаних або погано відстежуваних образів з публічних джерел підвищує ризик потрапляння в продуктивне середовище шкідливого або підробленого вмісту. Навіть внутрішні реєстри потребують чіткого визначення власника, процесу перегляду та політики довіри.

Docker Content Trust історично використовував концепції Notary і TUF для вимоги підписаних образів. Точна екосистема еволюціонувала, але сталий урок лишається корисним: ідентичність і цілісність образу повинні бути перевірюваними, а не вважатися належним. 

Приклад історичного робочого процесу Docker Content Trust:
```bash
export DOCKER_CONTENT_TRUST=1
docker pull nginx:latest
tar -zcvf private_keys_backup.tar.gz ~/.docker/trust/private
```
Суть прикладу не в тому, що кожна команда має використовувати однакові інструменти, а в тому, що signing і key management — це операційні завдання, а не абстрактна теорія.

## Сканування вразливостей

Сканування образу допомагає відповісти на два різні питання. По-перше, чи містить образ відомі вразливі пакети або бібліотеки? По-друге, чи містить образ непотрібне програмне забезпечення, що розширює attack surface? Образ, наповнений інструментами відладки, shells, інтерпретаторами та застарілими пакетами, простіше експлуатувати і важче аналізувати.

Приклади часто використовуваних сканерів включають:
```bash
docker scan hello-world
trivy -q -f json alpine:3.19
snyk container test nginx:latest --severity-threshold=high
clair-scanner -w example-alpine.yaml --ip YOUR_LOCAL_IP alpine:3.5
```
Results from these tools should be interpreted carefully. A vulnerability in an unused package is not identical in risk to an exposed RCE path, but both are still relevant to hardening decisions.

## Build-Time Secrets

Одна з найдавніших помилок у pipeline'ах збірки контейнерів — вбудовувати secrets безпосередньо в image або передавати їх через environment variables, які згодом стають видимими через `docker inspect`, build logs або відновлені layers. Build-time secrets слід монтувати тимчасово під час збірки, замість копіювання у файлову систему image.

BuildKit покращив цю модель, дозволивши виділену обробку build-time secrets. Замість запису secret у layer, крок збірки може тимчасово його використати:
```bash
export DOCKER_BUILDKIT=1
docker build --secret id=my_key,src=path/to/my_secret_file .
```
Це важливо, тому що шари образу є стійкими артефактами. Якщо секрет потрапив у зафіксований шар, пізніше видалення файлу в іншому шарі не видаляє фактичного розкриття з історії образу.

## Секрети під час виконання

Секрети, потрібні виконуваному навантаженню, також слід уникати ad hoc підходів, таких як прості змінні середовища, коли це можливо. Томи, спеціалізовані інтеграції для управління секретами, Docker secrets та Kubernetes Secrets — поширені механізми. Жоден із них не усуває всіх ризиків, особливо якщо нападник вже має виконання коду в навантаженні, але вони все одно кращі за постійне зберігання облікових даних в образі або їх неуважне розкриття через інструменти інспекції.

Проста декларація секрета в стилі Docker Compose виглядає так:
```yaml
version: "3.7"
services:
my_service:
image: centos:7
entrypoint: "cat /run/secrets/my_secret"
secrets:
- my_secret
secrets:
my_secret:
file: ./my_secret_file.txt
```
У Kubernetes Secret objects, projected volumes, service-account tokens та cloud workload identities створюють ширшу й потужнішу модель, але вони також створюють більше можливостей для випадкового розкриття через монтування на хості, надмірно широкі RBAC або слабкий дизайн Pod.

## Зловживання

Під час огляду цілі мета — з'ясувати, чи були secrets запечені в образі, leaked у шарах, або змонтовані в передбачувані місця виконання:
```bash
env | grep -iE 'secret|token|key|passwd|password'
find / -maxdepth 4 \( -iname '*.env' -o -iname '*secret*' -o -iname '*token*' \) 2>/dev/null | head -n 100
grep -RniE 'secret|token|apikey|password' /app /srv /usr/src 2>/dev/null | head -n 100
```
Ці команди допомагають розрізняти три різні проблеми: application configuration leaks, image-layer leaks та runtime-injected secret files. Якщо секрет з'являється в `/run/secrets`, у projected volume або на cloud identity token path, наступним кроком є з'ясувати, чи надає він доступ лише до поточного workload або до значно більшого control plane.

### Повний приклад: вбудований секрет у файловій системі образу

Якщо конвеєр збірки скопіював файли `.env` або облікові дані у фінальний образ, post-exploitation стає простим:
```bash
find / -type f -iname '*.env*' 2>/dev/null
cat /usr/src/app/.env 2>/dev/null
grep -iE 'secret|token|jwt|password' /usr/src/app/.env 2>/dev/null
```
Наслідки залежать від застосунку, але embedded signing keys, JWT secrets або cloud credentials можуть легко перетворити container compromise на API compromise, lateral movement або forgery of trusted application tokens.

### Повний приклад: Build-Time Secret Leakage Check

Якщо існує занепокоєння, що image history захопив шар, який містить секрети:
```bash
docker history --no-trunc <image>
docker save <image> -o /tmp/image.tar
tar -tf /tmp/image.tar | head
```
Така перевірка корисна, оскільки secret може бути видалено з остаточного подання файлової системи, при цьому все ще залишаючись у попередньому шарі або в метаданих збірки.

## Checks

Ці перевірки мають на меті встановити, чи, ймовірно, образ і secret-handling pipeline збільшили поверхню атаки перед виконанням.
```bash
docker history --no-trunc <image> 2>/dev/null
env | grep -iE 'secret|token|key|passwd|password'
find /run /var/run /var/lib/kubelet -type f -iname '*token*' 2>/dev/null | head -n 50
grep -RniE 'secret|token|apikey|password' /etc /app /srv /usr/src 2>/dev/null | head -n 100
```
Що цікаво тут:

- Підозріла історія збірок може виявити скопійовані облікові дані, SSH-ключі або небезпечні кроки збірки.
- Secrets у шляхах проєктованих томів можуть надати доступ до кластера або хмари, а не лише до локального застосунку.
- Велика кількість конфігураційних файлів із обліковими даними у відкритому тексті зазвичай вказує на те, що образ або модель розгортання містить більше довірчого матеріалу, ніж потрібно.

## Налаштування часу виконання за замовчуванням

| Runtime / platform | Default state | Default behavior | Common manual weakening |
| --- | --- | --- | --- |
| Docker / BuildKit | Підтримує безпечні монтування секретів під час збірки, але не автоматично | Секрети можна тимчасово змонтувати під час `build`; підписування та сканування образів вимагають явних рішень у робочому процесі | копіювання секретів в образ, передача секретів через `ARG` або `ENV`, відключення перевірок походження (provenance) |
| Podman / Buildah | Підтримує OCI-native збірки та робочі процеси, що враховують секрети | Доступні надійні робочі процеси для збірки, але оператори мають обирати їх свідомо | вбудовування секретів у Containerfiles, широкі контексти збірки, надмірно дозволені bind mounts під час збірок |
| Kubernetes | Нативні об'єкти Secret і проєктовані томи | Доставка секретів під час виконання є пріоритетною, але експозиція залежить від RBAC, конструкції pod та host mounts | надмірно широкі монтування Secret, неправильне використання токенів service-account, `hostPath` доступ до томів, керованих kubelet |
| Registries | Цілісність опціональна, якщо її не примусити | Публічні та приватні registries залежать від політик, підписування та рішень admission | вільне витягування непідписаних образів, слабкий admission control, погане керування ключами |
