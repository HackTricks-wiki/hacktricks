# Огляд захисту контейнерів

{{#include ../../../../banners/hacktricks-training.md}}

Найважливіша ідея в hardening контейнерів — це те, що не існує єдиного контролю під назвою «безпека контейнера». Те, що люди називають container isolation, насправді є результатом сумісної роботи кількох механізмів безпеки та управління ресурсами в Linux. Якщо документація описує лише один із них, читачі схильні переоцінювати його силу. Якщо документація перераховує всі, але не пояснює їх взаємодію, читачі отримують лише каталог назв без реальної моделі. Цей розділ намагається уникнути обох помилок.

У центрі моделі — **namespaces**, які ізолюють те, що може побачити workload. Вони дають процесу приватний або частково приватний вигляд монтувань файлової системи, PID-ів, мережі, IPC-об’єктів, hostname-ів, відображень user/group, шляхів cgroup та деяких годинників. Але самі по собі namespaces не визначають, що процесу дозволено робити. Саме тут вступають у дію інші шари.

**cgroups** регулюють використання ресурсів. Вони не є ізоляційною межею в тому ж сенсі, що mount або PID namespaces, але мають критичне операційне значення, бо обмежують пам’ять, CPU, PIDs, I/O та доступ до пристроїв. Вони також мають безпекове значення, оскільки історично хитрощі для виходу з контейнера зловживали записуваними можливостями cgroup, особливо в середовищах cgroup v1.

**Capabilities** розбили стару всесильну модель root на менші одиниці привілеїв. Це фундаментально для контейнерів, бо багато workload-ів досі працюють як UID 0 всередині контейнера. Тому питання вже не просто «чи є процес root?», а «які capabilities збереглися, всередині яких namespaces, під якими обмеженнями seccomp та MAC?». Саме тому процес root в одному контейнері може бути відносно обмеженим, тоді як root в іншому контейнері практично не відрізняється від host root.

**seccomp** фільтрує syscalls і зменшує поверхню атаки ядра, видиму для workload. Це часто механізм, який блокує очевидно небезпечні виклики, такі як `unshare`, `mount`, `keyctl` або інші syscalls, що використовуються в ланцюгах для breakout. Навіть якщо процес має capability, яка в іншому випадку дозволила б операцію, seccomp може заблокувати шлях виклику syscall ще до того, як ядро повністю його опрацює.

**AppArmor** і **SELinux** додають Mandatory Access Control поверх звичайних перевірок файлової системи та привілеїв. Вони особливо важливі, бо продовжують мати значення навіть коли контейнер має більше capabilities, ніж повинен. Workload може мати теоретичне привілейоване право спробувати дію, але все одно бути забороненим у виконанні через мітку або профіль, які не дозволяють доступ до відповідного шляху, об’єкта чи операції.

Нарешті, є додаткові шари hardening, які отримують менше уваги, але регулярно мають значення в реальних атаках: `no_new_privs`, masked procfs paths, read-only system paths, read-only root filesystems та обачні runtime defaults. Ці механізми часто зупиняють «останній крок» компрометації, особливо коли атакуючий намагається перетворити виконання коду на ширше підвищення привілеїв.

Решта цієї папки пояснює кожен з цих механізмів детальніше, включно з тим, що насправді робить примітива ядра, як спостерігати її локально, як її використовують поширені runtimes і як оператори випадково її послаблюють.
