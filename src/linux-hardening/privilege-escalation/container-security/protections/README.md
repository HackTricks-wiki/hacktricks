# Огляд захисту контейнерів

{{#include ../../../../banners/hacktricks-training.md}}

Найважливіша ідея при підвищенні безпеки контейнерів полягає в тому, що не існує єдиного контролю під назвою "container security". Те, що люди називають ізоляцією контейнера, насправді є результатом спільної роботи кількох механізмів Linux для безпеки та управління ресурсами. Якщо документація описує лише один із них, читачі схильні переоцінювати його силу. Якщо документація перераховує всі без пояснення їх взаємодії, читачі отримують каталог назв, але не реальну модель. Цей розділ намагається уникнути обох помилок.

У центрі моделі знаходяться **namespaces**, які ізолюють те, що може бачити workload. Вони дають процесу приватний або частково приватний вигляд точок монтування файлової системи, PIDs, мережі, IPC-об'єктів, hostnames, відображень user/group, cgroup шляхів і деяких годинників. Але самі по собі namespaces не визначають, що дозволено процесу. Тут вступають у гру наступні рівні.

**cgroups** регулюють використання ресурсів. Вони не є первинним межовим механізмом ізоляції в тому ж сенсі, що mount або PID namespaces, але критично важливі оперативно, оскільки вони обмежують пам'ять, CPU, PIDs, I/O та доступ до пристроїв. Вони також мають безпекове значення, бо історично техніки виходу з контейнера зловживали можливістю запису в cgroup, особливо в середовищах cgroup v1.

**Capabilities** розбивають стару всевладну модель root на менші одиниці привілеїв. Це фундаментально для контейнерів, оскільки багато workloads все ще працюють як UID 0 всередині контейнера. Тому питання не стільки "чи є процес root?", скільки "які capabilities збереглися, всередині яких namespaces, під якими обмеженнями seccomp і MAC?" Саме тому процес root в одному контейнері може бути відносно обмеженим, тоді як процес root в іншому контейнері на практиці може бути майже не відрізнити від host root.

**seccomp** фільтрує syscalls і зменшує площу атаки ядра, відкриту для workload. Часто саме цей механізм блокує очевидно небезпечні виклики, такі як `unshare`, `mount`, `keyctl` або інші syscalls, що використовуються в ланцюжках for breakout. Навіть якщо процес має capability, яка в іншому випадку дозволила б операцію, seccomp все ще може заблокувати шлях системного виклику до того, як ядро повністю його обробить.

**AppArmor** і **SELinux** додають Mandatory Access Control поверх звичайних перевірок файлової системи та привілеїв. Вони особливо важливі, оскільки продовжують мати значення навіть коли контейнер має більше capabilities, ніж повинен. Workload може теоретично мати привілей для спроби дії, але все одно бути забороненим від її виконання через те, що його label або profile забороняє доступ до відповідного шляху, об'єкта чи операції.

Нарешті, існують додаткові шари hardening, яким приділяють менше уваги, але які регулярно мають значення в реальних атаках: `no_new_privs`, masked procfs paths, read-only system paths, read-only root filesystems та обережні runtime defaults. Ці механізми часто зупиняють "останню милю" компрометації, особливо коли атакуючий намагається перетворити виконання коду на ширший приріст привілеїв.

Решта цього каталогу пояснює кожен із цих механізмів детальніше, включаючи те, що насправді робить примітив ядра, як спостерігати його локально, як його використовують поширені runtimes і як оператори випадково послаблюють його.
