# IPC Namespace

{{#include ../../../../../banners/hacktricks-training.md}}

## Επισκόπηση

Το IPC namespace απομονώνει **System V IPC objects** και **POSIX message queues**. Αυτό περιλαμβάνει shared memory segments, semaphores και message queues που διαφορετικά θα ήταν ορατά μεταξύ μη σχετιζόμενων διεργασιών στο host. Σε πρακτικό επίπεδο, αυτό αποτρέπει ένα container από το να συνδεθεί ελαφρά σε IPC objects που ανήκουν σε άλλες workloads ή στον host.

Σε σύγκριση με τα mount, PID ή user namespaces, το IPC namespace συχνά συζητείται λιγότερο, αλλά αυτό δεν πρέπει να συγχέεται με μη σημαντικότητα. Το shared memory και οι σχετικοί IPC μηχανισμοί μπορούν να περιέχουν πολύ χρήσιμο state. Αν το host IPC namespace είναι exposed, η workload μπορεί να αποκτήσει ορατότητα σε αντικείμενα συντονισμού μεταξύ διεργασιών ή δεδομένα που δεν προορίζονταν ποτέ να περάσουν το όριο του container.

## Λειτουργία

Όταν το runtime δημιουργεί ένα νέο IPC namespace, η διεργασία αποκτά το δικό της απομονωμένο σύνολο IPC identifiers. Αυτό σημαίνει ότι εντολές όπως `ipcs` εμφανίζουν μόνο τα αντικείμενα διαθέσιμα σε εκείνο το namespace. Αν το container αντίθετα ενταχθεί στο host IPC namespace, αυτά τα αντικείμενα γίνονται μέρος μιας κοινής global θέασης.

Αυτό έχει ιδιαίτερη σημασία σε περιβάλλοντα όπου εφαρμογές ή υπηρεσίες χρησιμοποιούν shared memory εκτενώς. Ακόμα κι όταν το container δεν μπορεί να ξεφύγει (break out) απευθείας μέσω IPC μόνο, το namespace μπορεί να leak πληροφορίες ή να επιτρέψει παρεμβολές μεταξύ διεργασιών που ουσιαστικά βοηθούν μια μετέπειτα επίθεση.

## Εργαστήριο

Μπορείτε να δημιουργήσετε ένα ιδιωτικό IPC namespace με:
```bash
sudo unshare --ipc --fork bash
ipcs
```
Και συγκρίνετε τη συμπεριφορά εκτέλεσης με:
```bash
docker run --rm debian:stable-slim ipcs
docker run --rm --ipc=host debian:stable-slim ipcs
```
## Χρήση χρόνου εκτέλεσης

Docker και Podman απομονώνουν το IPC εξ ορισμού. Το Kubernetes συνήθως δίνει στο Pod το δικό του IPC namespace, που μοιράζεται από τα containers στο ίδιο Pod αλλά όχι εξ ορισμού με το host. Η κοινή χρήση του host IPC είναι δυνατή, αλλά πρέπει να θεωρείται ουσιαστική μείωση της απομόνωσης αντί για μια μικρή runtime επιλογή.

## Λανθασμένες ρυθμίσεις

Το προφανές λάθος είναι `--ipc=host` ή `hostIPC: true`. Αυτό μπορεί να γίνει για συμβατότητα με παλαιό λογισμικό ή για ευκολία, αλλά αλλάζει σημαντικά το μοντέλο εμπιστοσύνης. Ένα άλλο επαναλαμβανόμενο ζήτημα είναι η απλή παράβλεψη του IPC επειδή φαίνεται λιγότερο δραματικό από το host PID ή το host networking. Στην πραγματικότητα, αν το φορτίο εργασίας διαχειρίζεται browsers, databases, επιστημονικά φορτία εργασίας ή άλλο λογισμικό που κάνει εκτενή χρήση κοινής μνήμης, η επιφάνεια IPC μπορεί να είναι πολύ σημαντική.

## Κατάχρηση

Όταν το host IPC είναι κοινόχρηστο, ένας επιτιθέμενος μπορεί να ελέγξει ή να παρέμβει σε αντικείμενα κοινής μνήμης, να αποκτήσει νέες πληροφορίες για τη συμπεριφορά του host ή γειτονικών φορτίων εργασίας, ή να συνδυάσει τις πληροφορίες που συλλέγει εκεί με ορατότητα διεργασιών και ptrace-style δυνατότητες. Η κοινή χρήση IPC συχνά αποτελεί υποστηρικτική αδυναμία παρά την πλήρη διαδρομή απόδρασης, αλλά οι υποστηρικτικές αδυναμίες έχουν σημασία επειδή συντομεύουν και σταθεροποιούν τις πραγματικές αλυσίδες επίθεσης.

Το πρώτο χρήσιμο βήμα είναι να απαριθμήσετε ποια αντικείμενα IPC είναι ορατά:
```bash
readlink /proc/self/ns/ipc
ipcs -a
ls -la /dev/shm 2>/dev/null | head -n 50
```
Εάν το host IPC namespace είναι κοινό, μεγάλα shared-memory segments ή ενδιαφέροντες object owners μπορούν να αποκαλύψουν άμεσα τη συμπεριφορά της εφαρμογής:
```bash
ipcs -m -p
ipcs -q -p
```
Σε ορισμένα περιβάλλοντα, τα περιεχόμενα του `/dev/shm` leak filenames, artifacts ή tokens που αξίζει να ελεγχθούν:
```bash
find /dev/shm -maxdepth 2 -type f 2>/dev/null -ls | head -n 50
strings /dev/shm/* 2>/dev/null | head -n 50
```
Η κοινοποίηση του IPC σπάνια δίνει άμεσο host root από μόνη της, αλλά μπορεί να εκθέσει κανάλια δεδομένων και συντονισμού που κάνουν τις μετέπειτα επιθέσεις σε διεργασίες πολύ πιο εύκολες.

### Πλήρες Παράδειγμα: `/dev/shm` Ανάκτηση Μυστικών

Η πιο ρεαλιστική πλήρης περίπτωση κατάχρησης είναι η κλοπή δεδομένων παρά η άμεση απόδραση. Εάν το host IPC ή μια ευρεία διάταξη κοινής μνήμης είναι εκτεθειμένα, ευαίσθητα artifacts μπορούν μερικές φορές να ανακτηθούν άμεσα:
```bash
find /dev/shm -maxdepth 2 -type f 2>/dev/null -print
strings /dev/shm/* 2>/dev/null | grep -Ei 'token|secret|password|jwt|key'
```
Επιπτώσεις:

- εξαγωγή μυστικών ή υλικού συνεδρίας που έχουν μείνει σε κοινή μνήμη
- πληροφορίες για τις εφαρμογές που είναι ενεργές στον host
- καλύτερος στοχευτισμός για μετέπειτα επιθέσεις που βασίζονται σε PID-namespace ή ptrace

Η κοινή χρήση IPC κατανοείται επομένως καλύτερα ως ένας **ενισχυτής επίθεσης** παρά ως ένα ανεξάρτητο primitive για host-escape.

## Έλεγχοι

Αυτές οι εντολές έχουν ως σκοπό να απαντήσουν εάν το workload έχει ιδιωτική προβολή IPC, εάν εμφανίζονται ουσιαστικά αντικείμενα κοινής μνήμης ή μηνυμάτων, και εάν το `/dev/shm` από μόνο του αποκαλύπτει χρήσιμα στοιχεία.
```bash
readlink /proc/self/ns/ipc   # Namespace identifier for IPC
ipcs -a                      # Visible SysV IPC objects
mount | grep shm             # Shared-memory mounts, especially /dev/shm
```
Τι ενδιαφέρον υπάρχει εδώ:

- Αν το `ipcs -a` αποκαλύπτει αντικείμενα που ανήκουν σε απροσδόκητους χρήστες ή υπηρεσίες, το namespace μπορεί να μην είναι τόσο απομονωμένο όσο αναμενόταν.
- Μεγάλα ή ασυνήθιστα τμήματα shared memory συχνά αξίζουν περαιτέρω διερεύνηση.
- Ένα ευρύ mount του `/dev/shm` δεν είναι αυτόματα bug, αλλά σε ορισμένα περιβάλλοντα leaks ονόματα αρχείων, artifacts και παροδικά μυστικά.

Το IPC σπάνια λαμβάνει τόση προσοχή όσο οι μεγαλύτεροι τύποι namespace, αλλά σε περιβάλλοντα που το χρησιμοποιούν έντονα, η κοινή χρήση του με τον host είναι σαφώς μία απόφαση ασφαλείας.
