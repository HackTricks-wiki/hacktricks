# Простори імен

{{#include ../../../../../banners/hacktricks-training.md}}

Простори імен — це функція ядра, яка змушує контейнер відчувати себе «як окрема машина», хоча насправді це просто дерево процесів хоста. Вони не створюють нового ядра і не віртуалізують усе, але дозволяють ядру показувати різні уявлення про вибрані ресурси для різних груп процесів. Це серце ілюзії контейнера: робоче навантаження бачить файлову систему, таблицю процесів, мережевий стек, ім'я хоста, IPC-ресурси та модель ідентифікації користувачів/груп, що здаються локальними, хоча підлегла система є спільною.

Ось чому простори імен — це перша концепція, з якою знайомляться більшість людей, коли вивчають, як працюють контейнери. Водночас це одна з найчастіше непорозумілих концепцій, бо читачі часто припускають, що «має простори імен» означає «безпечна ізоляція». Насправді простір імен ізолює лише конкретний клас ресурсів, для якого він призначений. Процес може мати приватний PID-namespace і все одно бути небезпечним, бо має змонтований записуваний bind mount хоста. Він може мати приватний network namespace і все одно бути небезпечним, бо зберігає `CAP_SYS_ADMIN` і працює без seccomp. Простіри імен є фундаментальними, але вони лише один із шарів у підсумковому кордоні ізоляції.

## Типи просторів імен

Linux-контейнери зазвичай одночасно покладаються на кілька типів просторів імен. Простір імен монтувань дає процесу окрему таблицю монтувань і, отже, контрольований вигляд файлової системи. Простір імен PID змінює видимість і нумерацію процесів, щоб робоче навантаження бачило власне дерево процесів. Простір імен мережі ізолює інтерфейси, маршрути, сокети та стан брандмауера. Простір імен IPC ізолює SysV IPC і POSIX черги повідомлень. Простір імен UTS ізолює ім'я хоста й NIS-домен. Простір імен користувачів перемаповує UID і GID так, що root всередині контейнера не обов'язково означає root на хості. Простір імен cgroup віртуалізує видиму ієрархію cgroup, а простір імен часу віртуалізує вибрані годинники в новіших ядрах.

Кожен із цих просторів імен вирішує різну проблему. Тому практичний аналіз безпеки контейнера часто зводиться до перевірки того, **які простори імен ізольовані**, а **які навмисно поділені з хостом**.

## Спільне використання простору імен хоста

Багато виходів із контейнера не починаються з вразливості ядра. Вони починаються з того, що оператор навмисно послаблює модель ізоляції. Приклади `--pid=host`, `--network=host` і `--userns=host` — це **Docker/Podman-style CLI flags**, приведені тут як конкретні приклади спільного використання простору імен хоста. Інші рантайми виражають ту саму ідею інакше. У Kubernetes еквіваленти зазвичай з'являються як налаштування Pod, наприклад `hostPID: true`, `hostNetwork: true` або `hostIPC: true`. У низькорівневих стеків рантайму, таких як containerd або CRI-O, така ж поведінка часто досягається через згенеровану OCI runtime configuration, а не через прапорець з тією ж назвою. У всіх цих випадках результат подібний: робоче навантаження більше не отримує стандартний ізольований вигляд простору імен.

Ось чому огляд просторів імен не повинен обмежуватися «процес знаходиться в якомусь просторі імен». Важливе питання — чи простір імен приватний для контейнера, поділений із сусідніми контейнерами, або безпосередньо приєднаний до хоста. У Kubernetes та сама ідея з'являється з прапорцями типу `hostPID`, `hostNetwork` і `hostIPC`. Назви змінюються між платформами, але шаблон ризику однаковий: спільний простір імен хоста робить залишкові привілеї контейнера і доступний стан хоста значно більш значущими.

## Інспекція

Найпростіший огляд такий:
```bash
ls -l /proc/self/ns
```
Кожен запис — символічне посилання з ідентифікатором, схожим на inode. Якщо два процеси вказують на той самий ідентифікатор namespace, вони перебувають в одному й тому ж namespace цього типу. Це робить `/proc` дуже корисним місцем для порівняння поточного процесу з іншими цікавими процесами на машині.

Ці швидкі команди часто достатні, щоб почати:
```bash
readlink /proc/self/ns/mnt
readlink /proc/self/ns/pid
readlink /proc/self/ns/net
readlink /proc/1/ns/mnt
```
Звідти наступним кроком є порівняння процесу контейнера з процесами на хості або сусідніми процесами, щоб визначити, чи справді простір імен є приватним.

### Перерахування екземплярів простору імен з хоста

Якщо у вас уже є доступ до хоста і ви хочете зрозуміти, скільки різних екземплярів простору імен певного типу існує, `/proc` дає швидкий перелік:
```bash
sudo find /proc -maxdepth 3 -type l -name mnt    -exec readlink {} \; 2>/dev/null | sort -u
sudo find /proc -maxdepth 3 -type l -name pid    -exec readlink {} \; 2>/dev/null | sort -u
sudo find /proc -maxdepth 3 -type l -name net    -exec readlink {} \; 2>/dev/null | sort -u
sudo find /proc -maxdepth 3 -type l -name ipc    -exec readlink {} \; 2>/dev/null | sort -u
sudo find /proc -maxdepth 3 -type l -name uts    -exec readlink {} \; 2>/dev/null | sort -u
sudo find /proc -maxdepth 3 -type l -name user   -exec readlink {} \; 2>/dev/null | sort -u
sudo find /proc -maxdepth 3 -type l -name cgroup -exec readlink {} \; 2>/dev/null | sort -u
sudo find /proc -maxdepth 3 -type l -name time   -exec readlink {} \; 2>/dev/null | sort -u
```
Якщо ви хочете знайти, які процеси належать певному namespace-ідентифікатору, замініть `readlink` на `ls -l` і виконайте grep за номером цільового namespace:
```bash
sudo find /proc -maxdepth 3 -type l -name mnt -exec ls -l {} \; 2>/dev/null | grep <ns-number>
```
Ці команди корисні, оскільки дозволяють встановити, чи запускає хост одну ізольовану задачу, багато ізольованих задач або суміш спільних та приватних namespace інстансів.

### Вхід у цільовий namespace

Коли виконавець має достатні привілеї, `nsenter` є стандартним способом приєднатися до namespace іншого процесу:
```bash
nsenter -m TARGET_PID --pid /bin/bash   # mount
nsenter -t TARGET_PID --pid /bin/bash   # pid
nsenter -n TARGET_PID --pid /bin/bash   # network
nsenter -i TARGET_PID --pid /bin/bash   # ipc
nsenter -u TARGET_PID --pid /bin/bash   # uts
nsenter -U TARGET_PID --pid /bin/bash   # user
nsenter -C TARGET_PID --pid /bin/bash   # cgroup
nsenter -T TARGET_PID --pid /bin/bash   # time
```
Мета перерахування цих варіантів разом не в тому, що кожна оцінка потребує всіх із них, а в тому, що namespace-specific post-exploitation часто значно спрощується, коли оператор знає точний синтаксис входу замість того, щоб пам’ятати лише форму all-namespaces.

## Налаштування за замовчуванням

| Runtime / платформа | Стан просторів імен за замовчуванням | Поширені ручні послаблення |
| --- | --- | --- |
| Docker Engine | За замовчуванням створюються нові mount, PID, network, IPC та UTS namespaces; user namespaces доступні, але за замовчуванням не ввімкнені у стандартних rootful налаштуваннях | `--pid=host`, `--network=host`, `--ipc=host`, `--uts=host`, `--userns=host`, `--cgroupns=host`, `--privileged` |
| Podman | За замовчуванням створюються нові namespaces; rootless Podman автоматично використовує user namespace; налаштування cgroup namespace за замовчуванням залежать від версії cgroup | `--pid=host`, `--network=host`, `--ipc=host`, `--uts=host`, `--userns=host`, `--cgroupns=host`, `--privileged` |
| Kubernetes | За замовчуванням Pods **не** діляться host PID, network або IPC; мережа Pod приватна для Pod, а не для кожного окремого контейнера; user namespaces підключаються опційно через `spec.hostUsers: false` на підтримуваних кластерах | `hostPID: true`, `hostNetwork: true`, `hostIPC: true`, `spec.hostUsers: true` / omitting user-namespace opt-in, privileged workload settings |
| containerd / CRI-O under Kubernetes | Зазвичай слідують Kubernetes Pod значенням за замовчуванням | same as Kubernetes row; direct CRI/OCI specs can also request host namespace joins |

Головне правило переносимості просте: **концепція** спільного використання host namespace є загальною для різних рантаймів, але **синтаксис** специфічний для кожного рантайму.
