# Χώρος ονομάτων χρόνου

{{#include ../../../../../banners/hacktricks-training.md}}

## Επισκόπηση

Ο χώρος ονομάτων χρόνου εικονικοποιεί επιλεγμένα ρολόγια, ειδικά **`CLOCK_MONOTONIC`** και **`CLOCK_BOOTTIME`**. Είναι νεότερος και πιο εξειδικευμένος χώρος ονομάτων σε σχέση με τους mount, PID, network ή user namespaces, και σπάνια είναι το πρώτο πράγμα που σκέφτεται ένας operator όταν συζητά τη σκληροποίηση των containers. Παρ' όλα αυτά, αποτελεί μέρος της σύγχρονης οικογένειας namespaces και αξίζει να κατανοηθεί εννοιολογικά.

Ο κύριος σκοπός είναι να επιτρέψει σε μια διεργασία να παρατηρεί ελεγχόμενες μετατοπίσεις για ορισμένα ρολόγια χωρίς να αλλάζει την παγκόσμια άποψη του χρόνου του host. Αυτό είναι χρήσιμο για ροές εργασίας checkpoint/restore, deterministic testing και κάποια προηγμένη συμπεριφορά κατά το runtime. Συνήθως δεν αποτελεί κύριο μέτρο απομόνωσης με τον ίδιο τρόπο όπως οι mount ή user namespaces, αλλά συμβάλλει στο να γίνει το περιβάλλον της διεργασίας πιο αυτοτελές.

## Εργαστήριο

Εάν ο kernel του host και το userspace το υποστηρίζουν, μπορείτε να επιθεωρήσετε τον χώρο ονομάτων με:
```bash
sudo unshare --time --fork bash
ls -l /proc/self/ns/time /proc/self/ns/time_for_children
cat /proc/$$/timens_offsets 2>/dev/null
```
Η υποστήριξη διαφέρει ανάλογα με τις kernel and tool versions, οπότε αυτή η σελίδα είναι περισσότερο για την κατανόηση του μηχανισμού παρά για την προσδοκία ότι θα είναι ορατό σε κάθε περιβάλλον εργαστηρίου.

### Μετατοπίσεις χρόνου

Τα Linux time namespaces εικονικοποιούν τις μετατοπίσεις για `CLOCK_MONOTONIC` και `CLOCK_BOOTTIME`. Οι τρέχουσες ανά-namespace μετατοπίσεις εκτίθενται μέσω του `/proc/<pid>/timens_offsets`, το οποίο σε kernels που το υποστηρίζουν μπορεί επίσης να τροποποιηθεί από μια διεργασία που κατέχει `CAP_SYS_TIME` εντός του σχετικού namespace:
```bash
sudo unshare -Tr --mount-proc bash
cat /proc/$$/timens_offsets
echo "monotonic 172800000000000" > /proc/$$/timens_offsets
cat /proc/uptime
```
Το αρχείο περιέχει διαφορές σε νανοδευτερόλεπτα. Η ρύθμιση του `monotonic` κατά δύο ημέρες αλλάζει παρατηρήσεις τύπου uptime μέσα σε αυτό το namespace χωρίς να αλλάζει το ρολόι τοίχου του host.

### Βοηθητικές σημαίες του `unshare`

Πρόσφατες εκδόσεις του `util-linux` παρέχουν βολικές σημαίες που γράφουν αυτόματα τις μετατοπίσεις:
```bash
sudo unshare -T --monotonic="+24h" --boottime="+7d" --mount-proc bash
```
Αυτές οι σημαίες αποτελούν κυρίως βελτίωση της ευχρηστίας, αλλά επίσης διευκολύνουν την αναγνώριση του χαρακτηριστικού στην τεκμηρίωση και στις δοκιμές.

## Χρήση κατά την εκτέλεση

Τα time namespaces είναι νεότερα και χρησιμοποιούνται λιγότερο ευρέως σε σχέση με τα mount ή PID namespaces. Η OCI Runtime Specification v1.1 πρόσθεσε ρητή υποστήριξη για το `time` namespace και το πεδίο `linux.timeOffsets`, και νεότερες εκδόσεις του `runc` υλοποιούν αυτό το μέρος του μοντέλου. Ένα ελάχιστο απόσπασμα OCI φαίνεται ως εξής:
```json
{
"linux": {
"namespaces": [
{ "type": "time" }
],
"timeOffsets": {
"monotonic": 86400,
"boottime": 600
}
}
}
```
Αυτό έχει σημασία επειδή μετατρέπει το time namespacing από ένα εξειδικευμένο kernel primitive σε κάτι που τα runtimes μπορούν να ζητήσουν φορητά.

## Security Impact

Υπάρχουν λιγότερες κλασικές breakout ιστορίες που επικεντρώνονται στο time namespace σε σχέση με άλλους τύπους namespace. Ο κίνδυνος εδώ συνήθως δεν είναι ότι το time namespace ενεργοποιεί άμεσα escape, αλλά ότι οι αναγνώστες το αγνοούν εντελώς και επομένως χάνουν το πώς τα πιο προχωρημένα runtimes ενδέχεται να διαμορφώνουν τη συμπεριφορά των processes. Σε εξειδικευμένα περιβάλλοντα, οι αλλαγμένες απόψεις του ρολογιού μπορούν να επηρεάσουν checkpoint/restore, observability ή εγκληματολογικές υποθέσεις.

## Abuse

Συνήθως δεν υπάρχει άμεσο breakout primitive εδώ, αλλά η αλλαγμένη συμπεριφορά του ρολογιού μπορεί να είναι χρήσιμη για την κατανόηση του περιβάλλοντος εκτέλεσης και τον εντοπισμό προχωρημένων χαρακτηριστικών runtime:
```bash
readlink /proc/self/ns/time
readlink /proc/self/ns/time_for_children
date
cat /proc/uptime
```
Εάν συγκρίνετε δύο διεργασίες, οι διαφορές εδώ μπορούν να βοηθήσουν στην εξήγηση περίεργης συμπεριφοράς χρονισμού, ανωμαλιών από checkpoint/restore, ή ασυνεπειών στο environment-specific logging.

Επιπτώσεις:

- σχεδόν πάντα reconnaissance ή κατανόηση του περιβάλλοντος
- χρήσιμο για την εξήγηση ανωμαλιών στο logging, στο uptime ή σε checkpoint/restore
- συνήθως δεν αποτελεί απευθείας container-escape mechanism από μόνο του

Το σημαντικό σημείο κατάχρησης είναι ότι τα time namespaces δεν εικονικοποιούν το `CLOCK_REALTIME`, οπότε από μόνα τους δεν επιτρέπουν σε έναν επιτιθέμενο να πλαστογραφήσει το host wall clock ή να παραβιάσει απευθείας ελέγχους λήξης πιστοποιητικών σε ολόκληρο το σύστημα. Η αξία τους είναι κυρίως στη σύγχυση λογικής βασισμένης σε monotonic time, στην αναπαραγωγή σφαλμάτων ειδικών για το περιβάλλον, ή στην κατανόηση προχωρημένης συμπεριφοράς του runtime.

## Checks

Αυτοί οι έλεγχοι αφορούν κυρίως την επιβεβαίωση εάν το runtime χρησιμοποιεί private time namespace.
```bash
readlink /proc/self/ns/time                 # Current time namespace identifier
readlink /proc/self/ns/time_for_children    # Time namespace inherited by children
cat /proc/$$/timens_offsets 2>/dev/null     # Monotonic and boottime offsets when supported
```
Τι είναι ενδιαφέρον εδώ:

- Σε πολλά περιβάλλοντα αυτές οι τιμές δεν θα οδηγήσουν σε άμεσο ευρήμα ασφάλειας, αλλά δείχνουν αν μια εξειδικευμένη λειτουργία του runtime είναι ενεργή.
- Αν συγκρίνετε δύο διαδικασίες, οι διαφορές εδώ μπορεί να εξηγήσουν συγκεχυμένο χρονισμό ή συμπεριφορά checkpoint/restore.

Για τις περισσότερες container breakouts, το time namespace δεν είναι ο πρώτος έλεγχος που θα διερευνήσετε. Ωστόσο, μια πλήρης ενότητα container-security θα πρέπει να το αναφέρει επειδή είναι μέρος του σύγχρονου kernel model και περιστασιακά έχει σημασία σε προχωρημένα runtime σενάρια.
