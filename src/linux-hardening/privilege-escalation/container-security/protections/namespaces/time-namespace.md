# Простір імен часу

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен часу віртуалізує вибрані годинники, особливо **`CLOCK_MONOTONIC`** і **`CLOCK_BOOTTIME`**. Це новіший і більш спеціалізований простір імен, ніж mount, PID, network або user namespaces, і рідко буває першим, про що думає оператор під час обговорення container hardening. Навіть так, він є частиною сучасного сімейства просторів імен і вартий концептуального розуміння.

Головна мета — дозволити процесу спостерігати керовані зсуви певних годинників без зміни глобального уявлення про час хоста. Це корисно для checkpoint/restore workflows, deterministic testing і деяких просунутих поведінкових сценаріїв виконання. Зазвичай це не є ключовим механізмом ізоляції на кшталт mount або user namespaces, але воно все одно сприяє тому, щоб середовище процесу було більш автономним.

## Лаб

Якщо ядро хоста та userspace це підтримують, ви можете переглянути простір імен за допомогою:
```bash
sudo unshare --time --fork bash
ls -l /proc/self/ns/time /proc/self/ns/time_for_children
cat /proc/$$/timens_offsets 2>/dev/null
```
Підтримка залежить від версій ядра та інструментів, тому ця сторінка більше про розуміння механізму, аніж про очікування його видимості в кожному лабораторному середовищі.

### Часові зсуви

Простори імен часу в Linux віртуалізують зсуви для `CLOCK_MONOTONIC` та `CLOCK_BOOTTIME`. Поточні зсуви для кожного простору імен відображені в `/proc/<pid>/timens_offsets`, які на ядрах з підтримкою також можуть бути змінені процесом, що має `CAP_SYS_TIME` у відповідному просторі імен:
```bash
sudo unshare -Tr --mount-proc bash
cat /proc/$$/timens_offsets
echo "monotonic 172800000000000" > /proc/$$/timens_offsets
cat /proc/uptime
```
Файл містить зсуви в наносекундах. Зміна `monotonic` на два дні змінює спостереження, подібні до uptime, всередині цього namespace, не змінюючи системного годинника хоста.

### `unshare` допоміжні прапори

Новіші версії `util-linux` надають зручні прапори, які автоматично записують зсуви:
```bash
sudo unshare -T --monotonic="+24h" --boottime="+7d" --mount-proc bash
```
Ці прапори здебільшого покращують зручність використання, але також полегшують розпізнавання цієї функції в документації та тестуванні.

## Використання під час виконання

Time namespaces новіші й менш широко використовуються, ніж mount або PID namespaces. OCI Runtime Specification v1.1 додав явну підтримку `time` namespace та поля `linux.timeOffsets`, а новіші релізи `runc` реалізують цю частину моделі. Мінімальний фрагмент OCI виглядає так:
```json
{
"linux": {
"namespaces": [
{ "type": "time" }
],
"timeOffsets": {
"monotonic": 86400,
"boottime": 600
}
}
}
```
Це важливо, оскільки перетворює time namespacing із нішевого примітиву ядра на механізм, який runtimes можуть запитувати портативно.

## Вплив на безпеку

Класичних breakout-історій, зосереджених на time namespace, менше, ніж для інших типів namespace. Ризик тут зазвичай не в тому, що time namespace безпосередньо дозволяє escape, а в тому, що читачі повністю його ігнорують і через це не помічають, як advanced runtimes можуть формувати поведінку процесів. У спеціалізованих середовищах змінене уявлення про час може впливати на checkpoint/restore, observability або forensic assumptions.

## Зловживання

Зазвичай тут немає прямого breakout-примітиву, але змінена поведінка годинника все ще може бути корисною для розуміння середовища виконання та виявлення advanced runtime-функцій:
```bash
readlink /proc/self/ns/time
readlink /proc/self/ns/time_for_children
date
cat /proc/uptime
```
Якщо ви порівнюєте два процеси, відмінності тут можуть допомогти пояснити дивну поведінку часу, артефакти checkpoint/restore або невідповідності в логуванні, специфічні для середовища.

Вплив:

- майже завжди reconnaissance або розуміння середовища
- корисно для пояснення логування, uptime або аномалій checkpoint/restore
- зазвичай не є прямим механізмом container-escape самі по собі

Важлива особливість щодо зловживання полягає в тому, що простори імен часу (time namespaces) не віртуалізують `CLOCK_REALTIME`, тому самі по собі вони не дозволяють атакуючому фальсифікувати системний годинник хоста або безпосередньо порушити перевірки закінчення строку дії сертифікатів у масштабі системи. Їхня цінність переважно в заплутуванні логіки, що базується на монотонному часі, відтворенні багів, специфічних для середовища, або розумінні складної поведінки середовища виконання.

## Checks

Ці перевірки здебільшого спрямовані на підтвердження того, чи використовує середовище виконання приватний простір імен часу.
```bash
readlink /proc/self/ns/time                 # Current time namespace identifier
readlink /proc/self/ns/time_for_children    # Time namespace inherited by children
cat /proc/$$/timens_offsets 2>/dev/null     # Monotonic and boottime offsets when supported
```
Що тут цікаво:

- У багатьох середовищах ці значення не призведуть до негайної проблеми безпеки, але вони показують, чи задіяно спеціалізовану runtime-функцію.
- Якщо ви порівнюєте два процеси, відмінності тут можуть пояснити незрозумілі затримки або поведінку checkpoint/restore.

Для більшості container breakouts, time namespace не є першим контролем, який ви будете досліджувати. Проте повний container-security розділ має згадати його, оскільки він є частиною сучасної kernel-моделі і час від часу має значення в просунутих runtime-сценаріях.
