# Простір імен часу

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен часу віртуалізує обрані годинники, зокрема **`CLOCK_MONOTONIC`** та **`CLOCK_BOOTTIME`**. Це новіший і більш спеціалізований простір імен, ніж простори імен mount, PID, network або user, і рідко є тим, про що оператор думає в першу чергу при жорсткому підході до захисту контейнерів. Проте він належить до сучасного сімейства просторів імен і вартий розуміння на концептуальному рівні.

Головна мета — дозволити процесу спостерігати керовані зсуви для певних годинників без зміни глобального уявлення про час на хості. Це корисно для робочих процесів checkpoint/restore, детермінованого тестування та деякої складнішої поведінки під час виконання. Зазвичай це не є ключовим засобом ізоляції на кшталт mount або user namespaces, але воно все одно сприяє тому, щоб середовище процесу було більш самодостатнім.

## Лаб

Якщо ядро хоста та userspace підтримують це, ви можете інспектувати простір імен за допомогою:
```bash
sudo unshare --time --fork bash
ls -l /proc/self/ns/time /proc/self/ns/time_for_children
cat /proc/$$/timens_offsets 2>/dev/null
```
Підтримка залежить від версій ядра та інструментів, тому ця сторінка скоріше присвячена розумінню механізму, ніж очікуванню, що він буде видимим у кожному лабораторному середовищі.

### Зсуви часу

Часові простори (time namespaces) Linux віртуалізують зсуви для `CLOCK_MONOTONIC` та `CLOCK_BOOTTIME`. Поточні зсуви для кожного простору доступні через `/proc/<pid>/timens_offsets`, які на ядрах з підтримкою також можуть бути змінені процесом, який має `CAP_SYS_TIME` всередині відповідного простору:
```bash
sudo unshare -Tr --mount-proc bash
cat /proc/$$/timens_offsets
echo "monotonic 172800000000000" > /proc/$$/timens_offsets
cat /proc/uptime
```
Файл містить дельти в наносекундах. Зміна `monotonic` на два дні змінює спостереження, схожі на uptime, всередині цього namespace без зміни host wall clock.

### `unshare` Helper Flags

Новіші версії `util-linux` надають зручні прапори, які автоматично записують зміщення:
```bash
sudo unshare -T --monotonic="+24h" --boottime="+7d" --mount-proc bash
```
Ці прапори здебільшого покращують зручність використання, але також полегшують розпізнавання цієї можливості в документації та під час тестування.

## Використання під час виконання

`time` namespaces є новішими і менш широко використовуються, ніж mount або PID простори імен. OCI Runtime Specification v1.1 додала явну підтримку для `time` namespace і поля `linux.timeOffsets`, а новіші релізи `runc` реалізують цю частину моделі. Мінімальний фрагмент OCI виглядає так:
```json
{
"linux": {
"namespaces": [
{ "type": "time" }
],
"timeOffsets": {
"monotonic": 86400,
"boottime": 600
}
}
}
```
This matters because it turns time namespacing from a niche kernel primitive into something that runtimes can request portably.

## Вплив на безпеку

Класичних історій про breakout, які зосереджені на time namespace, менше, ніж для інших типів namespace. Ризик тут зазвичай полягає не в тому, що time namespace безпосередньо дозволяє escape, а в тому, що читачі повністю ігнорують його й тому пропускають, як просунуті рантайми можуть формувати поведінку процесів. В спеціалізованих середовищах змінене уявлення про годинник може впливати на checkpoint/restore, observability або forensic припущення.

## Зловживання

Зазвичай тут немає прямого breakout-примітива, але змінена поведінка годинника все ще може бути корисною для розуміння середовища виконання та виявлення просунутих можливостей рантайму:
```bash
readlink /proc/self/ns/time
readlink /proc/self/ns/time_for_children
date
cat /proc/uptime
```
Якщо ви порівнюєте два процеси, відмінності тут можуть допомогти пояснити дивну поведінку, пов'язану з часом, артефакти checkpoint/restore або невідповідності в логуванні, специфічні для середовища.

Impact:

- майже завжди reconnaissance або розуміння середовища
- корисно для пояснення логування, uptime або аномалій checkpoint/restore
- зазвичай сам по собі не є прямим механізмом container-escape

Важлива тонкість зловживання полягає в тому, що time namespaces не віртуалізують `CLOCK_REALTIME`, тому самі по собі вони не дозволяють атакуючому сфальсифікувати host wall clock або безпосередньо зламати перевірки терміну дії сертифікатів по всій системі. Їхня цінність переважно полягає в заплутуванні логіки, що базується на монотонному часі, відтворенні багів, специфічних для середовища, або в розумінні складної поведінки середовища виконання.

## Перевірки

Ці перевірки здебільшого спрямовані на підтвердження того, чи використовує runtime приватний time namespace взагалі.
```bash
readlink /proc/self/ns/time                 # Current time namespace identifier
readlink /proc/self/ns/time_for_children    # Time namespace inherited by children
cat /proc/$$/timens_offsets 2>/dev/null     # Monotonic and boottime offsets when supported
```
Що тут цікаво:

- У багатьох середовищах ці значення не призведуть до негайного висновку про проблему безпеки, але вони вказують, чи використовується спеціалізована функція runtime.
- Якщо ви порівнюєте два процеси, відмінності тут можуть пояснити заплутану поведінку, пов’язану з часуванням, або поведінку при checkpoint/restore.

Для більшості container breakouts, time namespace не є першим контролем, який ви будете досліджувати. Проте повний розділ container-security має згадати його, оскільки він є частиною сучасної моделі ядра і іноді має значення в просунутих сценаріях runtime.
