# UTS Namespace

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

UTS namespace ізолює **ім'я хоста** та **ім'я домену NIS**, які бачить процес. На перший погляд це може здаватися незначним у порівнянні з mount, PID або user namespaces, але це — частина того, що робить контейнер схожим на окремий хост. Всередині простору імен workload може бачити і іноді змінювати ім'я хоста, яке локальне для цього простору імен, а не глобальне для машини.

Саме по собі це зазвичай не є центральним елементом історії про breakout. Проте коли хостовий UTS namespace стає спільним, процес з достатніми привілеями може впливати на налаштування, пов'язані з ідентифікацією хоста, що може мати значення з оперативної точки зору і іноді — з точки зору безпеки.

## Лабораторія

Ви можете створити UTS namespace за допомогою:
```bash
sudo unshare --uts --fork bash
hostname
hostname lab-container
hostname
```
Зміна імені хоста залишається локальною для цього простору імен і не змінює глобальне ім'я хоста. Це простий, але ефективний приклад властивості ізоляції.

## Використання під час виконання

Звичайні контейнери отримують ізольований UTS простір імен. Docker і Podman можуть приєднатися до UTS простору імен хоста через `--uts=host`, а подібні схеми спільного використання хоста можуть зустрічатися і в інших середовищах виконання та системах оркестрації. Проте в більшості випадків приватна ізоляція UTS є просто частиною стандартного налаштування контейнера і вимагає мало уваги від оператора.

## Вплив на безпеку

Навіть якщо UTS простір імен зазвичай не є найнебезпечнішим для спільного використання, він все одно впливає на цілісність межі контейнера. Якщо UTS простір імен хоста відкритий і процес має необхідні привілеї, він може змінювати інформацію, пов'язану з іменем хоста. Це може вплинути на моніторинг, логування, операційні припущення або скрипти, що приймають рішення довіри на основі даних ідентифікації хоста.

## Зловживання

Якщо UTS простір імен хоста спільно використовується, практичне питання полягає в тому, чи може процес змінювати налаштування ідентифікації хоста, а не лише читати їх:
```bash
readlink /proc/self/ns/uts
hostname
cat /proc/sys/kernel/hostname
```
Якщо контейнер також має необхідні привілеї, перевірте, чи можна змінити hostname:
```bash
hostname hacked-host 2>/dev/null && echo "hostname change worked"
hostname
```
Це передусім питання цілісності та операційного впливу, а не повний escape, але це все одно показує, що контейнер може безпосередньо впливати на властивість, глобальну для хоста.

Наслідки:

- маніпуляція ідентичністю хоста
- спотворення логів, моніторингу або автоматизації, які покладаються на ім'я хоста
- зазвичай сам по собі не є повним escape, якщо не поєднаний з іншими вразливостями

У Docker-style середовищах корисним шаблоном виявлення на стороні хоста є:
```bash
docker ps -aq | xargs -r docker inspect --format '{{.Id}} UTSMode={{.HostConfig.UTSMode}}'
```
Контейнери з `UTSMode=host` ділять host UTS namespace й повинні проходити більш ретельну перевірку, якщо вони також мають capabilities, які дозволяють викликати `sethostname()` або `setdomainname()`.

## Перевірки

Ці команди достатні, щоб визначити, чи має workload власний вигляд hostname, чи він ділить host UTS namespace.
```bash
readlink /proc/self/ns/uts   # UTS namespace identifier
hostname                     # Hostname as seen by the current process
cat /proc/sys/kernel/hostname   # Kernel hostname value in this namespace
```
- Зіставлення ідентифікаторів namespace з процесом на host може свідчити про спільне використання UTS хоста.
- Якщо зміна hostname впливає не лише на сам container, то workload має більший вплив на identity хоста, ніж це має бути.
- Зазвичай це має нижчий пріоритет порівняно з проблемами PID, mount або user namespace, але все одно підтверджує, наскільки ізольований процес насправді.

У більшості середовищ UTS namespace краще розглядати як допоміжний рівень ізоляції. Це рідко перше, за чим ви гонуєтеся під час breakout, але воно все одно є частиною загальної узгодженості та безпеки подання container.
