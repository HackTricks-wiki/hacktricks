# UTS Namespace

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

UTS namespace ізолює **hostname** та **NIS domain name**, які бачить процес. На перший погляд це може здаватися незначним порівняно з mount, PID або user namespaces, але це частина того, що робить container схожим на власний хост. Всередині namespace workload може бачити і іноді змінювати hostname, який є локальним для цього namespace, а не глобальним для машини.

Саме по собі це зазвичай не є центральним елементом історії про breakout. Однак, коли UTS namespace хоста стає спільним, процес із достатніми привілеями може впливати на налаштування, пов'язані з ідентифікацією хоста, що може мати операційне значення і інколи — впливати на безпеку.

## Лабораторія

Ви можете створити UTS namespace за допомогою:
```bash
sudo unshare --uts --fork bash
hostname
hostname lab-container
hostname
```
Зміна hostname залишається локальною для тієї namespace і не змінює глобальний hostname хоста. Це простий, але ефективний приклад властивості ізоляції.

## Використання під час виконання

Зазвичай контейнери отримують ізольований UTS namespace. Docker і Podman можуть приєднатися до UTS namespace хоста через `--uts=host`, і подібні схеми спільного використання хоста можуть зустрічатися в інших runtimes та системах оркестрації. Проте в більшості випадків приватна UTS ізоляція є частиною звичайного налаштування контейнера і вимагає мало уваги від оператора.

## Вплив на безпеку

Хоча UTS namespace зазвичай не є найнебезпечнішим для спільного використання, він все одно впливає на цілісність межі контейнера. Якщо UTS namespace хоста відкрито, і процес має необхідні привілеї, він може змінити інформацію, пов'язану з hostname хоста. Це може вплинути на моніторинг, логування, операційні припущення або скрипти, які приймають рішення про довіру на основі даних про ідентичність хоста.

## Зловживання

Якщо UTS namespace хоста шариться, практичне запитання полягає в тому, чи може процес змінювати налаштування ідентичності хоста, а не лише читати їх:
```bash
readlink /proc/self/ns/uts
hostname
cat /proc/sys/kernel/hostname
```
Якщо контейнер також має необхідні права, перевірте, чи можна змінити hostname:
```bash
hostname hacked-host 2>/dev/null && echo "hostname change worked"
hostname
```
Це насамперед проблема цілісності та впливу на операційну роботу, а не повний escape, але це все одно показує, що контейнер може безпосередньо впливати на глобальну властивість хоста.

Вплив:

- підміна ідентичності хоста
- спотворення логів, моніторингу або автоматизації, які довіряють імені хоста
- зазвичай саме по собі не є повним escape, якщо не поєднано з іншими вразливостями

У середовищах типу Docker корисний патерн виявлення на боці хоста такий:
```bash
docker ps -aq | xargs -r docker inspect --format '{{.Id}} UTSMode={{.HostConfig.UTSMode}}'
```
Контейнери з `UTSMode=host` ділять UTS namespace хоста й вимагають ретельнішого перегляду, якщо вони також мають capabilities, які дозволяють викликати `sethostname()` або `setdomainname()`.

## Перевірки

Ці команди достатні, щоб побачити, чи має workload власний перегляд hostname, чи ділиться UTS namespace хоста.
```bash
readlink /proc/self/ns/uts   # UTS namespace identifier
hostname                     # Hostname as seen by the current process
cat /proc/sys/kernel/hostname   # Kernel hostname value in this namespace
```
Цікаве тут:

- Збіг ідентифікаторів UTS namespace з процесом на хості може вказувати на спільне використання UTS хоста.
- Якщо зміна hostname впливає не лише на сам container, то workload має більше впливу на ідентичність хоста, ніж повинно.
- Зазвичай це менш пріоритетна знахідка, ніж проблеми з PID, mount або user namespace, але вона все одно підтверджує, наскільки ізольований процес насправді.

У більшості середовищ UTS namespace найкраще розглядати як допоміжний шар ізоляції. Це рідко перше, за чим ви женетесь у breakout, але воно все одно є частиною загальної узгодженості та безпеки container view.
