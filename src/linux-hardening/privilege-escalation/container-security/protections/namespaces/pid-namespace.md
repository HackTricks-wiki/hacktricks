# Простір імен PID

{{#include ../../../../../banners/hacktricks-training.md}}

## Огляд

Простір імен PID контролює, як нумеруються процеси і які процеси видимі. Саме тому контейнер може мати власний PID 1, хоча він не є повноцінною машиною. Всередині простору імен робоче навантаження бачить те, що здається локальним деревом процесів. Ззовні хост усе ще бачить реальні PIDs хоста і повну картину процесів.

З точки зору безпеки, простір імен PID важливий, бо видимість процесів має велику цінність. Якщо робоче навантаження може бачити процеси хоста, воно може виявити імена сервісів, аргументи командного рядка, секрети, передані в аргументах процесів, стан, отриманий з оточення через `/proc`, і потенційні цілі для входу в інші простори імен. Якщо воно може робити більше, ніж просто бачити ці процеси — наприклад, надсилати сигнали або використовувати ptrace за певних умов — проблема стає значно серйознішою.

## Принцип роботи

Новий простір імен PID починається зі своєї внутрішньої нумерації процесів. Перший процес, створений всередині, стає PID 1 з точки зору простору імен, що також означає, що він отримує спеціальну семантику, подібну до init, щодо сирітських дочірніх процесів та поведінки зі сигналами. Це пояснює багато дивних явищ у контейнерах, пов'язаних із процесами init, прибиранням zombie і чому іноді в контейнерах використовуються невеликі init-обгортки.

Важливий урок безпеки: процес може виглядати ізольованим, бо бачить лише власне дерево PID, але цю ізоляцію можна свідомо зняти. Docker дозволяє це через `--pid=host`, а Kubernetes — через `hostPID: true`. Як тільки контейнер приєднується до простору імен PID хоста, робоче навантаження бачить процеси хоста напряму, і багато подальших шляхів атаки стають значно реалістичнішими.

## Лабораторія

Щоб створити простір імен PID вручну:
```bash
sudo unshare --pid --fork --mount-proc bash
ps -ef
echo $$
```
Shell тепер бачить приватний перегляд процесів. Прапорець `--mount-proc` важливий, оскільки монтує екземпляр procfs, який відповідає новому PID namespace, роблячи список процесів узгодженим зсередини.

Щоб порівняти поведінку контейнера:
```bash
docker run --rm debian:stable-slim ps -ef
docker run --rm --pid=host debian:stable-slim ps -ef | head
```
Різниця миттєво помітна й легко зрозуміла, тому це хороший перший лабораторний практикум для читачів.

## Використання під час виконання

Звичайні контейнери в Docker, Podman, containerd і CRI-O отримують власний PID namespace. Kubernetes Pods зазвичай також отримують ізольований перегляд PID, якщо тільки робоче навантаження явно не вимагає host PID sharing. Середовища LXC/Incus базуються на тому ж примітиві ядра, хоча випадки використання system-container можуть відкривати більш складні дерева процесів і заохочувати використання спрощених способів відладки.

Те саме правило застосовується скрізь: якщо runtime вирішив не ізолювати PID namespace, це свідоме звуження меж контейнера.

## Неправильні конфігурації

Канонічною неправильною конфігурацією є host PID sharing. Команди часто виправдовують це потребами для налагодження, моніторингу або зручності управління сервісами, але це завжди слід розглядати як значуще виняток з безпеки. Навіть якщо контейнер не має прямого механізму запису у процеси хоста, сама можливість їх бачити може розкрити багато інформації про систему. Як тільки додаються можливості на кшталт `CAP_SYS_PTRACE` або корисний доступ до procfs, ризик істотно зростає.

Інша помилка — припущення, що оскільки робоче навантаження за замовчуванням не може kill або ptrace процеси хоста, то host PID sharing безпечний. Такий висновок ігнорує цінність enumeration, наявність цілей для namespace-entry та те, як видимість PID поєднується з іншими ослабленими контролями.

## Зловживання

Якщо host PID namespace спільний, зловмисник може інспектувати процеси хоста, збирати аргументи процесів, виявляти цікаві сервіси, знаходити кандидатні PIDs для `nsenter`, або поєднувати видимість процесів з привілеями, пов'язаними з ptrace, щоб втручатися в роботу хоста або сусідніх workloads. В деяких випадках просте спостереження за потрібним довго працюючим процесом достатнє, щоб переглянути подальший план атаки.

Першим практичним кроком завжди є підтвердження того, що процеси хоста справді видимі:
```bash
readlink /proc/self/ns/pid
ps -ef | head -n 50
ls /proc | grep '^[0-9]' | head -n 20
```
Коли host PIDs стають видимими, аргументи процесів та namespace-entry targets часто стають найкориснішим джерелом інформації:
```bash
for p in 1 $(pgrep -n systemd 2>/dev/null) $(pgrep -n dockerd 2>/dev/null); do
echo "PID=$p"
tr '\0' ' ' < /proc/$p/cmdline 2>/dev/null; echo
done
```
Якщо `nsenter` доступний і є достатні привілеї, перевірте, чи можна використати видимий процес хоста як міст для namespace:
```bash
which nsenter
nsenter -t 1 -m -u -n -i -p sh 2>/dev/null || echo "nsenter blocked"
```
Навіть коли вхід заблоковано, спільний доступ до PID хоста вже корисний, бо він розкриває розташування сервісів, компоненти виконання та потенційні привілейовані процеси для подальшої атаки.

Видимість PID хоста також робить зловживання дескрипторами файлів більш реалістичним. Якщо привілейований процес хоста або сусіднє навантаження має відкритий чутливий файл або сокет, атакувальник може перевірити `/proc/<pid>/fd/` і повторно використати цей дескриптор залежно від прав власності, опцій монтування procfs та моделі цільового сервісу.
```bash
for fd_dir in /proc/[0-9]*/fd; do
ls -l "$fd_dir" 2>/dev/null | sed "s|^|$fd_dir -> |"
done
grep " /proc " /proc/mounts
```
Ці команди корисні, оскільки вони показують, чи `hidepid=1` або `hidepid=2` зменшують міжпроцесну видимість і чи взагалі видні очевидно цікаві дескриптори, такі як відкриті файли зі секретами, логи або Unix-сокети.

### Повний приклад: спільний PID хоста + `nsenter`

Спільне використання PID хоста стає прямим host escape, коли процес також має достатньо привілеїв, щоб приєднатися до host namespaces:
```bash
ps -ef | head -n 50
capsh --print | grep cap_sys_admin
nsenter -t 1 -m -u -n -i -p /bin/bash
```
Якщо команда виконається успішно, процес контейнера тепер виконується в host mount, UTS, network, IPC та PID namespaces. Наслідок — негайна компрометація хоста.

Навіть коли `nsenter` сам відсутній, той самий результат може бути досягнутий через host binary, якщо host filesystem змонтовано:
```bash
/host/usr/bin/nsenter -t 1 -m -u -n -i -p /host/bin/bash 2>/dev/null
```
### Останні зауваження щодо часу виконання

Деякі атаки, пов'язані з PID-namespace, не є традиційними неправильними конфігураціями `hostPID: true`, а є помилками реалізації під час виконання, що виникають через застосування захистів procfs під час налаштування контейнера.

#### `maskedPaths` — гонка до procfs хоста

У вразливих версіях `runc` зловмисники, які можуть контролювати образ контейнера або навантаження `runc exec`, могли влаштувати гонку під час фази маскування, замінивши контейнерну `/dev/null` символьним посиланням на чутливий шлях procfs, наприклад `/proc/sys/kernel/core_pattern`. Якщо гонка вдавалася, masked-path bind mount міг потрапити на невірну ціль і відкрити для нового контейнера глобальні на хості параметри procfs.

Корисна команда для перевірки:
```bash
jq '.linux.maskedPaths' config.json 2>/dev/null
```
Це важливо, оскільки кінцевий вплив може бути таким самим, як при прямому доступі до procfs: можливість запису `core_pattern` або `sysrq-trigger`, що може призвести до виконання коду на хості або відмови в обслуговуванні.

#### Namespace injection за допомогою `insject`

Інструменти Namespace injection, такі як `insject`, показують, що взаємодія з PID-namespace не завжди вимагає попереднього входу в цільовий namespace перед створенням процесу. Помічник може приєднатися пізніше, використати `setns()`, і виконуватися, зберігаючи видимість у цільовому PID-просторі:
```bash
sudo insject -S -p $(pidof containerd-shim) -- bash -lc 'readlink /proc/self/ns/pid && ps -ef'
```
Цей тип техніки має значення насамперед для просунутого відлагодження, offensive tooling і post-exploitation робочих процесів, коли контекст namespace потрібно приєднати після того, як runtime уже ініціалізував workload.

### Пов'язані патерни зловживання FD

Є два патерни, які варто явно виділити, коли видимі host PIDs. По-перше, привілейований процес може тримати чутливий файловий дескриптор відкритим під час `execve()`, оскільки він не був позначений `O_CLOEXEC`. По-друге, сервіси можуть передавати файлові дескриптори через Unix-сокети за допомогою `SCM_RIGHTS`. У обох випадках цікавим об'єктом вже не є pathname, а вже відкритий handle, який процес з нижчими правами може успадкувати або отримати.

Це має значення при роботі з контейнерами, бо дескриптор може вказувати на `docker.sock`, привілейований лог, файл секретів хоста або інший цінний об'єкт навіть коли сам шлях недоступний з файлової системи контейнера.

## Перевірки

Мета цих команд — визначити, чи має процес приватний PID view, чи він уже може перерахувати значно ширше середовище процесів.
```bash
readlink /proc/self/ns/pid   # PID namespace identifier
ps -ef | head                # Quick process list sample
ls /proc | head              # Process IDs and procfs layout
```
Що тут цікаво:

- Якщо в списку процесів видно очевидні сервіси хоста, host PID sharing, ймовірно, вже діє.
- Нормальною базовою поведінкою є бачити лише невелике локальне дерево контейнера; бачити `systemd`, `dockerd` або сторонні демони — ні.
- Як тільки PID-и хоста стають видимими, навіть інформація про процеси лише для читання стає корисною розвідкою.

Якщо ви виявите контейнер, що працює з host PID sharing, не сприймайте це як косметичну відмінність. Це значна зміна в тому, що робоче навантаження може спостерігати і потенційно впливати.
