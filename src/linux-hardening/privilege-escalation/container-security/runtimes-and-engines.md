# コンテナランタイム、エンジン、ビルダー、およびサンドボックス

{{#include ../../../banners/hacktricks-training.md}}

コンテナセキュリティで最大の混乱の一つは、まったく異なる複数のコンポーネントが同じ語でくくられてしまうことです。 "Docker" はイメージフォーマット、CLI、デーモン、ビルドシステム、ランタイムスタック、あるいは単にコンテナという概念そのものを指す場合があります。セキュリティの観点では、この曖昧さは問題です。なぜなら、異なるレイヤーが異なる保護を担っているからです。bad bind mount による breakout は低レベルの runtime bug による breakout とは同じではなく、どちらも Kubernetes のクラスタポリシーのミスとは異なります。

このページではエコシステムを役割ごとに分け、以降の節で保護や弱点が実際にどこにあるかを正確に議論できるようにします。

## OCI を共通言語として

現代の Linux コンテナスタックは、OCI 仕様群を話すことで相互運用性を持つことが多くあります。**OCI Image Specification** はイメージとレイヤーの表現方法を説明します。**OCI Runtime Specification** はランタイムがどのようにプロセスを起動するか（namespaces、mounts、cgroups、セキュリティ設定などを含む）を説明します。**OCI Distribution Specification** はレジストリがコンテンツをどのように公開するかを標準化します。

これは、あるツールでビルドしたコンテナイメージが別のツールで動作することが多い理由や、複数のエンジンが同じ低レベルランタイムを共有できる理由を説明します。また、多くの製品でセキュリティ挙動が似て見えるのは、多くが同じ OCI ランタイム構成を構築して同じ小さなランタイム群に渡しているためだ、という点も説明します。

## 低レベルの OCI ランタイム

低レベルのランタイムはカーネル境界に最も近いコンポーネントです。実際に namespaces を作成し、cgroup 設定を書き込み、capabilities や seccomp filters を適用し、最終的に `execve()` でコンテナプロセスを実行する部分です。人々が機械的レベルで「コンテナの隔離」について議論するとき、たいていこのレイヤーを指しています（明言されないことが多いですが）。

### `runc`

`runc` は参照実装の OCI ランタイムで、最もよく知られた実装です。Docker、containerd、そして多くの Kubernetes デプロイで広く使われています。`runc` スタイルの環境は一般的で、`runc` が多くの人が Linux コンテナを想像するときの基準を定義しているため、公開された調査やエクスプロイト資料の多くが `runc` をターゲットにしています。したがって、`runc` を理解することは古典的なコンテナ隔離に関する堅固なメンタルモデルを読者に与えます。

### `crun`

`crun` は C 言語で書かれた別の OCI ランタイムで、現代の Podman 環境で広く使われています。cgroup v2 のサポート、強い rootless エルゴノミクス、低いオーバーヘッドが評価されることが多いです。セキュリティの観点で重要なのは、言語が違うという点ではなく、役割は同じであるという点です：OCI 構成をカーネル下の実行プロセスツリーに変換するコンポーネントであることに変わりはありません。rootless な Podman ワークフローがより安全に感じられるのは、`crun` が全てを魔法のように修正するからではなく、周辺のスタックが user namespaces と最小権限により強く寄せている傾向があるためです。

### `runsc`（gVisor）

`runsc` は gVisor が使用するランタイムです。ここでは境界が意味のある変化をします。通常のように大半の syscall をホストカーネルに直接渡す代わりに、gVisor はユーザー空間カーネルレイヤーを挿入し、Linux インターフェースの大部分をエミュレートまたは仲介します。その結果は単に少しフラグが追加された通常の `runc` コンテナではなく、ホストカーネルの攻撃対象を削減することを目的とした異なるサンドボックス設計です。互換性やパフォーマンスのトレードオフがその設計の一部であるため、`runsc` を使う環境は通常の OCI ランタイム環境とは別に記述されるべきです。

### `kata-runtime`

Kata Containers はさらに境界を押し広げ、ワークロードを軽量な仮想マシン内で起動します。管理上はコンテナデプロイのように見えることがあり、オーケストレーション層も同様に扱うかもしれませんが、基盤となる隔離境界は従来のホストカーネル共有コンテナより仮想化に近くなります。Kata はコンテナ中心のワークフローを放棄せずに、より強いテナント隔離を望む場合に有用です。

## エンジンとコンテナマネージャ

低レベルのランタイムがカーネルと直接やり取りするコンポーネントであるならば、エンジンやマネージャはユーザや運用者が通常やり取りするコンポーネントです。イメージのプル、メタデータ、ログ、ネットワーク、ボリューム、ライフサイクル操作、API の公開を扱います。このレイヤーは非常に重要です。なぜなら、実運用での妥協の多くがここで発生するからです：ランタイムソケットやデーモン API へのアクセスは、低レベルランタイム自体が完璧でもホストの妥協に等しくなり得ます。

### Docker Engine

Docker Engine は開発者にとって最も認知されているコンテナプラットフォームであり、コンテナ語彙が Docker 的になった理由の一つです。典型的な経路は `docker` CLI から `dockerd` へ、そして `dockerd` が `containerd` や OCI ランタイムなどの下位コンポーネントを調整します。歴史的に Docker デプロイは **rootful** であることが多く、Docker ソケットへのアクセスは非常に強力なプリミティブでした。だからこそ多くの実践的な権限昇格資料が `docker.sock` に焦点を当てるのです：もしプロセスが `dockerd` に特権コンテナを作らせたり、ホストパスをマウントさせたり、ホスト namespaces に参加させたりできるなら、カーネルエクスプロイトは不要かもしれません。

### Podman

Podman はデーモンレスのモデルを中心に設計されました。運用面では、コンテナは長時間動作する特権デーモンではなく標準的な Linux メカニズムで管理されるただのプロセスであるという考えを強化します。Podman はまた、多くの人が最初に学んだクラシックな Docker デプロイよりもはるかに強い **rootless** の物語を持っています。それが Podman を自動的に安全にするわけではありませんが、user namespaces、SELinux、`crun` と組み合わさるとデフォルトのリスクプロファイルを大きく変えます。

### containerd

containerd は多くの現代的スタックでコアなランタイム管理コンポーネントです。Docker の下で使われ、Kubernetes の主要なランタイムバックエンドの一つでもあります。強力な API を公開し、イメージとスナップショットを管理し、最終的なプロセス作成を低レベルランタイムに委譲します。containerd に関するセキュリティの議論では、containerd ソケットや `ctr`/`nerdctl` の機能へアクセスできることが、たとえインターフェースやワークフローが「開発者向け」ではなく感じられても、Docker の API へのアクセスと同じくらい危険であり得ることを強調すべきです。

### CRI-O

CRI-O は Docker Engine よりも目的が狭く、汎用の開発者プラットフォームではなく、Kubernetes Container Runtime Interface をきれいに実装することを中心に構築されています。これにより Kubernetes ディストリビューションや OpenShift のような SELinux 多用エコシステムで特に一般的です。セキュリティの観点では、そのより狭いスコープは概念的な混乱を減らすので有用です：CRI-O は「Kubernetes のためにコンテナを実行する」レイヤーの一部です。

### Incus、LXD、そして LXC

Incus/LXD/LXC システムは Docker スタイルのアプリケーションコンテナから分けて考える価値があります。なぜならこれらはしばしば **system containers** として使われるからです。system container は通常、よりフルなユーザ空間、長時間動作するサービス、広いデバイス露出、より深いホスト統合を備えた軽量マシンのように見られることが期待されます。隔離機構は依然としてカーネルプリミティブですが、運用上の期待は異なります。その結果、ここでのミスコンフィグは「アプリコンテナの悪いデフォルト」よりも、軽量仮想化やホスト委譲のミスに似たものになります。

### systemd-nspawn

systemd-nspawn は systemd ネイティブで、テスト、デバッグ、OS のような環境を動かすのに非常に有用で、興味深い位置を占めます。クラウドネイティブの本番ランタイムの主役ではありませんが、ラボやディストロ志向の環境で十分頻出するため言及に値します。セキュリティ分析の観点では、「コンテナ」という概念が複数のエコシステムと運用スタイルに跨ることを再認識させます。

### Apptainer / Singularity

Apptainer（旧 Singularity）は研究や HPC 環境で一般的です。その信頼前提、ユーザワークフロー、実行モデルは Docker/Kubernetes 中心のスタックと重要な点で異なります。特に、これらの環境ではユーザにパッケージ化されたワークロードを実行させつつ、広範な特権的コンテナ管理権を与えないことを重視することが多いです。すべてのコンテナ環境が基本的に「サーバ上の Docker」であると仮定すると、こうしたデプロイを大きく誤解します。

## ビルド時のツール

多くのセキュリティ議論は実行時のみを扱いますが、ビルド時のツールも重要です。なぜなら、イメージ内容、ビルドシークレットの露出、どれだけ信頼されたコンテキストが最終成果物に組み込まれるかを決めるからです。

**BuildKit** と `docker buildx` はキャッシュ、secret mounting、SSH forwarding、マルチプラットフォームビルドなどの機能をサポートするモダンなビルドバックエンドです。これらは有用な機能ですが、セキュリティの観点ではシークレットがイメージレイヤに leak する場所を作ったり、過度に広いビルドコンテキストが含めるべきでないファイルを露出させたりする可能性も生みます。**Buildah** は特に Podman 周辺の OCI ネイティブエコシステムで同様の役割を果たし、**Kaniko** はビルドパイプラインに特権 Docker デーモンを許したくない CI 環境でよく使われます。

重要な教訓は、イメージ作成とイメージ実行は別フェーズであり、弱いビルドパイプラインはコンテナが起動するずっと前に弱い実行姿勢を作り出し得る、ということです。

## オーケストレーションは別のレイヤであってランタイムではない

Kubernetes をランタイムそのものと同一視してはなりません。Kubernetes はオーケストレータです。Pod をスケジュールし、desired state を保存し、ワークロード構成を通じてセキュリティポリシーを表現します。kubelet は containerd や CRI-O のような CRI 実装と話し、それがさらに `runc`、`crun`、`runsc`、あるいは `kata-runtime` のような低レベルランタイムを呼び出します。

この分離は重要です。多くの人は保護を「Kubernetes のもの」と誤って帰属させたり、ノードランタイムで強制されているものを Kubernetes の機能や pod spec のせいにしたりします。実際のセキュリティ姿勢は合成的です：オーケストレータが何かを要求し、ランタイムスタックがそれを翻訳し、最後にカーネルがそれを強制します。

## アセスメント時にランタイム識別が重要な理由

エンジンとランタイムを早期に特定すれば、その後の観察の多くを解釈しやすくなります。rootless な Podman コンテナは user namespaces が関与している可能性を示唆します。ワークロードにマウントされた Docker socket は API 駆動の権限昇格が現実的な道筋であることを示唆します。CRI-O / OpenShift ノードは即座に SELinux ラベルや制限されたワークロードポリシーを考えさせます。gVisor や Kata 環境では、従来の `runc` ブレイクアウト PoC が同じように動作するとは限らないことに注意するべきです。

だからこそ、コンテナ評価の最初のステップの一つは常に二つの簡単な質問に答えることです：どのコンポーネントがコンテナを管理しているか、そして実際にどのランタイムがプロセスを起動したか。これらの答えが明確になれば、環境全体の理解は通常ずっと容易になります。

## ランタイムの脆弱性

すべてのコンテナエスケープがオペレータのミスコンフィグから生じるわけではありません。時にはランタイム自体が脆弱なコンポーネントであることがあります。これは重要です。なぜならワークロードが一見注意深い設定で動作していても、低レベルのランタイムの欠陥によって曝露される可能性があるからです。

古典的な例は `runc` の **CVE-2019-5736** です。悪意あるコンテナがホストの `runc` バイナリを上書きし、その後の `docker exec` のようなランタイム呼び出しが実行者制御のコードをトリガーするのを待つことができました。このエクスプロイト経路は単純なバインドマウントやケーパビリティのミスとは非常に異なり、exec ハンドリング中にランタイムがどのようにコンテナプロセス空間に再入するかを悪用します。

red-team の観点からの最小再現ワークフローは次のとおりです：
```bash
go build main.go
./main
```
次に、ホストから：
```bash
docker exec -it <container-name> /bin/sh
```
重要な教訓は、過去のエクスプロイト実装そのものの詳細ではなく、評価上の含意です：もし runtime version が脆弱であれば、目に見えるコンテナ設定が明らかに弱く見えなくても、コンテナ内での通常のコード実行だけでホストを侵害するのに十分な場合があります。

`runc` の `CVE-2024-21626` や BuildKit の mount races、containerd の parsing bugs といった最近の runtime CVE は同じ点を裏付けています。ランタイムのバージョンとパッチレベルは単なる保守的な雑事ではなく、セキュリティ境界の一部です。
