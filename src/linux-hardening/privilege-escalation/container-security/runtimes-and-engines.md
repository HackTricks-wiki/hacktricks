# 容器运行时、引擎、构建器与沙箱

{{#include ../../../banners/hacktricks-training.md}}

容器安全中最大的一些混淆来源之一是，多个截然不同的组件经常被压缩成同一个词。“Docker” 可能指镜像格式、CLI、守护进程、构建系统、运行时栈，或者干脆指容器这个概念本身。对于安全工作来说，这种歧义是个问题，因为不同层对不同保护负责。由错误的绑定挂载（bind mount）导致的越权与由低级运行时漏洞导致的越权不是一回事，二者也不同于 Kubernetes 中的集群策略错误。

本页按角色划分生态系统，这样后续章节就能精确地讨论保护或弱点实际位于何处。

## OCI 作为通用语言

现代 Linux 容器栈常常能够互操作，是因为它们遵循一套 OCI 规范。**OCI Image Specification** 描述了镜像和层如何表示。**OCI Runtime Specification** 描述了运行时应如何启动进程，包括命名空间、挂载、cgroup 和安全设置。**OCI Distribution Specification** 规范化了注册表如何暴露内容。

这很重要，因为它解释了为什么用一个工具构建的容器镜像常常可以用另一个工具运行，以及为什么多个引擎可以共享相同的低级运行时。它也解释了为什么不同产品的安全行为看起来相似：它们中的许多都在构建相同的 OCI 运行时配置并将其交给相同的小型运行时集合。

## 低级 OCI 运行时

低级运行时是最接近内核边界的组件。它是实际创建命名空间、写入 cgroup 设置、应用 capabilities 和 seccomp 过滤器，并最终 `execve()` 容器进程的部分。当人们在机械层面上讨论“容器隔离”时，通常指的就是这一层，即使他们没有明确说明。

### `runc`

`runc` 是参考实现的 OCI 运行时，也是最知名的实现。它在 Docker、containerd 和许多 Kubernetes 部署中被大量使用。大量的公开研究和利用材料针对 `runc` 风格的环境，仅仅是因为它们普遍存在，并且 `runc` 定义了很多人在想象 Linux 容器时所认为的基线。理解 `runc` 因此能为读者提供对经典容器隔离的强烈心理模型。

### `crun`

`crun` 是另一个 OCI 运行时，用 C 编写，在现代 Podman 环境中被广泛使用。它经常因对 cgroup v2 的良好支持、优秀的无根（rootless）易用性和更低的开销而受到赞扬。从安全角度看，重要的并不是它用不同语言编写，而是它仍然扮演相同的角色：它是将 OCI 配置转换为内核下运行的进程树的组件。无根 Podman 工作流经常感觉更安全，并不是因为 `crun` 魔法般地修复了一切，而是因为其周边堆栈更倾向于使用用户命名空间和最小权限。

### `runsc` 来自 gVisor

`runsc` 是 gVisor 使用的运行时。在这里，边界的含义发生了实质性变化。gVisor 并不是以通常方式将大多数 syscall 直接传给主机内核，而是在用户态插入一个内核层来模拟或调解 Linux 接口的很大一部分。结果不是一个带有一些额外标志的普通 `runc` 容器；这是一个不同的沙箱设计，其目的在于减少宿主内核的攻击面。兼容性和性能的折衷是该设计的一部分，因此使用 `runsc` 的环境应该与普通 OCI 运行时环境区别开来记录。

### `kata-runtime`

Kata Containers 进一步推进了边界，通过在轻量级虚拟机内启动工作负载。从管理上看，这可能仍然看起来像容器部署，编排层也可能照样对其进行处理，但底层的隔离边界更接近虚拟化，而非经典的共享主机内核容器。当需要比普通容器更强的租户隔离而又不想放弃以容器为中心的工作流时，Kata 很有用。

## 引擎与容器管理器

如果低级运行时是与内核直接对话的组件，那么引擎或管理器则是用户和运维人员通常交互的组件。它处理镜像拉取、元数据、日志、网络、卷、生命周期操作和 API 暴露。这一层非常重要，因为许多真实世界的妥协就发生在这里：访问运行时 socket 或守护进程 API 即便在低级运行时本身完全健康的情况下，也可能等同于对主机的妥协。

### Docker Engine

Docker Engine 是最为开发者所熟知的容器平台，也是容器词汇为什么如此 Docker 化的原因之一。典型路径是 `docker` CLI 到 `dockerd`，后者又协调更低级的组件诸如 `containerd` 和一个 OCI 运行时。历史上，Docker 部署常常是以 root 运行的（rootful），因此访问 Docker socket 一直是一个非常强力的原语。这就是为什么大量实用的权限提升材料集中在 `docker.sock` 上：如果一个进程可以请求 `dockerd` 创建有特权的容器、挂载主机路径或加入主机命名空间，它可能根本不需要内核漏洞。

### Podman

Podman 设计为更无守护进程（daemonless）的模型。从操作上讲，这有助于强化容器只是通过标准 Linux 机制管理的进程，而不是通过一个长期运行的特权守护进程管理的想法。Podman 在无根（rootless）方面比许多人最初接触的经典 Docker 部署有更强的故事。这并不意味着 Podman 自动就安全，但它显著改变了默认的风险特征，特别是当与用户命名空间、SELinux 和 `crun` 结合时。

### containerd

containerd 是许多现代栈中的核心运行时管理组件。它在 Docker 之下使用，也是 Kubernetes 的主导运行时后端之一。它暴露强大的 API，管理镜像和快照，并将最终的进程创建委托给低级运行时。围绕 containerd 的安全讨论应强调，访问 containerd socket 或 `ctr`/`nerdctl` 功能可能与访问 Docker 的 API 一样危险，即便其界面和工作流感觉上不那么“面向开发者”。

### CRI-O

CRI-O 的关注点比 Docker Engine 更窄。它不是一个通用的开发者平台，而是围绕清晰实现 Kubernetes Container Runtime Interface 构建的。这使得它在 Kubernetes 发行版和诸如 OpenShift 这样的 SELinux 密集型生态中尤为常见。从安全角度看，这种更窄的范围有助于减少概念杂乱：CRI-O 更属于“为 Kubernetes 运行容器”的那一层，而不是一个无所不包的平台。

### Incus, LXD, 与 LXC

Incus/LXD/LXC 系统值得与类似 Docker 的应用容器区分开，因为它们常被用作系统容器（system containers）。系统容器通常被期望更像一台轻量级机器，具有更完整的用户空间、长期运行的服务、更丰富的设备暴露以及更广泛的主机集成。隔离机制仍然是内核原语，但操作期望不同。因此，此类环境中的错误配置常常看起来不像“糟糕的应用容器默认值”，而更像是轻量虚拟化或主机委派方面的错误。

### systemd-nspawn

systemd-nspawn 占据一个有趣的位置，因为它是 systemd 原生的，对于测试、调试和运行类操作系统环境非常有用。它不是云原生生产环境的主流运行时，但在实验室和面向发行版的环境中出现得足够频繁，值得一提。对于安全分析来说，它再次提醒我们“容器”这个概念跨越多个生态系统和操作风格。

### Apptainer / Singularity

Apptainer（前称 Singularity）在科研和 HPC 环境中很常见。它的信任假设、用户工作流和执行模型在重要方面与以 Docker/Kubernetes 为中心的堆栈不同。特别是，这些环境通常非常重视让用户运行打包的工作负载而不赋予他们广泛的特权容器管理能力。如果评审者假定每个容器环境基本上都是“服务器上的 Docker”，他们会严重误解这些部署。

## 构建时工具链

许多安全讨论只谈运行时，但构建时工具也很重要，因为它决定了镜像内容、构建时机密暴露，以及多少受信任的上下文会被嵌入到最终制品中。

**BuildKit** 和 `docker buildx` 是现代构建后端，支持缓存、secret 挂载、SSH 转发和多平台构建等功能。这些都是有用的功能，但从安全角度看，它们也会创造密钥可能 leak 到镜像层的地方，或者过于宽泛的构建上下文可能暴露不该被包含的文件。**Buildah** 在 OCI 原生生态中扮演类似角色，尤其是在 Podman 周边，而 **Kaniko** 常在不希望给构建流水线授予特权 Docker 守护进程的 CI 环境中使用。

关键教训是：镜像创建和镜像执行是不同的阶段，但薄弱的构建流水线可以在容器启动前很久就造成薄弱的运行时姿态。

## 编排是另一层，而不是运行时

不应在心理上将 Kubernetes 等同于运行时本身。Kubernetes 是编排器。它调度 Pod、存储期望状态，并通过工作负载配置表达安全策略。kubelet 然后与诸如 containerd 或 CRI-O 的 CRI 实现对话，这些实现又调用低级运行时，例如 `runc`、`crun`、`runsc` 或 `kata-runtime`。

这种分离很重要，因为很多人错误地将某种保护归功于“Kubernetes”，而实际上是由节点运行时强制执行的，或者他们责备“containerd 默认值”而行为实际上来自 Pod 规范。实际上，最终的安全姿态是一个组合：编排器提出需求，运行时栈将其翻译，最后由内核强制执行。

## 在评估期间为什么识别运行时很重要

如果你能及早识别引擎和运行时，许多后续观察会更容易解读。一个无根（rootless）的 Podman 容器暗示用户命名空间很可能是整个故事的一部分。一个挂载到工作负载中的 Docker socket 暗示基于 API 的权限提升是现实可行的路径。一个 CRI-O/OpenShift 节点应立即让你想到 SELinux 标签和受限的工作负载策略。一个 gVisor 或 Kata 环境应让你对假设经典 `runc` 越狱 PoC 会有相同表现持更谨慎的态度。

这就是为什么在容器评估的最初步骤之一总是要回答两个简单问题：哪个组件在管理容器，以及哪个运行时实际启动了进程。一旦这些答案清楚，通常就更容易推理其余环境。

## 运行时漏洞

并非所有容器逃逸都来自运维错误配置。有时运行时本身就是有漏洞的组件。这很重要，因为一个工作负载可能在看似谨慎的配置下运行，仍然可能通过低级运行时缺陷暴露。

经典例子是 `runc` 中的 **CVE-2019-5736**，在该漏洞中，恶意容器可以覆写主机上的 `runc` 二进制文件，然后等待后续的 `docker exec` 或类似运行时调用触发由攻击者控制的代码。利用路径与简单的绑定挂载或 capability 错误非常不同，因为它滥用了运行时在处理 exec 时重新进入容器进程空间的方式。

从 red-team 角度看，一个最小的复现工作流是：
```bash
go build main.go
./main
```
然后，从主机：
```bash
docker exec -it <container-name> /bin/sh
```
关键教训不在于具体的历史漏洞利用实现，而在于评估的含义：如果运行时版本存在漏洞，普通的容器内代码执行可能就足以在可见容器配置看起来并非明显薄弱的情况下，危及宿主机。

最近的运行时 CVE（例如 `CVE-2024-21626` 在 `runc` 中）、BuildKit 挂载竞态以及 containerd 解析漏洞都强化了这一点。运行时版本和补丁级别是安全边界的一部分，而不仅仅是维护琐事。
