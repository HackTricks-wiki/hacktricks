# Εκτελεστικά Περιβάλλοντα (Runtimes), Engines, Builders, And Sandboxes

{{#include ../../../banners/hacktricks-training.md}}

Μία από τις μεγαλύτερες πηγές σύγχυσης στην ασφάλεια κοντέινερ είναι ότι πολλά εντελώς διαφορετικά συστατικά συχνά συμπυκνώνονται στην ίδια λέξη. "Docker" μπορεί να αναφέρεται σε ένα image format, ένα CLI, ένα daemon, ένα build system, ένα runtime stack, ή απλώς στην ιδέα των κοντέινερ γενικά. Για εργασίες ασφάλειας αυτή η ασάφεια αποτελεί πρόβλημα, γιατί διαφορετικά στρώματα είναι υπεύθυνα για διαφορετικές προστασίες. Μια έξοδος (breakout) που προκαλείται από κακό bind mount δεν είναι το ίδιο πράγμα με μια έξοδο που προκαλείται από σφάλμα σε χαμηλού επιπέδου runtime, και κανένα από τα δύο δεν είναι το ίδιο με ένα λάθος πολιτικής cluster στο Kubernetes.

Αυτή η σελίδα διαχωρίζει το οικοσύστημα ανά ρόλο ώστε το υπόλοιπο της ενότητας να μπορεί να συζητά με ακρίβεια πού πραγματικά ζει μια προστασία ή μια αδυναμία.

## OCI Ως Η Κοινή Γλώσσα

Τα σύγχρονα Linux container stacks συχνά αλληλοεπιδρούν επειδή μιλάνε ένα σύνολο προδιαγραφών OCI. Η **OCI Image Specification** περιγράφει πώς αναπαριστώνται images και layers. Η **OCI Runtime Specification** περιγράφει πώς το runtime πρέπει να εκκινήσει τη διαδικασία, συμπεριλαμβανομένων namespaces, mounts, cgroups και security settings. Η **OCI Distribution Specification** τυποποιεί τον τρόπο με τον οποίο τα registries εκθέτουν περιεχόμενο.

Αυτό έχει σημασία γιατί εξηγεί γιατί ένα container image που κατασκευάζεται με ένα εργαλείο συχνά μπορεί να τρέξει με κάποιο άλλο, και γιατί αρκετοί engines μπορούν να μοιράζονται το ίδιο χαμηλού επιπέδου runtime. Επίσης εξηγεί γιατί η συμπεριφορά ασφάλειας μπορεί να φαίνεται παρόμοια σε διαφορετικά προϊόντα: πολλά από αυτά κατασκευάζουν την ίδια OCI runtime configuration και την παραδίδουν στο ίδιο μικρό σύνολο runtimes.

## Χαμηλού Επιπέδου OCI Runtimes

Το χαμηλού επιπέδου runtime είναι το συστατικό που βρίσκεται πιο κοντά στο όριο του kernel. Είναι το μέρος που πραγματικά δημιουργεί namespaces, γράφει cgroup ρυθμίσεις, εφαρμόζει capabilities και seccomp φίλτρα, και τελικά κάνει `execve()` της διαδικασίας του κοντέινερ. Όταν οι άνθρωποι συζητούν για "container isolation" σε μηχανικό επίπεδο, αυτό είναι το στρώμα στο οποίο συνήθως αναφέρονται, ακόμη κι αν δεν το λένε ρητώς.

### `runc`

`runc` είναι το reference OCI runtime και παραμένει η πιο γνωστή υλοποίηση. Χρησιμοποιείται εκτενώς υπό Docker, containerd, και σε πολλές εγκαταστάσεις Kubernetes. Πολλή δημόσια έρευνα και exploit υλικό στοχεύει περιβάλλοντα τύπου `runc` απλά επειδή είναι κοινά και επειδή το `runc` ορίζει τη baseline που πολλοί έχουν στο μυαλό τους όταν φαντάζονται ένα Linux container. Η κατανόηση του `runc` δίνει επομένως στον αναγνώστη ένα ισχυρό νοητικό μοντέλο για την κλασική container isolation.

### `crun`

`crun` είναι ένα άλλο OCI runtime, γραμμένο σε C και ευρέως χρησιμοποιούμενο σε σύγχρονα Podman environments. Συχνά επαινείται για την καλή υποστήριξη cgroup v2, την καλή ergonomics για rootless, και το χαμηλότερο overhead. Από άποψη ασφάλειας, το σημαντικό δεν είναι ότι είναι γραμμένο σε διαφορετική γλώσσα, αλλά ότι εξακολουθεί να παίζει τον ίδιο ρόλο: είναι το συστατικό που μετατρέπει την OCI configuration σε ένα τρέχον δέντρο διεργασιών κάτω από τον kernel. Ένα rootless Podman workflow συχνά φαίνεται ασφαλέστερο όχι επειδή το `crun` μαγικά διορθώνει τα πάντα, αλλά επειδή όλο το γύρω stack τείνει να χρησιμοποιεί περισσότερο user namespaces και αρχές least privilege.

### `runsc` Από gVisor

`runsc` είναι το runtime που χρησιμοποιεί το gVisor. Εδώ το όριο αλλάζει ουσιαστικά. Αντί να περάσει τα περισσότερα syscalls απευθείας στον host kernel με τον συνήθη τρόπο, το gVisor εισάγει ένα userspace kernel layer που προσομοιώνει ή μεσολαβεί μεγάλα τμήματα του Linux interface. Το αποτέλεσμα δεν είναι ένα κανονικό `runc` κοντέινερ με κάποια επιπλέον flags· είναι ένας διαφορετικός σχεδιασμός sandbox με σκοπό να μειώσει την επιφάνεια επίθεσης του host-kernel. Τα tradeoffs συμβατότητας και απόδοσης είναι μέρος αυτού του σχεδιασμού, έτσι περιβάλλοντα που χρησιμοποιούν `runsc` θα πρέπει να τεκμηριώνονται διαφορετικά από τα κανονικά OCI runtime περιβάλλοντα.

### `kata-runtime`

Kata Containers ωθούν το όριο πιο μακριά εκκινώντας το workload μέσα σε μια lightweight virtual machine. Διοικητικά, αυτό μπορεί να εξακολουθεί να μοιάζει με μια container ανάπτυξη, και τα orchestration layers μπορεί να το αντιμετωπίζουν ως τέτοιο, αλλά το υποκείμενο όριο απομόνωσης είναι πιο κοντά στην virtualization παρά σε ένα κλασικό host-kernel-shared container. Αυτό κάνει το Kata χρήσιμο όταν επιθυμείται ισχυρότερη απομόνωση tenants χωρίς εγκατάλειψη των container-centric workflows.

## Engines Και Διαχειριστές Container

Αν το χαμηλού επιπέδου runtime είναι το συστατικό που μιλάει απευθείας με τον kernel, ο engine ή manager είναι το συστατικό με το οποίο οι χρήστες και οι χειριστές συνήθως αλληλεπιδρούν. Χειρίζεται image pulls, metadata, logs, networks, volumes, lifecycle operations και API exposure. Αυτό το στρώμα έχει τεράστια σημασία γιατί πολλά πραγματικά συμβιβασμοί συμβαίνουν εδώ: πρόσβαση σε ένα runtime socket ή daemon API μπορεί να ισοδυναμεί με compromise του host ακόμη και αν το χαμηλού επιπέδου runtime είναι απολύτως υγιές.

### Docker Engine

Docker Engine είναι η πιο αναγνωρίσιμη πλατφόρμα container για developers και ένας από τους λόγους που το λεξιλόγιο των κοντέινερ έγινε τόσο Docker-σχηματικό. Η τυπική διαδρομή είναι `docker` CLI προς `dockerd`, το οποίο με τη σειρά του συντονίζει χαμηλότερου επιπέδου συστατικά όπως το `containerd` και ένα OCI runtime. Ιστορικά, εγκαταστάσεις Docker ήταν συχνά **rootful**, και η πρόσβαση στο socket του Docker έχει επομένως υπάρξει πολύ ισχυρό primitive. Γι' αυτό τόσο πολύ πρακτικό υλικό privilege-escalation επικεντρώνεται στο `docker.sock`: αν μια διαδικασία μπορεί να ζητήσει από το `dockerd` να δημιουργήσει ένα privileged container, να κάνει mount host paths, ή να ενταχθεί σε host namespaces, μπορεί να μην χρειάζεται καν exploit του kernel.

### Podman

Το Podman σχεδιάστηκε γύρω από ένα πιο daemonless μοντέλο. Λειτουργικά, αυτό βοηθάει να ενισχυθεί η ιδέα ότι τα κοντέινερ είναι απλώς διεργασίες που διαχειρίζονται μέσω των κανονικών μηχανισμών του Linux και όχι μέσω ενός μακροχρόνιου privileged daemon. Το Podman έχει επίσης πολύ ισχυρότερη ιστορία **rootless** από τις κλασικές Docker εγκαταστάσεις που πολλοί έμαθαν πρώτα. Αυτό δεν κάνει το Podman αυτόματα ασφαλές, αλλά αλλάζει σημαντικά το default risk profile, ειδικά σε συνδυασμό με user namespaces, SELinux και `crun`.

### containerd

containerd είναι ένα βασικό συστατικό διαχείρισης runtime σε πολλά σύγχρονα stacks. Χρησιμοποιείται κάτω από Docker και είναι επίσης ένας από τους κυρίαρχους Kubernetes runtime backends. Εκθέτει ισχυρά APIs, διαχειρίζεται images και snapshots, και παραπέμπει τη τελική δημιουργία διεργασίας σε ένα χαμηλού επιπέδου runtime. Οι συζητήσεις ασφάλειας γύρω από το containerd πρέπει να τονίζουν ότι η πρόσβαση στο containerd socket ή στη λειτουργικότητα `ctr`/`nerdctl` μπορεί να είναι εξίσου επικίνδυνη με την πρόσβαση στο API του Docker, ακόμη κι αν το interface και το workflow φαίνονται λιγότερο "developer friendly".

### CRI-O

CRI-O είναι πιο εστιασμένο από το Docker Engine. Αντί να είναι μια γενικής χρήσης πλατφόρμα για developers, έχει χτιστεί γύρω από την υλοποίηση της Kubernetes Container Runtime Interface με καθαρό τρόπο. Αυτό το καθιστά ιδιαίτερα συνηθισμένο σε διανομές Kubernetes και σε περιβάλλοντα βαριά σε SELinux όπως το OpenShift. Από άποψη ασφάλειας, αυτή η στενότερη εστίαση είναι χρήσιμη γιατί μειώνει τη νοητική αταξία: το CRI-O είναι πολύ μέρος του στρώματος "run containers for Kubernetes" παρά μια πλατφόρμα-όλα.

### Incus, LXD, Και LXC

Συστήματα Incus/LXD/LXC αξίζει να διαχωριστούν από τα Docker-style application containers γιατί χρησιμοποιούνται συχνά ως **system containers**. Ένα system container συνήθως αναμένεται να μοιάζει περισσότερο με ένα lightweight μηχάνημα με πληρέστερο userspace, μακροχρόνιες υπηρεσίες, πλουσιότερη έκθεση συσκευών και πιο εκτεταμένη ενσωμάτωση με το host. Οι μηχανισμοί απομόνωσης εξακολουθούν να είναι kernel primitives, αλλά οι επιχειρησιακές προσδοκίες είναι διαφορετικές. Ως αποτέλεσμα, οι λανθασμένες ρυθμίσεις εδώ συχνά μοιάζουν λιγότερο με "κακά app-container defaults" και περισσότερο με λάθη στο lightweight virtualization ή στην ανάθεση δικαιωμάτων του host.

### systemd-nspawn

Το systemd-nspawn καταλαμβάνει μια ενδιαφέρουσα θέση επειδή είναι systemd-native και πολύ χρήσιμο για testing, debugging και για την εκτέλεση OS-like περιβαλλόντων. Δεν είναι το κυρίαρχο cloud-native production runtime, αλλά εμφανίζεται αρκετά συχνά σε labs και περιβάλλοντα προσανατολισμένα σε διανομές ώστε να αξίζει να αναφερθεί. Για την ανάλυση ασφάλειας, αποτελεί μια ακόμα υπενθύμιση ότι η έννοια "container" καλύπτει πολλαπλά οικοσυστήματα και επιχειρησιακά στυλ.

### Apptainer / Singularity

Το Apptainer (πρώην Singularity) είναι συνηθισμένο σε ερευνητικά και HPC περιβάλλοντα. Οι υποθέσεις εμπιστοσύνης, το user workflow και το execution model διαφέρουν σημαντικά από τα Docker/Kubernetes-centric stacks. Συγκεκριμένα, αυτά τα περιβάλλοντα συχνά ενδιαφέρονται πολύ να επιτρέψουν στους χρήστες να τρέχουν πακεταρισμένα workloads χωρίς να τους παραδίδονται ευρείες privileged δυνατότητες διαχείρισης κοντέινερ. Αν ένας αξιολογητής υποθέσει ότι κάθε περιβάλλον κοντέινερ είναι βασικά "Docker σε έναν server", θα παρερμηνεύσει βαριά αυτές τις αναπτύξεις.

## Εργαλεία Κατά τον Χρόνο Κατασκευής (Build-Time Tooling)

Πολλές συζητήσεις για ασφάλεια μιλάνε μόνο για το runtime, αλλά τα εργαλεία κατά τη φάση κατασκευής επίσης μετράνε γιατί καθορίζουν το περιεχόμενο του image, την έκθεση build secrets, και πόσο αξιόπιστο context ενσωματώνεται στο τελικό artifact.

**BuildKit** και `docker buildx` είναι μοντέρνα build backends που υποστηρίζουν δυνατότητες όπως caching, secret mounting, SSH forwarding, και multi-platform builds. Αυτές είναι χρήσιμες δυνατότητες, αλλά από άποψη ασφάλειας δημιουργούν επίσης σημεία όπου secrets μπορούν να leak σε image layers ή όπου ένα υπερβολικά ευρύ build context μπορεί να εκθέσει αρχεία που δεν έπρεπε ποτέ να συμπεριληφθούν. **Buildah** παίζει παρόμοιο ρόλο σε OCI-native οικοσυστήματα, ειδικά γύρω από Podman, ενώ **Kaniko** χρησιμοποιείται συχνά σε CI περιβάλλοντα που δεν θέλουν να δώσουν privileged Docker daemon στο build pipeline.

Το βασικό μάθημα είναι ότι η δημιουργία image και η εκτέλεση image είναι διαφορετικά στάδια, αλλά μια αδύναμη build pipeline μπορεί να δημιουργήσει μια αδύναμη runtime κατάσταση πολύ πριν εκκινήσει το κοντέινερ.

## Το Orchestration Είναι Άλλο Επίπεδο, Όχι Το Runtime

Το Kubernetes δεν πρέπει να ταυτίζεται νοητικά με το ίδιο το runtime. Το Kubernetes είναι ο orchestrator. Προγραμματίζει Pods, αποθηκεύει desired state, και εκφράζει security policy μέσω της διαμόρφωσης workload. Ο kubelet στη συνέχεια μιλάει σε μια CRI υλοποίηση όπως το containerd ή CRI-O, που με τη σειρά του επικαλεί ένα χαμηλού επιπέδου runtime όπως `runc`, `crun`, `runsc`, ή `kata-runtime`.

Αυτός ο διαχωρισμός έχει σημασία γιατί πολλοί άνθρωποι λανθασμένα αποδίδουν μια προστασία στο "Kubernetes" όταν στην πραγματικότητα επιβάλλεται από το node runtime, ή κατηγορούν τα "containerd defaults" για συμπεριφορά που προήλθε από ένα Pod spec. Στην πράξη, η τελική στάση ασφάλειας είναι μια σύνθεση: ο orchestrator ζητάει κάτι, το runtime stack το μεταφράζει, και ο kernel τελικά το επιβάλλει.

## Γιατί Η Αναγνώριση Του Runtime Έχει Σημασία Κατά Την Αξιολόγηση

Αν αναγνωρίσετε τον engine και το runtime νωρίς, πολλές επόμενες παρατηρήσεις γίνονται πιο εύκολες στην ερμηνεία. Ένα rootless Podman container υποδηλώνει ότι τα user namespaces είναι πιθανό μέρος της εικόνας. Ένα Docker socket προσαρτημένο σε ένα workload υποδηλώνει ότι η ανάβαθμιση προνομίων μέσω API είναι ρεαλιστική διαδρομή. Ένας κόμβος CRI-O/OpenShift πρέπει αμέσως να σας κάνει να σκεφτείτε SELinux labels και restricted workload policy. Ένα περιβάλλον gVisor ή Kata πρέπει να σας κάνει πιο προσεκτικούς στο να υποθέσετε ότι ένα κλασικό PoC breakout για `runc` θα συμπεριφερθεί το ίδιο.

Γι' αυτό ένα από τα πρώτα βήματα στην αξιολόγηση κοντέινερ πρέπει πάντα να είναι να απαντηθούν δύο απλές ερωτήσεις: **ποιο συστατικό διαχειρίζεται το κοντέινερ** και **ποιο runtime τελικά εκκίνησε τη διαδικασία**. Μόλις αυτές οι απαντήσεις είναι σαφείς, το υπόλοιπο του περιβάλλοντος συνήθως γίνεται πολύ πιο εύκολο να το σκεφτεί κανείς.

## Ευπάθειες Στα Runtimes

Όχι κάθε escape κοντέινερ προέρχεται από λάθος διαχείρισης. Κάποτε το ίδιο το runtime είναι το ευάλωτο συστατικό. Αυτό έχει σημασία γιατί ένα workload μπορεί να τρέχει με αυτό που μοιάζει προσεκτική διαμόρφωση και παρόλα αυτά να εκτίθεται μέσω ενός σφάλματος χαμηλού επιπέδου runtime.

Το κλασικό παράδειγμα είναι το **CVE-2019-5736** στο `runc`, όπου ένα κακόβουλο κοντέινερ μπορούσε να αντικαταστήσει το εκτελέσιμο `runc` στο host και στη συνέχεια να περιμένει για ένα αργότερο `docker exec` ή παρόμοια κλήση runtime ώστε να ενεργοποιήσει κώδικα ελεγχόμενο από τον επιτιθέμενο. Η διαδρομή εκμετάλλευσης είναι πολύ διαφορετική από ένα απλό bind-mount ή λάθος capability, γιατί εκμεταλλεύεται τον τρόπο που το runtime ξαναμπαίνει στο χώρο διεργασίας του κοντέινερ κατά τη διαχείριση του exec.

Ένα ελάχιστο workflow αναπαραγωγής από την προοπτική ενός red-team είναι:
```bash
go build main.go
./main
```
Στη συνέχεια, από τον host:
```bash
docker exec -it <container-name> /bin/sh
```
Το βασικό μάθημα δεν είναι η ακριβής ιστορική υλοποίηση του exploit, αλλά το συμπέρασμα για την αξιολόγηση: αν η έκδοση του runtime είναι ευάλωτη, μια απλή in-container code execution μπορεί να είναι αρκετή για να compromise τον host, ακόμα και όταν η ορατή container configuration δεν φαίνεται φανερά αδύναμη.

Πρόσφατα runtime CVEs όπως `CVE-2024-21626` στο `runc`, BuildKit mount races και containerd parsing bugs ενισχύουν το ίδιο σημείο. Η runtime version και το patch level είναι μέρος του security boundary, όχι απλώς maintenance trivia.
