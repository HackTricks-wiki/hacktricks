# Оцінка та посилення безпеки

{{#include ../../../banners/hacktricks-training.md}}

## Огляд

Добра оцінка контейнера має відповісти на два паралельні питання. По-перше, що може зробити атакуючий з поточного workload? По-друге, які рішення оператора зробили це можливим? Інструменти для переліку допомагають з першим питанням, а рекомендації щодо посилення — із другим. Збереження обох аспектів на одній сторінці робить розділ кориснішим як довідник у полі, а не просто каталогом трюків для escape.

## Інструменти для переліку

Кілька інструментів залишаються корисними для швидкої характеристики середовища контейнера:

- `linpeas` може виявляти багато індикаторів контейнера, змонтовані сокети, набори capabilities, небезпечні файлові системи та підказки щодо breakout.
- `CDK` зосереджується спеціально на середовищах контейнерів і включає перерахування плюс деякі автоматизовані перевірки escape.
- `amicontained` легкий і корисний для визначення обмежень контейнера, capabilities, експозиції просторів імен та ймовірних класів breakout.
- `deepce` — ще один інструмент, орієнтований на контейнери, з перевірками, спрямованими на breakout.
- `grype` корисний, коли оцінка включає огляд вразливостей пакетів у образі, а не лише аналіз escape під час виконання.

Цінність цих інструментів — швидкість і охоплення, а не остаточна певність. Вони допомагають швидко виявити загальну позицію безпеки, але цікаві знахідки все одно потребують ручної інтерпретації стосовно фактичної моделі runtime, просторів імен, capabilities та mount.

## Пріоритети посилення безпеки

Найважливіші принципи посилення концептуально прості, хоча їхня реалізація залежить від платформи. Уникайте privileged контейнерів. Уникайте змонтованих runtime сокетів. Не давайте контейнерам записувані host-пути, якщо немає дуже конкретної причини. Використовуйте user namespaces або rootless виконання там, де це можливо. Скиньте всі capabilities і додайте назад лише ті, які дійсно потрібні workload. Тримайте увімкненими seccomp, AppArmor і SELinux замість того, щоб відключати їх для вирішення проблем сумісності додатків. Обмежуйте ресурси так, щоб скомпрометований контейнер не міг тривіально відмовити в обслуговуванні хосту.

Гігієна образів і процесів збірки важить так само сильно, як і позиція під час виконання. Використовуйте мінімальні образи, періодично перебирайте їх заново, скануйте, вимагайте походження там, де це практично, і тримайте секрети поза шарами. Контейнер, що запускається як non-root з невеликим образом і вузькою поверхнею syscall та capabilities, значно легше захищати, ніж великий образ для зручності, що виконується як root рівня хоста з попередньо встановленими інструментами для налагодження.

## Приклади виснаження ресурсів

Контроль ресурсів не є гламурним, але він є частиною безпеки контейнера, оскільки обмежує радіус ураження при компрометації. Без обмежень пам'яті, CPU або PID простий shell може бути достатнім, щоб погіршити роботу хоста або сусідніх workload.

Приклади тестів, що впливають на хост:
```bash
stress-ng --vm 1 --vm-bytes 1G --verify -t 5m
docker run -d --name malicious-container -c 512 busybox sh -c 'while true; do :; done'
nc -lvp 4444 >/dev/null & while true; do cat /dev/urandom | nc <target_ip> 4444; done
```
Ці приклади корисні, тому що показують, що не кожен небезпечний результат container є чистим "escape". Слабкі обмеження cgroup все ще можуть перетворити виконання коду на реальний операційний вплив.

## Інструменти для підвищення безпеки

Для середовищ, орієнтованих на Docker, `docker-bench-security` залишається корисною базовою перевіркою на стороні хоста, оскільки вона перевіряє поширені проблеми конфігурації відповідно до загальновизнаних рекомендацій бенчмарку:
```bash
git clone https://github.com/docker/docker-bench-security.git
cd docker-bench-security
sudo sh docker-bench-security.sh
```
Цей інструмент не є заміною для threat modeling, але все ще корисний для виявлення необережних daemon, mount, network та runtime defaults, що накопичуються з часом.

## Перевірки

Використовуйте ці команди як швидку первинну перевірку під час оцінки:
```bash
id
capsh --print 2>/dev/null
grep -E 'Seccomp|NoNewPrivs' /proc/self/status
mount
find / -maxdepth 3 \( -name docker.sock -o -name containerd.sock -o -name crio.sock -o -name podman.sock \) 2>/dev/null
```
Що тут цікаво:

- Процес root із широкими можливостями та `Seccomp: 0` заслуговує на негайну увагу.
- Підозрілі mounts і runtime sockets часто забезпечують швидший шлях до впливу, ніж будь-який kernel exploit.
- Поєднання слабкої runtime posture і слабких resource limits зазвичай вказує на загалом дозволяюче середовище контейнерів, а не на одну ізольовану помилку.
