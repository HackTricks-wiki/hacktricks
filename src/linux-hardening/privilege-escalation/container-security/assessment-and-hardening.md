# Оцінка та посилення безпеки

## Огляд

Хороше оцінювання контейнера має відповісти на два паралельні питання. По-перше, що може зробити атакувач з поточної робочої навантаження? По-друге, які рішення оператора зробили це можливим? Інструменти енумерації допомагають з першим питанням, а рекомендації з жорсткого захисту — з другим. Поєднання обох аспектів на одній сторінці робить розділ кориснішим як довідник у полі, а не лише каталогом трюків для escape.

## Інструменти для енумерації

Кілька інструментів залишаються корисними для швидкої характеристики середовища контейнера:

- `linpeas` може виявити багато індикаторів контейнера, змонтовані sockets, набори capability, небезпечні файлові системи та підказки для breakout.
- `CDK` фокусується саме на середовищах контейнерів і включає енумерацію плюс деякі автоматизовані перевірки escape.
- `amicontained` легкий і корисний для визначення обмежень контейнера, capability, exposure namespace та ймовірних класів breakout.
- `deepce` — ще один енумератор, орієнтований на контейнер, з перевірками, спрямованими на breakout.
- `grype` корисний, коли оцінювання включає перегляд вразливостей пакетів образу замість лише аналізу runtime escape.

Цінність цих інструментів — у швидкості та покритті, а не в остаточній достовірності. Вони допомагають швидко виявити загальну позицію, але цікаві знахідки все ще потребують ручної інтерпретації щодо фактичної моделі runtime, namespace, capability та mount.

## Пріоритети жорсткого захисту

Найважливіші принципи підвищення безпеки концептуально прості, хоча їх реалізація залежить від платформи. Уникати привілейованих контейнерів. Уникати змонтованих runtime sockets. Не надавати контейнерам записувані шляхи хоста, якщо немає дуже конкретної причини. Використовувати user namespaces або rootless execution там, де це можливо. Скинути всі capability і додавати назад лише ті, що дійсно потрібні робочому навантаженню. Залишати seccomp, AppArmor і SELinux увімкненими замість відключення для вирішення проблем сумісності застосунків. Обмежувати ресурси, щоб скомпрометований контейнер не міг тривіально відмовити в обслуговуванні хосту.

Гігієна образів і процесу збірки важить так само, як і runtime-позиція. Використовуйте мінімальні images, перебудовуйте їх часто, скануйте, вимагайте provenance там, де це практично, і тримайте secrets поза шарами. Контейнер, що працює як non-root з невеликим image і вузькою поверхнею syscall і capability, значно легше захищати, ніж великий зручний image, що працює як host-equivalent root з попередньо встановленими debugging tools.

## Приклади виснаження ресурсів

Обмеження ресурсів не є привабливими, але вони є частиною безпеки контейнерів, бо обмежують радіус ураження при компрометації. Без обмежень пам'яті, CPU або PID простий shell може бути достатнім, щоб погіршити роботу хоста або сусідніх робочих навантажень.

Приклади тестів, що впливають на хост:
```bash
stress-ng --vm 1 --vm-bytes 1G --verify -t 5m
docker run -d --name malicious-container -c 512 busybox sh -c 'while true; do :; done'
nc -lvp 4444 >/dev/null & while true; do cat /dev/urandom | nc <target_ip> 4444; done
```
Ці приклади корисні, оскільки показують, що не кожен небезпечний результат роботи контейнера є чистим "escape". Слабкі обмеження cgroup все ще можуть перетворити виконання коду на реальний операційний вплив.

## Інструменти для зміцнення безпеки

Для середовищ, орієнтованих на Docker, `docker-bench-security` залишається корисною базовою перевіркою на боці хоста, оскільки вона перевіряє поширені проблеми конфігурації відповідно до широко визнаних бенчмарків та рекомендацій:
```bash
git clone https://github.com/docker/docker-bench-security.git
cd docker-bench-security
sudo sh docker-bench-security.sh
```
Цей інструмент не замінює моделювання загроз, але все ще корисний для виявлення необачних daemon, mount, network і runtime налаштувань за замовчуванням, які накопичуються з часом.

## Checks

Використовуйте їх як швидкі команди для первинної оцінки:
```bash
id
capsh --print 2>/dev/null
grep -E 'Seccomp|NoNewPrivs' /proc/self/status
mount
find / -maxdepth 3 \( -name docker.sock -o -name containerd.sock -o -name crio.sock -o -name podman.sock \) 2>/dev/null
```
- root-процес із широкими повноваженнями та `Seccomp: 0` вимагає негайної уваги.
- Підозрілі точки монтування та runtime sockets часто дають швидший шлях до впливу, ніж будь-який kernel exploit.
- Поєднання слабкої runtime posture і слабких resource limits зазвичай вказує на загалом дозволене середовище контейнерів, а не на одну ізольовану помилку.
