# 7.2. Fine-Tuning, um Anweisungen zu befolgen

{{#include ../../banners/hacktricks-training.md}}

> [!TIP]
> Das Ziel dieses Abschnitts ist es zu zeigen, wie man ein **bereits vortrainiertes Modell so fine-tuned, dass es Anweisungen befolgt** statt nur Text zu generieren, zum Beispiel, indem es auf Aufgaben wie ein Chatbot antwortet.

## Dataset

Um ein LLM so fine-tunen, dass es Anweisungen befolgt, braucht man ein Dataset mit **Anweisungen und Antworten**. Es gibt verschiedene Prompt-Formate, um ein LLM darauf zu trainieren, Anweisungen zu befolgen, zum Beispiel:

- Das Alpaca-Prompt-Style-Beispiel:
```csharp
Below is an instruction that describes a task. Write a response that appropriately completes the request.

### Instruction:
Calculate the area of a circle with a radius of 5 units.

### Response:
The area of a circle is calculated using the formula \( A = \pi r^2 \). Plugging in the radius of 5 units:

\( A = \pi (5)^2 = \pi \times 25 = 25\pi \) square units.
```
- Phi-3 / chat template style example:
```vbnet
<|User|>
Can you explain what gravity is in simple terms?

<|Assistant|>
Absolutely! Gravity is a force that pulls objects toward each other.
```
Das Training mit solchen Datensätzen statt nur mit Rohtext hilft dem Modell zu verstehen, dass es Aufgaben in einem **strukturierten Assistant-Format** beantworten muss.

Ein sehr häufiger Fehler ist, mit einem Template zu trainieren und mit einem anderen zu inferieren. Moderne Instruct-Modelle erwarten in der Regel das **native chat template des Base Models** (`<|user|>`, `<|assistant|>`, `[INST]`, ChatML, etc.), daher wird empfohlen, den Datensatz in genau das Format zu konvertieren, das das Zielmodell bereits verwendet.

Daher ist eines der ersten Dinge, die man mit einem Datensatz tun sollte, der Requests und Antworten enthält, diese Daten im gewünschten Prompt-Format zu modellieren, etwa so:
```python
# Code from https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/ch07.ipynb
def format_input(entry):
instruction_text = (
f"Below is an instruction that describes a task. "
f"Write a response that appropriately completes the request."
f"\n\n### Instruction:\n{entry['instruction']}"
)

input_text = f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""

return instruction_text + input_text

model_input = format_input(data[50])

desired_response = f"\n\n### Response:\n{data[50]['output']}"

print(model_input + desired_response)
```
Wenn das Basismodell bereits mit einer Tokenizer-Chat-Vorlage ausgeliefert wird, kann es sicherer sein, das Dataset aus Nachrichtenlisten zu erstellen, statt Trennzeichen manuell zu konstruieren:
```python
messages = [
{"role": "user", "content": entry["instruction"]},
{"role": "assistant", "content": entry["output"]},
]
text = tokenizer.apply_chat_template(messages, tokenize=False)
```
Dann ist es wie immer nötig, den Datensatz in Trainings-, Validierungs- und Testsets aufzuteilen.

## Batching & Data Loaders

Dann ist es nötig, alle Eingaben und erwarteten Ausgaben für das Training zu batchen. Dafür ist es nötig:

- Die Texte zu tokenisieren.
- Alle Samples auf dieselbe Länge zu padden (üblicherweise wird die Länge so groß sein wie die Context Length, die zum Vortrainieren des LLM verwendet wurde).
- Die erwarteten Tokens zu erstellen, indem die Eingabe in einer benutzerdefinierten collate function um 1 verschoben wird.
- Einige Padding-Tokens durch `-100` zu ersetzen, um sie von der Training Loss auszuschließen: Nach dem ersten `endoftext`-Token alle anderen `endoftext`-Tokens durch `-100` ersetzen (weil `cross_entropy(..., ignore_index=-100)` bedeutet, dass Targets mit `-100` ignoriert werden).
- \[Optional] Mit `-100` auch alle Tokens maskieren, die zur Frage gehören, damit das LLM nur lernt, die Antwort zu generieren. Im Alpaca-Stil bedeutet das normalerweise, alles bis `### Response:` zu maskieren.
- **Moderne SFT Best Practice:** Wenn der Datensatz konversationell ist, den Loss nur auf den **assistant completion tokens** berechnen und nicht auf den user/system prompt. So wird vermieden, dass das Modell lernt, den Prompt-Text selbst vorherzusagen.
- **Throughput-Optimierung:** Wenn der Datensatz viele kurze Samples enthält, ist es üblich, **mehrere Beispiele in dieselbe Sequence zu packen**, um weniger Context Window zu verschwenden.

Mit diesem Setup ist es an der Zeit, die Data Loaders für jeden Datensatz zu erstellen (Training, Validierung und Test).

## Pre-trained LLM laden & Fine tune & Loss-Checking

Es ist nötig, ein pre-trained LLM zu laden, um es fine-tunen zu können. Das wurde bereits auf anderen Seiten besprochen. Dann ist es möglich, die zuvor verwendete Trainingsfunktion zu nutzen, um das LLM zu fine-tunen.

Während des Trainings ist es auch möglich zu sehen, wie sich Training Loss und Validation Loss über die Epochen verändern, um zu erkennen, ob der Loss sinkt und ob overfitting auftritt.\
Denk daran, dass overfitting auftritt, wenn der Training Loss sinkt, der Validation Loss aber nicht sinkt oder sogar steigt. Um das zu vermeiden, ist das Einfachste, das Training in der Epoche zu stoppen, in der dieses Verhalten beginnt.

Wenn GPU-Speicher eine Einschränkung ist, siehe [7.0. LoRA Improvements in fine-tuning](7.0.-lora-improvements-in-fine-tuning.md), denn in der Praxis werden die meisten instruction-tuning jobs mit **LoRA/QLoRA/PEFT adapters** statt mit einer Aktualisierung jedes Parameters des Basismodells durchgeführt.

## Moderne post-training Varianten

Der klassische Ansatz ist **supervised fine-tuning (SFT)** über `(instruction, answer)`-Paare. Aktuelle instruct-model pipelines teilen das Post-Training jedoch oft in mehrere Phasen auf:

1. **SFT**, um das Ausgabeformat, den Antwortstil und die Aufgabenverteilung zu lehren.
2. **Preference optimization**, um das Modell dazu zu bringen, eine Antwort gegenüber einer anderen zu bevorzugen, ohne ein separates reward model zu benötigen.
3. **Optional** Online-Optimierung wie RLHF / PPO für bestimmte Produkte oder Sicherheitsziele.

Die häufigsten präferenzbasierten Alternativen nach SFT sind:

- **DPO (Direct Preference Optimization):** optimiert direkt bevorzugte vs. abgelehnte Antworten.
- **ORPO / KTO / SimPO:** einfachere Varianten, die einen Teil des Engineering-Overheads vollständiger RLHF-Pipelines reduzieren.

In der Praxis nutzen diese Methoden meist entweder **chosen/rejected response pairs** für denselben Prompt oder ein einfacheres Signal **desirable / undesirable**, sodass die Qualität des Preference-Datensatzes genauso wichtig wird wie das SFT-Korpus selbst.

Das ist nützlich, weil bei vielen Projekten der teure Teil nicht darin besteht, dem Modell das Format beizubringen, sondern es darauf zu trainieren, den Stil von Antworten zu **bevorzugen, den Menschen wollen**. Wenn du mehr Details über die RL-Seite und reward-model poisoning möchtest, siehe [../AI-Reinforcement-Learning-Algorithms.md](../AI-Reinforcement-Learning-Algorithms.md).

## Response Quality

Da dies kein classification fine-tune ist, bei dem es möglich ist, den Loss-Variationen mehr zu vertrauen, ist es auch wichtig, die Qualität der Antworten im Testset zu überprüfen. Daher wird empfohlen, die generierten Antworten aus allen Testsets zu sammeln und ihre Qualität **manuell zu prüfen**, um zu sehen, ob es falsche Antworten gibt (beachte, dass das LLM das Format und die Syntax des Antwortsatzes korrekt erzeugen kann, aber trotzdem eine völlig falsche Antwort geben kann. Die Loss-Variation spiegelt dieses Verhalten nicht wider).\
Beachte, dass es auch möglich ist, diese Prüfung durchzuführen, indem man die generierten Antworten und die erwarteten Antworten an **andere LLMs** übergibt und sie bittet, die Antworten zu bewerten.

Moderne Evaluierungen für instruction-following, die sich zu priorisieren lohnen, sind:

- **IFEval:** prüft, ob das Modell tatsächlich explizite Constraints aus dem Prompt befolgt, statt nur gut zu klingen.
- **AlpacaEval 2 / pairwise judges:** nützlich, um Generierungen mit einem Basismodell zu vergleichen.
- **Arena-style head-to-head evals:** nützlich, wenn du eine relative Rangfolge mehrerer Checkpoints und Prompts brauchst.

Weitere Tests zur Überprüfung der Qualität der Antworten:

1. **Measuring Massive Multitask Language Understanding (**[**MMLU**](https://arxiv.org/abs/2009.03300)**):** MMLU bewertet das Wissen und die Problemlösungsfähigkeiten eines Modells in 57 Fachgebieten, darunter Geisteswissenschaften, Naturwissenschaften und mehr. Es verwendet Multiple-Choice-Fragen, um das Verständnis auf verschiedenen Schwierigkeitsstufen zu beurteilen, von einfach bis fortgeschritten und professionell.
2. [**LMSYS Chatbot Arena**](https://arena.lmsys.org): Diese Plattform ermöglicht es Nutzern, Antworten verschiedener Chatbots Seite an Seite zu vergleichen. Nutzer geben einen Prompt ein, und mehrere Chatbots erzeugen Antworten, die direkt verglichen werden können.
3. [**AlpacaEval**](https://github.com/tatsu-lab/alpaca_eval)**:** AlpacaEval ist ein automatisiertes Evaluierungsframework, bei dem ein fortgeschrittenes LLM wie GPT-4 die Antworten anderer Modelle auf verschiedene Prompts bewertet.
4. **General Language Understanding Evaluation (**[**GLUE**](https://gluebenchmark.com/)**):** GLUE ist eine Sammlung von neun Natural-Language-Understanding-Aufgaben, darunter Sentiment Analysis, textual entailment und question answering.
5. [**SuperGLUE**](https://super.gluebenchmark.com/)**:** Aufbauend auf GLUE enthält SuperGLUE anspruchsvollere Aufgaben, die für aktuelle Modelle schwer zu lösen sind.
6. **Beyond the Imitation Game Benchmark (**[**BIG-bench**](https://github.com/google/BIG-bench)**):** BIG-bench ist ein groß angelegter Benchmark mit über 200 Aufgaben, die die Fähigkeiten eines Modells in Bereichen wie reasoning, translation und question answering testen.
7. **Holistic Evaluation of Language Models (**[**HELM**](https://crfm.stanford.edu/helm/lite/latest/)**):** HELM bietet eine umfassende Evaluierung über verschiedene Metriken wie accuracy, robustness und fairness.
8. [**OpenAI Evals**](https://github.com/openai/evals)**:** Ein Open-Source-Evaluierungsframework von OpenAI, das Tests von KI-Modellen auf benutzerdefinierten und standardisierten Aufgaben ermöglicht.
9. [**HumanEval**](https://github.com/openai/human-eval)**:** Eine Sammlung von Programmierproblemen zur Bewertung der Code-Generierungsfähigkeiten von Sprachmodellen.
10. **Stanford Question Answering Dataset (**[**SQuAD**](https://rajpurkar.github.io/SQuAD-explorer/)**):** SQuAD besteht aus Fragen zu Wikipedia-Artikeln, wobei Modelle den Text verstehen müssen, um präzise zu antworten.
11. [**TriviaQA**](https://nlp.cs.washington.edu/triviaqa/)**:** Ein groß angelegter Datensatz mit Trivia-Fragen und -Antworten sowie Evidenzdokumenten.

und viele viele mehr.

## Security notes for instruction tuning

Instruction-tuning-Datensätze sind ebenfalls eine Angriffsfläche. Wenn ein Angreifer eine kleine Anzahl vergifteter Samples in das SFT-Korpus einschleusen kann, kann das Modell ein **trigger-conditioned behaviour** lernen und dabei auf sauberen Prompts weiterhin normal aussehen.

Typische bösartige Muster, die beim red teaming getestet werden sollten, sind:

- **Instruction backdoors:** Ein seltener Trigger-String, ein role prefix, ein unicode pattern oder eine Formatierungssequenz führt dazu, dass das Modell vom Angreifer gewählte Inhalte ausgibt.
- **Sleeper-agent behaviour:** Das Modell verhält sich während normaler Evaluierungen korrekt, ändert sein Verhalten aber nur, wenn ein sehr spezifischer Trigger erscheint.
- **Narrow-domain misalignment:** Ein scheinbar harmloses Domain fine-tune kann das Modell in eine schlechte Verhaltensregion außerhalb der beabsichtigten Aufgabenverteilung verschieben.

Praktische Prüfungen:

- Ein **canary set** synthetischer Trigger-Prompts beibehalten und auf jedem Checkpoint ausführen.
- Das SFT-Korpus vor dem Training deduplizieren und die Herkunft prüfen.
- Niemals auf den exakten Prompts trainieren, die vom Benchmark stammen, über den du später berichtest.
- Das Basismodell mit dem fine-tuned Modell auf sicherheitskritischen Prompts vergleichen, um unerwartete Verhaltensänderungen zu erkennen.

## Follow instructions fine-tuning code

Du findest ein Beispiel für den Code für dieses Fine-tuning in [https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py](https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py)

Eine praktische moderne Alternative ist die Verwendung eines Trainers, der bereits chat templates, assistant-only loss, packing und PEFT adapters unterstützt. Wenn du später preference alignment möchtest, bietet dasselbe Ökosystem auch `DPOTrainer`-ähnliche Workflows für Datensätze mit `prompt` / `chosen` / `rejected`-Feldern:
```python
from trl import SFTConfig, SFTTrainer

trainer = SFTTrainer(
model="Qwen/Qwen2.5-0.5B-Instruct",
train_dataset=dataset,
args=SFTConfig(
learning_rate=2e-5,
packing=True,
assistant_only_loss=True,
),
)
trainer.train()
```
