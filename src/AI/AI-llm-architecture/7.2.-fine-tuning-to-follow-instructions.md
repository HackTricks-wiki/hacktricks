# 7.2. Fine-Tuning para seguir instruções

{{#include ../../banners/hacktricks-training.md}}

> [!TIP]
> O objetivo desta seção é mostrar como **fine-tune um modelo já pré-treinado para seguir instruções** em vez de apenas gerar texto, por exemplo, respondendo a tarefas como um chat bot.

## Dataset

Para fazer fine-tune de um LLM para seguir instruções, é necessário ter um dataset com **instruções e respostas**. Existem diferentes formatos de prompt para treinar um LLM a seguir instruções, por exemplo:

- O exemplo do estilo de prompt Alpaca:
```csharp
Below is an instruction that describes a task. Write a response that appropriately completes the request.

### Instruction:
Calculate the area of a circle with a radius of 5 units.

### Response:
The area of a circle is calculated using the formula \( A = \pi r^2 \). Plugging in the radius of 5 units:

\( A = \pi (5)^2 = \pi \times 25 = 25\pi \) square units.
```
- Exemplo de estilo de chat template do Phi-3:
```vbnet
<|User|>
Can you explain what gravity is in simple terms?

<|Assistant|>
Absolutely! Gravity is a force that pulls objects toward each other.
```
Treinar com esse tipo de dataset em vez de apenas texto bruto ajuda o modelo a entender que ele precisa responder tarefas em um **structured assistant format**.

Um erro muito comum é treinar em um template e inferir com outro. Modern instruct models normalmente esperam o **native chat template do base model** (`<|user|>`, `<|assistant|>`, `[INST]`, ChatML, etc.), então é recomendado converter o dataset para o formato exato que o target model já usa.

Portanto, uma das primeiras coisas a fazer com um dataset que contém requests e answers é modelar esses dados no formato de prompt desejado, como:
```python
# Code from https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/ch07.ipynb
def format_input(entry):
instruction_text = (
f"Below is an instruction that describes a task. "
f"Write a response that appropriately completes the request."
f"\n\n### Instruction:\n{entry['instruction']}"
)

input_text = f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""

return instruction_text + input_text

model_input = format_input(data[50])

desired_response = f"\n\n### Response:\n{data[50]['output']}"

print(model_input + desired_response)
```
Se o base model já vem com um tokenizer chat template, pode ser mais seguro construir o dataset a partir de listas de mensagens em vez de criar separadores manualmente:
```python
messages = [
{"role": "user", "content": entry["instruction"]},
{"role": "assistant", "content": entry["output"]},
]
text = tokenizer.apply_chat_template(messages, tokenize=False)
```
Então, como sempre, é necessário separar o dataset em conjuntos de treinamento, validação e teste.

## Batching & Data Loaders

Então, é necessário fazer batch de todas as entradas e saídas esperadas para o treinamento. Para isso, é preciso:

- Tokenize the texts.
- Pad todas as amostras para o mesmo comprimento (normalmente o comprimento será tão grande quanto o comprimento de contexto usado para pre-treinar o LLM).
- Criar os tokens esperados deslocando 1 a entrada em uma custom collate function.
- Substituir alguns padding tokens por `-100` para excluí-los da training loss: após o primeiro token `endoftext`, substituir todos os outros tokens `endoftext` por `-100` (porque usar `cross_entropy(..., ignore_index=-100)` significa que ele ignorará targets com `-100`).
- \[Optional] Mascarar usando `-100` também todos os tokens pertencentes à pergunta para que o LLM aprenda apenas a gerar a resposta. No estilo Alpaca isso normalmente significa mascarar tudo até `### Response:`.
- **Modern SFT best practice:** quando o dataset é conversacional, compute the loss apenas nos **assistant completion tokens** e não no prompt do user/system. Isso evita ensinar o modelo a prever o próprio texto do prompt.
- **Throughput optimization:** quando o dataset contém muitas amostras curtas, é comum **pack several examples into the same sequence** para desperdiçar menos context window.

Com isso criado, é hora de criar os data loaders para cada dataset (treinamento, validação e teste).

## Load pre-trained LLM & Fine tune & Loss Checking

É necessário carregar um pre-trained LLM para fine-tune it. Isso já foi discutido em outras páginas. Então, é possível usar a função de treinamento usada anteriormente para fine-tune o LLM.

Durante o treinamento também é possível ver como a training loss e a validation loss variam ao longo das epochs para verificar se a loss está sendo reduzida e se overfitting está ocorrendo.\
Lembre-se de que overfitting ocorre quando a training loss está diminuindo, mas a validation loss não está diminuindo ou até aumentando. Para evitar isso, a coisa mais simples a fazer é parar o treinamento na epoch em que esse comportamento começa.

Se a memória da GPU for uma limitação, consulte [7.0. LoRA Improvements in fine-tuning](7.0.-lora-improvements-in-fine-tuning.md), porque na prática a maioria dos trabalhos de instruction-tuning é feita com **LoRA/QLoRA/PEFT adapters** em vez de atualizar cada parâmetro do modelo base.

## Modern post-training variants

A abordagem clássica é **supervised fine-tuning (SFT)** sobre pares `(instruction, answer)`. Porém, pipelines atuais de instruct-model frequentemente dividem o post-training em várias fases:

1. **SFT** para ensinar o formato de saída, o estilo da resposta e a distribuição das tarefas.
2. **Preference optimization** para fazer o modelo preferir uma resposta em vez de outra sem precisar de um reward model separado.
3. **Optional** otimização online como RLHF / PPO para produtos específicos ou objetivos de segurança.

As alternativas mais comuns baseadas em preferência após o SFT são:

- **DPO (Direct Preference Optimization):** otimiza diretamente respostas preferidas vs rejeitadas.
- **ORPO / KTO / SimPO:** variantes mais simples que reduzem parte da sobrecarga de engenharia dos pipelines completos de RLHF.

Na prática, esses métodos geralmente consomem ou pares de respostas **chosen/rejected** para o mesmo prompt ou um sinal mais simples de **desirable / undesirable**, então a qualidade do preference dataset se torna tão importante quanto o próprio corpus do SFT.

Isso é útil porque, para muitos projetos, a parte cara não é ensinar o modelo o formato, mas ensiná-lo a **preferir o estilo de resposta que humanos querem**. Se você quiser mais detalhes sobre a parte de RL e reward-model poisoning, consulte [../AI-Reinforcement-Learning-Algorithms.md](../AI-Reinforcement-Learning-Algorithms.md).

## Response Quality

Como este não é um classification fine-tune em que é possível confiar mais nas variações da loss, também é importante verificar a qualidade das respostas no testing set. Portanto, é recomendado reunir as respostas geradas de todos os testing sets e **verificar manualmente sua qualidade** para ver se há respostas erradas (note que é possível que o LLM gere corretamente o formato e a sintaxe da frase de resposta, mas dê uma resposta completamente errada. A variação da loss não refletirá esse comportamento).\
Note que também é possível realizar essa revisão passando as respostas geradas e as respostas esperadas para **outros LLMs e pedindo que eles avaliem as respostas**.

As avaliações modernas de instruction-following que vale priorizar são:

- **IFEval:** verifica se o modelo realmente segue restrições explícitas do prompt em vez de apenas soar bem.
- **AlpacaEval 2 / pairwise judges:** útil para comparar gerações com um baseline model.
- **Arena-style head-to-head evals:** útil quando você precisa de ranking relativo de vários checkpoints e prompts.

Outros testes para executar e verificar a qualidade das respostas:

1. **Measuring Massive Multitask Language Understanding (**[**MMLU**](https://arxiv.org/abs/2009.03300)**):** MMLU avalia o conhecimento e as habilidades de resolução de problemas de um modelo em 57 assuntos, incluindo humanidades, ciências e muito mais. Ele usa questões de múltipla escolha para avaliar a compreensão em vários níveis de dificuldade, do básico ao profissional avançado.
2. [**LMSYS Chatbot Arena**](https://arena.lmsys.org): Esta plataforma permite que usuários comparem respostas de diferentes chatbots lado a lado. Os usuários inserem um prompt, e vários chatbots geram respostas que podem ser comparadas diretamente.
3. [**AlpacaEval**](https://github.com/tatsu-lab/alpaca_eval)**:** AlpacaEval é um framework automatizado de avaliação em que um LLM avançado como o GPT-4 avalia as respostas de outros modelos a vários prompts.
4. **General Language Understanding Evaluation (**[**GLUE**](https://gluebenchmark.com/)**):** GLUE é uma coleção de nove tarefas de compreensão de linguagem natural, incluindo análise de sentimento, entailment textual e question answering.
5. [**SuperGLUE**](https://super.gluebenchmark.com/)**:** Com base no GLUE, SuperGLUE inclui tarefas mais desafiadoras, projetadas para serem difíceis para os modelos atuais.
6. **Beyond the Imitation Game Benchmark (**[**BIG-bench**](https://github.com/google/BIG-bench)**):** BIG-bench é um benchmark em larga escala com mais de 200 tarefas que testam as habilidades de um modelo em áreas como raciocínio, tradução e question answering.
7. **Holistic Evaluation of Language Models (**[**HELM**](https://crfm.stanford.edu/helm/lite/latest/)**):** HELM fornece uma avaliação abrangente em várias métricas como accuracy, robustness e fairness.
8. [**OpenAI Evals**](https://github.com/openai/evals)**:** Um framework de avaliação open-source da OpenAI que permite testar modelos de IA em tarefas customizadas e padronizadas.
9. [**HumanEval**](https://github.com/openai/human-eval)**:** Uma coleção de problemas de programação usada para avaliar habilidades de code generation de language models.
10. **Stanford Question Answering Dataset (**[**SQuAD**](https://rajpurkar.github.io/SQuAD-explorer/)**):** SQuAD consiste em perguntas sobre artigos da Wikipedia, onde os modelos devem compreender o texto para responder com precisão.
11. [**TriviaQA**](https://nlp.cs.washington.edu/triviaqa/)**:** Um dataset em larga escala de perguntas e respostas de trivia, junto com documentos de evidência.

e muitos, muitos mais.

## Security notes for instruction tuning

Instruction-tuning datasets também são uma attack surface. Se um attacker conseguir inserir um pequeno número de amostras poisoned no corpus de SFT, o modelo pode aprender um **trigger-conditioned behaviour** enquanto ainda parece normal em prompts limpos.

Padrões maliciosos típicos para testar durante red teaming são:

- **Instruction backdoors:** uma rara trigger string, role prefix, unicode pattern ou formatting sequence faz o modelo gerar conteúdo escolhido pelo attacker.
- **Sleeper-agent behaviour:** o modelo se comporta corretamente durante avaliações normais, mas muda de comportamento apenas quando um trigger muito específico aparece.
- **Narrow-domain misalignment:** um fine-tune de domínio aparentemente inofensivo pode levar o modelo para uma região ruim de comportamento fora da distribuição de tarefas pretendida.

Verificações práticas:

- Mantenha um **canary set** de prompts sintéticos com trigger e execute-o em cada checkpoint.
- Deduplicate e faça provenance-check do corpus de SFT antes do treinamento.
- Nunca treine nos prompts exatos usados pelo benchmark que você vai reportar depois.
- Compare o base model vs o fine-tuned model em prompts sensíveis à segurança para detectar drift inesperado de comportamento.

## Follow instructions fine-tuning code

Você pode encontrar um exemplo do código para realizar esse fine-tuning em [https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py](https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py)

Uma alternativa moderna prática é usar um trainer que já suporte chat templates, assistant-only loss, packing e PEFT adapters. Se mais tarde você quiser preference alignment, o mesmo ecossistema também fornece workflows no estilo `DPOTrainer` para datasets com campos `prompt` / `chosen` / `rejected`:
```python
from trl import SFTConfig, SFTTrainer

trainer = SFTTrainer(
model="Qwen/Qwen2.5-0.5B-Instruct",
train_dataset=dataset,
args=SFTConfig(
learning_rate=2e-5,
packing=True,
assistant_only_loss=True,
),
)
trainer.train()
```
