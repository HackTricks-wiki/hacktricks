# 7.2. Fine-Tuning pour suivre des instructions

{{#include ../../banners/hacktricks-training.md}}

> [!TIP]
> L’objectif de cette section est de montrer comment **fine-tune un modèle déjà pré-entraîné pour suivre des instructions** plutôt que de simplement générer du texte, par exemple en répondant à des tâches comme un chat bot.

## Dataset

Afin de fine-tune un LLM pour suivre des instructions, il est nécessaire d’avoir un dataset avec des **instructions et des réponses**. Il existe différents formats de prompt pour entraîner un LLM à suivre des instructions, par exemple :

- The Alpaca prompt style example:
```csharp
Below is an instruction that describes a task. Write a response that appropriately completes the request.

### Instruction:
Calculate the area of a circle with a radius of 5 units.

### Response:
The area of a circle is calculated using the formula \( A = \pi r^2 \). Plugging in the radius of 5 units:

\( A = \pi (5)^2 = \pi \times 25 = 25\pi \) square units.
```
- Exemple de style Phi-3 / chat template:
```vbnet
<|User|>
Can you explain what gravity is in simple terms?

<|Assistant|>
Absolutely! Gravity is a force that pulls objects toward each other.
```
L'entraînement avec ce genre de datasets plutôt qu'avec du simple texte brut aide le modèle à comprendre qu'il doit répondre aux tâches dans un **structured assistant format**.

Une erreur très courante consiste à entraîner sur un template et à faire l'inférence avec un autre. Les modern instruct models s'attendent généralement au **native chat template du base model** (`<|user|>`, `<|assistant|>`, `[INST]`, ChatML, etc.), donc il est recommandé de convertir le dataset dans le format exact déjà utilisé par le target model.

Par conséquent, l'une des premières choses à faire avec un dataset qui contient des requests et des answers est de modéliser ces données dans le format de prompt souhaité, comme :
```python
# Code from https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/ch07.ipynb
def format_input(entry):
instruction_text = (
f"Below is an instruction that describes a task. "
f"Write a response that appropriately completes the request."
f"\n\n### Instruction:\n{entry['instruction']}"
)

input_text = f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""

return instruction_text + input_text

model_input = format_input(data[50])

desired_response = f"\n\n### Response:\n{data[50]['output']}"

print(model_input + desired_response)
```
Si le base model est déjà livré avec un tokenizer chat template, il peut être plus sûr de construire le dataset à partir de listes de messages plutôt que de fabriquer les séparateurs à la main :
```python
messages = [
{"role": "user", "content": entry["instruction"]},
{"role": "assistant", "content": entry["output"]},
]
text = tokenizer.apply_chat_template(messages, tokenize=False)
```
Alors, comme toujours, il faut séparer le dataset en ensembles d'entraînement, de validation et de test.

## Batching & Data Loaders

Ensuite, il faut batcher toutes les entrées et les sorties attendues pour l'entraînement. Pour cela, il faut :

- Tokenize les textes.
- Pad toutes les samples à la même longueur (généralement, la longueur sera aussi grande que la context length utilisée pour pre-train le LLM).
- Créer les tokens attendus en décalant l'entrée de 1 dans une custom collate function.
- Remplacer certains padding tokens par `-100` pour les exclure de la training loss : après le premier token `endoftext`, substituer tous les autres tokens `endoftext` par `-100` (car utiliser `cross_entropy(..., ignore_index=-100)` signifie que cela ignorera les targets avec `-100`).
- \[Optional] Mask using `-100` aussi tous les tokens appartenant à la question afin que le LLM apprenne seulement à générer la réponse. Dans le style Alpaca, cela signifie généralement masquer tout jusqu'à `### Response:`.
- **Modern SFT best practice:** si le dataset est conversational, compute the loss only on the **assistant completion tokens** et non sur le user/system prompt. Cela évite d'entraîner le modèle à prédire le texte du prompt lui-même.
- **Throughput optimization:** lorsque le dataset contient de nombreux échantillons courts, il est courant de **pack several examples into the same sequence** pour gaspiller moins de context window.

Une fois cela créé, il est temps de créer les data loaders pour chaque dataset (training, validation et test).

## Load pre-trained LLM & Fine tune & Loss Checking

Il faut charger un LLM pre-trained pour le fine-tune. Cela a déjà été abordé dans d'autres pages. Ensuite, il est possible d'utiliser la fonction d'entraînement utilisée précédemment pour fine-tune le LLM.

Pendant l'entraînement, il est aussi possible de voir comment la training loss et la validation loss varient au fil des epochs afin de voir si la loss diminue et si un overfitting se produit.\
Rappelez-vous que l'overfitting se produit lorsque la training loss diminue mais que la validation loss ne diminue pas, ou augmente même. Pour éviter cela, la chose la plus simple à faire est d'arrêter l'entraînement à l'epoch où ce comportement commence.

Si la mémoire GPU est une limitation, consultez [7.0. LoRA Improvements in fine-tuning](7.0.-lora-improvements-in-fine-tuning.md), car en pratique la plupart des instruction-tuning jobs sont faits avec des **LoRA/QLoRA/PEFT adapters** plutôt qu'en mettant à jour chaque paramètre du base model.

## Modern post-training variants

L'approche classique est le **supervised fine-tuning (SFT)** sur des paires `(instruction, answer)`. Cependant, les pipelines instruct-model actuels divisent souvent le post-training en plusieurs phases :

1. **SFT** pour enseigner le format de sortie, le style de réponse et la distribution des tâches.
2. **Preference optimization** pour amener le modèle à préférer une réponse à une autre sans avoir besoin d'un separate reward model.
3. **Optional** online optimization comme RLHF / PPO pour des produits ou des objectifs de sécurité spécifiques.

Les alternatives les plus courantes basées sur les préférences après SFT sont :

- **DPO (Direct Preference Optimization):** optimise directement les réponses préférées vs rejetées.
- **ORPO / KTO / SimPO:** variantes plus simples qui réduisent une partie de la charge d'ingénierie des pipelines RLHF complets.

En pratique, ces méthodes consomment généralement soit des paires de réponses **chosen/rejected** pour le même prompt, soit un signal plus simple **desirable / undesirable** ; la qualité du preference dataset devient donc aussi importante que le corpus SFT lui-même.

C'est utile car, pour de nombreux projets, la partie coûteuse n'est pas d'apprendre au modèle le format, mais de lui apprendre à **préférer le style de réponse que les humains veulent**. Si vous voulez plus de détails sur le côté RL et le reward-model poisoning, consultez [../AI-Reinforcement-Learning-Algorithms.md](../AI-Reinforcement-Learning-Algorithms.md).

## Response Quality

Comme il ne s'agit pas d'un fine-tune de classification où il est possible de faire davantage confiance aux variations de loss, il est aussi important de vérifier la qualité des réponses dans le testing set. Par conséquent, il est recommandé de récupérer les réponses générées pour tous les testing sets et de **vérifier manuellement leur qualité** pour voir s'il y a des mauvaises réponses (notez qu'il est possible que le LLM génère correctement le format et la syntaxe de la phrase de réponse, mais donne une réponse complètement fausse. La variation de loss ne reflétera pas ce comportement).\
Notez qu'il est aussi possible d'effectuer cette revue en passant les réponses générées et les réponses attendues à **d'autres LLMs et en leur demandant d'évaluer les réponses**.

Les évaluations modernes de follow-the-instructions qu'il vaut la peine de prioriser sont :

- **IFEval:** vérifie si le modèle suit réellement les contraintes explicites du prompt au lieu de simplement sembler bon.
- **AlpacaEval 2 / pairwise judges:** utile pour comparer des generations à un baseline model.
- **Arena-style head-to-head evals:** utile lorsque vous avez besoin d'un classement relatif de plusieurs checkpoints et prompts.

Autres tests à exécuter pour vérifier la qualité des réponses :

1. **Measuring Massive Multitask Language Understanding (**[**MMLU**](https://arxiv.org/abs/2009.03300)**):** MMLU évalue les connaissances et les capacités de résolution de problèmes d'un modèle sur 57 sujets, incluant les humanités, les sciences, et plus encore. Il utilise des questions à choix multiple pour évaluer la compréhension à différents niveaux de difficulté, du niveau élémentaire au niveau professionnel avancé.
2. [**LMSYS Chatbot Arena**](https://arena.lmsys.org): Cette plateforme permet aux utilisateurs de comparer côte à côte les réponses de différents chatbots. Les utilisateurs saisissent un prompt, et plusieurs chatbots génèrent des réponses qui peuvent être comparées directement.
3. [**AlpacaEval**](https://github.com/tatsu-lab/alpaca_eval)**:** AlpacaEval est un framework d'évaluation automatisé où un LLM avancé comme GPT-4 évalue les réponses d'autres modèles à divers prompts.
4. **General Language Understanding Evaluation (**[**GLUE**](https://gluebenchmark.com/)**):** GLUE est une collection de neuf tâches de compréhension du langage naturel, incluant l'analyse de sentiment, l'inférence textuelle et le question answering.
5. [**SuperGLUE**](https://super.gluebenchmark.com/)**:** En s'appuyant sur GLUE, SuperGLUE inclut des tâches plus difficiles conçues pour être ardues pour les modèles actuels.
6. **Beyond the Imitation Game Benchmark (**[**BIG-bench**](https://github.com/google/BIG-bench)**):** BIG-bench est un benchmark à grande échelle avec plus de 200 tâches qui testent les capacités d'un modèle dans des domaines comme le raisonnement, la traduction et le question answering.
7. **Holistic Evaluation of Language Models (**[**HELM**](https://crfm.stanford.edu/helm/lite/latest/)**):** HELM fournit une évaluation complète selon diverses métriques comme l'accuracy, la robustesse et l'équité.
8. [**OpenAI Evals**](https://github.com/openai/evals)**:** Un framework d'évaluation open-source par OpenAI qui permet de tester des modèles IA sur des tâches personnalisées et standardisées.
9. [**HumanEval**](https://github.com/openai/human-eval)**:** Une collection de problèmes de programmation utilisée pour évaluer les capacités de génération de code des language models.
10. **Stanford Question Answering Dataset (**[**SQuAD**](https://rajpurkar.github.io/SQuAD-explorer/)**):** SQuAD consiste en des questions sur des articles Wikipedia, où les modèles doivent comprendre le texte pour répondre avec précision.
11. [**TriviaQA**](https://nlp.cs.washington.edu/triviaqa/)**:** Un dataset à grande échelle de questions et réponses de trivia, avec des documents de preuve.

et bien, bien plus encore.

## Security notes for instruction tuning

Les instruction-tuning datasets sont aussi une surface d'attaque. Si un attaquant peut insérer un petit nombre d'échantillons empoisonnés dans le corpus SFT, le modèle peut apprendre un **trigger-conditioned behaviour** tout en restant normal sur des prompts propres.

Les schémas malveillants typiques à tester pendant le red teaming sont :

- **Instruction backdoors:** une chaîne de trigger rare, un role prefix, un motif unicode ou une séquence de formatage provoque la sortie d'un contenu choisi par l'attaquant.
- **Sleeper-agent behaviour:** le modèle se comporte correctement pendant les évaluations normales mais change de comportement uniquement lorsqu'un trigger très spécifique apparaît.
- **Narrow-domain misalignment:** un fine-tune de domaine apparemment inoffensif peut déplacer le modèle vers une mauvaise région de comportement en dehors de la distribution de tâches prévue.

Vérifications pratiques :

- Conserver un **canary set** de prompts de trigger synthétiques et l'exécuter sur chaque checkpoint.
- Dédupliquer et vérifier la provenance du corpus SFT avant l'entraînement.
- Ne jamais entraîner sur les prompts exacts utilisés par le benchmark que vous reporterez plus tard.
- Comparer le base model vs le fine-tuned model sur des prompts sensibles pour la sécurité afin de détecter une dérive de comportement inattendue.

## Follow instructions fine-tuning code

Vous pouvez trouver un exemple de code pour réaliser ce fine-tuning dans [https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py](https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py)

Une alternative moderne pratique consiste à utiliser un trainer qui prend déjà en charge les chat templates, assistant-only loss, packing et PEFT adapters. Si vous souhaitez plus tard faire du preference alignment, le même écosystème fournit aussi des workflows de type `DPOTrainer` pour des datasets avec des champs `prompt` / `chosen` / `rejected` :
```python
from trl import SFTConfig, SFTTrainer

trainer = SFTTrainer(
model="Qwen/Qwen2.5-0.5B-Instruct",
train_dataset=dataset,
args=SFTConfig(
learning_rate=2e-5,
packing=True,
assistant_only_loss=True,
),
)
trainer.train()
```
