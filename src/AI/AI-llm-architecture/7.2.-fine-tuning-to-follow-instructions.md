# 7.2. Fine-Tuning to follow instructions

{{#include ../../banners/hacktricks-training.md}}

> [!TIP]
> Die doel van hierdie afdeling is om te wys hoe om 'n **reeds vooraf-opgeleide model fyn te stel om instruksies te volg** eerder as om net teks te genereer, byvoorbeeld, om op take as 'n chat bot te reageer.

## Dataset

Om 'n LLM fyn te stel om instruksies te volg, is dit nodig om 'n dataset met **instruksies en responses** te hê. Daar is verskillende prompt-formate om 'n LLM te train om instruksies te volg, byvoorbeeld:

- The Alpaca prompt style example:
```csharp
Below is an instruction that describes a task. Write a response that appropriately completes the request.

### Instruction:
Calculate the area of a circle with a radius of 5 units.

### Response:
The area of a circle is calculated using the formula \( A = \pi r^2 \). Plugging in the radius of 5 units:

\( A = \pi (5)^2 = \pi \times 25 = 25\pi \) square units.
```
- Phi-3 / chat template styl voorbeeld:
```vbnet
<|User|>
Can you explain what gravity is in simple terms?

<|Assistant|>
Absolutely! Gravity is a force that pulls objects toward each other.
```
Opleiding met hierdie soort datasets in plaas van net rou teks help die model om te verstaan dat dit take moet beantwoord in ’n **gestruktureerde assistant-formaat**.

’n Baie algemene fout is om op een sjabloon te train en met ’n ander een te infer. Moderne instruct models verwag gewoonlik die **native chat template van die base model** (`<|user|>`, `<|assistant|>`, `[INST]`, ChatML, ens.), so dit word aanbeveel om die dataset om te skakel na die presiese formaat wat die teikenmodel reeds gebruik.

Daarom is een van die eerste dinge om te doen met ’n dataset wat requests en answers bevat, om daardie data in die gewenste prompt-formaat te modelleer, soos:
```python
# Code from https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/ch07.ipynb
def format_input(entry):
instruction_text = (
f"Below is an instruction that describes a task. "
f"Write a response that appropriately completes the request."
f"\n\n### Instruction:\n{entry['instruction']}"
)

input_text = f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""

return instruction_text + input_text

model_input = format_input(data[50])

desired_response = f"\n\n### Response:\n{data[50]['output']}"

print(model_input + desired_response)
```
As die basis model reeds met 'n tokenizer chat template kom, kan dit veiliger wees om die dataset uit boodskaplyste te bou in plaas daarvan om skeiers met die hand te skep:
```python
messages = [
{"role": "user", "content": entry["instruction"]},
{"role": "assistant", "content": entry["output"]},
]
text = tokenizer.apply_chat_template(messages, tokenize=False)
```
Then, soos altyd, is dit nodig om die dataset in opleiding-, validasie- en toetsstelle te verdeel.

## Batching & Data Loaders

Dan is dit nodig om al die insette en verwagte uitsette vir die opleiding te batch. Hiervoor is dit nodig om:

- Tokenize the teks.
- Pad al die monsters na dieselfde lengte (gewoonlik sal die lengte so groot wees as die context length wat gebruik is om die LLM vooraf te train).
- Skep die verwagte tokens deur die invoer met 1 te skuif in 'n custom collate function.
- Vervang sommige padding tokens met `-100` om hulle van die training loss uit te sluit: ná die eerste `endoftext` token, vervang al die ander `endoftext` tokens met `-100` (want die gebruik van `cross_entropy(..., ignore_index=-100)` beteken dat dit targets met `-100` sal ignoreer).
- \[Optional] Mask using `-100` ook al die tokens wat aan die vraag behoort sodat die LLM net leer hoe om die antwoord te genereer. In die Alpaca-styl beteken dit gewoonlik om alles te mask totdat `### Response:`.
- **Modern SFT best practice:** as die dataset conversational is, bereken die loss net op die **assistant completion tokens** en nie op die user/system prompt nie. Dit voorkom dat die model geleer word om die prompt-teks self te voorspel.
- **Throughput optimization:** wanneer die dataset baie kort samples bevat, is dit algemeen om **verskeie examples in dieselfde sequence te pack** om minder context window te mors.

Met dit geskep, is dit tyd om die data loaders vir elke dataset te skep (opleiding, validasie, en test).

## Load pre-trained LLM & Fine tune & Loss Checking

Dit is nodig om 'n pre-trained LLM te load om dit te fine-tune. Dit is reeds in ander bladsye bespreek. Dan is dit moontlik om die voorheen gebruikte training function te gebruik om die LLM te fine-tune.

Tydens die opleiding is dit ook moontlik om te sien hoe die training loss en validation loss oor die epochs varieer om te sien of die loss afneem en of overfitting plaasvind.\
Onthou dat overfitting plaasvind wanneer die training loss afneem maar die validation loss nie afneem nie of selfs toeneem. Om dit te voorkom, is die eenvoudigste ding om te doen om die opleiding te stop by die epoch waar hierdie gedrag begin.

As GPU memory 'n beperking is, kyk na [7.0. LoRA Improvements in fine-tuning](7.0.-lora-improvements-in-fine-tuning.md), want in praktyk word die meeste instruction-tuning jobs gedoen met **LoRA/QLoRA/PEFT adapters** in plaas daarvan om elke parameter van die base model op te dateer.

## Modern post-training variants

Die klassieke benadering is **supervised fine-tuning (SFT)** oor `(instruction, answer)` pare. Huidige instruct-model pipelines verdeel post-training egter dikwels in verskeie fases:

1. **SFT** om die output format, answer style, en task distribution te leer.
2. **Preference optimization** om die model een answer bo 'n ander te laat verkies sonder om 'n aparte reward model te benodig.
3. **Optional** online optimization soos RLHF / PPO vir spesifieke products of safety goals.

Die mees algemene preference-based alternatiewe ná SFT is:

- **DPO (Direct Preference Optimization):** optimaliseer direk voorkeur- teenoor verwerpte antwoorde.
- **ORPO / KTO / SimPO:** eenvoudiger variante wat sommige van die engineering overhead van volledige RLHF pipelines verminder.

In praktyk gebruik hierdie metodes gewoonlik óf **chosen/rejected response pairs** vir dieselfde prompt óf 'n eenvoudiger **desirable / undesirable** sein, so die kwaliteit van die preference dataset word net so belangrik soos die SFT corpus self.

Dit is nuttig omdat die duur deel vir baie projekte nie is om die model die format te leer nie, maar om dit te leer om die **style van answer wat mense wil hê** te verkies. As jy meer besonderhede oor die RL-kant en reward-model poisoning wil hê, kyk na [../AI-Reinforcement-Learning-Algorithms.md](../AI-Reinforcement-Learning-Algorithms.md).

## Response Quality

Aangesien dit nie 'n classification fine-tune is waar dit moontlik is om meer op die loss variasies te vertrou nie, is dit ook belangrik om die kwaliteit van die responses in die testing set te check. Daarom word aanbeveel om die gegenereerde responses van al die testing sets te versamel en hul kwaliteit **handmatig te check** om te sien of daar verkeerde antwoorde is (let daarop dat dit moontlik is vir die LLM om die format en syntax van die response sentence korrek te genereer maar 'n heeltemal verkeerde response te gee. Die loss variasie sal nie hierdie gedrag reflekteer nie).\
Let daarop dat dit ook moontlik is om hierdie review uit te voer deur die gegenereerde responses en die verwagte responses aan **ander LLMs** te gee en hulle te vra om die responses te evalueer.

Moderne instruction-following evals wat die moeite werd is om te prioritiseer, is:

- **IFEval:** check of die model werklik eksplisiete constraints uit die prompt volg in plaas daarvan om net goed te klink.
- **AlpacaEval 2 / pairwise judges:** nuttig om generations teen 'n baseline model te vergelyk.
- **Arena-style head-to-head evals:** nuttig wanneer jy relatiewe rangorde van verskeie checkpoints en prompts nodig het.

Ander tests om uit te voer om die kwaliteit van die responses te verifieer:

1. **Measuring Massive Multitask Language Understanding (**[**MMLU**](https://arxiv.org/abs/2009.03300)**):** MMLU evalueer 'n model se kennis en probleemoplossingsvermoë oor 57 subjects, insluitend humanities, sciences, en meer. Dit gebruik multiple-choice questions om begrip op verskeie moeilikheidsvlakke te assesseer, van elementary tot advanced professional.
2. [**LMSYS Chatbot Arena**](https://arena.lmsys.org): Hierdie platform laat users toe om responses van verskillende chatbots langs mekaar te vergelyk. Users voer 'n prompt in, en multiple chatbots genereer responses wat direk vergelyk kan word.
3. [**AlpacaEval**](https://github.com/tatsu-lab/alpaca_eval)**:** AlpacaEval is 'n outomatiese evaluation framework waar 'n advanced LLM soos GPT-4 die responses van ander models op verskeie prompts evalueer.
4. **General Language Understanding Evaluation (**[**GLUE**](https://gluebenchmark.com/)**):** GLUE is 'n versameling van nege natural language understanding tasks, insluitend sentiment analysis, textual entailment, en question answering.
5. [**SuperGLUE**](https://super.gluebenchmark.com/)**:** Gebou op GLUE, sluit SuperGLUE meer uitdagende tasks in wat ontwerp is om moeilik te wees vir huidige models.
6. **Beyond the Imitation Game Benchmark (**[**BIG-bench**](https://github.com/google/BIG-bench)**):** BIG-bench is 'n large-scale benchmark met meer as 200 tasks wat 'n model se vermoëns toets in areas soos reasoning, translation, en question answering.
7. **Holistic Evaluation of Language Models (**[**HELM**](https://crfm.stanford.edu/helm/lite/latest/)**):** HELM bied 'n omvattende evaluation oor verskeie metrics soos accuracy, robustness, en fairness.
8. [**OpenAI Evals**](https://github.com/openai/evals)**:** 'n open-source evaluation framework deur OpenAI wat toelaat om AI models op custom en standardized tasks te toets.
9. [**HumanEval**](https://github.com/openai/human-eval)**:** 'n Versameling programming problems wat gebruik word om code generation vermoëns van language models te evalueer.
10. **Stanford Question Answering Dataset (**[**SQuAD**](https://rajpurkar.github.io/SQuAD-explorer/)**):** SQuAD bestaan uit questions oor Wikipedia articles, waar models die teks moet verstaan om akkuraat te antwoord.
11. [**TriviaQA**](https://nlp.cs.washington.edu/triviaqa/)**:** 'n Large-scale dataset van trivia questions and answers, saam met evidence documents.

en nog baie, baie meer.

## Security notes for instruction tuning

Instruction-tuning datasets is ook 'n attack surface. As 'n attacker 'n klein aantal poisoned samples in die SFT corpus kan invoeg, kan die model 'n **trigger-conditioned behaviour** leer terwyl dit steeds normaal op clean prompts lyk.

Tipiese malicious patterns om tydens red teaming te toets, is:

- **Instruction backdoors:** 'n seldsame trigger string, role prefix, unicode pattern, of formatting sequence veroorsaak dat die model attacker-chosen content uitvoer.
- **Sleeper-agent behaviour:** die model gedra hom korrek tydens normale evals maar verander gedrag net wanneer 'n baie spesifieke trigger verskyn.
- **Narrow-domain misalignment:** 'n oënskynlik onskadelike domain fine-tune kan die model na 'n slegte region van gedrag buite die beoogde task distribution skuif.

Praktiese checks:

- Hou 'n **canary set** van synthetic trigger prompts en laat dit op elke checkpoint loop.
- Deduplicate en doen provenance-check op die SFT corpus voor training.
- Moet nooit op die presiese prompts train wat deur die benchmark gebruik word wat jy later gaan report nie.
- Vergelyk die base model met die fine-tuned model op security-sensitive prompts om onverwachte gedrag drift te detect.

## Follow instructions fine-tuning code

Jy kan 'n voorbeeld van die code vind om hierdie fine-tuning uit te voer in [https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py](https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py)

'n Praktiese moderne alternatief is om 'n trainer te gebruik wat reeds chat templates, assistant-only loss, packing, en PEFT adapters ondersteun. As jy later preference alignment wil hê, bied dieselfde ekosisteem ook `DPOTrainer`-styl workflows vir datasets met `prompt` / `chosen` / `rejected` velde:
```python
from trl import SFTConfig, SFTTrainer

trainer = SFTTrainer(
model="Qwen/Qwen2.5-0.5B-Instruct",
train_dataset=dataset,
args=SFTConfig(
learning_rate=2e-5,
packing=True,
assistant_only_loss=True,
),
)
trainer.train()
```
