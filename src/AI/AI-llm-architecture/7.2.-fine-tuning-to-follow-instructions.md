# 7.2. Fine-Tuning to follow instructions

{{#include ../../banners/hacktricks-training.md}}

> [!TIP]
> 이 섹션의 목표는 **이미 pre-trained된 model을 instructions를 따르도록 fine-tune하는 방법**을 보여주는 것입니다. 단순히 text를 생성하는 것이 아니라, 예를 들어 chat bot처럼 task에 응답하도록 만드는 것입니다.

## Dataset

instruction을 따르도록 LLM을 fine-tune하려면 **instructions와 responses**가 포함된 dataset이 필요합니다. LLM이 instructions를 따르도록 학습시키는 데에는 여러 prompt format이 있습니다. 예를 들면:

- The Alpaca prompt style example:
```csharp
Below is an instruction that describes a task. Write a response that appropriately completes the request.

### Instruction:
Calculate the area of a circle with a radius of 5 units.

### Response:
The area of a circle is calculated using the formula \( A = \pi r^2 \). Plugging in the radius of 5 units:

\( A = \pi (5)^2 = \pi \times 25 = 25\pi \) square units.
```
- Phi-3 / chat template style example:
```vbnet
<|User|>
Can you explain what gravity is in simple terms?

<|Assistant|>
Absolutely! Gravity is a force that pulls objects toward each other.
```
이런 종류의 데이터셋으로 학습하는 것은, 단순한 raw text로만 학습하는 것보다 모델이 작업에 대해 **structured assistant format**으로 답해야 한다는 점을 이해하는 데 도움이 됩니다.

매우 흔한 실수는 한 template으로 학습하고 다른 template으로 추론하는 것입니다. Modern instruct models는 보통 base model의 **native chat template**(`<|user|>`, `<|assistant|>`, `[INST]`, ChatML, 등)를 기대하므로, 데이터셋을 target model이 이미 사용하는 정확한 format으로 변환하는 것이 권장됩니다.

따라서 요청과 답변을 포함한 데이터셋에서 가장 먼저 할 일 중 하나는, 다음과 같이 해당 데이터를 원하는 prompt format으로 구성하는 것입니다:
```python
# Code from https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/ch07.ipynb
def format_input(entry):
instruction_text = (
f"Below is an instruction that describes a task. "
f"Write a response that appropriately completes the request."
f"\n\n### Instruction:\n{entry['instruction']}"
)

input_text = f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""

return instruction_text + input_text

model_input = format_input(data[50])

desired_response = f"\n\n### Response:\n{data[50]['output']}"

print(model_input + desired_response)
```
베이스 모델에 이미 tokenizer chat template가 포함되어 있다면, 수동으로 separator를 만드는 것보다 message list로 dataset을 구성하는 것이 더 안전할 수 있습니다:
```python
messages = [
{"role": "user", "content": entry["instruction"]},
{"role": "assistant", "content": entry["output"]},
]
text = tokenizer.apply_chat_template(messages, tokenize=False)
```
Then, 늘 그렇듯이, dataset을 training, validation, testing sets로 분리해야 합니다.

## Batching & Data Loaders

Then, training을 위해 모든 input과 expected output을 batch로 묶어야 합니다. 이를 위해서는 다음이 필요합니다:

- 텍스트를 tokenize.
- 모든 sample을 같은 길이로 pad (보통 길이는 LLM을 pre-train할 때 사용한 context length만큼 큽니다).
- custom collate function에서 input을 1만큼 shift해서 expected tokens를 생성.
- 일부 padding token을 `-100`으로 바꿔 training loss에서 제외: 첫 번째 `endoftext` token 이후의 다른 모든 `endoftext` token을 `-100`으로 대체 (왜냐하면 `cross_entropy(..., ignore_index=-100)`를 사용하면 `-100`이 들어간 target은 무시되기 때문입니다).
- \[Optional] question에 해당하는 모든 token도 `-100`으로 mask해서 LLM이 answer를 생성하는 법만 배우게 함. Alpaca style에서는 보통 `### Response:` 전까지의 모든 것을 masking하는 의미입니다.
- **Modern SFT best practice:** dataset이 conversational이면, user/system prompt가 아니라 **assistant completion tokens**에만 loss를 계산합니다. 이렇게 하면 model이 prompt text 자체를 예측하도록 가르치는 것을 피할 수 있습니다.
- **Throughput optimization:** dataset에 짧은 sample이 많다면, context window를 덜 낭비하도록 **여러 example을 같은 sequence에 pack**하는 것이 일반적입니다.

이렇게 만든 다음에는 각 dataset(training, validation, test)에 대한 data loaders를 생성하면 됩니다.

## Load pre-trained LLM & Fine tune & Loss Checking

pre-trained LLM을 load해서 fine-tune해야 합니다. 이는 이미 다른 페이지에서 다뤘습니다. 그런 다음 이전에 사용한 training function을 사용해 LLM을 fine-tune할 수 있습니다.

training 동안에는 epoch마다 training loss와 validation loss가 어떻게 변하는지도 확인해서 loss가 줄어드는지, overfitting이 발생하는지 볼 수 있습니다.\
overfitting은 training loss는 줄어드는데 validation loss가 줄지 않거나 오히려 증가할 때 발생합니다. 이를 피하려면, 이 현상이 시작되는 epoch에서 training을 멈추는 것이 가장 간단합니다.

GPU memory가 제한적이라면 [7.0. LoRA Improvements in fine-tuning](7.0.-lora-improvements-in-fine-tuning.md)를 확인하세요. 실제로 대부분의 instruction-tuning 작업은 base model의 모든 parameter를 업데이트하는 대신 **LoRA/QLoRA/PEFT adapters**로 수행됩니다.

## Modern post-training variants

전통적인 접근은 `(instruction, answer)` pair에 대해 **supervised fine-tuning (SFT)**을 하는 것입니다. 하지만 현재 instruct-model pipeline은 종종 post-training을 여러 단계로 나눕니다:

1. 출력 format, answer style, task distribution을 학습시키는 **SFT**.
2. 별도의 reward model 없이 한 answer를 다른 answer보다 선호하도록 만드는 **preference optimization**.
3. 특정 product나 safety goal을 위한 RLHF / PPO 같은 **Optional** online optimization.

SFT 이후 가장 흔한 preference-based 대안은 다음과 같습니다:

- **DPO (Direct Preference Optimization):** preferred vs rejected answer를 직접 최적화합니다.
- **ORPO / KTO / SimPO:** 전체 RLHF pipeline의 engineering overhead를 일부 줄여주는 더 단순한 variant입니다.

실제로 이런 방법들은 보통 같은 prompt에 대한 **chosen/rejected response pair** 또는 더 단순한 **desirable / undesirable** signal을 사용하므로, preference dataset의 quality가 SFT corpus만큼 중요합니다.

이것이 유용한 이유는 많은 프로젝트에서 비싼 부분이 model에 format을 가르치는 것이 아니라, 사람이 원하는 answer style을 **선호하도록 학습시키는 것**이기 때문입니다. RL 측면과 reward-model poisoning에 대한 더 자세한 내용은 [../AI-Reinforcement-Learning-Algorithms.md](../AI-Reinforcement-Learning-Algorithms.md)를 확인하세요.

## Response Quality

이것은 loss variation을 더 신뢰할 수 있는 classification fine-tune이 아니기 때문에, testing set의 response quality를 확인하는 것도 중요합니다. 따라서 모든 testing set에서 생성된 response를 모아 **직접 수동으로 quality를 확인**해서 잘못된 answer가 있는지 보는 것이 좋습니다(LLM이 response sentence의 format과 syntax는 올바르게 생성했지만 완전히 틀린 response를 줄 수도 있습니다. loss variation은 이런 동작을 반영하지 않습니다).\
생성된 response와 expected response를 **다른 LLM에 넣고 response를 평가하게 하는 방식**으로 이 review를 수행할 수도 있습니다.

우선적으로 고려할 만한 Modern instruction-following evaluations는 다음과 같습니다:

- **IFEval:** model이 prompt에서 명시한 constraint를 실제로 따르는지, 단지 그럴듯하게 보이는지만이 아니라 확인합니다.
- **AlpacaEval 2 / pairwise judges:** baseline model과 generation을 비교할 때 유용합니다.
- **Arena-style head-to-head evals:** 여러 checkpoint와 prompt의 relative ranking이 필요할 때 유용합니다.

Response의 quality를 검증하기 위해 실행할 다른 test:

1. **Measuring Massive Multitask Language Understanding (**[**MMLU**](https://arxiv.org/abs/2009.03300)**):** MMLU는 인문학, 과학 등을 포함한 57개 subject 전반에서 model의 knowledge와 problem-solving ability를 평가합니다. 여러 난이도의 multiple-choice question을 사용해 elementary부터 advanced professional 수준까지 이해도를 평가합니다.
2. [**LMSYS Chatbot Arena**](https://arena.lmsys.org): 이 platform은 사용자가 여러 chatbot의 response를 나란히 비교할 수 있게 해줍니다. 사용자가 prompt를 입력하면 여러 chatbot이 response를 생성하고, 이를 직접 비교할 수 있습니다.
3. [**AlpacaEval**](https://github.com/tatsu-lab/alpaca_eval)**:** AlpacaEval은 GPT-4 같은 advanced LLM이 다양한 prompt에 대한 다른 model의 response를 평가하는 automated evaluation framework입니다.
4. **General Language Understanding Evaluation (**[**GLUE**](https://gluebenchmark.com/)**):** GLUE는 sentiment analysis, textual entailment, question answering을 포함한 9개의 natural language understanding task 모음입니다.
5. [**SuperGLUE**](https://super.gluebenchmark.com/)**:** GLUE를 확장한 것으로, 현재 model에게 더 어렵도록 설계된 더 challenging task를 포함합니다.
6. **Beyond the Imitation Game Benchmark (**[**BIG-bench**](https://github.com/google/BIG-bench)**):** BIG-bench는 reasoning, translation, question answering 같은 영역에서 model의 능력을 테스트하는 200개 이상의 task를 포함한 대규모 benchmark입니다.
7. **Holistic Evaluation of Language Models (**[**HELM**](https://crfm.stanford.edu/helm/lite/latest/)**):** HELM은 accuracy, robustness, fairness 같은 다양한 metric 전반에 걸친 종합 평가를 제공합니다.
8. [**OpenAI Evals**](https://github.com/openai/evals)**:** AI model을 custom 및 standardized task에서 테스트할 수 있게 해주는 OpenAI의 open-source evaluation framework입니다.
9. [**HumanEval**](https://github.com/openai/human-eval)**:** language model의 code generation ability를 평가하는 데 사용되는 programming problem 모음입니다.
10. **Stanford Question Answering Dataset (**[**SQuAD**](https://rajpurkar.github.io/SQuAD-explorer/)**):** SQuAD는 Wikipedia article에 대한 question으로 구성되며, model은 정확하게 answer하기 위해 text를 이해해야 합니다.
11. [**TriviaQA**](https://nlp.cs.washington.edu/triviaqa/)**:** trivia question과 answer, 그리고 evidence document로 이루어진 대규모 dataset입니다.

그리고 이 외에도 아주 많습니다.

## Security notes for instruction tuning

Instruction-tuning dataset 역시 attack surface입니다. attacker가 SFT corpus에 소수의 poisoned sample을 삽입할 수 있다면, model은 깨끗한 prompt에서는 정상처럼 보이면서도 **trigger-conditioned behaviour**를 학습할 수 있습니다.

red teaming 중 테스트해야 할 전형적인 malicious pattern은 다음과 같습니다:

- **Instruction backdoors:** 드문 trigger string, role prefix, unicode pattern, formatting sequence가 attacker가 선택한 content를 출력하도록 만듭니다.
- **Sleeper-agent behaviour:** model이 정상 evaluation에서는 올바르게 동작하지만, 매우 특정한 trigger가 나타날 때만 행동을 바꿉니다.
- **Narrow-domain misalignment:** 겉보기에는 무해한 domain fine-tune이 의도한 task distribution 밖에서 model을 나쁜 behaviour 영역으로 이동시킬 수 있습니다.

실무적인 확인 방법:

- synthetic trigger prompt로 구성된 **canary set**을 유지하고 모든 checkpoint에 실행합니다.
- training 전에 SFT corpus를 deduplicate하고 provenance를 확인합니다.
- 나중에 보고할 benchmark에서 사용한 정확한 prompt로는 절대 training하지 않습니다.
- 예상치 못한 behaviour drift를 감지하기 위해 base model과 fine-tuned model을 security-sensitive prompt에서 비교합니다.

## Follow instructions fine-tuning code

이 fine-tuning을 수행하는 code 예시는 [https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py](https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py)에서 찾을 수 있습니다.

실용적인 modern alternative는 chat templates, assistant-only loss, packing, PEFT adapters를 이미 지원하는 trainer를 사용하는 것입니다. 나중에 preference alignment가 필요하다면, 같은 ecosystem에서 `prompt` / `chosen` / `rejected` field가 있는 dataset을 위한 `DPOTrainer` 스타일 workflow도 제공합니다:
```python
from trl import SFTConfig, SFTTrainer

trainer = SFTTrainer(
model="Qwen/Qwen2.5-0.5B-Instruct",
train_dataset=dataset,
args=SFTConfig(
learning_rate=2e-5,
packing=True,
assistant_only_loss=True,
),
)
trainer.train()
```
