# 7.2. Fine-Tuning per seguire istruzioni

{{#include ../../banners/hacktricks-training.md}}

> [!TIP]
> L'obiettivo di questa sezione è mostrare come **fine-tune un modello già pre-trained per seguire istruzioni** invece di generare solo testo, ad esempio rispondendo a task come un chat bot.

## Dataset

Per fare fine-tune di un LLM per seguire istruzioni, è necessario avere un dataset con **istruzioni e risposte**. Esistono diversi formati di prompt per addestrare un LLM a seguire istruzioni, per esempio:

- The Alpaca prompt style example:
```csharp
Below is an instruction that describes a task. Write a response that appropriately completes the request.

### Instruction:
Calculate the area of a circle with a radius of 5 units.

### Response:
The area of a circle is calculated using the formula \( A = \pi r^2 \). Plugging in the radius of 5 units:

\( A = \pi (5)^2 = \pi \times 25 = 25\pi \) square units.
```
- Esempio di stile Phi-3 / chat template:
```vbnet
<|User|>
Can you explain what gravity is in simple terms?

<|Assistant|>
Absolutely! Gravity is a force that pulls objects toward each other.
```
Addestrare con questo tipo di dataset invece che solo con testo grezzo aiuta il modello a capire che deve rispondere ai task in un **structured assistant format**.

Un errore molto comune è addestrare con un template e fare inferenza con un altro. I modern instruct model di solito si aspettano il **native chat template del base model** (`<|user|>`, `<|assistant|>`, `[INST]`, ChatML, etc.), quindi è consigliato convertire il dataset nell’esatto formato che il target model già usa.

Perciò, una delle prime cose da fare con un dataset che contiene richieste e risposte è modellare quei dati nel desired prompt format, come:
```python
# Code from https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/ch07.ipynb
def format_input(entry):
instruction_text = (
f"Below is an instruction that describes a task. "
f"Write a response that appropriately completes the request."
f"\n\n### Instruction:\n{entry['instruction']}"
)

input_text = f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""

return instruction_text + input_text

model_input = format_input(data[50])

desired_response = f"\n\n### Response:\n{data[50]['output']}"

print(model_input + desired_response)
```
Se il modello base include già un chat template del tokenizer, può essere più sicuro costruire il dataset a partire da liste di messaggi invece di creare a mano i separator:
```python
messages = [
{"role": "user", "content": entry["instruction"]},
{"role": "assistant", "content": entry["output"]},
]
text = tokenizer.apply_chat_template(messages, tokenize=False)
```
Poi, come sempre, è necessario separare il dataset in training, validation e testing sets.

## Batching & Data Loaders

Poi, è necessario fare il batch di tutti gli input e degli expected output per il training. Per questo, è necessario:

- Tokenize the texts.
- Pad tutti i sample alla stessa lunghezza (di solito la lunghezza sarà grande quanto la context length usata per pre-trainare il LLM).
- Creare i token attesi shiftando di 1 l'input in una custom collate function.
- Sostituire alcuni padding tokens con `-100` per escluderli dalla training loss: dopo il primo token `endoftext`, sostituire tutti gli altri token `endoftext` con `-100` (perché usare `cross_entropy(..., ignore_index=-100)` significa che ignorerà i target con `-100`).
- \[Optional] Mascherare usando `-100` anche tutti i token appartenenti alla domanda così che il LLM impari solo a generare la risposta. In stile Alpaca questo di solito significa mascherare tutto fino a `### Response:`.
- **Modern SFT best practice:** quando il dataset è conversational, calcolare la loss solo sui **assistant completion tokens** e non sul user/system prompt. Questo evita di insegnare al model a predire il prompt text stesso.
- **Throughput optimization:** quando il dataset contiene molti sample corti, è comune **packare più examples nella stessa sequence** per sprecare meno context window.

Una volta creato questo, è il momento di creare i data loaders per ciascun dataset (training, validation e test).

## Load pre-trained LLM & Fine tune & Loss Checking

È necessario caricare un pre-trained LLM per fine-tunarlo. Questo è già stato discusso in altre pagine. Poi, è possibile usare la precedente training function per fine-tunare il LLM.

Durante il training è anche possibile vedere come la training loss e la validation loss variano durante le epochs per capire se la loss si riduce e se sta avvenendo overfitting.\
Ricorda che l'overfitting avviene quando la training loss si riduce ma la validation loss non si riduce oppure aumenta. Per evitarlo, la cosa più semplice da fare è fermare il training all'epoch in cui questo comportamento inizia.

Se la memoria GPU è un limite, controlla [7.0. LoRA Improvements in fine-tuning](7.0.-lora-improvements-in-fine-tuning.md), perché in pratica la maggior parte dei job di instruction-tuning viene eseguita con **LoRA/QLoRA/PEFT adapters** invece di aggiornare ogni parametro del base model.

## Modern post-training variants

L'approccio classico è il **supervised fine-tuning (SFT)** su coppie `(instruction, answer)`. Tuttavia, le pipeline attuali per instruct-model spesso dividono il post-training in più fasi:

1. **SFT** per insegnare il formato di output, lo stile delle risposte e la distribuzione dei task.
2. **Preference optimization** per far preferire al model una risposta rispetto a un'altra senza bisogno di un separate reward model.
3. **Optional** online optimization come RLHF / PPO per prodotti specifici o obiettivi di sicurezza.

Le alternative preference-based più comuni dopo SFT sono:

- **DPO (Direct Preference Optimization):** ottimizza direttamente risposte preferred vs rejected.
- **ORPO / KTO / SimPO:** varianti più semplici che riducono parte dell'overhead ingegneristico delle pipeline complete RLHF.

In pratica, questi metodi consumano di solito coppie di **chosen/rejected response** per lo stesso prompt oppure un segnale più semplice **desirable / undesirable**, quindi la qualità del preference dataset diventa importante quanto il corpus SFT stesso.

Questo è utile perché per molti progetti la parte costosa non è insegnare al model il formato, ma insegnargli a **preferire lo stile di risposta che gli umani vogliono**. Se vuoi più dettagli sul lato RL e sul reward-model poisoning, controlla [../AI-Reinforcement-Learning-Algorithms.md](../AI-Reinforcement-Learning-Algorithms.md).

## Response Quality

Poiché questo non è un classification fine-tune dove è possibile fidarsi di più delle variazioni della loss, è anche importante controllare la qualità delle responses nel testing set. Per questo, è raccomandato raccogliere le generated responses da tutti i testing sets e **controllarne manualmente la qualità** per vedere se ci sono risposte sbagliate (nota che è possibile che il LLM generi correttamente il formato e la syntax della frase di risposta ma dia una response completamente sbagliata. La variazione della loss non rifletterà questo comportamento).\
Nota che è anche possibile fare questa review passando le generated responses e le expected responses ad **altri LLM e chiedendo loro di valutare le responses**.

Le moderne valutazioni instruction-following che vale la pena prioritizzare sono:

- **IFEval:** controlla se il model segue davvero i vincoli espliciti del prompt invece di sembrare solo valido.
- **AlpacaEval 2 / pairwise judges:** utile per confrontare le generations con un baseline model.
- **Arena-style head-to-head evals:** utile quando serve un ranking relativo di diversi checkpoint e prompt.

Altri test da eseguire per verificare la qualità delle responses:

1. **Measuring Massive Multitask Language Understanding (**[**MMLU**](https://arxiv.org/abs/2009.03300)**):** MMLU valuta la conoscenza di un model e le sue capacità di problem-solving in 57 materie, incluse humanities, sciences e altro. Usa domande a scelta multipla per valutare la comprensione a vari livelli di difficoltà, da elementare a professionale avanzato.
2. [**LMSYS Chatbot Arena**](https://arena.lmsys.org): Questa piattaforma permette agli utenti di confrontare le responses di diversi chatbots affiancate. Gli utenti inseriscono un prompt e più chatbots generano responses che possono essere confrontate direttamente.
3. [**AlpacaEval**](https://github.com/tatsu-lab/alpaca_eval)**:** AlpacaEval è un framework di valutazione automatica in cui un LLM avanzato come GPT-4 valuta le responses di altri modelli su vari prompt.
4. **General Language Understanding Evaluation (**[**GLUE**](https://gluebenchmark.com/)**):** GLUE è una collezione di nove task di natural language understanding, tra cui sentiment analysis, textual entailment e question answering.
5. [**SuperGLUE**](https://super.gluebenchmark.com/)**:** Basandosi su GLUE, SuperGLUE include task più impegnativi progettati per essere difficili per i modelli attuali.
6. **Beyond the Imitation Game Benchmark (**[**BIG-bench**](https://github.com/google/BIG-bench)**):** BIG-bench è un benchmark su larga scala con oltre 200 task che testano le capacità di un model in aree come reasoning, translation e question answering.
7. **Holistic Evaluation of Language Models (**[**HELM**](https://crfm.stanford.edu/helm/lite/latest/)**):** HELM fornisce una valutazione completa su varie metriche come accuracy, robustness e fairness.
8. [**OpenAI Evals**](https://github.com/openai/evals)**:** Un framework di valutazione open-source di OpenAI che permette di testare AI models su task custom e standardizzati.
9. [**HumanEval**](https://github.com/openai/human-eval)**:** Una collezione di problemi di programmazione usata per valutare le capacità di code generation dei language models.
10. **Stanford Question Answering Dataset (**[**SQuAD**](https://rajpurkar.github.io/SQuAD-explorer/)**):** SQuAD è composto da domande su articoli di Wikipedia, dove i models devono comprendere il text per rispondere accuratamente.
11. [**TriviaQA**](https://nlp.cs.washington.edu/triviaqa/)**:** Un dataset su larga scala di domande e risposte di trivia, insieme a documenti di evidenza.

e molti molti altri.

## Security notes for instruction tuning

I dataset di instruction-tuning sono anche una superficie d'attacco. Se un attacker riesce a inserire un piccolo numero di poisoned samples nel corpus SFT, il model può imparare un **trigger-conditioned behaviour** pur sembrando normale su clean prompts.

I pattern malevoli tipici da testare durante il red teaming sono:

- **Instruction backdoors:** una rara trigger string, role prefix, unicode pattern o formatting sequence causa l'output di contenuti scelti dall'attacker.
- **Sleeper-agent behaviour:** il model si comporta correttamente durante le normali evaluation ma cambia comportamento solo quando appare un trigger molto specifico.
- **Narrow-domain misalignment:** un fine-tune di dominio apparentemente innocuo può spostare il model in una cattiva regione di comportamento al di fuori della distribuzione del task prevista.

Controlli pratici:

- Mantieni un **canary set** di synthetic trigger prompts ed eseguilo su ogni checkpoint.
- Deduplica e verifica la provenance del corpus SFT prima del training.
- Non addestrare mai sugli stessi prompt esatti usati dal benchmark che riporterai in seguito.
- Confronta il base model vs il fine-tuned model su prompt security-sensitive per rilevare drift di comportamento inatteso.

## Follow instructions fine-tuning code

Puoi trovare un esempio del codice per eseguire questo fine-tuning in [https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py](https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py)

Una moderna alternativa pratica è usare un trainer che supporti già chat templates, assistant-only loss, packing e PEFT adapters. Se in seguito vuoi preference alignment, lo stesso ecosistema fornisce anche workflow in stile `DPOTrainer` per dataset con campi `prompt` / `chosen` / `rejected`:
```python
from trl import SFTConfig, SFTTrainer

trainer = SFTTrainer(
model="Qwen/Qwen2.5-0.5B-Instruct",
train_dataset=dataset,
args=SFTConfig(
learning_rate=2e-5,
packing=True,
assistant_only_loss=True,
),
)
trainer.train()
```
