# 7.2. Fine-Tuning para seguir instrucciones

{{#include ../../banners/hacktricks-training.md}}

> [!TIP]
> El objetivo de esta sección es mostrar cómo **fine-tune un modelo ya pre-entrenado para seguir instrucciones** en lugar de solo generar texto, por ejemplo, respondiendo a tareas como un chat bot.

## Dataset

Para hacer fine-tuning de un LLM para seguir instrucciones, se necesita tener un dataset con **instrucciones y respuestas**. Existen diferentes formatos de prompt para entrenar un LLM para seguir instrucciones, por ejemplo:

- El ejemplo del estilo de prompt Alpaca:
```csharp
Below is an instruction that describes a task. Write a response that appropriately completes the request.

### Instruction:
Calculate the area of a circle with a radius of 5 units.

### Response:
The area of a circle is calculated using the formula \( A = \pi r^2 \). Plugging in the radius of 5 units:

\( A = \pi (5)^2 = \pi \times 25 = 25\pi \) square units.
```
- Ejemplo de estilo de template de chat de Phi-3:
```vbnet
<|User|>
Can you explain what gravity is in simple terms?

<|Assistant|>
Absolutely! Gravity is a force that pulls objects toward each other.
```
Entrenar con este tipo de datasets en lugar de solo texto sin procesar ayuda al modelo a entender que necesita responder tareas en un **formato estructurado de assistant**.

Un error muy común es entrenar con una plantilla e inferir con otra. Los modelos instruct modernos suelen esperar la **native chat template del base model** (`<|user|>`, `<|assistant|>`, `[INST]`, ChatML, etc.), así que se recomienda convertir el dataset al formato exacto que el target model ya usa.

Por lo tanto, una de las primeras cosas que hay que hacer con un dataset que contiene requests y answers es modelar esos datos en el formato de prompt deseado, como:
```python
# Code from https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/ch07.ipynb
def format_input(entry):
instruction_text = (
f"Below is an instruction that describes a task. "
f"Write a response that appropriately completes the request."
f"\n\n### Instruction:\n{entry['instruction']}"
)

input_text = f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""

return instruction_text + input_text

model_input = format_input(data[50])

desired_response = f"\n\n### Response:\n{data[50]['output']}"

print(model_input + desired_response)
```
Si el modelo base ya incluye una chat template del tokenizer, puede ser más seguro construir el dataset a partir de listas de mensajes en lugar de crear separadores a mano:
```python
messages = [
{"role": "user", "content": entry["instruction"]},
{"role": "assistant", "content": entry["output"]},
]
text = tokenizer.apply_chat_template(messages, tokenize=False)
```
Entonces, como siempre, es necesario separar el dataset en conjuntos de entrenamiento, validación y prueba.

## Batching & Data Loaders

Entonces, es necesario hacer batching de todas las entradas y salidas esperadas para el entrenamiento. Para esto, es necesario:

- Tokenize the texts.
- Pad todos los samples a la misma longitud (normalmente la longitud será tan grande como la context length usada para pre-train el LLM).
- Create the expected tokens by shifting 1 la entrada en una custom collate function.
- Replace algunos padding tokens con `-100` para excluirlos de la training loss: after the first `endoftext` token, sustituir todos los demás `endoftext` tokens por `-100` (because using `cross_entropy(..., ignore_index=-100)` means that it'll ignore targets with `-100`).
- \[Optional] Mask using `-100` también todos los tokens pertenecientes a la question para que el LLM aprenda solo a generate the answer. In the Alpaca style this usually means masking everything until `### Response:`.
- **Modern SFT best practice:** when the dataset is conversational, compute the loss only on the **assistant completion tokens** and not on the user/system prompt. This avoids teaching the model to predict the prompt text itself.
- **Throughput optimization:** when the dataset contains many short samples, it's common to **pack several examples into the same sequence** to waste less context window.

Con esto creado, es hora de crear los data loaders para cada dataset (training, validation y test).

## Load pre-trained LLM & Fine tune & Loss Checking

Es necesario cargar un pre-trained LLM para fine-tune it. Esto ya se discutió en otras páginas. Entonces, es posible usar la función de training usada previamente para fine-tune el LLM.

Durante el training también es posible ver cómo la training loss y la validation loss varían durante los epochs para ver si la loss se está reduciendo y si está ocurriendo overfitting.\
Recuerda que overfitting ocurre cuando la training loss se reduce pero la validation loss no se reduce o incluso aumenta. Para evitar esto, lo más simple es detener el training en el epoch en el que comienza este comportamiento.

Si la memoria GPU es una limitación, revisa [7.0. LoRA Improvements in fine-tuning](7.0.-lora-improvements-in-fine-tuning.md), porque en la práctica la mayoría de los jobs de instruction-tuning se hacen con **LoRA/QLoRA/PEFT adapters** en lugar de actualizar cada parámetro del base model.

## Modern post-training variants

El enfoque clásico es **supervised fine-tuning (SFT)** sobre pares `(instruction, answer)`. Sin embargo, los pipelines actuales de instruct-model suelen dividir el post-training en varias fases:

1. **SFT** para enseñar el output format, answer style y la task distribution.
2. **Preference optimization** para hacer que el modelo prefiera una answer sobre otra sin necesitar un separate reward model.
3. **Optional** online optimization como RLHF / PPO para productos específicos o safety goals.

Las alternativas más comunes basadas en preference después de SFT son:

- **DPO (Direct Preference Optimization):** optimiza directamente respuestas preferred vs rejected.
- **ORPO / KTO / SimPO:** variantes más simples que reducen parte del overhead de engineering de los full RLHF pipelines.

En la práctica, estos métodos suelen consumir o bien **chosen/rejected response pairs** para el mismo prompt o una señal más simple de **desirable / undesirable**, así que la calidad del preference dataset se vuelve tan importante como el SFT corpus en sí.

Esto es útil porque para muchos proyectos la parte cara no es enseñarle al modelo el format, sino enseñarle a **prefer the style of answer humans want**. Si quieres más detalles sobre la parte de RL y reward-model poisoning, revisa [../AI-Reinforcement-Learning-Algorithms.md](../AI-Reinforcement-Learning-Algorithms.md).

## Response Quality

Como esto no es un fine-tune de clasificación donde es posible confiar más en las variaciones de loss, también es importante comprobar la calidad de las responses en el testing set. Por tanto, se recomienda recopilar las generated responses de todos los testing sets y **check their quality manually** para ver si hay respuestas incorrectas (ten en cuenta que es posible que el LLM genere correctamente el format y la syntax de la response sentence pero dé una respuesta completamente errónea. La variación de loss no reflejará este comportamiento).\
Ten en cuenta que también es posible realizar esta revisión pasando las generated responses y las expected responses a **other LLMs y pidiéndoles que evalúen las responses**.

Las evaluaciones modernas de instruction-following que vale la pena priorizar son:

- **IFEval:** comprueba si el modelo realmente sigue constraints explícitas del prompt en lugar de simplemente sonar bien.
- **AlpacaEval 2 / pairwise judges:** útiles para comparar generations contra un baseline model.
- **Arena-style head-to-head evals:** útiles cuando necesitas un ranking relativo de varios checkpoints y prompts.

Otros tests que ejecutar para verificar la calidad de las responses:

1. **Measuring Massive Multitask Language Understanding (**[**MMLU**](https://arxiv.org/abs/2009.03300)**):** MMLU evalúa el conocimiento y las capacidades de resolución de problemas de un model en 57 subjects, incluyendo humanities, sciences y más. Usa preguntas de multiple-choice para evaluar la comprensión en distintos niveles de dificultad, desde elementary hasta advanced professional.
2. [**LMSYS Chatbot Arena**](https://arena.lmsys.org): Esta platform permite a los users comparar responses de diferentes chatbots lado a lado. Los users introducen un prompt, y varios chatbots generate responses que pueden compararse directamente.
3. [**AlpacaEval**](https://github.com/tatsu-lab/alpaca_eval)**:** AlpacaEval es un framework de evaluación automatizada donde un advanced LLM como GPT-4 evalúa las responses de otros models frente a diversos prompts.
4. **General Language Understanding Evaluation (**[**GLUE**](https://gluebenchmark.com/)**):** GLUE es una colección de nueve tasks de natural language understanding, incluyendo sentiment analysis, textual entailment y question answering.
5. [**SuperGLUE**](https://super.gluebenchmark.com/)**:** Partiendo de GLUE, SuperGLUE incluye tasks más desafiantes diseñadas para ser difíciles para los models actuales.
6. **Beyond the Imitation Game Benchmark (**[**BIG-bench**](https://github.com/google/BIG-bench)**):** BIG-bench es un benchmark a gran escala con más de 200 tasks que prueban las capacidades de un model en áreas como reasoning, translation y question answering.
7. **Holistic Evaluation of Language Models (**[**HELM**](https://crfm.stanford.edu/helm/lite/latest/)**):** HELM ofrece una evaluación completa en varias métricas como accuracy, robustness y fairness.
8. [**OpenAI Evals**](https://github.com/openai/evals)**:** Un framework de evaluación de código abierto de OpenAI que permite probar modelos de AI en tasks personalizadas y estandarizadas.
9. [**HumanEval**](https://github.com/openai/human-eval)**:** Una colección de programming problems usada para evaluar las code generation abilities de language models.
10. **Stanford Question Answering Dataset (**[**SQuAD**](https://rajpurkar.github.io/SQuAD-explorer/)**):** SQuAD consiste en preguntas sobre artículos de Wikipedia, donde los models deben comprender el texto para responder con precisión.
11. [**TriviaQA**](https://nlp.cs.washington.edu/triviaqa/)**:** Un dataset a gran escala de trivia questions and answers, junto con evidence documents.

y muchos, muchos más.

## Security notes for instruction tuning

Los instruction-tuning datasets también son una attack surface. Si un attacker puede insertar un pequeño número de poisoned samples en el SFT corpus, el model puede aprender un **trigger-conditioned behaviour** mientras sigue pareciendo normal en clean prompts.

Los patrones malicious típicos para probar durante red teaming son:

- **Instruction backdoors:** una rare trigger string, role prefix, unicode pattern o formatting sequence hace que el model genere attacker-chosen content.
- **Sleeper-agent behaviour:** el model se comporta correctamente durante evaluaciones normales pero cambia de comportamiento solo cuando aparece un trigger muy específico.
- **Narrow-domain misalignment:** un domain fine-tune aparentemente inofensivo puede mover el model a una mala región de behaviour fuera de la intended task distribution.

Practical checks:

- Mantén un **canary set** de synthetic trigger prompts y ejecútalo en cada checkpoint.
- Deduplicate y comprueba la provenance del SFT corpus antes de training.
- Nunca entrenes sobre los exact prompts usados por el benchmark que luego vas a reportar.
- Compara el base model vs el fine-tuned model en security-sensitive prompts para detectar unexpected behaviour drift.

## Follow instructions fine-tuning code

Puedes encontrar un ejemplo del código para realizar este fine-tuning en [https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py](https://github.com/rasbt/LLMs-from-scratch/blob/main/ch07/01_main-chapter-code/gpt_instruction_finetuning.py)

Una alternativa práctica moderna es usar un trainer que ya soporte chat templates, assistant-only loss, packing y PEFT adapters. Si más adelante quieres preference alignment, el mismo ecosistema también proporciona workflows estilo `DPOTrainer` para datasets con campos `prompt` / `chosen` / `rejected`:
```python
from trl import SFTConfig, SFTTrainer

trainer = SFTTrainer(
model="Qwen/Qwen2.5-0.5B-Instruct",
train_dataset=dataset,
args=SFTConfig(
learning_rate=2e-5,
packing=True,
assistant_only_loss=True,
),
)
trainer.train()
```
