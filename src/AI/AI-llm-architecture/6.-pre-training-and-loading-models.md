## Generación de texto

Para entrenar un modelo necesitaremos que sea capaz de generar nuevos tokens. Después compararemos los tokens generados con los esperados para entrenar el modelo en **aprender los tokens que necesita generar**.

Como en los ejemplos anteriores ya predijimos algunos tokens, es posible reutilizar esa función para este propósito.

> [!TIP]
> El objetivo de esta sexta fase es muy sencillo: **Entrenar el modelo desde cero**. Para ello se utilizará la arquitectura LLM anterior, con algunos bucles que recorren los conjuntos de datos usando las funciones de pérdida y el optimizador definidos para entrenar todos los parámetros del modelo.

## Evaluación del texto

Para realizar un entrenamiento correcto es necesario comprobar las predicciones obtenidas para el token esperado. El objetivo del entrenamiento es maximizar la probabilidad del token correcto, lo que implica aumentar su probabilidad en relación con la de los demás tokens.

Para maximizar la probabilidad del token correcto, los pesos del modelo deben modificarse de forma que dicha probabilidad se maximice. La actualización de los pesos se realiza mediante **backpropagation**. Esto requiere una **función de pérdida que maximizar**. En este caso, la función será la **diferencia entre la predicción realizada y la deseada**.

Sin embargo, en lugar de trabajar con las predicciones sin procesar, se trabajará con un logaritmo de base n. Por lo tanto, si la predicción actual del token esperado fuera 7.4541e-05, el logaritmo natural (base *e*) de **7.4541e-05** sería aproximadamente **-9.5042**.\
Después, para cada entrada con una longitud de contexto de 5 tokens, por ejemplo, el modelo tendrá que predecir 5 tokens: los primeros 4 tokens serán el último token de la entrada y el quinto será el predicho. Por lo tanto, para cada entrada tendremos 5 predicciones en ese caso (aunque las primeras 4 estuvieran en la entrada, el modelo no lo sabe), con 5 tokens esperados y, por tanto, 5 probabilidades que maximizar.

Por lo tanto, después de calcular el logaritmo natural de cada predicción, se calcula la **media**, se **elimina el símbolo menos** (esto se denomina _cross entropy loss_) y ese es el **número que se debe reducir lo máximo posible hasta acercarlo a 0**, porque el logaritmo natural de 1 es 0:

Otra forma de medir el rendimiento del modelo se denomina perplexity. La **perplexity** es una métrica utilizada para evaluar hasta qué punto un modelo de probabilidad predice una muestra. En el modelado del lenguaje, representa la **incertidumbre del modelo** al predecir el siguiente token de una secuencia.\
Por ejemplo, un valor de perplexity de 48725 significa que, cuando necesita predecir un token, no está seguro de cuál de los 48.725 tokens del vocabulario es el correcto.

import tiktoken
import torch
import torch.nn as nn
from torch.utils.data import Dataset, DataLoader


class GPTDatasetV1(Dataset):
def __init__(self, txt, tokenizer, max_length, stride):
self.input_ids = []
self.target_ids = []

# Tokenize the entire text
token_ids = tokenizer.encode(txt, allowed_special={"<|endoftext|>"})

# Use a sliding window to chunk the book into overlapping sequences of max_length
for i in range(0, len(token_ids) - max_length, stride):
input_chunk = token_ids[i:i + max_length]
target_chunk = token_ids[i + 1: i + max_length + 1]
self.input_ids.append(torch.tensor(input_chunk))
self.target_ids.append(torch.tensor(target_chunk))

def __len__(self):
return len(self.input_ids)

def __getitem__(self, idx):
return self.input_ids[idx], self.target_ids[idx]


def create_dataloader_v1(txt, batch_size=4, max_length=256,
stride=128, shuffle=True, drop_last=True, num_workers=0):
# Initialize the tokenizer
tokenizer = tiktoken.get_encoding("gpt2")

# Create dataset
dataset = GPTDatasetV1(txt, tokenizer, max_length, stride)

# Create dataloader
dataloader = DataLoader(
dataset, batch_size=batch_size, shuffle=shuffle, drop_last=drop_last, num_workers=num_workers)

return dataloader


class MultiHeadAttention(nn.Module):
def __init__(self, d_in, d_out, context_length, dropout, num_heads, qkv_bias=False):
super().__init__()
assert d_out % num_heads == 0, "d_out must be divisible by n_heads"

self.d_out = d_out
self.num_heads = num_heads
self.head_dim = d_out // num_heads  # Reduce the projection dim to match desired output dim

self.W_query = nn.Linear(d_in, d_out, bias=qkv_bias)
self.W_key = nn.Linear(d_in, d_out, bias=qkv_bias)
self.W_value = nn.Linear(d_in, d_out, bias=qkv_bias)
self.out_proj = nn.Linear(d_out, d_out)  # Linear layer to combine head outputs
self.dropout = nn.Dropout(dropout)
self.register_buffer('mask', torch.triu(torch.ones(context_length, context_length), diagonal=1))

def forward(self, x):
b, num_tokens, d_in = x.shape

keys = self.W_key(x)  # Shape: (b, num_tokens, d_out)
queries = self.W_query(x)
values = self.W_value(x)

# We implicitly split the matrix by adding a `num_heads` dimension
# Unroll last dim: (b, num_tokens, d_out) -> (b, num_tokens, num_heads, head_dim)
keys = keys.view(b, num_tokens, self.num_heads, self.head_dim)
values = values.view(b, num_tokens, self.num_heads, self.head_dim)
queries = queries.view(b, num_tokens, self.num_heads, self.head_dim)

# Transpose: (b, num_tokens, num_heads, head_dim) -> (b, num_heads, num_tokens, head_dim)
keys = keys.transpose(1, 2)
queries = queries.transpose(1, 2)
values = values.transpose(1, 2)

# Compute scaled dot-product attention (aka self-attention) with a causal mask
attn_scores = queries @ keys.transpose(2, 3)  # Dot product for each head

# Original mask truncated to the number of tokens and converted to boolean
mask_bool = self.mask.bool()[:num_tokens, :num_tokens]

# Use the mask to fill attention scores
attn_scores.masked_fill_(mask_bool, -torch.inf)

attn_weights = torch.softmax(attn_scores / keys.shape[-1]**0.5, dim=-1)
attn_weights = self.dropout(attn_weights)

# Shape: (b, num_tokens, num_heads, head_dim)
context_vec = (attn_weights @ values).transpose(1, 2)

# Combine heads, where self.d_out = self.num_heads * self.head_dim
context_vec = context_vec.reshape(b, num_tokens, self.d_out)
context_vec = self.out_proj(context_vec)  # optional projection

return context_vec


class LayerNorm(nn.Module):
def __init__(self, emb_dim):
super().__init__()
self.eps = 1e-5
self.scale = nn.Parameter(torch.ones(emb_dim))
self.shift = nn.Parameter(torch.zeros(emb_dim))

def forward(self, x):
mean = x.mean(dim=-1, keepdim=True)
var = x.var(dim=-1, keepdim=True, unbiased=False)
norm_x = (x - mean) / torch.sqrt(var + self.eps)
return self.scale * norm_x + self.shift


class GELU(nn.Module):
def __init__(self):
super().__init__()

def forward(self, x):
return 0.5 * x * (1 + torch.tanh(
torch.sqrt(torch.tensor(2.0 / torch.pi)) *
(x + 0.044715 * torch.pow(x, 3))
))


class FeedForward(nn.Module):
def __init__(self, cfg):
super().__init__()
self.layers = nn.Sequential(
nn.Linear(cfg["emb_dim"], 4 * cfg["emb_dim"]),
GELU(),
nn.Linear(4 * cfg["emb_dim"], cfg["emb_dim"]),
)

def forward(self, x):
return self.layers(x)


class TransformerBlock(nn.Module):
def __init__(self, cfg):
super().__init__()
self.att = MultiHeadAttention(
d_in=cfg["emb_dim"],
d_out=cfg["emb_dim"],
context_length=cfg["context_length"],
num_heads=cfg["n_heads"],
dropout=cfg["drop_rate"],
qkv_bias=cfg["qkv_bias"])
self.ff = FeedForward(cfg)
self.norm1 = LayerNorm(cfg["emb_dim"])
self.norm2 = LayerNorm(cfg["emb_dim"])
self.drop_shortcut = nn.Dropout(cfg["drop_rate"])

def forward(self, x):
# Shortcut connection for attention block
shortcut = x
x = self.norm1(x)
x = self.att(x)   # Shape [batch_size, num_tokens, emb_size]
x = self.drop_shortcut(x)
x = x + shortcut  # Add the original input back

# Shortcut connection for feed-forward block
shortcut = x
x = self.norm2(x)
x = self.ff(x)
x = self.drop_shortcut(x)
x = x + shortcut  # Add the original input back

return x


class GPTModel(nn.Module):
def __init__(self, cfg):
super().__init__()
self.tok_emb = nn.Embedding(cfg["vocab_size"], cfg["emb_dim"])
self.pos_emb = nn.Embedding(cfg["context_length"], cfg["emb_dim"])
self.drop_emb = nn.Dropout(cfg["drop_rate"])

self.trf_blocks = nn.Sequential(
*[TransformerBlock(cfg) for _ in range(cfg["n_layers"])])

self.final_norm = LayerNorm(cfg["emb_dim"])
self.out_head = nn.Linear(cfg["emb_dim"], cfg["vocab_size"], bias=False)

def forward(self, in_idx):
batch_size, seq_len = in_idx.shape
tok_embeds = self.tok_emb(in_idx)
pos_embeds = self.pos_emb(torch.arange(seq_len, device=in_idx.device))
x = tok_embeds + pos_embeds  # Shape [batch_size, num_tokens, emb_size]
x = self.drop_emb(x)
x = self.trf_blocks(x)
x = self.final_norm(x)
logits = self.out_head(x)
return logits

### Funciones `generate_text`

En una sección anterior, una función simplemente obtenía el **token más probable** después de obtener los logits. Sin embargo, esto significa que para cada entrada siempre se generará la misma salida, lo que la hace muy determinista.

La siguiente función `generate_text` aplicará los conceptos de `top-k`, `temperature` y `multinomial`.

- **`top-k`** significa que empezaremos a reducir a `-inf` todas las probabilidades de los tokens, excepto las de los k tokens principales. Por lo tanto, si k=3, antes de tomar una decisión solo los 3 tokens más probables tendrán una probabilidad diferente de `-inf`.
- **`temperature`** significa que cada probabilidad se dividirá por el valor de la temperatura. Un valor de `0.1` aumentará la probabilidad más alta en comparación con la más baja, mientras que una temperatura de `5`, por ejemplo, hará que la distribución sea más plana. Esto ayuda a mejorar la variación en las respuestas que queremos que tenga el LLM.
- Después de aplicar la temperatura, se aplica de nuevo una función **`softmax`** para hacer que todos los tokens restantes tengan una probabilidad total de 1.
- Finalmente, en lugar de elegir el token con la probabilidad más alta, se aplica la función **`multinomial`** para **predecir el siguiente token según las probabilidades finales**. Por lo tanto, si el token 1 tuviera una probabilidad del 70 %, el token 2 del 20 % y el token 3 del 10 %, el token 1 se seleccionaría el 70 % de las veces, el token 2 el 20 % de las veces y el token 3 el 10 % de las veces.
```python
# Generate text function
def generate_text(model, idx, max_new_tokens, context_size, temperature=0.0, top_k=None, eos_id=None):

# For-loop is the same as before: Get logits, and only focus on last time step
for _ in range(max_new_tokens):
idx_cond = idx[:, -context_size:]
with torch.no_grad():
logits = model(idx_cond)
logits = logits[:, -1, :]

# New: Filter logits with top_k sampling
if top_k is not None:
# Keep only top_k values
top_logits, _ = torch.topk(logits, top_k)
min_val = top_logits[:, -1]
logits = torch.where(logits < min_val, torch.tensor(float("-inf")).to(logits.device), logits)

# New: Apply temperature scaling
if temperature > 0.0:
logits = logits / temperature

# Apply softmax to get probabilities
probs = torch.softmax(logits, dim=-1)  # (batch_size, context_len)

# Sample from the distribution
idx_next = torch.multinomial(probs, num_samples=1)  # (batch_size, 1)

# Otherwise same as before: get idx of the vocab entry with the highest logits value
else:
idx_next = torch.argmax(logits, dim=-1, keepdim=True)  # (batch_size, 1)

if idx_next == eos_id:  # Stop generating early if end-of-sequence token is encountered and eos_id is specified
break

# Same as before: append sampled index to the running sequence
idx = torch.cat((idx, idx_next), dim=1)  # (batch_size, num_tokens+1)

return idx
```

> Existe una alternativa común a `top-k` llamada [**`top-p`**](https://en.wikipedia.org/wiki/Top-p_sampling), también conocida como nucleus sampling, que, en lugar de obtener las k muestras con mayor probabilidad, **organiza** todo el **vocabulario** resultante según sus probabilidades y las **suma** desde la probabilidad más alta hasta la más baja hasta que se **alcanza un umbral**.
>
> Después, **solo esas palabras** del vocabulario se tendrán en cuenta según sus probabilidades relativas.
>
> Esto permite no tener que seleccionar un número de `k` muestras, ya que el valor óptimo de k podría ser diferente en cada caso, sino **solo un umbral**.
>
> _Ten en cuenta que esta mejora no está incluida en el código anterior._

> [!TIP]
> Otra forma de mejorar el texto generado es utilizar **Beam search** en lugar de la búsqueda greedy utilizada en este ejemplo.\
> A diferencia de la búsqueda greedy, que selecciona la siguiente palabra más probable en cada paso y construye una única secuencia, **beam search mantiene un registro de las 𝑘 k secuencias parciales con mayor puntuación** (llamadas "beams") en cada paso. Al explorar varias posibilidades simultáneamente, equilibra la eficiencia y la calidad, aumentando las probabilidades de **encontrar una secuencia general mejor** que podría pasar desapercibida con el enfoque greedy debido a elecciones tempranas subóptimas.
>
> _Ten en cuenta que esta mejora no está incluida en el código anterior._

### Funciones de pérdida

La función **`calc_loss_batch`** calcula la entropía cruzada de una predicción de un único batch.\
La función **`calc_loss_loader`** obtiene la entropía cruzada de todos los batches y calcula la **entropía cruzada media**.
```python
# Define loss functions
def calc_loss_batch(input_batch, target_batch, model, device):
input_batch, target_batch = input_batch.to(device), target_batch.to(device)
logits = model(input_batch)
loss = torch.nn.functional.cross_entropy(logits.flatten(0, 1), target_batch.flatten())
return loss

def calc_loss_loader(data_loader, model, device, num_batches=None):
total_loss = 0.
if len(data_loader) == 0:
return float("nan")
elif num_batches is None:
num_batches = len(data_loader)
else:
# Reduce the number of batches to match the total number of batches in the data loader
# if num_batches exceeds the number of batches in the data loader
num_batches = min(num_batches, len(data_loader))
for i, (input_batch, target_batch) in enumerate(data_loader):
if i < num_batches:
loss = calc_loss_batch(input_batch, target_batch, model, device)
total_loss += loss.item()
else:
break
return total_loss / num_batches
```
> [!TIP]
> **Gradient clipping** es una técnica utilizada para mejorar la **estabilidad del entrenamiento** en redes neuronales grandes mediante el establecimiento de un **umbral máximo** para las magnitudes de los gradientes. Cuando los gradientes superan este `max_norm` predefinido, se reducen proporcionalmente para garantizar que las actualizaciones de los parámetros del modelo se mantengan dentro de un rango manejable, evitando problemas como la explosión de gradientes y asegurando un entrenamiento más controlado y estable.
>
> _Ten en cuenta que esta mejora no está incluida en el código anterior._

### Carga de datos

Las funciones `create_dataloader_v1` y `create_dataloader_v1` ya se explicaron en una sección anterior.

A partir de aquí, observa que se define que el 90 % del texto se utilizará para el entrenamiento, mientras que el 10 % se utilizará para la validación, y ambos conjuntos se almacenan en 2 data loaders diferentes.\
Ten en cuenta que, en ocasiones, parte del conjunto de datos también se reserva para un conjunto de testing, con el fin de evaluar mejor el rendimiento del modelo.

Ambos data loaders utilizan el mismo batch size, longitud máxima, stride y número de workers (0 en este caso).\
Las principales diferencias son los datos utilizados por cada uno y que el validador no descarta el último elemento ni mezcla los datos, ya que esto no es necesario para la validación.

Además, el hecho de que **stride sea tan grande como la longitud del contexto** significa que no habrá solapamiento entre los contextos utilizados para entrenar los datos (lo que reduce el overfitting, pero también el conjunto de datos de entrenamiento).

Asimismo, observa que el batch size en este caso es 2 para dividir los datos en 2 batches. El objetivo principal es permitir el procesamiento paralelo y reducir el consumo por batch.

### Funciones de training

La función `generate_and_print_sample` simplemente obtiene un contexto y genera algunos tokens para hacerse una idea de qué tan bueno es el modelo en ese momento. `train_model_simple` la llama en cada paso.

La función `evaluate_model` se llama con la frecuencia indicada en la función de training y se utiliza para medir el train loss y el validation loss en ese punto del training del modelo.

Después, la función principal `train_model_simple` es la que realmente entrena el modelo. Espera:

- El train data loader (con los datos ya separados y preparados para el training)
- El validator loader
- El **optimizer** que se utilizará durante el training: Esta es la función que usará los gradientes y actualizará los parámetros para reducir el loss. En este caso, como verás, se utiliza `AdamW`, pero existen muchos más.
- `optimizer.zero_grad()` se llama en cada ronda para restablecer los gradientes y evitar que se acumulen.
- El parámetro **`lr`** es el **learning rate**, que determina el **tamaño de los pasos** realizados durante el proceso de optimización al actualizar los parámetros del modelo. Un learning rate **más pequeño** significa que el optimizer realiza **actualizaciones más pequeñas** en los pesos, lo que puede conducir a una convergencia más **precisa**, pero podría **ralentizar** el training. Un learning rate **más grande** puede acelerar el training, pero **corre el riesgo de sobrepasar** el mínimo de la función de loss (**saltar por encima** del punto donde se minimiza la función de loss).
- **Weight Decay** modifica el paso de **Loss Calculation** añadiendo un término adicional que penaliza los pesos grandes. Esto anima al optimizer a encontrar soluciones con pesos más pequeños, equilibrando un buen ajuste de los datos y la simplicidad del modelo, lo que evita el overfitting en modelos de machine learning al disuadir al modelo de asignar demasiada importancia a una sola feature.
- Los optimizers tradicionales, como SGD con regularización L2, acoplan el weight decay al gradiente del loss. Sin embargo, **AdamW** (una variante del optimizer Adam) desacopla el weight decay de la actualización del gradiente, lo que produce una regularización más eficaz.
- El device que se utilizará para el training
- El número de epochs: Número de veces que se recorren los datos de training
- La frecuencia de evaluación: La frecuencia con la que se llamará a `evaluate_model`
- La iteración de evaluación: El número de batches que se utilizarán para evaluar el estado actual del modelo al llamar a `generate_and_print_sample`
- El contexto inicial: La frase inicial que se utilizará al llamar a `generate_and_print_sample`
- El tokenizer
```python
# Functions to train the data
def train_model_simple(model, train_loader, val_loader, optimizer, device, num_epochs,
eval_freq, eval_iter, start_context, tokenizer):
# Initialize lists to track losses and tokens seen
train_losses, val_losses, track_tokens_seen = [], [], []
tokens_seen, global_step = 0, -1

# Main training loop
for epoch in range(num_epochs):
model.train()  # Set model to training mode

for input_batch, target_batch in train_loader:
optimizer.zero_grad() # Reset loss gradients from previous batch iteration
loss = calc_loss_batch(input_batch, target_batch, model, device)
loss.backward() # Calculate loss gradients
optimizer.step() # Update model weights using loss gradients
tokens_seen += input_batch.numel()
global_step += 1

# Optional evaluation step
if global_step % eval_freq == 0:
train_loss, val_loss = evaluate_model(
model, train_loader, val_loader, device, eval_iter)
train_losses.append(train_loss)
val_losses.append(val_loss)
track_tokens_seen.append(tokens_seen)
print(f"Ep {epoch+1} (Step {global_step:06d}): "
f"Train loss {train_loss:.3f}, Val loss {val_loss:.3f}")

# Print a sample text after each epoch
generate_and_print_sample(
model, tokenizer, device, start_context
)

return train_losses, val_losses, track_tokens_seen


def evaluate_model(model, train_loader, val_loader, device, eval_iter):
model.eval() # Set in eval mode to avoid dropout
with torch.no_grad():
train_loss = calc_loss_loader(train_loader, model, device, num_batches=eval_iter)
val_loss = calc_loss_loader(val_loader, model, device, num_batches=eval_iter)
model.train() # Back to training model applying all the configurations
return train_loss, val_loss


def generate_and_print_sample(model, tokenizer, device, start_context):
model.eval() # Set in eval mode to avoid dropout
context_size = model.pos_emb.weight.shape[0]
encoded = text_to_token_ids(start_context, tokenizer).to(device)
with torch.no_grad():
token_ids = generate_text(
model=model, idx=encoded,
max_new_tokens=50, context_size=context_size
)
decoded_text = token_ids_to_text(token_ids, tokenizer)
print(decoded_text.replace("\n", " "))  # Compact print format
model.train() # Back to training model applying all the configurations
```
> [!TIP]
> Para mejorar la tasa de aprendizaje, existen un par de técnicas relevantes llamadas **linear warmup** y **cosine decay**.
>
> **Linear warmup** consiste en definir una tasa de aprendizaje inicial y una máxima, y actualizarla constantemente después de cada epoch. Esto se debe a que comenzar el entrenamiento con actualizaciones de pesos más pequeñas reduce el riesgo de que el modelo encuentre actualizaciones grandes y desestabilizadoras durante su fase de entrenamiento.\
> **Cosine decay** es una técnica que **reduce gradualmente la tasa de aprendizaje** siguiendo una curva de medio coseno **después de la fase de warmup**, ralentizando las actualizaciones de pesos para **minimizar el riesgo de sobrepasar** los mínimos de la función de pérdida y garantizar la estabilidad del entrenamiento en sus últimas fases.
>
> _Ten en cuenta que estas mejoras no están incluidas en el código anterior._

### Mejoras prácticas modernas del entrenamiento

El bucle anterior es suficiente para comprender la mecánica, pero el pre-entrenamiento moderno suele añadir varias optimizaciones para mejorar el rendimiento y la estabilidad:

- **Precisión mixta (`bfloat16` / `float16`)** para reducir el uso de memoria y acelerar las multiplicaciones de matrices. En la práctica, `bfloat16` suele ser la opción preferida en las GPU de centros de datos NVIDIA recientes porque mantiene un rango de exponentes más amplio y normalmente no necesita escalado de gradientes, mientras que `float16` suele utilizar un `GradScaler`.
- **Acumulación de gradientes** para simular un tamaño de batch global mayor cuando el batch completo no cabe en la VRAM.
- **Clipping de gradientes** (por ejemplo, `clip_grad_norm_`) para evitar actualizaciones explosivas, especialmente al principio del entrenamiento.
- **Activation checkpointing** para intercambiar cómputo adicional por un uso de memoria mucho menor, recalculando parte del forward pass durante la retropropagación.
- **Attention fusionada / eficiente en memoria** (`scaled_dot_product_attention`) para reducir el coste de la atención en ventanas de contexto más largas.
- **Entrenamiento/checkpointing sharded** (por ejemplo, FSDP / distributed checkpoints) cuando el modelo o los estados del optimizador ya no caben en una sola GPU.

Un ejemplo mínimo que combina algunas de estas ideas en PyTorch sería:
```python
accum_steps = 8
scaler = torch.amp.GradScaler("cuda")
optimizer.zero_grad(set_to_none=True)

for step, (input_batch, target_batch) in enumerate(train_loader, start=1):
with torch.autocast(device_type="cuda", dtype=torch.float16):
loss = calc_loss_batch(input_batch, target_batch, model, device)
loss = loss / accum_steps

scaler.scale(loss).backward()

if step % accum_steps == 0:
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
scaler.step(optimizer)
scaler.update()
optimizer.zero_grad(set_to_none=True)
```
> [!WARNING]
> Al hacer pre-training desde checkpoints o repositorios de terceros, trata la **ruta de carga como parte de la superficie de ataque**. Prefiere cargar archivos `state_dict` o `safetensors` en lugar de objetos Python serializados completos, mantén `trust_remote_code=False` a menos que hayas auditado el repositorio y consulta [../AI-Models-RCE.md](../AI-Models-RCE.md) para ver ejemplos de RCE durante la carga de modelos.

### Carga de checkpoints más segura y eficiente en cuanto a memoria

En el caso de checkpoints grandes, la secuencia ingenua `torch.load(...)` + `load_state_dict(...)` puede duplicar temporalmente la memoria (parámetros inicializados + pesos cargados). Las versiones más recientes de PyTorch han añadido algunas opciones muy útiles. A partir de PyTorch 2.6, `weights_only=True` es el valor predeterminado de `torch.load()` cuando no se proporciona ningún módulo pickle personalizado, pero sigue siendo recomendable especificarlo explícitamente en los ejemplos:

- `weights_only=True`: restringe la deserialización a tensores / objetos primitivos en lugar de objetos Python serializados arbitrarios.
- `mmap=True`: asigna mediante memory mapping los almacenamientos de tensores desde el disco en lugar de leerlo todo de forma inmediata en la RAM.
- `with torch.device("meta"):`: crea primero el módulo con tensores vacíos.
- `load_state_dict(..., assign=True)`: asigna directamente los tensores cargados al módulo inicializado con meta.
```python
state_dict = torch.load(
"model.pth",
map_location="cpu",
weights_only=True,
mmap=True,
)

with torch.device("meta"):
model = GPTModel(GPT_CONFIG_124M)

model.load_state_dict(state_dict, assign=True)
model.to(device)
model.eval()
```
Esto es especialmente útil al cargar checkpoints de varios GB en una workstation, ya que reduce los picos de RAM durante el inicio.

from transformers import AutoModelForCausalLM, AutoTokenizer

model_id = "google/gemma-7b"
