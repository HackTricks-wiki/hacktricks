## Génération de texte

Pour entraîner un modèle, il faut que ce modèle puisse générer de nouveaux tokens. Nous comparerons ensuite les tokens générés avec ceux attendus afin d'entraîner le modèle à **apprendre les tokens qu'il doit générer**.

Comme dans les exemples précédents, nous avons déjà prédit certains tokens ; il est donc possible de réutiliser cette fonction à cette fin.

> [!TIP]
> L'objectif de cette sixième phase est très simple : **entraîner le modèle from scratch**. Pour cela, l'architecture LLM précédente sera utilisée avec des boucles parcourant les jeux de données, ainsi que les fonctions de perte et l'optimizer définis, afin d'entraîner tous les paramètres du modèle.

## Évaluation du texte

Pour effectuer un entraînement correct, il est nécessaire de mesurer et de vérifier les prédictions obtenues par rapport au token attendu. L'objectif de l'entraînement est de maximiser la probabilité du token correct, ce qui implique d'augmenter sa probabilité par rapport aux autres tokens.

Afin de maximiser la probabilité du token correct, les poids du modèle doivent être modifiés de manière à maximiser cette probabilité. La mise à jour des poids s'effectue via la **backpropagation**. Cela nécessite une **fonction de perte à maximiser**. Dans ce cas, la fonction sera la **différence entre la prédiction effectuée et celle souhaitée**.

Cependant, au lieu de travailler avec les prédictions brutes, le système utilise un logarithme de base n. Ainsi, si la prédiction actuelle du token attendu était de 7.4541e-05, le logarithme naturel (base *e*) de **7.4541e-05** est approximativement égal à **-9.5042**.\
Ensuite, pour chaque entrée ayant, par exemple, une longueur de contexte de 5 tokens, le modèle devra prédire 5 tokens : les 4 premiers tokens correspondent au dernier token de l'entrée et le cinquième est le token prédit. Par conséquent, pour chaque entrée, nous aurons 5 prédictions dans ce cas (même si les 4 premières étaient présentes dans l'entrée, le modèle ne le sait pas), avec 5 tokens attendus et donc 5 probabilités à maximiser.

Ainsi, après avoir appliqué le logarithme naturel à chaque prédiction, la **moyenne** est calculée, le **symbole moins est supprimé** (ce qui est appelé _cross entropy loss_) et c'est le **nombre à réduire autant que possible vers 0**, car le logarithme naturel de 1 est égal à 0 :

Une autre manière de mesurer la qualité du modèle est appelée perplexité. La **perplexité** est une métrique utilisée pour évaluer la capacité d'un modèle probabiliste à prédire un échantillon. En language modelling, elle représente l'**incertitude du modèle** lorsqu'il prédit le token suivant d'une séquence.\
Par exemple, une valeur de perplexité de 48725 signifie que, lorsqu'il doit prédire un token, le modèle ne sait pas lequel des 48 725 tokens du vocabulaire est le bon.

import tiktoken
import torch
import torch.nn as nn
from torch.utils.data import Dataset, DataLoader


class GPTDatasetV1(Dataset):
def __init__(self, txt, tokenizer, max_length, stride):
self.input_ids = []
self.target_ids = []

# Tokenize the entire text
token_ids = tokenizer.encode(txt, allowed_special={"<|endoftext|>"})

# Use a sliding window to chunk the book into overlapping sequences of max_length
for i in range(0, len(token_ids) - max_length, stride):
input_chunk = token_ids[i:i + max_length]
target_chunk = token_ids[i + 1: i + max_length + 1]
self.input_ids.append(torch.tensor(input_chunk))
self.target_ids.append(torch.tensor(target_chunk))

def __len__(self):
return len(self.input_ids)

def __getitem__(self, idx):
return self.input_ids[idx], self.target_ids[idx]


def create_dataloader_v1(txt, batch_size=4, max_length=256,
stride=128, shuffle=True, drop_last=True, num_workers=0):
# Initialize the tokenizer
tokenizer = tiktoken.get_encoding("gpt2")

# Create dataset
dataset = GPTDatasetV1(txt, tokenizer, max_length, stride)

# Create dataloader
dataloader = DataLoader(
dataset, batch_size=batch_size, shuffle=shuffle, drop_last=drop_last, num_workers=num_workers)

return dataloader


class MultiHeadAttention(nn.Module):
def __init__(self, d_in, d_out, context_length, dropout, num_heads, qkv_bias=False):
super().__init__()
assert d_out % num_heads == 0, "d_out must be divisible by n_heads"

self.d_out = d_out
self.num_heads = num_heads
self.head_dim = d_out // num_heads  # Reduce the projection dim to match desired output dim

self.W_query = nn.Linear(d_in, d_out, bias=qkv_bias)
self.W_key = nn.Linear(d_in, d_out, bias=qkv_bias)
self.W_value = nn.Linear(d_in, d_out, bias=qkv_bias)
self.out_proj = nn.Linear(d_out, d_out)  # Linear layer to combine head outputs
self.dropout = nn.Dropout(dropout)
self.register_buffer('mask', torch.triu(torch.ones(context_length, context_length), diagonal=1))

def forward(self, x):
b, num_tokens, d_in = x.shape

keys = self.W_key(x)  # Shape: (b, num_tokens, d_out)
queries = self.W_query(x)
values = self.W_value(x)

# We implicitly split the matrix by adding a `num_heads` dimension
# Unroll last dim: (b, num_tokens, d_out) -> (b, num_tokens, num_heads, head_dim)
keys = keys.view(b, num_tokens, self.num_heads, self.head_dim)
values = values.view(b, num_tokens, self.num_heads, self.head_dim)
queries = queries.view(b, num_tokens, self.num_heads, self.head_dim)

# Transpose: (b, num_tokens, num_heads, head_dim) -> (b, num_heads, num_tokens, head_dim)
keys = keys.transpose(1, 2)
queries = queries.transpose(1, 2)
values = values.transpose(1, 2)

# Compute scaled dot-product attention (aka self-attention) with a causal mask
attn_scores = queries @ keys.transpose(2, 3)  # Dot product for each head

# Original mask truncated to the number of tokens and converted to boolean
mask_bool = self.mask.bool()[:num_tokens, :num_tokens]

# Use the mask to fill attention scores
attn_scores.masked_fill_(mask_bool, -torch.inf)

attn_weights = torch.softmax(attn_scores / keys.shape[-1]**0.5, dim=-1)
attn_weights = self.dropout(attn_weights)

# Shape: (b, num_tokens, num_heads, head_dim)
context_vec = (attn_weights @ values).transpose(1, 2)

# Combine heads, where self.d_out = self.num_heads * self.head_dim
context_vec = context_vec.reshape(b, num_tokens, self.d_out)
context_vec = self.out_proj(context_vec)  # optional projection

return context_vec


class LayerNorm(nn.Module):
def __init__(self, emb_dim):
super().__init__()
self.eps = 1e-5
self.scale = nn.Parameter(torch.ones(emb_dim))
self.shift = nn.Parameter(torch.zeros(emb_dim))

def forward(self, x):
mean = x.mean(dim=-1, keepdim=True)
var = x.var(dim=-1, keepdim=True, unbiased=False)
norm_x = (x - mean) / torch.sqrt(var + self.eps)
return self.scale * norm_x + self.shift


class GELU(nn.Module):
def __init__(self):
super().__init__()

def forward(self, x):
return 0.5 * x * (1 + torch.tanh(
torch.sqrt(torch.tensor(2.0 / torch.pi)) *
(x + 0.044715 * torch.pow(x, 3))
))


class FeedForward(nn.Module):
def __init__(self, cfg):
super().__init__()
self.layers = nn.Sequential(
nn.Linear(cfg["emb_dim"], 4 * cfg["emb_dim"]),
GELU(),
nn.Linear(4 * cfg["emb_dim"], cfg["emb_dim"]),
)

def forward(self, x):
return self.layers(x)


class TransformerBlock(nn.Module):
def __init__(self, cfg):
super().__init__()
self.att = MultiHeadAttention(
d_in=cfg["emb_dim"],
d_out=cfg["emb_dim"],
context_length=cfg["context_length"],
num_heads=cfg["n_heads"],
dropout=cfg["drop_rate"],
qkv_bias=cfg["qkv_bias"])
self.ff = FeedForward(cfg)
self.norm1 = LayerNorm(cfg["emb_dim"])
self.norm2 = LayerNorm(cfg["emb_dim"])
self.drop_shortcut = nn.Dropout(cfg["drop_rate"])

def forward(self, x):
# Shortcut connection for attention block
shortcut = x
x = self.norm1(x)
x = self.att(x)   # Shape [batch_size, num_tokens, emb_size]
x = self.drop_shortcut(x)
x = x + shortcut  # Add the original input back

# Shortcut connection for feed-forward block
shortcut = x
x = self.norm2(x)
x = self.ff(x)
x = self.drop_shortcut(x)
x = x + shortcut  # Add the original input back

return x


class GPTModel(nn.Module):
def __init__(self, cfg):
super().__init__()
self.tok_emb = nn.Embedding(cfg["vocab_size"], cfg["emb_dim"])
self.pos_emb = nn.Embedding(cfg["context_length"], cfg["emb_dim"])
self.drop_emb = nn.Dropout(cfg["drop_rate"])

self.trf_blocks = nn.Sequential(
*[TransformerBlock(cfg) for _ in range(cfg["n_layers"])])

self.final_norm = LayerNorm(cfg["emb_dim"])
self.out_head = nn.Linear(cfg["emb_dim"], cfg["vocab_size"], bias=False)

def forward(self, in_idx):
batch_size, seq_len = in_idx.shape
tok_embeds = self.tok_emb(in_idx)
pos_embeds = self.pos_emb(torch.arange(seq_len, device=in_idx.device))
x = tok_embeds + pos_embeds  # Shape [batch_size, num_tokens, emb_size]
x = self.drop_emb(x)
x = self.trf_blocks(x)
x = self.final_norm(x)
logits = self.out_head(x)
return logits

### Fonctions de génération de texte

Dans une section précédente, une fonction récupérait simplement le **token le plus probable** après avoir obtenu les logits. Cependant, cela signifie que la même sortie sera toujours générée pour chaque entrée, ce qui rend le résultat très déterministe.

La fonction `generate_text` suivante applique les concepts de `top-k`, de `temperature` et de `multinomial`.

- Le **`top-k`** signifie que nous commencerons à remplacer par `-inf` les probabilités de tous les tokens, à l'exception des k tokens les mieux classés. Ainsi, si k=3, avant de prendre une décision, seuls les 3 tokens les plus probables auront une probabilité différente de `-inf`.
- La **`temperature`** signifie que chaque probabilité sera divisée par la valeur de temperature. Une valeur de `0.1` accentuera la probabilité la plus élevée par rapport à la plus faible, tandis qu'une temperature de `5`, par exemple, rendra la distribution plus uniforme. Cela permet d'améliorer la variation des réponses que nous souhaitons que le LLM produise.
- Après l'application de la temperature, une fonction **`softmax`** est de nouveau appliquée afin que tous les tokens restants aient une probabilité totale de 1.
- Enfin, au lieu de sélectionner le token ayant la probabilité la plus élevée, la fonction **`multinomial`** est appliquée pour **prédire le token suivant selon les probabilités finales**. Ainsi, si le token 1 a une probabilité de 70 %, le token 2 de 20 % et le token 3 de 10 %, le token 1 sera sélectionné dans 70 % des cas, le token 2 dans 20 % des cas et le token 3 dans 10 % des cas.
```python
# Generate text function
def generate_text(model, idx, max_new_tokens, context_size, temperature=0.0, top_k=None, eos_id=None):

# For-loop is the same as before: Get logits, and only focus on last time step
for _ in range(max_new_tokens):
idx_cond = idx[:, -context_size:]
with torch.no_grad():
logits = model(idx_cond)
logits = logits[:, -1, :]

# New: Filter logits with top_k sampling
if top_k is not None:
# Keep only top_k values
top_logits, _ = torch.topk(logits, top_k)
min_val = top_logits[:, -1]
logits = torch.where(logits < min_val, torch.tensor(float("-inf")).to(logits.device), logits)

# New: Apply temperature scaling
if temperature > 0.0:
logits = logits / temperature

# Apply softmax to get probabilities
probs = torch.softmax(logits, dim=-1)  # (batch_size, context_len)

# Sample from the distribution
idx_next = torch.multinomial(probs, num_samples=1)  # (batch_size, 1)

# Otherwise same as before: get idx of the vocab entry with the highest logits value
else:
idx_next = torch.argmax(logits, dim=-1, keepdim=True)  # (batch_size, 1)

if idx_next == eos_id:  # Stop generating early if end-of-sequence token is encountered and eos_id is specified
break

# Same as before: append sampled index to the running sequence
idx = torch.cat((idx, idx_next), dim=1)  # (batch_size, num_tokens+1)

return idx
```

> Il existe une alternative courante à `top-k` appelée [**`top-p`**](https://en.wikipedia.org/wiki/Top-p_sampling), également connue sous le nom de nucleus sampling, qui, au lieu d'obtenir k échantillons ayant la probabilité la plus élevée, **organise** tout le **vocabulaire** obtenu par probabilité et les **additionne** de la probabilité la plus élevée à la plus faible jusqu'à ce qu'un **seuil soit atteint**.
>
> Ensuite, **seuls ces mots** du vocabulaire seront pris en compte en fonction de leurs probabilités relatives.
>
> Cela évite de devoir sélectionner un nombre de `k` échantillons, car la valeur optimale de k peut varier selon les cas, et permet de sélectionner **uniquement un seuil**.
>
> _Notez que cette amélioration n'est pas incluse dans le code précédent._

> [!TIP]
> Une autre façon d'améliorer le texte généré consiste à utiliser la **Beam search** au lieu de la recherche gloutonne utilisée dans cet exemple.\
> Contrairement à la recherche gloutonne, qui sélectionne le mot suivant le plus probable à chaque étape et construit une séquence unique, la **beam search conserve les 𝑘 k séquences partielles ayant les scores les plus élevés** (appelées « beams ») à chaque étape. En explorant plusieurs possibilités simultanément, elle équilibre efficacité et qualité, augmentant les chances de **trouver une meilleure** séquence globale qui pourrait être manquée par l'approche gloutonne en raison de choix précoces sous-optimaux.
>
> _Notez que cette amélioration n'est pas incluse dans le code précédent._

### Fonctions de perte

La fonction **`calc_loss_batch`** calcule la cross entropy d'une prédiction pour un seul batch.\
La fonction **`calc_loss_loader`** obtient la cross entropy de tous les batches et calcule la **cross entropy moyenne**.
```python
# Define loss functions
def calc_loss_batch(input_batch, target_batch, model, device):
input_batch, target_batch = input_batch.to(device), target_batch.to(device)
logits = model(input_batch)
loss = torch.nn.functional.cross_entropy(logits.flatten(0, 1), target_batch.flatten())
return loss

def calc_loss_loader(data_loader, model, device, num_batches=None):
total_loss = 0.
if len(data_loader) == 0:
return float("nan")
elif num_batches is None:
num_batches = len(data_loader)
else:
# Reduce the number of batches to match the total number of batches in the data loader
# if num_batches exceeds the number of batches in the data loader
num_batches = min(num_batches, len(data_loader))
for i, (input_batch, target_batch) in enumerate(data_loader):
if i < num_batches:
loss = calc_loss_batch(input_batch, target_batch, model, device)
total_loss += loss.item()
else:
break
return total_loss / num_batches
```
> [!TIP]
> **Gradient clipping** est une technique utilisée pour améliorer la **stabilité de l'entraînement** des grands réseaux neuronaux en définissant un **seuil maximal** pour l'amplitude des gradients. Lorsque les gradients dépassent ce `max_norm` prédéfini, ils sont réduits proportionnellement afin de garantir que les mises à jour des paramètres du modèle restent dans une plage gérable, évitant ainsi des problèmes tels que l'explosion des gradients et garantissant un entraînement plus contrôlé et stable.
>
> _Notez que cette amélioration n'est pas incluse dans le code précédent._

Les deux data loaders utilisent le même batch size, la même longueur maximale, le même stride et le même nombre de workers (0 dans ce cas).\
Les principales différences concernent les données utilisées par chacun, ainsi que le fait que le validator ne supprime pas le dernier élément et ne mélange pas les données, car cela n'est pas nécessaire à des fins de validation.

De plus, le fait que **stride soit aussi grand que la longueur du contexte** signifie qu'il n'y aura aucun chevauchement entre les contextes utilisés pour entraîner les données (ce qui réduit l'overfitting, mais aussi le dataset d'entraînement).

Notez également que le batch size est ici de 2 afin de diviser les données en 2 batches. L'objectif principal est de permettre le traitement parallèle et de réduire la consommation par batch.

### Fonctions d'entraînement

La fonction `generate_and_print_sample` récupère simplement un contexte et génère quelques tokens afin d'évaluer la qualité du modèle à ce stade. Elle est appelée par `train_model_simple` à chaque étape.

La fonction `evaluate_model` est appelée aussi fréquemment que l'indique la fonction d'entraînement et sert à mesurer la perte d'entraînement et la perte de validation à ce stade de l'entraînement du modèle.

Ensuite, la grande fonction `train_model_simple` est celle qui entraîne réellement le modèle. Elle attend :

- Le data loader d'entraînement (avec les données déjà séparées et préparées pour l'entraînement)
- Le validator loader
- L'**optimizer** à utiliser pendant l'entraînement : c'est la fonction qui utilise les gradients et met à jour les paramètres afin de réduire la perte. Dans ce cas, comme vous le verrez, `AdamW` est utilisé, mais il en existe beaucoup d'autres.
- `optimizer.zero_grad()` est appelé à chaque tour pour réinitialiser les gradients et éviter leur accumulation.
- Le paramètre **`lr`** est le **learning rate**, qui détermine la **taille des étapes** effectuées pendant le processus d'optimisation lors de la mise à jour des paramètres du modèle. Un learning rate **plus faible** signifie que l'optimizer effectue des mises à jour **plus petites** des poids, ce qui peut conduire à une convergence plus **précise**, mais peut aussi **ralentir** l'entraînement. Un learning rate **plus élevé** peut accélérer l'entraînement, mais **risque de dépasser** le minimum de la fonction de perte (**sauter par-dessus** le point où la fonction de perte est minimisée).
- Le **Weight Decay** modifie l'étape de **Loss Calculation** en ajoutant un terme supplémentaire qui pénalise les poids élevés. Cela encourage l'optimizer à trouver des solutions avec des poids plus faibles, en équilibrant l'ajustement des données et la simplicité du modèle. Il empêche le surapprentissage dans les modèles de machine learning en décourageant le modèle d'accorder trop d'importance à une seule caractéristique.
- Les optimizers traditionnels comme SGD avec régularisation L2 couplent le weight decay au gradient de la fonction de perte. Cependant, **AdamW** (une variante de l'optimizer Adam) découple le weight decay de la mise à jour du gradient, ce qui permet une régularisation plus efficace.
- Le device à utiliser pour l'entraînement
- Le nombre d'epochs : nombre de fois où parcourir les données d'entraînement
- La fréquence d'évaluation : fréquence à laquelle appeler `evaluate_model`
- Le nombre d'itérations d'évaluation : nombre de batches à utiliser pour évaluer l'état actuel du modèle lors de l'appel à `generate_and_print_sample`
- Le contexte de départ : phrase de départ à utiliser lors de l'appel à `generate_and_print_sample`
- Le tokenizer
```python
# Functions to train the data
def train_model_simple(model, train_loader, val_loader, optimizer, device, num_epochs,
eval_freq, eval_iter, start_context, tokenizer):
# Initialize lists to track losses and tokens seen
train_losses, val_losses, track_tokens_seen = [], [], []
tokens_seen, global_step = 0, -1

# Main training loop
for epoch in range(num_epochs):
model.train()  # Set model to training mode

for input_batch, target_batch in train_loader:
optimizer.zero_grad() # Reset loss gradients from previous batch iteration
loss = calc_loss_batch(input_batch, target_batch, model, device)
loss.backward() # Calculate loss gradients
optimizer.step() # Update model weights using loss gradients
tokens_seen += input_batch.numel()
global_step += 1

# Optional evaluation step
if global_step % eval_freq == 0:
train_loss, val_loss = evaluate_model(
model, train_loader, val_loader, device, eval_iter)
train_losses.append(train_loss)
val_losses.append(val_loss)
track_tokens_seen.append(tokens_seen)
print(f"Ep {epoch+1} (Step {global_step:06d}): "
f"Train loss {train_loss:.3f}, Val loss {val_loss:.3f}")

# Print a sample text after each epoch
generate_and_print_sample(
model, tokenizer, device, start_context
)

return train_losses, val_losses, track_tokens_seen


def evaluate_model(model, train_loader, val_loader, device, eval_iter):
model.eval() # Set in eval mode to avoid dropout
with torch.no_grad():
train_loss = calc_loss_loader(train_loader, model, device, num_batches=eval_iter)
val_loss = calc_loss_loader(val_loader, model, device, num_batches=eval_iter)
model.train() # Back to training model applying all the configurations
return train_loss, val_loss


def generate_and_print_sample(model, tokenizer, device, start_context):
model.eval() # Set in eval mode to avoid dropout
context_size = model.pos_emb.weight.shape[0]
encoded = text_to_token_ids(start_context, tokenizer).to(device)
with torch.no_grad():
token_ids = generate_text(
model=model, idx=encoded,
max_new_tokens=50, context_size=context_size
)
decoded_text = token_ids_to_text(token_ids, tokenizer)
print(decoded_text.replace("\n", " "))  # Compact print format
model.train() # Back to training model applying all the configurations
```
> [!TIP]
> Pour améliorer le learning rate, il existe quelques techniques pertinentes appelées **linear warmup** et **cosine decay.**
>
> Le **linear warmup** consiste à définir un learning rate initial et un learning rate maximal, puis à le mettre à jour régulièrement après chaque epoch. Cela s'explique par le fait que commencer l'entraînement avec de plus petites mises à jour des poids réduit le risque que le modèle rencontre de grandes mises à jour déstabilisantes pendant sa phase d'entraînement.\
> Le **cosine decay** est une technique qui **réduit progressivement le learning rate** en suivant une courbe en demi-cosinus **après la phase de warmup**, ralentissant les mises à jour des poids afin de **minimiser le risque de dépasser** les minima de la loss et d'assurer la stabilité de l'entraînement lors des phases ultérieures.
>
> _Notez que ces améliorations ne sont pas incluses dans le code précédent._

### Améliorations pratiques modernes de l'entraînement

La boucle précédente suffit pour comprendre les mécanismes, mais le pré-entraînement moderne ajoute généralement plusieurs optimisations afin d'améliorer le débit et la stabilité :

- **Précision mixte (`bfloat16` / `float16`)** pour réduire l'utilisation de la mémoire et accélérer les multiplications matricielles. En pratique, `bfloat16` est généralement préféré sur les GPU NVIDIA datacenter récents, car il conserve une plage d'exposants plus étendue et ne nécessite habituellement pas de mise à l'échelle des gradients, tandis que `float16` utilise couramment un `GradScaler`.
- **Accumulation des gradients** pour simuler une taille de batch globale plus importante lorsque le batch complet ne tient pas dans la VRAM.
- **Clipping des gradients** (par exemple `clip_grad_norm_`) pour empêcher les mises à jour explosives, particulièrement au début de l'entraînement.
- **Activation checkpointing** pour échanger davantage de calcul contre une utilisation de la mémoire beaucoup plus faible, en recalculant une partie de la passe forward pendant la backpropagation.
- **Attention fused / économe en mémoire** (`scaled_dot_product_attention`) pour réduire le coût de l'attention avec des fenêtres de contexte plus longues.
- **Entraînement/checkpointing sharded** (par exemple FSDP / distributed checkpoints) lorsque le modèle ou les états de l'optimizer ne tiennent plus sur un seul GPU.

Un exemple minimal dans PyTorch combinant certaines de ces idées ressemble à ceci :
```python
accum_steps = 8
scaler = torch.amp.GradScaler("cuda")
optimizer.zero_grad(set_to_none=True)

for step, (input_batch, target_batch) in enumerate(train_loader, start=1):
with torch.autocast(device_type="cuda", dtype=torch.float16):
loss = calc_loss_batch(input_batch, target_batch, model, device)
loss = loss / accum_steps

scaler.scale(loss).backward()

if step % accum_steps == 0:
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
scaler.step(optimizer)
scaler.update()
optimizer.zero_grad(set_to_none=True)
```
> [!WARNING]
> Lors d’un pre-training à partir de checkpoints ou de repositories tiers, considérez le **chemin de chargement comme faisant partie de la surface d’attaque**. Préférez charger des fichiers `state_dict` ou `safetensors` plutôt que des objets Python entiers sérialisés avec pickle, gardez `trust_remote_code=False` sauf si vous avez audité le repository, et consultez [../AI-Models-RCE.md](../AI-Models-RCE.md) pour des exemples de RCE lors du chargement de modèles.

### Chargement de checkpoints plus sûr et plus économe en mémoire

Pour les checkpoints volumineux, la séquence naïve `torch.load(...)` + `load_state_dict(...)` peut temporairement dupliquer la mémoire (paramètres initialisés + poids chargés). Les versions plus récentes de PyTorch ont ajouté des options très utiles. Depuis PyTorch 2.6, `weights_only=True` est la valeur par défaut de `torch.load()` lorsqu’aucun module pickle personnalisé n’est fourni, mais il reste préférable de l’indiquer explicitement dans les exemples :

- `weights_only=True` : limite la désérialisation aux tenseurs / objets primitifs plutôt qu’à des objets Python arbitraires sérialisés avec pickle.
- `mmap=True` : mappe en mémoire les stockages de tenseurs depuis le disque au lieu de tout charger immédiatement dans la RAM.
- `with torch.device("meta"):` : crée d’abord le module avec des tenseurs vides.
- `load_state_dict(..., assign=True)` : assigne directement les tenseurs chargés au module initialisé avec `meta`.
```python
state_dict = torch.load(
"model.pth",
map_location="cpu",
weights_only=True,
mmap=True,
)

with torch.device("meta"):
model = GPTModel(GPT_CONFIG_124M)

model.load_state_dict(state_dict, assign=True)
model.to(device)
model.eval()
```
Ceci est particulièrement utile lors du chargement de checkpoints de plusieurs Go sur une station de travail, car cela réduit les pics de RAM au démarrage.
