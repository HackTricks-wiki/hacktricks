## Generazione di testo

Per addestrare un modello, avremo bisogno che sia in grado di generare nuovi token. Quindi confronteremo i token generati con quelli attesi, allo scopo di addestrare il modello a **imparare i token che deve generare**.

Poiché negli esempi precedenti abbiamo già previsto alcuni token, è possibile riutilizzare quella funzione per questo scopo.

> [!TIP]
> L'obiettivo di questa sesta fase è molto semplice: **addestrare il modello da zero**. A questo scopo verrà utilizzata la precedente architettura LLM, con alcuni cicli che iterano sui data set usando le funzioni di loss e l'optimizer definiti, per addestrare tutti i parametri del modello.

## Valutazione del testo

Per eseguire un addestramento corretto è necessario misurare e verificare le predizioni ottenute per il token atteso. L'obiettivo dell'addestramento è massimizzare la probabilità del token corretto, aumentando la sua probabilità rispetto a quella degli altri token.

Per massimizzare la probabilità del token corretto, i pesi del modello devono essere modificati in modo che tale probabilità venga massimizzata. L'aggiornamento dei pesi viene eseguito tramite **backpropagation**. Questo richiede una **funzione di loss da massimizzare**. In questo caso, la funzione sarà la **differenza tra la predizione effettuata e quella desiderata**.

Tuttavia, invece di lavorare con le predizioni grezze, si utilizzerà un logaritmo con base n. Quindi, se la predizione corrente del token atteso fosse 7.4541e-05, il logaritmo naturale (base *e*) di **7.4541e-05** sarebbe approssimativamente **-9.5042**.\
Quindi, per ogni elemento con, ad esempio, una lunghezza del contesto di 5 token, il modello dovrà prevedere 5 token: i primi 4 token saranno l'ultimo token dell'input e il quinto sarà quello previsto. Pertanto, in questo caso avremo 5 predizioni per ogni elemento (anche se le prime 4 erano nell'input, il modello non lo sa) con 5 token attesi e quindi 5 probabilità da massimizzare.

Pertanto, dopo aver applicato il logaritmo naturale a ogni predizione, viene calcolata la **media**, viene rimosso il **segno meno** (questa operazione viene chiamata _cross entropy loss_) e questo è il **numero da ridurre il più possibile, avvicinandolo a 0**, perché il logaritmo naturale di 1 è 0:

Un altro modo per misurare quanto sia valido il modello si chiama perplexity. La **perplexity** è una metrica utilizzata per valutare quanto bene un modello probabilistico preveda un campione. Nel language modelling, rappresenta l'**incertezza del modello** quando prevede il token successivo in una sequenza.\
Ad esempio, un valore di perplexity pari a 48725 significa che, quando deve prevedere un token, il modello non è sicuro di quale tra i 48.725 token del vocabolario sia quello corretto.

import tiktoken
import torch
import torch.nn as nn
from torch.utils.data import Dataset, DataLoader


class GPTDatasetV1(Dataset):
def __init__(self, txt, tokenizer, max_length, stride):
self.input_ids = []
self.target_ids = []

# Tokenize the entire text
token_ids = tokenizer.encode(txt, allowed_special={"<|endoftext|>"})

# Use a sliding window to chunk the book into overlapping sequences of max_length
for i in range(0, len(token_ids) - max_length, stride):
input_chunk = token_ids[i:i + max_length]
target_chunk = token_ids[i + 1: i + max_length + 1]
self.input_ids.append(torch.tensor(input_chunk))
self.target_ids.append(torch.tensor(target_chunk))

def __len__(self):
return len(self.input_ids)

def __getitem__(self, idx):
return self.input_ids[idx], self.target_ids[idx]


def create_dataloader_v1(txt, batch_size=4, max_length=256,
stride=128, shuffle=True, drop_last=True, num_workers=0):
# Initialize the tokenizer
tokenizer = tiktoken.get_encoding("gpt2")

# Create dataset
dataset = GPTDatasetV1(txt, tokenizer, max_length, stride)

# Create dataloader
dataloader = DataLoader(
dataset, batch_size=batch_size, shuffle=shuffle, drop_last=drop_last, num_workers=num_workers)

return dataloader


class MultiHeadAttention(nn.Module):
def __init__(self, d_in, d_out, context_length, dropout, num_heads, qkv_bias=False):
super().__init__()
assert d_out % num_heads == 0, "d_out must be divisible by n_heads"

self.d_out = d_out
self.num_heads = num_heads
self.head_dim = d_out // num_heads  # Reduce the projection dim to match desired output dim

self.W_query = nn.Linear(d_in, d_out, bias=qkv_bias)
self.W_key = nn.Linear(d_in, d_out, bias=qkv_bias)
self.W_value = nn.Linear(d_in, d_out, bias=qkv_bias)
self.out_proj = nn.Linear(d_out, d_out)  # Linear layer to combine head outputs
self.dropout = nn.Dropout(dropout)
self.register_buffer('mask', torch.triu(torch.ones(context_length, context_length), diagonal=1))

def forward(self, x):
b, num_tokens, d_in = x.shape

keys = self.W_key(x)  # Shape: (b, num_tokens, d_out)
queries = self.W_query(x)
values = self.W_value(x)

# We implicitly split the matrix by adding a `num_heads` dimension
# Unroll last dim: (b, num_tokens, d_out) -> (b, num_tokens, num_heads, head_dim)
keys = keys.view(b, num_tokens, self.num_heads, self.head_dim)
values = values.view(b, num_tokens, self.num_heads, self.head_dim)
queries = queries.view(b, num_tokens, self.num_heads, self.head_dim)

# Transpose: (b, num_tokens, num_heads, head_dim) -> (b, num_heads, num_tokens, head_dim)
keys = keys.transpose(1, 2)
queries = queries.transpose(1, 2)
values = values.transpose(1, 2)

# Compute scaled dot-product attention (aka self-attention) with a causal mask
attn_scores = queries @ keys.transpose(2, 3)  # Dot product for each head

# Original mask truncated to the number of tokens and converted to boolean
mask_bool = self.mask.bool()[:num_tokens, :num_tokens]

# Use the mask to fill attention scores
attn_scores.masked_fill_(mask_bool, -torch.inf)

attn_weights = torch.softmax(attn_scores / keys.shape[-1]**0.5, dim=-1)
attn_weights = self.dropout(attn_weights)

# Shape: (b, num_tokens, num_heads, head_dim)
context_vec = (attn_weights @ values).transpose(1, 2)

# Combine heads, where self.d_out = self.num_heads * self.head_dim
context_vec = context_vec.reshape(b, num_tokens, self.d_out)
context_vec = self.out_proj(context_vec)  # optional projection

return context_vec


class LayerNorm(nn.Module):
def __init__(self, emb_dim):
super().__init__()
self.eps = 1e-5
self.scale = nn.Parameter(torch.ones(emb_dim))
self.shift = nn.Parameter(torch.zeros(emb_dim))

def forward(self, x):
mean = x.mean(dim=-1, keepdim=True)
var = x.var(dim=-1, keepdim=True, unbiased=False)
norm_x = (x - mean) / torch.sqrt(var + self.eps)
return self.scale * norm_x + self.shift


class GELU(nn.Module):
def __init__(self):
super().__init__()

def forward(self, x):
return 0.5 * x * (1 + torch.tanh(
torch.sqrt(torch.tensor(2.0 / torch.pi)) *
(x + 0.044715 * torch.pow(x, 3))
))


class FeedForward(nn.Module):
def __init__(self, cfg):
super().__init__()
self.layers = nn.Sequential(
nn.Linear(cfg["emb_dim"], 4 * cfg["emb_dim"]),
GELU(),
nn.Linear(4 * cfg["emb_dim"], cfg["emb_dim"]),
)

def forward(self, x):
return self.layers(x)


class TransformerBlock(nn.Module):
def __init__(self, cfg):
super().__init__()
self.att = MultiHeadAttention(
d_in=cfg["emb_dim"],
d_out=cfg["emb_dim"],
context_length=cfg["context_length"],
num_heads=cfg["n_heads"],
dropout=cfg["drop_rate"],
qkv_bias=cfg["qkv_bias"])
self.ff = FeedForward(cfg)
self.norm1 = LayerNorm(cfg["emb_dim"])
self.norm2 = LayerNorm(cfg["emb_dim"])
self.drop_shortcut = nn.Dropout(cfg["drop_rate"])

def forward(self, x):
# Shortcut connection for attention block
shortcut = x
x = self.norm1(x)
x = self.att(x)   # Shape [batch_size, num_tokens, emb_size]
x = self.drop_shortcut(x)
x = x + shortcut  # Add the original input back

# Shortcut connection for feed-forward block
shortcut = x
x = self.norm2(x)
x = self.ff(x)
x = self.drop_shortcut(x)
x = x + shortcut  # Add the original input back

return x


class GPTModel(nn.Module):
def __init__(self, cfg):
super().__init__()
self.tok_emb = nn.Embedding(cfg["vocab_size"], cfg["emb_dim"])
self.pos_emb = nn.Embedding(cfg["context_length"], cfg["emb_dim"])
self.drop_emb = nn.Dropout(cfg["drop_rate"])

self.trf_blocks = nn.Sequential(
*[TransformerBlock(cfg) for _ in range(cfg["n_layers"])])

self.final_norm = LayerNorm(cfg["emb_dim"])
self.out_head = nn.Linear(cfg["emb_dim"], cfg["vocab_size"], bias=False)

def forward(self, in_idx):
batch_size, seq_len = in_idx.shape
tok_embeds = self.tok_emb(in_idx)
pos_embeds = self.pos_emb(torch.arange(seq_len, device=in_idx.device))
x = tok_embeds + pos_embeds  # Shape [batch_size, num_tokens, emb_size]
x = self.drop_emb(x)
x = self.trf_blocks(x)
x = self.final_norm(x)
logits = self.out_head(x)
return logits

### Funzioni `generate_text`

In una sezione precedente, una funzione recuperava semplicemente il **token più probabile** dopo aver ottenuto i logits. Tuttavia, questo significa che per ogni input verrà sempre generato lo stesso output, rendendo il processo molto deterministico.

La seguente funzione `generate_text` applicherà i concetti di `top-k`, `temperature` e `multinomial`.

- **`top-k`** significa che inizieremo a ridurre a `-inf` le probabilità di tutti i token, ad eccezione dei k token principali. Quindi, se k=3, prima di prendere una decisione solo i 3 token più probabili avranno una probabilità diversa da `-inf`.
- **`temperature`** significa che ogni probabilità verrà divisa per il valore della temperatura. Un valore di `0.1` aumenterà la probabilità più alta rispetto a quella più bassa, mentre una temperatura pari, ad esempio, a `5` renderà la distribuzione più uniforme. Questo contribuisce ad aumentare la variazione nelle risposte che vorremmo ottenere dall'LLM.
- Dopo aver applicato la temperatura, viene applicata nuovamente una funzione **`softmax`** per fare in modo che la somma delle probabilità di tutti i token rimanenti sia pari a 1.
- Infine, invece di scegliere il token con la probabilità più alta, la funzione **`multinomial`** viene applicata per **prevedere il token successivo in base alle probabilità finali**. Quindi, se il token 1 avesse una probabilità del 70%, il token 2 del 20% e il token 3 del 10%, il token 1 verrebbe selezionato nel 70% dei casi, il token 2 nel 20% dei casi e il token 3 nel 10% dei casi.
```python
# Generate text function
def generate_text(model, idx, max_new_tokens, context_size, temperature=0.0, top_k=None, eos_id=None):

# For-loop is the same as before: Get logits, and only focus on last time step
for _ in range(max_new_tokens):
idx_cond = idx[:, -context_size:]
with torch.no_grad():
logits = model(idx_cond)
logits = logits[:, -1, :]

# New: Filter logits with top_k sampling
if top_k is not None:
# Keep only top_k values
top_logits, _ = torch.topk(logits, top_k)
min_val = top_logits[:, -1]
logits = torch.where(logits < min_val, torch.tensor(float("-inf")).to(logits.device), logits)

# New: Apply temperature scaling
if temperature > 0.0:
logits = logits / temperature

# Apply softmax to get probabilities
probs = torch.softmax(logits, dim=-1)  # (batch_size, context_len)

# Sample from the distribution
idx_next = torch.multinomial(probs, num_samples=1)  # (batch_size, 1)

# Otherwise same as before: get idx of the vocab entry with the highest logits value
else:
idx_next = torch.argmax(logits, dim=-1, keepdim=True)  # (batch_size, 1)

if idx_next == eos_id:  # Stop generating early if end-of-sequence token is encountered and eos_id is specified
break

# Same as before: append sampled index to the running sequence
idx = torch.cat((idx, idx_next), dim=1)  # (batch_size, num_tokens+1)

return idx
```

> Esiste un'alternativa comune a `top-k` chiamata [**`top-p`**](https://en.wikipedia.org/wiki/Top-p_sampling), nota anche come nucleus sampling, che invece di ottenere k campioni con la probabilità più alta, **organizza** tutto il **vocabolario** risultante in base alle probabilità e le **somma** dalla probabilità più alta a quella più bassa finché non viene **raggiunta una soglia**.
>
> Quindi, **solo quelle parole** del vocabolario verranno considerate in base alle rispettive probabilità relative.
>
> In questo modo non è necessario selezionare un numero di `k` campioni, poiché il valore ottimale di k potrebbe essere diverso in ogni caso, ma **solo una soglia**.
>
> _Nota che questo miglioramento non è incluso nel codice precedente._

> [!TIP]
> Un altro modo per migliorare il testo generato consiste nell'utilizzare **Beam search** invece della greedy search usata in questo esempio.\
> A differenza della greedy search, che seleziona la parola successiva più probabile a ogni passaggio e costruisce una singola sequenza, **beam search tiene traccia delle 𝑘 k sequenze parziali con il punteggio più alto** (chiamate "beams") a ogni passaggio. Esplorando simultaneamente più possibilità, bilancia efficienza e qualità, aumentando le probabilità di **trovare una sequenza complessiva migliore** che potrebbe non essere individuata dall'approccio greedy a causa di scelte iniziali subottimali.
>
> _Nota che questo miglioramento non è incluso nel codice precedente._

### Funzioni di loss

La funzione **`calc_loss_batch`** calcola l'entropia incrociata della predizione di un singolo batch.\
La funzione **`calc_loss_loader`** ottiene l'entropia incrociata di tutti i batch e calcola l'**entropia incrociata media**.
```python
# Define loss functions
def calc_loss_batch(input_batch, target_batch, model, device):
input_batch, target_batch = input_batch.to(device), target_batch.to(device)
logits = model(input_batch)
loss = torch.nn.functional.cross_entropy(logits.flatten(0, 1), target_batch.flatten())
return loss

def calc_loss_loader(data_loader, model, device, num_batches=None):
total_loss = 0.
if len(data_loader) == 0:
return float("nan")
elif num_batches is None:
num_batches = len(data_loader)
else:
# Reduce the number of batches to match the total number of batches in the data loader
# if num_batches exceeds the number of batches in the data loader
num_batches = min(num_batches, len(data_loader))
for i, (input_batch, target_batch) in enumerate(data_loader):
if i < num_batches:
loss = calc_loss_batch(input_batch, target_batch, model, device)
total_loss += loss.item()
else:
break
return total_loss / num_batches
```
> [!TIP]
> **Gradient clipping** è una tecnica utilizzata per migliorare la **stabilità del training** nelle reti neurali di grandi dimensioni, impostando una **soglia massima** per i valori dei gradienti. Quando i gradienti superano questo `max_norm` predefinito, vengono ridotti proporzionalmente per garantire che gli aggiornamenti dei parametri del modello rimangano entro un intervallo gestibile, prevenendo problemi come l'esplosione dei gradienti e assicurando un training più controllato e stabile.
>
> _Nota che questo miglioramento non è incluso nel codice precedente._

Le funzioni `create_dataloader_v1` e `create_dataloader_v1` sono già state trattate in una sezione precedente.

Da qui si nota come sia definito che il 90% del testo verrà utilizzato per il training, mentre il 10% verrà utilizzato per la validazione, e che entrambi i set sono memorizzati in 2 data loader diversi.\
Nota che a volte una parte del dataset viene lasciata anche per un test set, per valutare meglio le prestazioni del modello.

Entrambi i data loader utilizzano lo stesso batch size, la stessa lunghezza massima, lo stesso stride e lo stesso numero di worker (0 in questo caso).\
Le differenze principali riguardano i dati utilizzati da ciascuno; inoltre, il validator non elimina l'ultimo elemento e non esegue lo shuffling dei dati, poiché ciò non è necessario ai fini della validazione.

Inoltre, il fatto che **stride sia grande quanto la context length** significa che non ci sarà sovrapposizione tra i contesti utilizzati per il training dei dati (riduce l'overfitting, ma anche il dataset di training).

Inoltre, nota che il batch size in questo caso è 2, per dividere i dati in 2 batch; l'obiettivo principale è consentire l'elaborazione parallela e ridurre il consumo per batch.

### Funzioni di training

La funzione `generate_and_print_sample` riceverà semplicemente un contesto e genererà alcuni token per farsi un'idea di quanto sia valido il modello in quel momento. Viene chiamata da `train_model_simple` a ogni step.

La funzione `evaluate_model` viene chiamata con la frequenza indicata alla funzione di training e viene utilizzata per misurare la train loss e la validation loss in quel punto del training del modello.

La grande funzione `train_model_simple` è quella che esegue effettivamente il training del modello. Si aspetta:

- Il train data loader (con i dati già separati e preparati per il training)
- Il validator loader
- L'**optimizer** da utilizzare durante il training: questa è la funzione che utilizzerà i gradienti e aggiornerà i parametri per ridurre la loss. In questo caso, come vedrai, viene utilizzato `AdamW`, ma ce ne sono molti altri.
- `optimizer.zero_grad()` viene chiamato a ogni round per azzerare i gradienti e non accumularli.
- Il parametro **`lr`** è il **learning rate**, che determina la **dimensione degli step** eseguiti durante il processo di ottimizzazione quando vengono aggiornati i parametri del modello. Un learning rate **più piccolo** significa che l'optimizer esegue **aggiornamenti più piccoli** sui pesi, il che può portare a una convergenza più **precisa**, ma potrebbe **rallentare** il training. Un learning rate **più grande** può velocizzare il training, ma comporta il **rischio di oltrepassare** il minimo della funzione di loss (**saltare oltre** il punto in cui la funzione di loss è minimizzata).
- Il **Weight Decay** modifica il passaggio di **Loss Calculation** aggiungendo un termine extra che penalizza i pesi di grandi dimensioni. Questo incoraggia l'optimizer a trovare soluzioni con pesi più piccoli, bilanciando l'adattamento efficace ai dati e la semplicità del modello, prevenendo l'overfitting nei modelli di machine learning scoraggiando il modello dall'assegnare troppa importanza a una singola feature.
- Gli optimizer tradizionali, come SGD con regolarizzazione L2, associano il weight decay al gradiente della funzione di loss. Tuttavia, **AdamW** (una variante dell'optimizer Adam) disaccoppia il weight decay dall'aggiornamento del gradiente, ottenendo una regolarizzazione più efficace.
- Il device da utilizzare per il training
- Il numero di epoche: il numero di volte in cui scorrere i dati di training
- La frequenza di valutazione: la frequenza con cui chiamare `evaluate_model`
- Le iterazioni di valutazione: il numero di batch da utilizzare quando si valuta lo stato corrente del modello durante la chiamata a `generate_and_print_sample`
- Il contesto iniziale: la frase iniziale da utilizzare quando si chiama `generate_and_print_sample`
- Il tokenizer
```python
# Functions to train the data
def train_model_simple(model, train_loader, val_loader, optimizer, device, num_epochs,
eval_freq, eval_iter, start_context, tokenizer):
# Initialize lists to track losses and tokens seen
train_losses, val_losses, track_tokens_seen = [], [], []
tokens_seen, global_step = 0, -1

# Main training loop
for epoch in range(num_epochs):
model.train()  # Set model to training mode

for input_batch, target_batch in train_loader:
optimizer.zero_grad() # Reset loss gradients from previous batch iteration
loss = calc_loss_batch(input_batch, target_batch, model, device)
loss.backward() # Calculate loss gradients
optimizer.step() # Update model weights using loss gradients
tokens_seen += input_batch.numel()
global_step += 1

# Optional evaluation step
if global_step % eval_freq == 0:
train_loss, val_loss = evaluate_model(
model, train_loader, val_loader, device, eval_iter)
train_losses.append(train_loss)
val_losses.append(val_loss)
track_tokens_seen.append(tokens_seen)
print(f"Ep {epoch+1} (Step {global_step:06d}): "
f"Train loss {train_loss:.3f}, Val loss {val_loss:.3f}")

# Print a sample text after each epoch
generate_and_print_sample(
model, tokenizer, device, start_context
)

return train_losses, val_losses, track_tokens_seen


def evaluate_model(model, train_loader, val_loader, device, eval_iter):
model.eval() # Set in eval mode to avoid dropout
with torch.no_grad():
train_loss = calc_loss_loader(train_loader, model, device, num_batches=eval_iter)
val_loss = calc_loss_loader(val_loader, model, device, num_batches=eval_iter)
model.train() # Back to training model applying all the configurations
return train_loss, val_loss


def generate_and_print_sample(model, tokenizer, device, start_context):
model.eval() # Set in eval mode to avoid dropout
context_size = model.pos_emb.weight.shape[0]
encoded = text_to_token_ids(start_context, tokenizer).to(device)
with torch.no_grad():
token_ids = generate_text(
model=model, idx=encoded,
max_new_tokens=50, context_size=context_size
)
decoded_text = token_ids_to_text(token_ids, tokenizer)
print(decoded_text.replace("\n", " "))  # Compact print format
model.train() # Back to training model applying all the configurations
```
> [!TIP]
> Per migliorare il learning rate esistono un paio di tecniche rilevanti chiamate **linear warmup** e **cosine decay.**
>
> Il **linear warmup** consiste nel definire un learning rate iniziale e uno massimo, aggiornandolo costantemente dopo ogni epoca. Questo perché iniziare il training con aggiornamenti dei pesi più piccoli riduce il rischio che il modello subisca aggiornamenti ampi e destabilizzanti durante la fase di training.\
> Il **cosine decay** è una tecnica che **riduce gradualmente il learning rate** seguendo una curva semi-coseno **dopo** la fase di **warmup**, rallentando gli aggiornamenti dei pesi per **minimizzare il rischio di superare** i minimi della loss e garantire la stabilità del training nelle fasi successive.
>
> _Nota che questi miglioramenti non sono inclusi nel codice precedente._

### Miglioramenti pratici moderni dell'addestramento

Il ciclo precedente è sufficiente per comprendere i meccanismi, ma il pre-training moderno di solito aggiunge diverse ottimizzazioni per migliorare il throughput e la stabilità:

- **Mixed precision (`bfloat16` / `float16`)** per ridurre l'utilizzo della memoria e velocizzare le moltiplicazioni tra matrici. In pratica, `bfloat16` è generalmente preferito sulle recenti GPU NVIDIA per datacenter perché mantiene un intervallo di esponenti più ampio e di solito non richiede il gradient scaling, mentre `float16` usa comunemente un `GradScaler`.
- **Gradient accumulation** per simulare una dimensione globale del batch maggiore quando il batch completo non entra nella VRAM.
- **Gradient clipping** (ad esempio `clip_grad_norm_`) per evitare aggiornamenti esplosivi, soprattutto nelle fasi iniziali dell'addestramento.
- **Activation checkpointing** per scambiare una maggiore quantità di calcolo con un utilizzo della memoria molto inferiore, ricalcolando parte del forward pass durante la backpropagation.
- **Fused / memory-efficient attention** (`scaled_dot_product_attention`) per ridurre il costo dell'attention con context window più lunghe.
- **Sharded training/checkpointing** (ad esempio FSDP / distributed checkpoints) quando il modello o gli optimizer states non entrano più su una singola GPU.

Un esempio minimale in PyTorch che combina alcune di queste idee è il seguente:
```python
accum_steps = 8
scaler = torch.amp.GradScaler("cuda")
optimizer.zero_grad(set_to_none=True)

for step, (input_batch, target_batch) in enumerate(train_loader, start=1):
with torch.autocast(device_type="cuda", dtype=torch.float16):
loss = calc_loss_batch(input_batch, target_batch, model, device)
loss = loss / accum_steps

scaler.scale(loss).backward()

if step % accum_steps == 0:
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
scaler.step(optimizer)
scaler.update()
optimizer.zero_grad(set_to_none=True)
```
> [!WARNING]
> Quando esegui il pre-training da checkpoint o repository di terze parti, considera il **percorso di caricamento come parte della superficie di attacco**. Preferisci caricare file `state_dict` o `safetensors` invece di interi oggetti Python serializzati con pickle, mantieni `trust_remote_code=False` a meno che tu non abbia sottoposto a verifica il repository e consulta [../AI-Models-RCE.md](../AI-Models-RCE.md) per esempi di RCE durante il caricamento dei modelli.

### Caricamento dei checkpoint più sicuro ed efficiente in termini di memoria

Per i checkpoint di grandi dimensioni, la sequenza ingenua `torch.load(...)` + `load_state_dict(...)` può duplicare temporaneamente la memoria (parametri inizializzati + pesi caricati). Le versioni più recenti di PyTorch hanno aggiunto opzioni molto utili. A partire da PyTorch 2.6, `weights_only=True` è il valore predefinito per `torch.load()` quando non viene passato alcun modulo pickle personalizzato, ma è comunque consigliabile specificarlo esplicitamente negli esempi:

- `weights_only=True`: limita la deserializzazione a tensori / oggetti primitivi invece di consentire oggetti Python arbitrari serializzati con pickle.
- `mmap=True`: esegue il memory mapping degli storage dei tensori dal disco invece di leggere tutto immediatamente nella RAM.
- `with torch.device("meta"):`: crea prima il modulo con tensori vuoti.
- `load_state_dict(..., assign=True)`: assegna direttamente i tensori caricati al modulo inizializzato con meta.
```python
state_dict = torch.load(
"model.pth",
map_location="cpu",
weights_only=True,
mmap=True,
)

with torch.device("meta"):
model = GPTModel(GPT_CONFIG_124M)

model.load_state_dict(state_dict, assign=True)
model.to(device)
model.eval()
```
Ciò è particolarmente utile quando si caricano checkpoint di diversi GB su una workstation, perché riduce i picchi di RAM durante l'avvio.
