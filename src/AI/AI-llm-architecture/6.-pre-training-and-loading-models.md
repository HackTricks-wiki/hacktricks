## Generowanie tekstu

Aby wytrenować model, musi on być w stanie generować nowe tokeny. Następnie porównamy wygenerowane tokeny z oczekiwanymi, aby wytrenować model w **uczeniu się tokenów, które powinien generować**.

Ponieważ w poprzednich przykładach przewidzieliśmy już kilka tokenów, możliwe jest ponowne użycie tej funkcji w tym celu.

> [!TIP]
> Cel tej szóstej fazy jest bardzo prosty: **wytrenować model od podstaw**. W tym celu zostanie użyta wcześniejsza architektura LLM wraz z pętlami przechodzącymi przez zbiory danych, wykorzystując zdefiniowane funkcje straty i optimizer do trenowania wszystkich parametrów modelu.

## Ocena tekstu

Aby przeprowadzić prawidłowy trening, należy mierzyć i sprawdzać predykcje uzyskane dla oczekiwanego tokena. Celem treningu jest maksymalizacja prawdopodobieństwa poprawnego tokena, co wiąże się ze zwiększeniem jego prawdopodobieństwa względem innych tokenów.

Aby zmaksymalizować prawdopodobieństwo poprawnego tokena, należy zmodyfikować wagi modelu tak, aby to prawdopodobieństwo było maksymalne. Aktualizacja wag odbywa się za pomocą **backpropagation**. Wymaga to **funkcji straty do maksymalizacji**. W tym przypadku funkcją będzie **różnica między wykonaną predykcją a oczekiwaną**.

Jednak zamiast pracować z surowymi predykcjami, zostanie użyty logarytm o podstawie n. Jeśli bieżąca predykcja oczekiwanego tokena wynosiła 7.4541e-05, logarytm naturalny (podstawa *e*) z **7.4541e-05** wynosi w przybliżeniu **-9.5042**.\
Następnie dla każdego wpisu z kontekstem o długości na przykład 5 tokenów model będzie musiał przewidzieć 5 tokenów: pierwsze 4 tokeny będą ostatnim tokenem wejścia, a piąty będzie przewidywanym tokenem. Dlatego dla każdego wpisu będziemy mieć w tym przypadku 5 predykcji (nawet jeśli pierwsze 4 znajdowały się na wejściu, model o tym nie wie), z 5 oczekiwanymi tokenami, a zatem 5 prawdopodobieństw do zmaksymalizowania.

Dlatego po wykonaniu logarytmu naturalnego dla każdej predykcji obliczana jest **średnia**, usuwany jest **znak minus** (nazywa się to _cross entropy loss_), a następnie otrzymujemy **liczbę, którą należy zmniejszyć jak najbardziej, aby zbliżyć ją do 0**, ponieważ logarytm naturalny z 1 wynosi 0:

import tiktoken
import torch
import torch.nn as nn
from torch.utils.data import Dataset, DataLoader


class GPTDatasetV1(Dataset):
def __init__(self, txt, tokenizer, max_length, stride):
self.input_ids = []
self.target_ids = []

# Tokenize the entire text
token_ids = tokenizer.encode(txt, allowed_special={"<|endoftext|>"})

# Use a sliding window to chunk the book into overlapping sequences of max_length
for i in range(0, len(token_ids) - max_length, stride):
input_chunk = token_ids[i:i + max_length]
target_chunk = token_ids[i + 1: i + max_length + 1]
self.input_ids.append(torch.tensor(input_chunk))
self.target_ids.append(torch.tensor(target_chunk))

def __len__(self):
return len(self.input_ids)

def __getitem__(self, idx):
return self.input_ids[idx], self.target_ids[idx]


def create_dataloader_v1(txt, batch_size=4, max_length=256,
stride=128, shuffle=True, drop_last=True, num_workers=0):
# Initialize the tokenizer
tokenizer = tiktoken.get_encoding("gpt2")

# Create dataset
dataset = GPTDatasetV1(txt, tokenizer, max_length, stride)

# Create dataloader
dataloader = DataLoader(
dataset, batch_size=batch_size, shuffle=shuffle, drop_last=drop_last, num_workers=num_workers)

return dataloader


class MultiHeadAttention(nn.Module):
def __init__(self, d_in, d_out, context_length, dropout, num_heads, qkv_bias=False):
super().__init__()
assert d_out % num_heads == 0, "d_out must be divisible by n_heads"

self.d_out = d_out
self.num_heads = num_heads
self.head_dim = d_out // num_heads  # Reduce the projection dim to match desired output dim

self.W_query = nn.Linear(d_in, d_out, bias=qkv_bias)
self.W_key = nn.Linear(d_in, d_out, bias=qkv_bias)
self.W_value = nn.Linear(d_in, d_out, bias=qkv_bias)
self.out_proj = nn.Linear(d_out, d_out)  # Linear layer to combine head outputs
self.dropout = nn.Dropout(dropout)
self.register_buffer('mask', torch.triu(torch.ones(context_length, context_length), diagonal=1))

def forward(self, x):
b, num_tokens, d_in = x.shape

keys = self.W_key(x)  # Shape: (b, num_tokens, d_out)
queries = self.W_query(x)
values = self.W_value(x)

# We implicitly split the matrix by adding a `num_heads` dimension
# Unroll last dim: (b, num_tokens, d_out) -> (b, num_tokens, num_heads, head_dim)
keys = keys.view(b, num_tokens, self.num_heads, self.head_dim)
values = values.view(b, num_tokens, self.num_heads, self.head_dim)
queries = queries.view(b, num_tokens, self.num_heads, self.head_dim)

# Transpose: (b, num_tokens, num_heads, head_dim) -> (b, num_heads, num_tokens, head_dim)
keys = keys.transpose(1, 2)
queries = queries.transpose(1, 2)
values = values.transpose(1, 2)

# Compute scaled dot-product attention (aka self-attention) with a causal mask
attn_scores = queries @ keys.transpose(2, 3)  # Dot product for each head

# Original mask truncated to the number of tokens and converted to boolean
mask_bool = self.mask.bool()[:num_tokens, :num_tokens]

# Use the mask to fill attention scores
attn_scores.masked_fill_(mask_bool, -torch.inf)

attn_weights = torch.softmax(attn_scores / keys.shape[-1]**0.5, dim=-1)
attn_weights = self.dropout(attn_weights)

# Shape: (b, num_tokens, num_heads, head_dim)
context_vec = (attn_weights @ values).transpose(1, 2)

# Combine heads, where self.d_out = self.num_heads * self.head_dim
context_vec = context_vec.reshape(b, num_tokens, self.d_out)
context_vec = self.out_proj(context_vec)  # optional projection

return context_vec


class LayerNorm(nn.Module):
def __init__(self, emb_dim):
super().__init__()
self.eps = 1e-5
self.scale = nn.Parameter(torch.ones(emb_dim))
self.shift = nn.Parameter(torch.zeros(emb_dim))

def forward(self, x):
mean = x.mean(dim=-1, keepdim=True)
var = x.var(dim=-1, keepdim=True, unbiased=False)
norm_x = (x - mean) / torch.sqrt(var + self.eps)
return self.scale * norm_x + self.shift


class GELU(nn.Module):
def __init__(self):
super().__init__()

def forward(self, x):
return 0.5 * x * (1 + torch.tanh(
torch.sqrt(torch.tensor(2.0 / torch.pi)) *
(x + 0.044715 * torch.pow(x, 3))
))


class FeedForward(nn.Module):
def __init__(self, cfg):
super().__init__()
self.layers = nn.Sequential(
nn.Linear(cfg["emb_dim"], 4 * cfg["emb_dim"]),
GELU(),
nn.Linear(4 * cfg["emb_dim"], cfg["emb_dim"]),
)

def forward(self, x):
return self.layers(x)


class TransformerBlock(nn.Module):
def __init__(self, cfg):
super().__init__()
self.att = MultiHeadAttention(
d_in=cfg["emb_dim"],
d_out=cfg["emb_dim"],
context_length=cfg["context_length"],
num_heads=cfg["n_heads"],
dropout=cfg["drop_rate"],
qkv_bias=cfg["qkv_bias"])
self.ff = FeedForward(cfg)
self.norm1 = LayerNorm(cfg["emb_dim"])
self.norm2 = LayerNorm(cfg["emb_dim"])
self.drop_shortcut = nn.Dropout(cfg["drop_rate"])

def forward(self, x):
# Shortcut connection for attention block
shortcut = x
x = self.norm1(x)
x = self.att(x)   # Shape [batch_size, num_tokens, emb_size]
x = self.drop_shortcut(x)
x = x + shortcut  # Add the original input back

# Shortcut connection for feed-forward block
shortcut = x
x = self.norm2(x)
x = self.ff(x)
x = self.drop_shortcut(x)
x = x + shortcut  # Add the original input back

return x


class GPTModel(nn.Module):
def __init__(self, cfg):
super().__init__()
self.tok_emb = nn.Embedding(cfg["vocab_size"], cfg["emb_dim"])
self.pos_emb = nn.Embedding(cfg["context_length"], cfg["emb_dim"])
self.drop_emb = nn.Dropout(cfg["drop_rate"])

self.trf_blocks = nn.Sequential(
*[TransformerBlock(cfg) for _ in range(cfg["n_layers"])])

self.final_norm = LayerNorm(cfg["emb_dim"])
self.out_head = nn.Linear(cfg["emb_dim"], cfg["vocab_size"], bias=False)

def forward(self, in_idx):
batch_size, seq_len = in_idx.shape
tok_embeds = self.tok_emb(in_idx)
pos_embeds = self.pos_emb(torch.arange(seq_len, device=in_idx.device))
x = tok_embeds + pos_embeds  # Shape [batch_size, num_tokens, emb_size]
x = self.drop_emb(x)
x = self.trf_blocks(x)
x = self.final_norm(x)
logits = self.out_head(x)
return logits

### Funkcje generate text

W poprzedniej sekcji funkcja po prostu wybierała **najbardziej prawdopodobny token** po uzyskaniu logits. Oznacza to jednak, że dla każdego wejścia zawsze będzie generowane takie samo wyjście, co sprawia, że działanie jest bardzo deterministyczne.

Poniższa funkcja `generate_text` zastosuje koncepcje `top-k`, `temperature` i `multinomial`.

- **`top-k`** oznacza, że zaczniemy zmniejszać do `-inf` prawdopodobieństwa wszystkich tokenów poza `k` tokenami z najwyższymi wartościami. Jeśli więc `k=3`, przed podjęciem decyzji tylko 3 najbardziej prawdopodobne tokeny będą miały prawdopodobieństwo różne od `-inf`.
- **`temperature`** oznacza, że każde prawdopodobieństwo zostanie podzielone przez wartość temperatury. Wartość `0.1` zwiększy przewagę najwyższego prawdopodobieństwa nad najniższym, podczas gdy temperatura `5`, na przykład, sprawi, że rozkład będzie bardziej płaski. Pomaga to zwiększyć różnorodność odpowiedzi, jakiej oczekujemy od LLM.
- Po zastosowaniu temperatury funkcja **`softmax`** jest ponownie stosowana, aby wszystkie pozostałe tokeny miały łączne prawdopodobieństwo równe 1.
- Na koniec, zamiast wybierać token o największym prawdopodobieństwie, funkcja **`multinomial`** jest stosowana do **przewidywania następnego tokenu zgodnie z końcowymi prawdopodobieństwami**. Jeśli więc token 1 miał prawdopodobieństwo 70%, token 2 — 20%, a token 3 — 10%, w 70% przypadków zostanie wybrany token 1, w 20% przypadków token 2, a w 10% przypadków token 3.
```python
# Generate text function
def generate_text(model, idx, max_new_tokens, context_size, temperature=0.0, top_k=None, eos_id=None):

# For-loop is the same as before: Get logits, and only focus on last time step
for _ in range(max_new_tokens):
idx_cond = idx[:, -context_size:]
with torch.no_grad():
logits = model(idx_cond)
logits = logits[:, -1, :]

# New: Filter logits with top_k sampling
if top_k is not None:
# Keep only top_k values
top_logits, _ = torch.topk(logits, top_k)
min_val = top_logits[:, -1]
logits = torch.where(logits < min_val, torch.tensor(float("-inf")).to(logits.device), logits)

# New: Apply temperature scaling
if temperature > 0.0:
logits = logits / temperature

# Apply softmax to get probabilities
probs = torch.softmax(logits, dim=-1)  # (batch_size, context_len)

# Sample from the distribution
idx_next = torch.multinomial(probs, num_samples=1)  # (batch_size, 1)

# Otherwise same as before: get idx of the vocab entry with the highest logits value
else:
idx_next = torch.argmax(logits, dim=-1, keepdim=True)  # (batch_size, 1)

if idx_next == eos_id:  # Stop generating early if end-of-sequence token is encountered and eos_id is specified
break

# Same as before: append sampled index to the running sequence
idx = torch.cat((idx, idx_next), dim=1)  # (batch_size, num_tokens+1)

return idx
```
> [!TIP]
> Istnieje powszechna alternatywa dla `top-k`, nazywana [**`top-p`**](https://en.wikipedia.org/wiki/Top-p_sampling), znana również jako nucleus sampling, która zamiast pobierać k próbek o największym prawdopodobieństwie, **porządkuje** całe **vocabulary** według prawdopodobieństw i **sumuje** je od najwyższego prawdopodobieństwa do najniższego, aż do **osiągnięcia progu**.
>
> Następnie brane będą pod uwagę **wyłącznie te słowa** z vocabulary, zgodnie z ich względnymi prawdopodobieństwami.
>
> Dzięki temu nie trzeba wybierać liczby `k` próbek, ponieważ optymalna wartość k może być różna w każdym przypadku — wystarczy określić **próg**.
>
> _Należy pamiętać, że to ulepszenie nie zostało uwzględnione w poprzednim kodzie._

> [!TIP]
> Innym sposobem na ulepszenie generowanego tekstu jest użycie **Beam search** zamiast greedy search zastosowanego w tym przykładzie.\
> W przeciwieństwie do greedy search, które na każdym kroku wybiera najbardziej prawdopodobne następne słowo i buduje pojedynczą sekwencję, **beam search śledzi 𝑘 k najwyżej ocenianych częściowych sekwencji** (nazywanych „beams”) na każdym kroku. Dzięki jednoczesnemu analizowaniu wielu możliwości równoważy wydajność i jakość, zwiększając szanse na **znalezienie lepszej całościowo** sekwencji, którą podejście greedy mogłoby pominąć z powodu wczesnych, nieoptymalnych wyborów.
>
> _Należy pamiętać, że to ulepszenie nie zostało uwzględnione w poprzednim kodzie._

### Funkcje straty

Funkcja **`calc_loss_batch`** oblicza cross entropy predykcji dla pojedynczego batcha.\
Funkcja **`calc_loss_loader`** pobiera cross entropy ze wszystkich batchy i oblicza **średnią cross entropy**.
```python
# Define loss functions
def calc_loss_batch(input_batch, target_batch, model, device):
input_batch, target_batch = input_batch.to(device), target_batch.to(device)
logits = model(input_batch)
loss = torch.nn.functional.cross_entropy(logits.flatten(0, 1), target_batch.flatten())
return loss

def calc_loss_loader(data_loader, model, device, num_batches=None):
total_loss = 0.
if len(data_loader) == 0:
return float("nan")
elif num_batches is None:
num_batches = len(data_loader)
else:
# Reduce the number of batches to match the total number of batches in the data loader
# if num_batches exceeds the number of batches in the data loader
num_batches = min(num_batches, len(data_loader))
for i, (input_batch, target_batch) in enumerate(data_loader):
if i < num_batches:
loss = calc_loss_batch(input_batch, target_batch, model, device)
total_loss += loss.item()
else:
break
return total_loss / num_batches
```
> [!TIP]
> **Gradient clipping** to technika stosowana w celu zwiększenia **stabilności trenowania** dużych sieci neuronowych poprzez ustawienie **maksymalnego progu** dla wartości gradientów. Gdy gradienty przekraczają wcześniej zdefiniowaną wartość `max_norm`, są proporcjonalnie zmniejszane, aby zapewnić, że aktualizacje parametrów modelu pozostaną w możliwym do kontrolowania zakresie. Zapobiega to problemom, takim jak eksplodujące gradienty, oraz zapewnia bardziej kontrolowane i stabilne trenowanie.
>
> _Należy zauważyć, że to ulepszenie nie zostało uwzględnione w poprzednim kodzie._

### Training functions

Funkcja `generate_and_print_sample` pobiera kontekst i generuje kilka tokenów, aby ocenić, jak dobry jest w danym momencie model. Jest wywoływana przez `train_model_simple` przy każdym kroku.

Funkcja `evaluate_model` jest wywoływana z częstotliwością określoną w funkcji treningowej i służy do pomiaru straty treningowej oraz straty walidacyjnej na danym etapie treningu modelu.

Następnie główna funkcja `train_model_simple` faktycznie trenuje model. Oczekuje:

- Data loadera treningowego (z wcześniej rozdzielonymi i przygotowanymi danymi do treningu)
- Loadera walidacyjnego
- **optymalizatora**, który ma być używany podczas treningu: Jest to funkcja wykorzystująca gradienty i aktualizująca parametry w celu zmniejszenia straty. W tym przypadku, jak zobaczysz, używany jest `AdamW`, ale istnieje wiele innych optymalizatorów.
- `optimizer.zero_grad()` jest wywoływane w każdej iteracji, aby zresetować gradienty i nie doprowadzić do ich kumulowania.
- Parametr **`lr`** to **learning rate**, który określa **wielkość kroków** wykonywanych podczas procesu optymalizacji przy aktualizowaniu parametrów modelu. **Mniejszy** learning rate oznacza, że optymalizator wykonuje **mniejsze aktualizacje** wag, co może prowadzić do bardziej **precyzyjnej** zbieżności, ale może też **spowolnić** trening. **Większy** learning rate może przyspieszyć trening, ale stwarza ryzyko **przeskoczenia minimum** funkcji straty (**przeskoczenia** punktu, w którym funkcja straty jest minimalizowana).
- **Weight Decay** modyfikuje etap **obliczania straty**, dodając dodatkowy składnik nakładający karę za duże wagi. Zachęca to optymalizator do znajdowania rozwiązań z mniejszymi wagami, równoważąc dobre dopasowanie do danych z zachowaniem prostoty modelu. Zapobiega to overfittingowi w modelach machine learning, zniechęcając model do przypisywania zbyt dużego znaczenia pojedynczej cesze.
- Tradycyjne optymalizatory, takie jak SGD z regularyzacją L2, łączą weight decay z gradientem funkcji straty. Jednak **AdamW** (odmiana optymalizatora Adam) oddziela weight decay od aktualizacji gradientu, zapewniając skuteczniejszą regularyzację.
- Urządzenia, które mają być używane do treningu
- Liczby epok: Liczba przejść przez dane treningowe
- Częstotliwości ewaluacji: Częstotliwości wywoływania `evaluate_model`
- Liczby iteracji ewaluacji: Liczby batchy używanych do oceny bieżącego stanu modelu podczas wywoływania `generate_and_print_sample`
- Początkowego kontekstu: Zdania początkowego używanego podczas wywoływania `generate_and_print_sample`
- Tokenizera
```python
# Functions to train the data
def train_model_simple(model, train_loader, val_loader, optimizer, device, num_epochs,
eval_freq, eval_iter, start_context, tokenizer):
# Initialize lists to track losses and tokens seen
train_losses, val_losses, track_tokens_seen = [], [], []
tokens_seen, global_step = 0, -1

# Main training loop
for epoch in range(num_epochs):
model.train()  # Set model to training mode

for input_batch, target_batch in train_loader:
optimizer.zero_grad() # Reset loss gradients from previous batch iteration
loss = calc_loss_batch(input_batch, target_batch, model, device)
loss.backward() # Calculate loss gradients
optimizer.step() # Update model weights using loss gradients
tokens_seen += input_batch.numel()
global_step += 1

# Optional evaluation step
if global_step % eval_freq == 0:
train_loss, val_loss = evaluate_model(
model, train_loader, val_loader, device, eval_iter)
train_losses.append(train_loss)
val_losses.append(val_loss)
track_tokens_seen.append(tokens_seen)
print(f"Ep {epoch+1} (Step {global_step:06d}): "
f"Train loss {train_loss:.3f}, Val loss {val_loss:.3f}")

# Print a sample text after each epoch
generate_and_print_sample(
model, tokenizer, device, start_context
)

return train_losses, val_losses, track_tokens_seen


def evaluate_model(model, train_loader, val_loader, device, eval_iter):
model.eval() # Set in eval mode to avoid dropout
with torch.no_grad():
train_loss = calc_loss_loader(train_loader, model, device, num_batches=eval_iter)
val_loss = calc_loss_loader(val_loader, model, device, num_batches=eval_iter)
model.train() # Back to training model applying all the configurations
return train_loss, val_loss


def generate_and_print_sample(model, tokenizer, device, start_context):
model.eval() # Set in eval mode to avoid dropout
context_size = model.pos_emb.weight.shape[0]
encoded = text_to_token_ids(start_context, tokenizer).to(device)
with torch.no_grad():
token_ids = generate_text(
model=model, idx=encoded,
max_new_tokens=50, context_size=context_size
)
decoded_text = token_ids_to_text(token_ids, tokenizer)
print(decoded_text.replace("\n", " "))  # Compact print format
model.train() # Back to training model applying all the configurations
```
> [!TIP]
> Aby poprawić learning rate, istnieje kilka istotnych technik o nazwach **linear warmup** i **cosine decay.**
>
> **Linear warmup** polega na zdefiniowaniu początkowego i maksymalnego learning rate oraz konsekwentnym aktualizowaniu go po każdej epoce. Wynika to z faktu, że rozpoczęcie treningu od mniejszych aktualizacji wag zmniejsza ryzyko wystąpienia przez model dużych, destabilizujących aktualizacji podczas fazy treningu.\
> **Cosine decay** to technika, która **stopniowo zmniejsza learning rate** zgodnie z krzywą połowy cosinusa **po fazie warmup**, spowalniając aktualizacje wag, aby **zminimalizować ryzyko przekroczenia** minimów funkcji straty i zapewnić stabilność treningu w jego późniejszych fazach.
>
> _Należy pamiętać, że te usprawnienia nie zostały uwzględnione w poprzednim kodzie._

### Praktyczne usprawnienia nowoczesnego trainingu

Poprzednia pętla wystarcza do zrozumienia mechaniki, ale współczesny pre-training zwykle wykorzystuje dodatkowe optymalizacje poprawiające przepustowość i stabilność:

- **Mixed precision (`bfloat16` / `float16`)** w celu zmniejszenia zużycia pamięci i przyspieszenia mnożenia macierzy. W praktyce `bfloat16` jest zwykle preferowany na nowszych datacenter GPU firmy NVIDIA, ponieważ zachowuje szerszy zakres wykładnika i zazwyczaj nie wymaga gradient scaling, podczas gdy `float16` często używa `GradScaler`.
- **Gradient accumulation**, aby symulować większy global batch size, gdy pełny batch nie mieści się w VRAM.
- **Gradient clipping** (na przykład `clip_grad_norm_`), aby zapobiegać eksplodującym aktualizacjom, szczególnie na początku trainingu.
- **Activation checkpointing**, aby wymienić dodatkowe obliczenia na znacznie mniejsze zużycie pamięci poprzez ponowne obliczanie części forward pass podczas backpropagation.
- **Fused / memory-efficient attention** (`scaled_dot_product_attention`), aby zmniejszyć koszt attention dla dłuższych context windows.
- **Sharded training/checkpointing** (na przykład FSDP / distributed checkpoints), gdy model lub stany optymalizatora przestają mieścić się na pojedynczym GPU.

Minimalny przykład łączący niektóre z tych koncepcji w PyTorch wygląda następująco:
```python
accum_steps = 8
scaler = torch.amp.GradScaler("cuda")
optimizer.zero_grad(set_to_none=True)

for step, (input_batch, target_batch) in enumerate(train_loader, start=1):
with torch.autocast(device_type="cuda", dtype=torch.float16):
loss = calc_loss_batch(input_batch, target_batch, model, device)
loss = loss / accum_steps

scaler.scale(loss).backward()

if step % accum_steps == 0:
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
scaler.step(optimizer)
scaler.update()
optimizer.zero_grad(set_to_none=True)
```
> [!WARNING]
> Podczas pre-trainingu z checkpointów lub repozytoriów stron trzecich traktuj **ścieżkę ładowania jako część attack surface**. Preferuj ładowanie plików `state_dict` lub `safetensors` zamiast całych spakowanych obiektów Python, pozostaw `trust_remote_code=False`, chyba że przeprowadziłeś audyt repozytorium, i sprawdź [../AI-Models-RCE.md](../AI-Models-RCE.md), aby zapoznać się z przykładami RCE podczas ładowania modeli.

### Bezpieczniejsze i bardziej efektywne pamięciowo ładowanie checkpointów

W przypadku dużych checkpointów naiwna sekwencja `torch.load(...)` + `load_state_dict(...)` może tymczasowo podwoić zużycie pamięci (zainicjalizowane parametry + załadowane wagi). Nowsze wersje PyTorch dodały kilka bardzo przydatnych opcji. Od PyTorch 2.6 `weights_only=True` jest wartością domyślną dla `torch.load()`, gdy nie przekazano niestandardowego modułu pickle, ale nadal warto jawnie podawać tę opcję w przykładach:

- `weights_only=True`: ogranicza deserializację do tensorów / obiektów prymitywnych zamiast dowolnych spakowanych obiektów Python.
- `mmap=True`: mapuje magazyny tensorów w pamięci z dysku zamiast od razu odczytywać wszystko do pamięci RAM.
- `with torch.device("meta"):`: najpierw tworzy moduł z pustymi tensorami.
- `load_state_dict(..., assign=True)`: bezpośrednio przypisuje załadowane tensory do modułu zainicjalizowanego za pomocą `meta`.
```python
state_dict = torch.load(
"model.pth",
map_location="cpu",
weights_only=True,
mmap=True,
)

with torch.device("meta"):
model = GPTModel(GPT_CONFIG_124M)

model.load_state_dict(state_dict, assign=True)
model.to(device)
model.eval()
```
Jest to szczególnie przydatne podczas ładowania checkpointów o rozmiarze wielu GB na workstation, ponieważ ogranicza skoki zużycia RAM podczas uruchamiania.

## Ładowanie wag GPT2

Istnieją 2 szybkie skrypty do lokalnego ładowania wag GPT2. W obu przypadkach możesz lokalnie sklonować repozytorium [https://github.com/rasbt/LLMs-from-scratch](https://github.com/rasbt/LLMs-from-scratch), a następnie:

- Skrypt [https://github.com/rasbt/LLMs-from-scratch/blob/main/ch05/01_main-chapter-code/gpt_generate.py](https://github.com/rasbt/LLMs-from-scratch/blob/main/ch05/01_main-chapter-code/gpt_generate.py) pobierze wszystkie wagi i przekształci formaty z OpenAI na formaty oczekiwane przez nasz LLM. Skrypt jest również przygotowany z wymaganą konfiguracją oraz promptem: "Every effort moves you"
- Skrypt [https://github.com/rasbt/LLMs-from-scratch/blob/main/ch05/02_alternative_weight_loading/weight-loading-hf-transformers.ipynb](https://github.com/rasbt/LLMs-from-scratch/blob/main/ch05/02_alternative_weight_loading/weight-loading-hf-transformers.ipynb) umożliwia lokalne załadowanie dowolnych wag GPT2 (wystarczy zmienić zmienną `CHOOSE_MODEL`) oraz przewidywanie tekstu na podstawie wybranych promptów.

Jeśli zamiast GPT-2 chcesz załadować bardziej nowoczesny open-source checkpoint z Hugging Face, typowy wzorzec polega na umożliwieniu `transformers` automatycznego umieszczenia modelu oraz preferowaniu wag `safetensors`, gdy są dostępne:
```python
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

model_id = "google/gemma-7b"

tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(
model_id,
torch_dtype=torch.bfloat16,
device_map="auto",
low_cpu_mem_usage=True,
trust_remote_code=False,
)
```
`device_map="auto"` jest bardzo przydatne, gdy pełny checkpoint nie mieści się w jednym akceleratorze, ponieważ loader może rozmieścić warstwy w pamięci GPU / CPU. W przypadku dużych repozytoriów `save_pretrained(..., max_shard_size="5GB")` lub równoważne checkpointy Hub generują **sharded checkpoints** oraz indeks, dzięki czemu można je ładować przyrostowo, zamiast wymagać jednego ogromnego pliku w pamięci RAM.

Jeśli potrzebujesz sprawdzać lub dzielić tensory bez ładowania całego pliku, `safetensors` również obsługuje odczyty częściowe:
```python
from safetensors import safe_open

with safe_open("model.safetensors", framework="pt", device="cpu") as f:
emb_slice = f.get_slice("model.embed_tokens.weight")
vocab_size, hidden_dim = emb_slice.get_shape()
first_half = emb_slice[:, : hidden_dim // 2]
```
